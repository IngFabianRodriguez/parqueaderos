# RF-140 — Búsqueda de Cliente por Placa/Nombre/Teléfono e Historial

## Metadata

- **Módulo**: 11 — App Operador
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: customer-service, ticket-service, vehicle-service, operator-app

---

## User Story

**Como** operador de parqueadero **quiero** buscar un cliente por nombre, placa o teléfono y ver su historial completo **para** atender rápidamente consultas, resolver problemas y tener contexto del cliente.

---

## Objetivo

El operador tiene una función de búsqueda integrada en la app que le permite encontrar clientes (propietarios de vehículos) usando cualquiera de tres criterios: nombre, placa o número de teléfono. Una vez encontrado, el sistema muestra el perfil del cliente con su historial completo de movimientos, pagos, bloqueos y observaciones.

---

## Actores

- **Actor principal**: Operador de parqueadero (`operator_role`)
- **Actor secundario**: Supervisor de sede

---

## Precondiciones

- El operador tiene sesión activa en la app.
- Existe al menos un cliente registrado en el sistema.

---

## Postcondiciones

### En caso de éxito

- Se muestra el perfil del cliente con todos sus datos.
- Se muestra el historial completo de sus movimientos.

### En caso de falla

- Si no hay resultados, se muestra mensaje "No se encontró ningún cliente con esos datos".
- Si hay múltiples resultados, se muestra lista para seleccionar.

---

## Flujo Funcional

1. El operador toca la barra de búsqueda en la app (dashboard o menú).
2. El operador ingresa al menos 3 caracteres en cualquiera de los campos:
   - Nombre (búsqueda parcial, sin distinción de mayúsculas/minúsculas).
   - Placa (búsqueda exacta o parcial).
   - Teléfono (búsqueda exacta).
3. El sistema muestra resultados en tiempo real a medida que se escribe (mínimo 3 caracteres).
4. El operador selecciona un cliente de la lista.
5. Se muestra el perfil del cliente con:
   - Datos personales: nombre, teléfono, email.
   - Vehículos asociados: lista de placas.
   - Estado de cuenta: tiene saldo pendiente, tiene bloqueo activo.
   - Historial de movimientos: últimos 50 registros.
6. El operador puede navegar a acciones desde el perfil: ver ticket abierto, desbloquear vehículo, etc.

---

## Criterios de Búsqueda

| Campo | Mínimo Caracteres | Tipo de Búsqueda | Prioridad |
|-------|-------------------|------------------|-----------|
| nombre | 3 | Parcial (contiene) | Baja |
| placa | 3 | Exacta o parcial (contiene) | Alta |
| telefono | 3 | Exacta | Alta |

**Nota**: La búsqueda debe devolver resultados al escribir 3 o más caracteres. Si el campo tiene menos de 3 caracteres, no se ejecuta búsqueda.

---

## Datos de Salida — Perfil del Cliente

| Campo | Tipo | Descripción |
|-------|------|-------------|
| cliente_id | UUID | Identificador único |
| nombre | String | Nombre completo |
| telefono | String | Número de teléfono |
| email | String | Correo electrónico |
| fecha_registro | DateTime | Cuándo se registró |
| tiene_bloqueo_activo | Boolean | Si tiene algún vehículo bloqueado |
| saldo_pendiente | Decimal | Monto total pendiente de pago |
| tipo_cliente | Enum | FRECUENTE, OCASIONAL, CORPORATIVO |
| vehiculos | Array | Lista de vehículos asociados |

---

## Datos de Salida — Vehículo

| Campo | Tipo | Descripción |
|-------|------|-------------|
| vehiculo_id | UUID | Identificador |
| placa | String | Placa del vehículo |
| marca | String | Marca (opcional) |
| modelo | String | Modelo (opcional) |
| color | String | Color (opcional) |
| estado_bloqueo | Enum | ACTIVO, INACTIVO |
| observaciones | String | Notas sobre el vehículo |

---

## Datos de Salida — Historial de Movimientos

| Campo | Tipo | Descripción |
|-------|------|-------------|
| fecha_hora | DateTime | Timestamp del movimiento |
| tipo | Enum | ENTRADA, SALIDA, PAGO, BLOQUEO, DESBLOQUEO |
| detalle | String | Descripción breve |
| monto | Decimal | Monto pagado (si aplica) |
| ticket_id | UUID | Ticket asociado (si aplica) |
| operador | String | Nombre del operador |

**Se muestran los últimos 50 registros; se pueden cargar más con paginación.**

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Búsqueda con < 3 caracteres | No se ejecuta; se muestra mensaje "Ingrese al menos 3 caracteres" |
| Sin resultados | Mensaje "No se encontró ningún cliente" |
| Múltiples resultados (>10) | Se muestra paginado; el primero es el más relevante |
| Cliente sin vehículos | Se muestra perfil vacío de vehículos |
| Cliente con saldo pendiente | Se muestra cartel amarillo con el monto |
| Cliente con bloqueo activo | Se muestra cartel rojo con información del bloqueo |
| Historial vacío | Mensaje "Este cliente aún no tiene movimientos" |
| Error de servidor | Retry automático; mensaje tras 3 intentos fallidos |

---

## Criterios de Aceptación

1. La búsqueda devuelve resultados en < 1 segundo para textos de 3+ caracteres.
2. La búsqueda por placa es la más rápida y precisa.
3. El operador puede buscar sin saber exactamente qué criterio usar.
4. El perfil del cliente muestra de un vistazo si tiene bloqueos o deudas pendientes.
5. El historial muestra los últimos 50 movimientos sin paginación visible.
6. Se puede acceder a cualquier ticket abierto desde el perfil del cliente.
7. Los datos del cliente se actualizan en tiempo real si hay cambios.
8. La búsqueda funciona offline si se tienen datos en caché de clientes frecuentes.

## Endpoints API Involucrados

- `GET /api/v1/operator/customers/search?q={query}&type={nombre|placa|telefono}` — Búsqueda
- `GET /api/v1/operator/customers/{id}` — Perfil completo
- `GET /api/v1/operator/customers/{id}/vehicles` — Vehículos del cliente
- `GET /api/v1/operator/customers/{id}/history?limit=50` — Historial

---

## Métricas de Éxito

- Búsqueda devuelve resultados en < 1 segundo.
- Perfil carga en < 2 segundos.
- Historial carga en < 3 segundos.
- 0 errores de búsqueda con textos válidos.

---

## Notas

- La barra de búsqueda debe estar visible en el dashboard para acceso rápido.
- Se debe recordar las últimas 10 búsquedas del operador para acceso offline.
- El historial debe ser exportable a CSV si el supervisor lo solicita.
- Si el cliente tiene múltiples vehículos, se debe poder ver el estado de cada uno por separado.
