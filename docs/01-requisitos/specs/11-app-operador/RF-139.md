# RF-139 — Resolución y Historial de Alertas en App

## Metadata

- **Módulo**: 11 — App Operador
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: alert-service, operator-app, notification-service

---

## User Story

**Como** operador de parqueadero **quiero** resolver o descartar alertas desde la app y ver el historial de alertas de las últimas 24 horas **para** mantener el control de situaciones críticas y tener trazabilidad de las decisiones tomadas.

---

## Objetivo

El operador puede interactuar con las alertas activas: resolverlas (tomando acción correctiva) o descartarlas (con justificación). Además, el operador tiene acceso a un historial de alertas de las últimas 24 horas con el detalle de cada una, incluyendo quién la resolvió, cuándo y con qué acción.

---

## Actores

- **Actor principal**: Operador de parqueadero (`operator_role`)
- **Actor secundario**: Supervisor de sede

---

## Precondiciones

- El operador tiene sesión activa en la app.
- Existe al menos una alerta activa en el sistema.

---

## Postcondiciones

### En caso de éxito

- La alerta se marca como RESUELTA o DESCARTADA con la justificación proporcionada.
- La alerta desaparece del panel de alertas activas.
- Se registra en el historial con todos los datos.

### En caso de falla

- Si la alerta ya fue resuelta por otro operador, se muestra mensaje y se refresca la lista.
- Si el operador no tiene permisos, se muestra error.

---

## Flujo Funcional — Resolver Alerta

1. El operador toca una alerta en el panel de alertas.
2. Se muestra el detalle de la alerta: tipo, vehículo/placa, tiempo, contexto.
3. El operador selecciona la acción correctiva tomada:
   - **Pago registrado**: El vehículo pagó y se cerró el ticket.
   - **Vehículo salió**: El vehículo salió sin pago (tomar nota).
   - **Bloqueo aplicado**: Se bloqueó el vehículo (RF-141).
   - **Talanquera reiniciada**: Se reinició/reparó la talanquera.
   - **Otra**:Descripción manual.
4. El operador añade una observación opcional.
5. El operador toca "Resolver Alerta".
6. La alerta se marca como RESUELTA con timestamp y operador.

---

## Flujo Funcional — Descartar Alerta

1. El operador toca "Descartar" en una alerta.
2. Se pide seleccionar el motivo del descarte:
   - No aplica (falso positivo).
   - Ya resuelta por otro medio.
   - Requiere supervisor.
   -Otra.
3. El operador añade una observación opcional.
4. El operador confirma el descarte.
5. La alerta se marca como DESCARTADA.

---

## Estados de Alerta

| Estado | Descripción | Visible en Panel |
|--------|-------------|------------------|
| ACTIVA | Alerta sin atender | Sí (en color) |
| RESUELTA | Atendida por operador | Solo en historial |
| DESCARTADA | Descartada con justificación | Solo en historial |
| EXPIRADA | Superó ventana sin acción | Solo en historial |

---

## Historial de Alertas (Últimas 24h)

**Acceso**: El operador toca "Ver Historial" en el panel de alertas.

**Contenido por alerta**:
- Timestamp de creación.
- Tipo de alerta.
- Placa / ID relacionado.
- Estado final (RESUELTA / DESCARTADA).
- Operador que la atendió.
- Acción tomada.
- Observación.

**Filtrado**: Se puede filtrar por tipo de alerta y estado.

---

## Datos de Entrada (Resolver)

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| alerta_id | UUID | ID de la alerta | Sí |
| accion_tomada | Enum | PAGO_REGISTRADO, VEHICULO_SALIO, BLOQUEO_APLICADO, TALANQUERA_REINICIADA, OTRA | Sí |
| observacion | String | Nota adicional | No |

---

## Datos de Entrada (Descartar)

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| alerta_id | UUID | ID de la alerta | Sí |
| motivo_descarte | Enum | NO_APLICA, YA_RESUELTA, REQUIERE_SUPERVISOR, OTRA | Sí |
| observacion | String | Nota adicional | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| alerta_id | UUID | Identificador |
| tipo | Enum | MORA_VEHICULO, GATE_OFFLINE, ALERTA_MORA |
| placa | String | Placa asociada |
| ticket_id | UUID | Ticket asociado (si aplica) |
| gate_id | UUID | Talanquera asociada (si aplica) |
| estado | Enum | ACTIVA, RESUELTA, DESCARTADA, EXPIRADA |
| created_at | DateTime | Cuándo se generó |
| resolved_at | DateTime | Cuándo se resolvió |
| resolved_by | UUID | Operador que resolvió |
| accion_tomada | Enum | Acción seleccionada |
| observacion | String | Nota del operador |
| tiempo_resolucion | Integer | Minutos desde creación hasta resolución |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Alerta ya resuelta por otro | Se muestra "Esta alerta ya fue resuelta por [operador]" |
| Operador sin permisos | Se muestra "No tienes permisos para resolver alertas" |
| Sin conexión al resolver | Se almacena localmente; se sube cuando haya conexión |
| Historial vacío | Mensaje "No hay alertas en las últimas 24 horas" |
| Más de 100 alertas | Paginación de 20 en 20; máximo 100 visibles |
| Intentos múltiples de resolución | Solo la primera resolución cuenta; las siguientes muestran error |

---

## Criterios de Aceptación

1. El operador puede resolver una alerta en menos de 10 segundos.
2. El operador puede descartar una alerta con justificación obligatoria.
3. Cada resolución/descarte queda registrado con operador, timestamp y acción.
4. Las alertas resueltas desaparecen del panel activo.
5. El historial muestra las alertas de las últimas 24 horas.
6. Las alertas se pueden filtrar por tipo y estado en el historial.
7. Si el operador está offline, la acción se guarda localmente y se sincroniza después.
8. Las alertas no resueltas en 24h se marcan como EXPIRADAS automáticamente.

## Endpoints API Involucrados

- `GET /api/v1/operator/alerts/active` — Lista de alertas activas
- `GET /api/v1/operator/alerts/history?hours=24` — Historial de alertas
- `POST /api/v1/operator/alerts/{id}/resolve` — Resolver alerta
- `POST /api/v1/operator/alerts/{id}/dismiss` — Descartar alerta

---

## Métricas de Éxito

- Resolución de alerta en < 10 segundos.
- Historial carga en < 2 segundos.
- Sincronización offline en < 30 segundos al recuperar conexión.
- 0 alertas perdidas entre resolución y registro.

---

## Notas

- Las alertas resueltas deben aparecer en el historial con toda la información para auditoría.
- Se debe poderExportar el historial a PDF o CSV para reportes del supervisor.
- Si una alerta de mora se resuelve con "Pago registrado", debe cerrar efectivamente el ticket asociado.
