# RF-135 — Registro de Pago en App Operador

## Metadata

- **Módulo**: 11 — App Operador
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: payment-service, ticket-service, operator-app, cash-service

---

## User Story

**Como** operador de parqueadero **quiero** registrar el pago de un vehículo (efectivo o transferencia) desde la app **para** cerrar el ticket y liberar el espacio sin necesidad de un sistema POS externo.

---

## Objetivo

Después de calcular el valor a pagar (RF-134), el operador registra el pago directamente desde la app móvil. El sistema acepta efectivo (con cálculo de cambio si aplica) y transferencia (con registro del método). Una vez confirmado el pago, el ticket se cierra y el espacio se libera.

---

## Actores

- **Actor principal**: Operador de parqueadero (`operator_role`)
- **Actor secundario**: Supervisor de sede, Cajero

---

## Precondiciones

- Existe un ticket ABIERTO con cálculo de salida realizado (RF-134).
- El monto a pagar está definido y visible para el operador.

---

## Postcondiciones

### En caso de éxito

- El ticket pasa a estado CERRADO.
- El pago se registra en el sistema.
- El espacio se libera.
- Se imprime/comparte el comprobante si se solicita.

### En caso de falla

- Si el pago falla, el ticket permanece ABIERTO.
- No se libera el espacio.

---

## Flujo Funcional

1. Desde la pantalla de resumen de salida (RF-134), el operador toca "Registrar Pago".
2. El operador selecciona el método de pago:
   - **Efectivo**: ingresa el monto recibido; el sistema calcula el cambio.
   - **Transferencia**: registra el número de referencia de la transferencia.
3. El operador confirma los datos del pago.
4. El operador toca "Confirmar Pago".
5. El sistema valida:
   - El monto recibido >= total a pagar (efectivo).
   - La referencia de transferencia no esté duplicada en las últimas 2 horas.
6. El sistema registra el pago y actualiza el ticket a CERRADO.
7. El espacio se marca como disponible.
8. Se muestra confirmación con: ticket cerrado, monto pagado, método, número de comprobante.
9. Opcionalmente, el operador puede imprimir o enviar el comprobante por WhatsApp/email.

---

## Métodos de Pago Soportados

| Método | Datos Requeridos | Validación |
|--------|------------------|------------|
| EFECTIVO | monto_recibido | monto_recibido >= total_pagar |
| TRANSFERENCIA | referencia, banco, tipo_banco | referencia única en ventana de 2h |
| NEQUI | numero_telefono, referencia | referencia única en ventana de 2h |
| DAVIPLATA | numero_telefono, referencia | referencia única en ventana de 2h |
| TARJETA | últimos_4_digitos, tipo | sin validación adicional (integración futura) |

---

## Validaciones

| Campo | Regla |
|-------|-------|
| monto_recibido | Requerido si método=EFECTIVO; debe ser >= total_pagar |
| monto_recibido | No puede ser negativo |
| referencia | Requerido si método=TRANSFERENCIA/NEQUI/DAVIPLATA; máximo 20 caracteres |
| banco | Opcional para transferencia; requerido según configuración de sede |

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| ticket_id | UUID | ID del ticket a cerrar | Sí |
| metodo_pago | Enum | EFECTIVO, TRANSFERENCIA, NEQUI, DAVIPLATA, TARJETA | Sí |
| monto_recibido | Decimal | Monto recibido (efectivo) | No |
| monto_cambio | Decimal | Cambio a devolver (calculado) | No |
| referencia | String | Número de referencia de transferencia | No |
| banco | String | Nombre del banco (transferencia) | No |
| telefono | String | Número de teléfono (Nequi/Daviplata) | No |
| operador_id | UUID | ID del operador que registra | Sí (del token) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| pago_id | UUID | Identificador del pago |
| ticket_id | UUID | Identificador del ticket |
| numero_comprobante | String | Número de comprobante (ej: "PAG-2024-00847") |
| monto_pagado | Decimal | Monto total pagado |
| monto_cambio | Decimal | Cambio devuelto (0 si no aplica) |
| metodo_pago | Enum | Método utilizado |
| referencia | String | Referencia del pago |
| fecha_hora | DateTime | Timestamp del pago |
| ticket_estado | Enum | CERRADO |
| espacio_liberado | Boolean | Si el espacio fue liberado |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Monto recibido < total | Se muestra error "Monto insuficiente. Faltan $X" y no se permite continuar |
| Referencia duplicada | Se muestra error "Esta referencia ya fue usada. Verifique" |
| Transferencia sin referencia | Se muestra error de validación |
| Ticket bloqueado | No se permite registrar pago; se muestra "Ticket bloqueado. Contacte supervisor" |
| Ticket ya cerrado | Se muestra error "Este ticket ya fue pagado" |
| Error de sistema durante pago | El pago no se registra; el ticket permanece ABIERTO; se muestra mensaje de error |
| Operador cancela a mitad | No se registra nada; se regresa a la pantalla de resumen |
| Método=EFECTIVO sin cambio | Se muestra "No hay cambio. Pago exacto" |

---

## Criterios de Aceptación

1. El operador puede registrar un pago en menos de 10 segundos.
2. El sistema calcula correctamente el cambio para pagos en efectivo.
3. El ticket pasa a estado CERRADO inmediatamente tras la confirmación.
4. El espacio se libera al mismo tiempo que el cierre del ticket.
5. No se permite registrar un pago sobre un ticket ya cerrado.
6. El comprobante contiene: número de ticket, placa, hora entrada/salida, monto, método, operador.
7. La referencia de transferencia se valida contra duplicados en las últimas 2 horas.
8. Los ingresos del día se actualizan en el dashboard tras el pago.

---

## Métricas de Éxito

- Registro de pago en < 3 segundos.
- 0 pagos duplicates con misma referencia en ventana de 2 horas.
- Cierre de ticket y liberación de espacio simultáneos en < 1 segundo.
- Actualización del dashboard en < 5 segundos tras el pago.

---

## Notas

- Para pagos en efectivo, siempre se debe mostrar el cálculo de cambio al operador antes de confirmar.
- El comprobante debe poder enviarse por WhatsApp con un botón directo.
- Si la sede requiere imprimmir ticket en cada salida, se envía a una impresora Bluetooth conectada.
