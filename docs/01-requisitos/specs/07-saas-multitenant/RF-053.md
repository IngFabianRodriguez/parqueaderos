# RF-053 — Rate Limits por API Key Según el Plan del Tenant

## Metadata

- **Módulo**: SaaS Multi-Tenant — API Keys y Acceso Externo
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: api-gateway, rate-limit-service, tenant-service

---

## User Story

**Como** administrador de un tenant **quiero** conocer los límites de requests por minuto que aplica a cada API key **para** diseñar mis integraciones dentro de esos límites y escalar mi uso si necesito más.

---

## Objetivo

El sistema debe definir rate limits por API key basados en el plan del tenant, aplicando límites diferentes según el plan (Starter, Professional, Enterprise, Custom). Cada API key tendrá un límite de requests por minuto (RPM) y por mes (RPMonth), y el API Gateway rechazará requests que excedan esos límites con código 429 Too Many Requests.

---

## Actores

- **Actor principal**: Sistema (aplica rate limits en el API Gateway)
- **Actor secundario**: Integración del tenant (consume el API dentro de los límites)

---

## Precondiciones

- La API key existe y está activa (RF-051).
- El tenant tiene un plan con límite de rate limit asociado (RF-037).
- Los rate limits se configuran por key en la tabla `api_keys`.

---

## Postcondiciones

### En caso de éxito

- El request se procesa normalmente; el contador de uso se incrementa.
- El header `X-RateLimit-Remaining` indica los requests restantes en la ventana actual.
- El header `X-RateLimit-Reset` indica cuándo se reinicia la ventana.

### En caso de exceder el límite

- El API Gateway retorna `429 Too Many Requests`.
- El header `Retry-After` indica los segundos hasta que se pueda hacer el siguiente request.
- Se genera el evento `rate_limit_exceeded` para métricas.

---

## Rate Limits por Plan

| Plan | Requests por Minuto (RPM) | Requests por Mes |
|------|---------------------------|------------------|
| Starter | 60 | 10,000 |
| Professional | 300 | 100,000 |
| Enterprise | 1,000 | 1,000,000 |
| Custom | 10,000 | Ilimitado |

---

## Límites Adicionales por Scope

Para evitar que un scope consuma todo el presupuesto de la key, se aplican sub-límites por scope:

| Scope | % del rate limit total aplicable |
|-------|----------------------------------|
| `disponibilidad:read` | 30% del total (lecturas frecuentes) |
| `transacciones:write` | 20% del total (escrituras menos frecuentes) |
| `pagos:write` | 10% del total (crítico) |
| Otros scopes | 40% del total |

---

## Flujo Funcional

1. API Gateway recibe request con `Authorization: Bearer pk_live_xxx`.
2. Se extrae la key; se buscan rate limits asociados al plan del tenant.
3. Se verifica si la key tiene un override de rate limit (casos especiales configurados manualmente).
4. Se determina la ventana de rate limit (`window = 60 segundos` para RPM).
5. Se incrementa el contador en Redis o similar: `rate_limit:{key_id}:{window}` con TTL 60s.
6. Se compara `count` vs `limit`:
   - **Si count <= limit**: request pasa; se añaden headers.
   - **Si count > limit**: request rechaza con 429.
7. Headers de respuesta:
   - `X-RateLimit-Limit`: límite de la ventana (ej: 300)
   - `X-RateLimit-Remaining`: requests restantes (limit - count)
   - `X-RateLimit-Reset`: timestamp Unix de fin de ventana
   - `Retry-After`: solo presente en 429, segundos hasta retry

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tenant_id | UUID | Identificador del tenant (determina el plan) | Sí |
| key_id | UUID | Identificador de la key | Sí |

---

## Datos de Salida (headers)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| X-RateLimit-Limit | integer | Límite de requests en la ventana |
| X-RateLimit-Remaining | integer | Requests restantes |
| X-RateLimit-Reset | integer | Timestamp Unix de fin de ventana |
| Retry-After | integer | Segundos de espera (solo en 429) |

---

## Response 429 (Excedió rate limit)

```json
{
  "error": "rate_limit_exceeded",
  "message": "Has excedido el límite de requests por minuto. Espera 30 segundos.",
  "limit": 300,
  "retry_after": 30,
  "plan": "Professional",
  "upgrade_url": "https://app.parkcore.co/settings/subscription"
}
```

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Límite de RPM alcanzaod | Retorna 429; el cliente debe esperar y reintentar después de `Retry-After`. |
| Límite mensual alcanzado | Retorna 429 con mensaje diferente: \"Límite mensual alcanzado. Espera hasta [fecha].\" |
| Límite por scope excedido | Retorna 429 específico: \"Scope transacciones:write excedido. Usa menos frecuencia.\" |
| Redis caído (no puede contar) | Fail-open: permitir el request; loguear alerta de dependencia caída. |
| Key sin rate limit configurado | Aplicar el límite default del plan del tenant. |
| Plan del tenant cambia (upgrade) | Los rate limits se recalculan con los del nuevo plan en el siguiente request. |

---

## Criterios de Aceptación

1. Cada request al API Gateway incrementa el contador de rate limit de la key correspondiente.
2. Cuando el contador alcanza el límite, el siguiente request retorna 429 sin procesar.
3. Los headers de rate limit se incluyen en todas las respuestas (tanto éxito como rechazo parcial).
4. Los rate limits se basan en el plan del tenant; Starter tiene 60 RPM, Professional 300, Enterprise 1,000, Custom 10,000.
5. Un tenant puede tener keys con rate limits diferentes si algunas keys tienen overrides manuales.
6. El evento `rate_limit_exceeded` se genera cada vez que se rechaza un request por límite.
7. El contador se resetea al inicio de cada ventana de 60 segundos.

## Endpoints API Involucrados

---

## Métricas de Éxito

- Rate limit rejections (429) como % del total de requests: < 2%.
- Time de validación de rate limit: < 3ms por request.
- Accuracy del contador: 100% (usando Redis atomic INCR).
- Alertas cuando rate limit usage > 80% del límite: para anticipar problemas.

---

## Notas

- El rate limit se aplica por API key, no por IP ni por tenant. Cada key tiene su propio bucket.
- El límite mensual es soft: se puede exceder pero se factura como overage (RF-039).
- Para casos de emergency/beta, el equipo de ParkCore puede crear keys con `rate_limit_override` más alto.
- Los headers de rate limit son estándar HTTP (`X-RateLimit-*`) para compatibilidad con cliente HTTP estándar.
- El `Retry-After` header usa segundos, no timestamp, que es el formato recomendado por RFC 7231.
