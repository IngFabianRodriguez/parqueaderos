# RF-048 — Aplicación de Branding del Tenant en Emails Transaccionales

## Metadata

- **Módulo**: SaaS Multi-Tenant — Branding y Personalización
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: notification-service, branding-service, tenant-service, email-service

---

## User Story

**Como** operador/administrador de un tenant **quiero** que los emails enviados por ParkCore (notificaciones de entrada/salida, facturas, alertas) muestren mi logo y colores **para** que los destinatarios perciban la comunicación como oficial de mi empresa.

---

## Objetivo

El sistema debe inyectar automáticamente el branding del tenant (logo, colores, email_remitente) en todos los emails transaccionales que se originen desde el tenant, incluyendo el email Remitente (from address), el logo en el header, los colores en los botones y el pie de página.

---

## Actores

- **Actor principal**: Sistema (arma y envía los emails con branding del tenant)
- **Actor secundario**: Destinatario del email (cliente final, operador del tenant)

---

## Precondiciones

- El tenant tiene branding configurado en `tenant_branding`.
- El email transaccional a enviar tiene una plantilla base en el sistema.
- El remitente del email está configurado en `notification_channels`.

---

## Postcondiciones

### En caso de éxito

- El email se envía con el from address del tenant (ej: \"Parking Plus <notificaciones@parkingplus.co>\").
- El header del email muestra el logo del tenant con link al dashboard.
- Los botones y enlaces usan el color primario del tenant.
- El footer del email muestra el nombre de la empresa y dirección del tenant.
- El logo del tenant aparece también como imagen open-graph en el email.

### En caso de falla

- Si el tenant no tiene branding, se usa el branding default de ParkCore.
- Si el logo no carga en el email (cliente email no soporta imágenes), el email sigue siendo funcional.

---

## Flujo Funcional

1. Un evento del sistema dispara el envío de un email transaccional (ej: `invoice.created`, `vehicle.entered`, `payment.confirmed`).
2. El `notification-service` recibe el evento con `tenant_id`, `template_name`, `recipient_email`, `template_data`.
3. El servicio consulta `GET /api/v1/tenants/{tenant_id}/branding` para obtener logo y colores.
4. El servicio consulta `GET /api/v1/tenants/{tenant_id}/notification-channels` para obtener el `from_address` configurado.
5. El sistema procesa la plantilla del email:
   a. Reemplaza variables de plantilla: `{{cliente_nombre}}`, `{{monto}}`, `{{sede}}`, etc.
   b. Inyecta logo del tenant en el header: `<img src="{logo_url}" alt="{tenant_name}">`.
   c. Aplica colores del tenant en los estilos inline: `color: {primary_color}` para botones.
   d. Genera el footer con nombre de empresa, dirección y datos del tenant.
6. El `from_address` se configura como: `{tenant_display_name} <{tenant_email_slug}@parkcore.co>` o el custom domain si está configurado (RF-049).
7. El email se envía via email provider (SendGrid, AWS SES, etc.) con tracking de apertura y clicks.
8. Se genera evento `email_sent` con: `tenant_id`, `recipient`, `template_name`, `delivered_at`.

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tenant_id | UUID | Identificador del tenant | Sí |
| template_name | string | Nombre de la plantilla (invoice, entry_notification, etc.) | Sí |
| recipient_email | string | Email del destinatario | Sí |
| template_data | object | Datos para interpolar en la plantilla | Sí |
| from_address_override | string | Sobrescribir from address (solo para emails críticos) | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| email_id | string | ID del email enviado (del provider) |
| from_address | string | Address de remitente usado |
| logo_url | string | URL del logo inyectado |
| delivered_at | datetime | Timestamp de envío |
| open_tracked | boolean | Si el email tiene tracking de apertura |
| click_tracked | boolean | Si el email tiene tracking de clicks |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Tenant no tiene branding | Usar default ParkCore; email se envía normalmente. |
| Logo no carga en el email | Mostrar texto con nombre del tenant como fallback; no bloquear envío. |
| Custom domain configurado | Usar `notificaciones@{custom_domain}` como from address. |
| Email provider falla | Guardar email en cola para reintento; generar evento `email_failed`. |
| Destinatario tiene cliente email que bloquea imágenes | El email es funcional sin imágenes; los botones siguen siendo visibles. |
| Template no existe | Usar plantilla genérica; loguear error como `template_not_found`. |

---

## Criterios de Aceptación

1. Todos los emails transaccionales llevan el logo del tenant en el header.
2. El `from_address` de los emails muestra el nombre del tenant, no \"ParkCore\" genérico.
3. Los botones de los emails usan el `primary_color` del tenant.
4. El footer del email muestra el nombre y datos de la empresa del tenant.
5. Los colores y logo del tenant se aplican sin demora perceptible en el envío.
6. Si el tenant tiene custom domain, el from address usa ese dominio.
7. Los eventos `email_sent`, `email_delivered`, `email_opened`, `email_bounced` se registran en la tabla de eventos SaaS.

- `POST /api/v1/notifications/email` — Enviar email transaccional con branding

---

## Métricas de Éxito

- Emails enviados con branding correcto > 99.5%.
- Tiempo de inyección de branding < 200ms.
- Tasa de emails marcados como spam < 0.1% (beneficio de usar from address del tenant).
- Tracking de apertura activo en 100% de los emails.

---

## Notas

- El branding de email se implementa usando Email CSS inlining (los estilos se aplican inline porque muchos clientes de email no soportan CSS externo o `<style>` en el `<head>`).
- El logo se incluye como imagen embebida (base64) para algunos clientes de email que bloquean imágenes externas; para otros se usa URL pública con fallback.
- El `from_address` del tenant puede ser override por el sistema en casos de emergencia (ej: suspensión del tenant) donde ParkCore necesita comunicarse directamente.
- Los colores del tenant en emails deben tener contraste suficiente para texto sobre fondo blanco; el sistema valida esto al guardar el branding.
