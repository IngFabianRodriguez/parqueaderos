# RF-039 — Tracking de Uso Excedente y Facturación al Cierre del Ciclo

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, usage-tracking-service, tenant-service

---

## User Story

**Como** administrador de ParkCore **quiero** que cuando un tenant exceda los límites de su plan (transacciones, sedes, API calls) se.trackee el uso excedente y se incluya en la factura del mes **para** cobrar justamente por lo que se usa.

---

## Objetivo

El sistema debe trackear el uso de recursos de cada tenant en tiempo real, comparar con los límites de su plan, y al cierre del ciclo de facturación calcular los excedentes para agregarlos a la invoice mensual como cargos adicionales (overage).

---

## Actores

- **Actor principal**: Sistema (tracking automático)
- **Actor secundario**: Tenant admin (recibe notificación de uso cercano al límite)

---

## Precondiciones

- El tenant tiene estado `active` con una suscripción activa.
- El plan del tenant tiene límites definidos (transactions, sedes, users, API calls).

---

## Postcondiciones

### En caso de tracking normal

- El sistema incrementа el contador de uso cada vez que ocurre el evento relevante.
- Los contadores se almacenan por tenant por mes (`YYYY-MM`).
- Al acercarse al límite (80%), se envía notificación al tenant.

### En caso de excedente al cierre del ciclo

- El sistema calcula el uso total del mes vs el límite incluido en el plan.
- Los excedentes se convierten en cargos adicionales en la invoice.
- Se genera el evento `overage_charged` con detalle de los excedentes.
- Se envía notificación al tenant con el resumen de excedentes facturados.

---

## Flujo Funcional

### Tracking continuo

1. Cada vez que ocurre un evento de uso, el servicio correspondiente incrementa el contador.
   - Transaction creada → `usage_transactions += 1`
   - Nueva sede creada → `usage_sedes += 1`
   - Nuevo usuario creado → `usage_users += 1`
   - API call recibida → `usage_api_calls += 1`
2. El contador se almacena en tabla `usage_metrics` con: `tenant_id`, `YYYY_MM`, `resource_type`, `count`.
3. Cada vez que se incrementa, el sistema verifica si el tenant está cerca del límite (80%):
   - Si `usage >= limit * 0.8`: enviar email de advertencia "Estás usando X% de tu límite de Y."
   - Si `usage >= limit`: se permite continuar pero se marca para billing como overage.
4. Los contadores se resetean al inicio de cada mes (nuevo `YYYY_MM`).

### Cálculo de excedentes al cierre del ciclo

1. El job de billing se ejecuta el día `current_period_end` (o el día siguiente si es festivo).
2. Para cada tenant activo:
   a. Consultar límites del plan en `plans`.
   b. Consultar uso total del mes en `usage_metrics`.
   c. Para cada recurso donde `usage > limit`: calcular `overage = usage - limit`.
   d. Calcular cargo overage: `overage_units * precio_por_unit` (definido en `plans.overage_pricing`).
   e. Agregar los cargos overage a la invoice del mes.
3. Generar línea de invoice para cada tipo de overage con descripción detallada.
4. Generar evento `overage_charged` con: `tenant_id`, `resource_type`, `units_over`, `unit_price`, `total_charge`.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tenant_id | UUID | Identificador del tenant | Sí |
| resource_type | enum | Tipo de recurso: transactions, sedes, users, api_calls | Sí |
| increment | integer | Cantidad a incrementar (normalmente 1) | Sí |
| timestamp | datetime | Momento del evento | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| usage_current | integer | Uso actual del recurso |
| usage_limit | integer | Límite del plan |
| usage_percentage | decimal | Porcentaje de uso |
| overage_units | integer | Unidades excedentes (0 si no hay) |
| overage_charge | decimal | Cargo adicional por excedentes |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Límite de usuarios alcanzado al crear nuevo usuario | El sistema permite crear el usuario y lo marca como overage; se factura al final del mes. |
| Plan sin límites (Custom) | Los límites están vacíos o en null; no se aplica tracking de overage. |
| Uso en el mes exactamente igual al límite | No hay overage; uso_percentage = 100%. |
| Usage tracking falla (servicio caído) | El contador se reconstruye batch desde los logs de transacciones al reiniciar; no se pierde uso. |
| Overage de API calls muy alto ( > 10x límite) | Se aplica un teto máximo de overage facturable (ej: no más de 5x el precio base); el resto se trunca. |

---

## Criterios de Aceptación

1. Cada evento de uso (transacción creada, usuario añadido) incrementa el contador correspondiente en menos de 1 segundo.
2. El contador de uso se muestra en el dashboard del tenant_admin ( Settings > Uso).
3. Se envía email de advertencia cuando el uso alcanza el 80% del límite.
4. Al cierre del ciclo, los excedentes se calculan correctamente y se agregan a la invoice.
5. La línea de invoice muestra: recurso excedido, unidades, precio unitario, total.
6. El evento `overage_charged` se genera por cada tipo de recurso con excedente.
7. Los contadores se resetean al inicio de cada mes; el historial se mantiene para reportes.

---

## Métricas de Éxito

- Tiempo de incremento de contador < 100ms
- Overage facturado como % del MRR total: benchmark < 5%
- Accuracy de contadores: 100% (auditado vs logs)
- Email de advertencia enviado cuando corresponde: 100%

---

## Notas

- El tracking de uso se hace async para no impactar latencia de las operaciones principales.
- Los contadores se almacenan en una tabla particionada por mes para performance.
- Para planes con límites muy altos (Enterprise, Custom), el tracking de uso sigue activo para métricas internas pero no se factura overage.
- El precio por unidad de overage se define en la tabla `plans.overage_pricing` y se muestra al tenant en los detalles de su plan.
