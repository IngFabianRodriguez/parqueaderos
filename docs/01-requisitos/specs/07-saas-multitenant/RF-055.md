# RF-055 — Registro de Último Uso de API Key para Auditoría

## Metadata

- **Módulo**: SaaS Multi-Tenant — API Keys y Acceso Externo
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: api-gateway, auth-service, audit-service

---

## User Story

**Como** administrador de un tenant **quiero** ver cuándo se usó por última vez cada API key **para** detectar llaves expuestas o integraciones abandonadas y hacer limpieza de seguridad.

---

## Objetivo

El sistema debe registrar el timestamp del último uso de cada API key y mantener un log de las últimas N requests realizados con esa key, permitiendo al admin consultar el historial de uso y detectar actividad anómala.

---

## Actores

- **Actor principal**: Tenant admin (consulta logs de uso)
- **Actor secundario**: Sistema (registra cada request)

---

## Precondiciones

- La API key existe y está activa o fue revocada recientemente.

---

## Postcondiciones

### En caso de registro de uso

- El contador de requests se incrementa.
- El campo `last_used_at` se actualiza al timestamp actual.
- Los logs de las últimas N requests se appendean a `api_key_logs`.

---

## Flujo Funcional

1. Cada request que pasa la validación de la API key (RF-051, RF-052, RF-053) dispara un registro de uso.
2. El API Gateway escribe en Redis o base de datos:
   a. `api_keys.last_used_at = NOW()`.
   b. `api_keys.request_count += 1`.
   c. Se appendea a `api_key_logs`: `{timestamp, method, path, status_code, response_time_ms}`.
3. Se mantiene un buffer circular de los últimos 1,000 requests por key en Redis para acceso rápido.
4. Periódicamente (cada hora), se sincroniza el contador a la base de datos principal para persistencia.
5. El admin consulta `GET /api/v1/api-keys/{key_id}/usage` y ve:
   - Resumen: total requests, último uso, requests en las últimas 24h.
   - Logs recientes: lista de últimos 100 requests con detalle.

---

## Datos de Entrada (registro automático)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| key_id | UUID | Identificador de la key usada |
| timestamp | datetime | Momento del request |
| method | string | HTTP method (GET, POST, etc.) |
| path | string | Endpoint solicitado |
| status_code | integer | Código de respuesta HTTP |
| response_time_ms | integer | Tiempo de respuesta en ms |
| ip_address | string | IP origen del request |
| user_agent | string | User agent del cliente |

---

## Datos de Salida (consulta de uso)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| key_id | UUID | Identificador de la key |
| total_requests | integer | Total de requests desde la creación |
| last_used_at | datetime | Timestamp del último request |
| requests_last_24h | integer | Requests en las últimas 24 horas |
| requests_last_7d | integer | Requests en los últimos 7 días |
| avg_response_time_ms | integer | Tiempo promedio de respuesta |
| recent_logs | array | Últimos 100 requests con detalle |
| error_rate | decimal | Porcentaje de requests que retornaron 4xx/5xx |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Redis caído durante el registro | Fail-open: el request pasa; el log se escribe async a DB cuando Redis恢复. |
| Key no usada nunca | `last_used_at = null`; `total_requests = 0`. |
| Logs de key revocada | Se mantienen accesibles por 2 años para auditoría. |
| Overflow del buffer circular | Los logs más antiguos se descartan; solo se mantienen los últimos 1,000. |
| Uso anómalo detectado ( spike) | Se genera alerta al admin; posible indicador de key expuesta. |

---

## Criterios de Aceptación

1. Cada request exitoso con API key actualiza `last_used_at` en menos de 50ms.
2. El admin puede ver el resumen de uso desde Settings > API Keys en tiempo real.
3. Los logs de uso muestran los últimos 100 requests con method, path, timestamp, status.
4. Se genera alerta si el uso de una key aumenta > 10x en 1 hora vs el promedio histórico.
5. Los logs de keys revocadas persisten por 2 años para auditoría.
6. El total de requests en la UI coincide con los logs de auditoría.

## Endpoints API Involucrados

- `GET /api/v1/api-keys/{key_id}/usage` — Ver resumen de uso y logs
- `GET /api/v1/api-keys/{key_id}/logs` — Ver logs detallados con paginación

---

## Métricas de Éxito

- Keys sin uso en > 90 días: identificar y potencialmente revocar.
- Detección de keys expuestas: > 80% de las keys expuestas se detectan por spike en logs.
- Tiempo de registro de uso: < 50ms por request.

---

## Notas

- El registro de uso es async para no impactar la latencia del request principal.
- Se usa Redis para el buffer circular de logs por performance; la persistencia va a PostgreSQL periódicamente.
- Las alertas de uso anómalo se configuran por key, no globalmente.
