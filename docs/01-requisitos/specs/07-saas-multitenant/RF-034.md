# RF-034 — Aislamiento Completo de Datos Entre Tenants

## Metadata

- **Módulo**: SaaS Multi-Tenant — Seguridad y Aislamiento
- **Prioridad**: Crítica
- **Tipo**: No-Funcional (Seguridad)
- **Servicios afectados**: todos los servicios (database layer, api-gateway, auth-service)

---

## User Story

**Como** administrador de un tenant **quiero** estar seguro de que mis datos (transacciones, clientes, ingresos) son invisibles para todos los demás tenants **para** proteger la confidencialidad de mi información de negocio.

---

## Objetivo

El sistema debe garantizar que los datos de cada tenant están completamente aislados unos de otros, de manera que ninguna query, API o proceso pueda acceder, modificar o consultar datos de un tenant diferente. El aislamiento debe ser enforceable tanto a nivel de base de datos ( row-level security) como a nivel de aplicación ( middleware de tenant) y a nivel de red.

---

## Actores

- **Actor principal**: Sistema ( enforce de aislamiento en todas las capas)
- **Actor secundario**: Todos los usuarios autenticados de todos los tenants

---

## Precondiciones

- El tenant tiene un `tenant_id` único asignado en el momento de su creación.
- El usuario está autenticado con un JWT que contiene el `tenant_id` en sus claims.
- El token JWT ha sido validado por el auth-service.

---

## Postcondiciones

### En caso de éxito

- Cada consulta a la base de datos está automáticamente filtrada por `tenant_id` del usuario autenticado.
- Los datos de un tenant no son visibles en los resultados de queries de otro tenant.
- Los logs de auditoría no contienen datos cruzados entre tenants.
- Los webhooks salientes solo contienen datos del tenant que los disparó.

### En caso de falla (intento de acceso cruzado)

- El sistema retorna error 403 Forbidden.
- Se genera un evento de seguridad `unauthorized_cross_tenant_access` con: `tenant_id_actual`, `tenant_id_objetivo`, `recurso_solicitado`, `timestamp`, `ip`.
- Se envía alerta al equipo de seguridad si el patrón se repite ( > 3 intentos en 1 hora).

---

## Flujo Funcional

### Capa de aplicación: Middleware de Tenant

1. Todo request HTTP pasa por el middleware de autenticación.
2. El middleware extrae el `tenant_id` del JWT (`sub` o claim `tenant_id`).
3. El middleware inyecta el `tenant_id` en el contexto de la request (`req.tenantId`).
4. Cada handler de API recibe el contexto con el `tenant_id` y lo usa en todas las queries.
5. Si un handler intenta consultar un recurso sin el `tenant_id` del contexto, se retorna error 400.

### Capa de base de datos: Row-Level Security (RLS)

1. Todas las tablas del schema de negocio tienen columna `tenant_id` (UUID, not null).
2. Se habilita Row-Level Security en PostgreSQL para todas las tablas del schema `parkcore`.
3. Cada tabla tiene policies de RLS que filtran rows WHERE `tenant_id = current_setting('app.tenant_id')`.
4. La aplicación setea el setting `app.tenant_id` al inicio de cada conexión DB con el valor del JWT del usuario actual.
5. Queries desde otros contextos ( migrations, seeders) setean `app.tenant_id` explícitamente o usan bypass de RLS solo para superuser de DB (restringido a pipelines de deployment).

### Capa de API: Validación de ownership

1. Cada endpoint GET/ PUT/ DELETE valida que el recurso solicitado pertenece al `tenant_id` del usuario.
2. Si se intenta acceder a un recurso de otro tenant (por ejemplo, modificando el `tenant_id` en el body), se retorna 403.
3. Los IDs de recursos (transaction_id, user_id, sede_id) incluyen el `tenant_id` en su composición ( ULID con prefix) para dificultar enumeración.

### Capa de eventos y webhooks

1. Los eventos internos ( Kafka topics) incluyen `tenant_id` como key de-partition para garantizar que consumers de otro tenant no reciban los eventos.
2. Los webhooks salientes se emiten solo con datos del tenant correspondiente; el `tenant_id` se valida antes de construir el payload.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| JWT sin `tenant_id` claim | Retornar 401 Unauthorized; no permitir acceso a ningún recurso. |
| Token JWT válido pero tenant_id no existe en DB | Retornar 401; invalidate el token; forzar re-login. |
| Request intenta modificar recurso de otro tenant | Retornar 403 Forbidden; registrar en auditoría de seguridad. |
| Query intenta acceder a tabla sin `tenant_id` (error de código) | El RLS de Postgres retorna 0 rows; no se expone error al usuario; se alerta al equipo de desarrollo. |
| Conexión de base de datos sin tenant context set | La conexión falla con error; no se ejecuta la query. |
| Admin de tenant A intenta inferir existencia de tenant B por tiempos de respuesta | El aislamiento es total; no hay diferencias medibles. |
| Auditor intenta acceder a datos de otro tenant | Retornar 403; el rol auditor solo lee datos del tenant de su propio tenant_admin. |

---

## Criterios de Aceptación

1. Cada tabla en el schema `parkcore` tiene columna `tenant_id` not null con índice B-tree.
2. PostgreSQL RLS está habilitado y configurado en todas las tablas del schema de negocio.
3. Un query hecho con `tenant_id = A` retorna 0 rows si el recurso pertenece a `tenant_id = B`.
4. Intentar hacer `UPDATE tenants SET tenant_id = B WHERE id = A` retorna error de permisos.
5. El middleware de tenant retorna 403 si el JWT no contiene `tenant_id`.
6. Los eventos de Kafka usan `tenant_id` como key de partición.
7. Un intento de acceso cruzado genera un evento de seguridad con todos los datos de diagnóstico.
8. Los logs de auditoría de un tenant no contienen nombres, emails o datos de otros tenants.
9. Los rate limits por API key (RF-053) se aplican por tenant, no globalmente.

## Métricas de Éxito

- 0 incidentes de cross-tenant data access en producción (medido mensualmente).
- RLS policies activas en 100% de las tablas del schema de negocio.
- Tiempo de query no aumenta más de 5% por overhead de RLS.
- Cada evento de seguridad `unauthorized_cross_tenant_access` es revisado en menos de 1 hora.

---

## Notas

- La decisión de diseño de usar `tenant_id` como columna en todas las tablas (en lugar de esquemas separados por tenant) es para facilitar migraciones y mantener la arquitectura más simple. El RLS de Postgres hace transparente el filtro para la aplicación.
- El uso de ULID con prefix de tenant (ej: `TNT1-xxxxxxxx`) para los IDs públicos dificulta la enumeración de recursos, pero no es el mecanismo principal de seguridad — el RLS es.
- El aislamiento es contra todo tipo de datos: transacciones, clientes, facturas, reportes, configuraciones, usuarios, API keys.
- En caso de bug en el código que intente hacer query sin filtrar por `tenant_id`, RLS proteje que no se devuelvan datos de otro tenant — el query retorna 0 rows y la aplicación retorna 404 (no error de aislamiento).
- Los datos de métricas SaaS agregadas (MRR, Churn) no son datos de un tenant específico; estos se almacenan en un schema separado `saas_metrics` sin `tenant_id`.
