# RF-047 — Aplicación de Branding del Tenant en el Dashboard del Operador

## Metadata

- **Módulo**: SaaS Multi-Tenant — Branding y Personalización
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: branding-service, frontend-service, tenant-service

---

## User Story

**Como** operador de un tenant **quiero** ver el dashboard con los colores y logo de mi empresa **para** que la experiencia se sienta propia y no como una herramienta genérica.

---

## Objetivo

El sistema debe aplicar dinámicamente el branding configurado (logo, color primario, color secundario) del tenant en todas las pantallas del dashboard que ven los operadores y usuarios del tenant, incluyendo la barra superior, botones primarios, acentos visuales y favicon.

---

## Actores

- **Actor principal**: Operador / usuario del tenant (consume el branding)
- **Actor secundario**: Sistema (inyecta el branding dinámicamente)

---

## Precondiciones

- El tenant tiene branding configurado en `tenant_branding` (o valores por defecto si no hay).
- El operador tiene sesión activa en el dashboard del tenant.

---

## Postcondiciones

### En caso de éxito

- El dashboard del operador muestra el logo del tenant en la barra superior.
- Los botones primarios y elementos de acción usan el color primario del tenant.
- Los acentos y bordes usan el color secundario del tenant.
- El favicon del navegador muestra el logo del tenant.
- Los emails disparados desde el dashboard incluyen el logo del tenant.

### En caso de falla

- Si no hay branding configurado, se usan los valores por defecto de ParkCore.
- Si el logo_url está caído o retorna 404, se muestra el logo default de ParkCore.

---

## Flujo Funcional

1. El operador accede a `https://{slug}.parkcore.co/dashboard`.
2. El frontend consulta `GET /api/v1/tenants/{tenant_id}/branding` para obtener logo y colores.
3. El frontend inyecta el branding dinámicamente:
   a. La barra superior muestra el logo del tenant (izquierda).
   b. Los CSS variables (`--primary-color`, `--secondary-color`) se establecen con los valores del tenant.
   c. El `<link rel="icon">` se actualiza con el logo del tenant.
4. Todos los componentes del dashboard leen las CSS variables para sus estilos:
   - Botones primarios: `background-color: var(--primary-color)`
   - Hover states: `var(--primary-color)` con 10% más oscuro
   - Borde de elementos activos: `var(--secondary-color)`
   - Fondos de cards: `var(--secondary-color)` con 5% opacidad
5. El branding se cachea en el navegador por 1 hora (`Cache-Control: max-age=3600`).
6. Si el operador cambia de sede dentro del mismo tenant, el branding persiste (es por tenant, no por sede).

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tenant_id | UUID | Identificador del tenant | Sí |
| session_token | string | Token JWT de la sesión del operador | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| logo_url | string | URL del logo del tenant |
| primary_color | string | Color primario (#RRGGBB) |
| secondary_color | string | Color secundario (#RRGGBB) |
| favicon_url | string | URL del favicon ( mismo logo en 32x32) |
| tenant_name | string | Nombre de la empresa del tenant |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Branding no configurado | Usar valores por defecto: logo ParkCore, #2563EB, #1E40AF. |
| logo_url retorna 404 | Mostrar logo default de ParkCore; loguear warning. |
| Logo carga lento (> 2s) | Mostrar placeholder con iniciales del tenant hasta que cargue. |
| Colores no tienen contraste WCAG AA | El sistema permite guardar pero muestra warning \"Los colores elegidos pueden afectar la accesibilidad\". |
| Custom domain configurado (RF-049) | El branding también se aplica en el custom domain. |

---

## Criterios de Aceptación

1. El logo del tenant aparece en la barra superior del dashboard en menos de 1 segundo después de cargar.
2. Los botones primarios del dashboard usan el `primary_color` del tenant.
3. Los acentos y bordes usan el `secondary_color` del tenant.
4. El favicon del navegador se actualiza con el logo del tenant.
5. Los cambios de branding se reflejan inmediatamente sin necesidad de recargar la página.
6. El branding es consistente en todas las páginas del dashboard del tenant.
7. Si el tenant tiene custom domain, el branding se aplica también en ese dominio.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/branding` — Obtener branding del tenant
- `GET /api/v1/assets/{asset_id}` — Servir logo con Cache-Control

---

## Métricas de Éxito

- Tiempo de aplicación del branding post-carga de página < 500ms.
- Branding aplicado en 100% de las páginas del dashboard del tenant.
- Logo default usado cuando hay falla < 0.1% de las cargas.

---

## Notas

- El branding se aplica vía CSS custom properties (variables CSS), lo que permite cambios sin recompilar el frontend.
- El logo se sirve con CDN cache para minimizar latencia.
- Los colores del tenant no afectan la estructura del layout, solo los colores de elementos.
- El tenant_admin ve una opción adicional en Settings > Branding: \"Ver como operador\" para previsualizar.
