# RF-033 — Múltiples Usuarios por Tenant

## Metadata

- **Módulo**: SaaS Multi-Tenant — Gestión de Usuarios
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: auth-service, tenant-service, user-service

---

## User Story

**Como** administrador de un tenant **quiero** invitar a mis colaboradores (managers, operadores) a mi cuenta **para** que puedan gestionar el parqueadero conmigo con sus propios accesos individuales.

---

## Objetivo

El sistema debe permitir que cada tenant tenga múltiples usuarios, cada uno con rol específico dentro de la organización del tenant, credenciales independientes y acceso exclusivamente a los recursos (sedes, zonas) que su rol permita.

---

## Actores

- **Actor principal**: Tenant admin (creador y administrador de usuarios del tenant)
- **Actor secundario**: Usuarios invitados (managers, operadores, auditores)

---

## Precondiciones

- El tenant existe y tiene estado `trial` o `active`.
- El usuario admin tiene sesión activa y rol `tenant_admin`.
- El email del usuario a invitar no está registrado en ningún tenant de la plataforma.

---

## Postcondiciones

### En caso de éxito

- Se crea un usuario en el tenant con rol asignado.
- El usuario recibe email de invitación con enlace para crear contraseña.
- El usuario invitado puede iniciar sesión con su email y contraseña.
- El usuario invitado ve únicamente las sedes y módulos que su rol permite.

### En caso de falla

- El email ya existe en la plataforma: se muestra error indicando que el usuario ya tiene cuenta y debe contactar al admin del tenant para ser añadido.
- El tenant no tiene slots disponibles según su plan: se muestra error de límite de usuarios alcanzado.

---

## Flujo Funcional

### Creación de usuario por tenant_admin

1. Tenant admin accede a Settings > Usuarios > "Invitar usuario".
2. Ingresa: nombre completo, email, rol (manager, operador, auditor).
3. Sistema valida que el email no esté registrado en otro tenant.
4. Sistema valida que el plan del tenant permite la cantidad de usuarios invitados (límite según plan: Starter 3, Professional 10, Enterprise 50, Custom ilimitado).
5. Sistema crea usuario con: `email`, `role`, `tenant_id`, `status = pending_invitation`, `invitation_token = UUID()`, `invitation_sent_at = NOW()`.
6. Sistema envía email de invitación con enlace: `/invite/{token}`.
7. Usuario hace clic en enlace; sistema muestra formulario para crear contraseña.
8. Usuario ingresa contraseña; sistema valida requisitos.
9. Sistema encripta contraseña con bcrypt y guarda; actualiza `status = active`, `password_set_at = NOW()`.
10. Sistema genera sesión JWT y redirige al dashboard del tenant.
11. Usuario aparece en la lista de usuarios del tenant con estado "Activo".

### Roles y permisos

- `tenant_admin`: acceso total a todas las sedes, usuarios, configuraciones, facturación del tenant.
- `manager`: acceso a operaciones de todas las sedes asignadas, reportes, pero no a configuración de billing ni invitación de usuarios.
- `operador`: acceso limitado a la sede asignada; puede abrir/cerrar talanquera, registrar entrada/salida manual, ver dashboard de ocupación.
- `auditor`: solo acceso de lectura a reportes y transacciones; no puede hacer operaciones ni ver datos sensibles de usuarios.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| nombre_completo | string(100) | Nombre del usuario a invitar | Sí |
| email | string(255) | Email del usuario (debe ser único en la plataforma) | Sí |
| rol | enum | Rol: tenant_admin, manager, operador, auditor | Sí |
| sedes_asignadas | array[UUID] | Lista de sedes a las que tiene acceso (para manager/operador) | No (default: todas las sedes) |
| telefono | string(20) | Teléfono de contacto | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| user_id | UUID | Identificador del usuario creado |
| email | string | Email del usuario |
| rol | string | Rol asignado |
| status | enum | `pending_invitation`, `active`, `disabled` |
| invitation_sent_at | datetime | Fecha en que se envió la invitación |
| invitation_token | string | Token único para el enlace de invitación |
|Ultimo acceso | datetime | Última fecha de login del usuario (actualiza en cada login) |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Email ya existe en otro tenant | Retornar error 409 "Este email ya está registrado. Pide al usuario que use otro email o contacta a soporte." |
| Plan no permite más usuarios | Retornar error 403 "Has alcanzado el límite de usuarios de tu plan. Haz upgrade para añadir más." |
| Invitación no aceptada en 7 días | El sistema marca la invitación como `expired`; el admin puede reenviar. |
| Admin intenta invitarse a sí mismo | No tiene sentido; el fundador ya tiene cuenta. Mostrar error "Ya tienes acceso completo como admin." |
| Usuario invitado intenta acceder antes de crear contraseña | El enlace de invitación muestra formulario de creación de contraseña; no puede hacer login hasta completarlo. |
| Admin elimina usuario activo | El usuario pierde acceso inmediatamente; sus datos permanecen en auditoría. |

---

## Criterios de Aceptación

1. Un tenant_admin puede crear hasta el número de usuarios permitido por su plan.
2. Cada usuario tiene credenciales independientes (email + contraseña).
3. Un mismo email no puede pertenecer a dos tenants distintos.
4. Los usuarios ven exclusivamente las sedes y módulos que su rol permite.
5. La invitación expira a los 7 días si no es aceptada.
6. El usuario puede iniciar sesión solo después de crear su contraseña.
7. Un tenant_admin puede desactivar o eliminar usuarios de su tenant.
8. Los usuarios eliminados no pueden hacer login pero sus registros de auditoría persisten.

## Endpoints API Involucrados

- `POST /api/v1/tenants/{tenant_id}/users` — Crear usuario invitado
- `GET /api/v1/tenants/{tenant_id}/users` — Listar usuarios del tenant
- `PUT /api/v1/tenants/{tenant_id}/users/{user_id}` — Editar rol o estado de usuario
- `DELETE /api/v1/tenants/{tenant_id}/users/{user_id}` — Eliminar usuario
- `POST /api/v1/invitations/{token}/accept` — Aceptar invitación y crear contraseña

---

## Métricas de Éxito

- Emails de invitación aceptados en menos de 24 horas > 60%
- Tasa de invitación rechazada/expirada < 30%
- Tiempo promedio de aceptación de invitación < 4 horas

---

## Notas

- El límite de usuarios por plan se cuenta en usuarios activos (no pendientes de invitación).
- Un usuario puede pertenecer a un solo tenant a la vez (no puede ser manager en dos tenants distintos).
- El rol `tenant_admin` solo puede haber uno por tenant. Para transferir adminismo, el admin actual debe cambiar el rol de otro usuario a `tenant_admin` y renunciar al suyo.
- Los usuarios de un tenant ven sus propias métricas de actividad (último acceso, sesiones activas).
