# RF-040 — Upgrade de Plan con Prorrateo Inmediato

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, stripe-integration, tenant-service

---

## User Story

**Como** administrador de un tenant **quiero** hacer upgrade de mi plan (por ejemplo, de Starter a Professional) y que el cobro sea inmediato y prorrateado **para** acceder a las nuevas features sin esperar al siguiente ciclo de facturación.

---

## Objetivo

El sistema debe procesar el upgrade de plan de un tenant de forma inmediata, calculando el prorrateo de los días no consumidos del plan actual y cobrando la diferencia para el resto del ciclo, usando Stripe para manejar el cambio de precio y la facturación.

---

## Actores

- **Actor principal**: Tenant admin
- **Actor secundario**: Sistema (Stripe), Billing service

---

## Precondiciones

- El tenant tiene estado `active` con una suscripción activa.
- El plan destino es de mayor precio que el plan actual (upgrade, no downgrade).
- El tenant tiene un método de pago vigente en Stripe.

---

## Postcondiciones

### En caso de éxito

- El plan del tenant se actualiza inmediatamente al plan seleccionado.
- Se cobra la diferencia prorrateada al día de hoy.
- La nueva invoice muestra el cargo de upgrade con descripción detallada.
- Los feature flags del nuevo plan se activan al instante.
- Se genera el evento `plan_upgraded` con plan anterior, plan nuevo, y monto cobrado.
- Se envía email de confirmación de upgrade al tenant_admin.

### En caso de falla

- Si el pago del prorrateo falla, el plan no cambia; el tenant permanece en su plan actual.
- Se muestra mensaje de error de Stripe.

---

## Flujo Funcional

1. Tenant admin accede a Settings > Suscripción > "Hacer upgrade".
2. Sistema muestra el plan actual y los planes disponibles con la diferencia de precio mensual.
3. Tenant admin selecciona el plan destino y confirma.
4. Sistema calcula días restantes del ciclo actual: `dias_restantes = current_period_end - hoy`.
5. Sistema calcula prorrateo: `(precio_nuevo_mensual - precio_actual_mensual) * (dias_restantes / dias_total_ciclo)`.
6. Sistema crea Stripe Checkout Session para el cargo prorrateado (o usa `subscriptions.update` con `proration_behavior: create_prorations`).
7. Tenant es redirigido a Stripe para confirmar el pago.
8. Al confirmar el pago, Stripe envía webhook `invoice.payment_succeeded`.
9. Sistema recibe el webhook; identifica el `subscription_id` y el `tenant_id`.
10. Sistema actualiza `subscriptions.plan_id` al nuevo plan.
11. Sistema activa los feature flags del nuevo plan (RF-056).
12. Sistema genera evento `plan_upgraded` con: `tenant_id`, `previous_plan`, `new_plan`, `amount_charged`, `prorated_days`.
13. Sistema envía email de confirmación con resumen del upgrade y los nuevos feature flags.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tenant_id | UUID | Identificador del tenant | Sí |
| new_plan | enum | Plan destino (debe ser más caro que el actual) | Sí |
| billing_cycle | string | Ciclo del plan (monthly/annual) | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| previous_plan | string | Plan que tenía antes |
| new_plan | string | Plan activado |
| amount_charged | decimal | Monto cobrado por el prorrateo |
| prorated_days | integer | Días prorrateados |
| effective_date | datetime | Fecha de inicio del nuevo plan |
| next_billing_date | datetime | Fecha del próximo cobro ( permanece igual) |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Pago de upgrade falla | Tenant permanece en plan actual; se muestra error y se invita a actualizar método de pago. |
| Upgrade a Custom | No hay upgrade directo; el sistema muestra formulario de contacto con ventas. |
| El plan destino tiene límite de sedes menor que el actual | Permitir upgrade; el límite nuevo aplica pero los datos existentes no se eliminan. |
| Upgrade desde trial (sin suscripción activa) | El prorrateo no aplica porque no hay ciclo actual; se cobra el plan completo del primer ciclo. |

---

## Criterios de Aceptación

1. Un upgrade de plan se procesa en menos de 10 segundos después de confirmar el pago.
2. El monto cobrado corresponde exactamente al prorrateo: diferencia de precio multiplicada por días restantes del ciclo.
3. El nuevo plan y sus feature flags están activos inmediatamente después de confirmar el pago.
4. La invoice muestra una línea de "Upgrade prorrateado" con descripción clara.
5. El evento `plan_upgraded` se genera con los datos completos del cambio.
6. El `current_period_end` no cambia; la próxima renovación sigue en la fecha original.
7. Un tenant no puede hacer downgrade desde este endpoint; para eso está RF-041.

---

## Métricas de Éxito

- Tiempo de activación post-pago < 5 segundos
- Tasa de upgrade completados vs iniciados > 90%
- Precisión del cálculo de prorrateo: verificado contra Stripe (error < $0.01)
- Count de upgrades coincide con `plan_upgraded` events

---

## Notas

- Stripe tiene un mecanismo nativo de prorrateo con `proration_behavior: create_prorations`. Usar ese mecanismo para que Stripe genere las invoices de prorrateo automáticamente.
- El cargo de prorrateo no es un cargo recurrente; es un cargo único por la diferencia del periodo actual.
- Los feature flags del nuevo plan se activan inmediatamente al recibir el webhook, no esperar confirmación manual.
- Para planes annual con upgrade, el prorrateo se calcula sobre el precio monthly equivalente del plan annual.
