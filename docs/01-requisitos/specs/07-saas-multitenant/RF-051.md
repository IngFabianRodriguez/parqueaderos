# RF-051 — Generación de API Keys por Tenant

## Metadata

- **Módulo**: SaaS Multi-Tenant — API Keys y Acceso Externo
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: api-gateway, auth-service, tenant-service

---

## User Story

**Como** administrador de un tenant **quiero** generar API keys para que mi sistema externo (app de clientes, integración con ERP, dashboard propio) pueda conectarse a ParkCore **para** automatizar operaciones, consultar datos y realizar acciones programáticamente sin usar el dashboard web.

---

## Objetivo

El sistema debe permitir que cada tenant genere una o múltiples API keys desde su panel de administración, donde cada key tenga un nombre descriptivo, fecha de creación, y los permisos (scopes) que la key puede usar. Las keys se usan para autenticar requests al API de ParkCore con el formato `Bearer <api_key>`.

---

## Actores

- **Actor principal**: Tenant admin (crea y gestiona las API keys)
- **Actor secundario**: Sistema externo del tenant (consume el API con la key)

---

## Precondiciones

- El tenant tiene estado `active` (trial o suspended no puede crear keys nuevas).
- El tenant admin tiene sesión activa.
- El plan del tenant permite el número de API keys solicitado (límite según plan: Starter 2, Professional 5, Enterprise 20, Custom ilimitado).

---

## Postcondiciones

### En caso de éxito

- Se genera una API key única (256 bits, codificada en base64url) para el tenant.
- La key se muestra una sola vez al admin; después solo se ve la fecha de creación y el hash.
- La key puede usarse inmediatamente para autenticación en requests al API.
- Se genera el evento `api_key_created`.

### En caso de falla

- Si el plan no permite más keys, se muestra error indicando el límite y la opción de hacer upgrade.
- Si el tenant está suspended, no puede crear keys nuevas.

---

## Flujo Funcional

### Creación de API key

1. Tenant admin accede a Settings > API Keys > \"Nueva API Key\".
2. Sistema muestra formulario:
   - **Nombre**: texto descriptivo (ej: \"App Clientes Production\", \"ERP SAP Integration\").
   - **Scopes**: checklist de permisos disponibles (disponibilidad:read, reservas:write, etc.).
   - **Expira en**: opcional, fecha de expiración (null = nunca expira).
   - **IP whitelist**: opcional, lista de IPs permitidas (vacío = cualquier IP).
3. Tenant admin completa el formulario y hace clic en \"Crear\".
4. Sistema genera la API key:
   - Genera 32 bytes random (256 bits).
   - Codifica en base64url: `pk_live_{random_base64url}`.
   - Calcula SHA-256 hash de la key para almacenamiento.
   - Almacena en `api_keys`: `key_id`, `tenant_id`, `name`, `scopes`, `expires_at`, `ip_whitelist`, `key_hash`, `created_at`.
5. Sistema muestra la key completa **una sola vez**: `pk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`.
6. Sistema genera evento `api_key_created` con `tenant_id`, `key_id`, `name`, `scopes`.
7. Tenant admin copia la key y la guarda de forma segura (no se puede recuperar después).

### Uso de API key

1. Sistema externo hace request: `GET /api/v1/disponibilidad?sede_id=xxx` con header `Authorization: Bearer pk_live_xxxxxxxx`.
2. API Gateway intercepta el request:
   a. Extrae la key del header.
   b. Busca en `api_keys` por `key_hash` (el hash se recalcula para buscar).
   c. Verifica: tenant_id, scopes, expiración, IP whitelist.
   d. Si todo OK: permite el request, inyecta `tenant_id` y `scopes` en el contexto.
   e. Si falla: retorna 401 Unauthorized.

### Listado y revocación

1. Tenant admin accede a Settings > API Keys.
2. Sistema muestra lista de keys: nombre, scopes, creación, último uso, expiración.
3. Tenant admin puede:
   - Revocar key: marcar como `revoked`; ya no funciona.
   - Rotar key: genera nueva key con los mismos scopes; la anterior se revoca.
   - Ver logs de uso de cada key (RF-055).

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| name | string(100) | Nombre descriptivo de la key | Sí |
| scopes | array[string] | Lista de scopes solicitados | Sí (mínimo 1) |
| expires_at | datetime | Fecha de expiración (null = no expira) | No |
| ip_whitelist | array[string] | IPs permitidas (vacío = todas) | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| api_key | string | La key completa (solo se muestra una vez) |
| key_id | UUID | Identificador de la key (para gestión) |
| name | string | Nombre descriptivo |
| scopes | array | Permisos de la key |
| created_at | datetime | Fecha de creación |
| expires_at | datetime | Fecha de expiración |
| last_used_at | datetime | Último uso (null si nunca se usó) |
| status | enum | `active`, `revoked`, `expired` |

---

## Scopes Disponibles

| Scope | Descripción |
|-------|-------------|
| `disponibilidad:read` | Consultar espacios disponibles |
| `reservas:read` | Leer reservas |
| `reservas:write` | Crear/cancelar reservas |
| `transacciones:read` | Leer transacciones |
| `transacciones:write` | Registrar ingresos/salidas manualmente |
| `clientes:read` | Leer datos de clientes |
| `clientes:write` | Crear/editar clientes |
| `pagos:read` | Leer información de pagos |
| `pagos:write` | Procesar pagos |
| `reportes:read` | Acceder a reportes |
| `webhooks:write` | Configurar webhooks |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Key mostrada al admin por segunda vez | No se puede recuperar; debe crear nueva key y rotar. |
| Plan no permite más keys | Error 403 con mensaje de límite y opción de upgrade. |
| Key vencida | Retorna 401; el admin debe rotar la key. |
| Request desde IP no whitelist | Retorna 403 \"IP no autorizada para esta key.\" |
| Scope no permitido en la key | Retorna 403 \"No tienes permiso para este recurso.\" |
| Tenant suspended | Las keys existentes funcionan para read-only; no se pueden crear nuevas. |
| Key roubada o expuesta | El admin la revoca inmediatamente; todas las requests con esa key retornan 401. |

---

## Criterios de Aceptación

1. Un tenant admin puede crear hasta el número de API keys permitido por su plan.
2. La API key se muestra solo una vez al momento de la creación; después solo se ve metadata.
3. Cada key tiene scopes granulares que limitan qué endpoints puede usar.
4. Las keys pueden tener expiración opcional; keys vencidas se rechazan con 401.
5. Las keys pueden tener IP whitelist opcional; requests desde IPs no autorizadas se rechazan con 403.
6. Un admin puede revoke una key en cualquier momento; la key deja de funcionar inmediatamente.
7. Las keys activas tienen rate limits según el plan del tenant (RF-053).

---

## Endpoints API Involucrados

- `POST /api/v1/api-keys` — Crear nueva API key
- `GET /api/v1/api-keys` — Listar keys del tenant
- `DELETE /api/v1/api-keys/{key_id}` — Revocar key
- `POST /api/v1/api-keys/{key_id}/rotate` — Rotar key (crear nueva, revocar vieja)
- `GET /api/v1/api-keys/{key_id}/usage` — Ver logs de uso

---

## Métricas de Éxito

- Keys creadas vs usadas (> 80% de keys tienen al menos un request en 30 días).
- Revocación inmediata: key deja de funcionar < 1 segundo después de revoke.
- Tasa de keys vencidas sin rotación < 10%.

---

## Notas

- La API key se transmite en el header `Authorization: Bearer {key}`; nunca en la URL como query param (por seguridad).
- El hash de la key se almacena, no la key en texto plano. Solo el admin ve la key en texto plano una vez.
- Si el tenant hace upgrade de plan, mantiene las keys existentes; el nuevo límite aplica para keys nuevas.
- Para producción, usar keys con prefijo `pk_live_`; para testing en sandbox, `pk_test_`.
- El prefijo `pk_live_` vs `pk_test_` se asigna según el entorno donde se creó la key (producción vs staging/sandbox).
