# RF-031 — Periodo de Trial de 14 Días Sin Cobro

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: tenant-service, billing-service, notification-service

---

## User Story

**Como** nuevo cliente de ParkCore **quiero** probar la plataforma durante 14 días sin costo **para** evaluar si se ajusta a mis necesidades antes de comprometerme con un pago.

---

## Objetivo

Todo nuevo tenant creado en la plataforma debe iniciar automáticamente en un periodo de prueba de 14 días durante el cual no se genera ningún cargo, no se requiere información de pago por adelantado, y el acceso a las funcionalidades está sujeto únicamente a los límites del plan Starter trial.

---

## Actores

- **Actor principal**: Tenant nuevo (usuario fundador)
- **Actor secundario**: Sistema (conversión automática de trial a suscripción), Billing service

---

## Precondiciones

- El tenant ha sido creado exitosamente (RF-029).
- El tenant tiene estado `trial` en la base de datos.
- Han pasado 0 días desde la creación del tenant.

---

## Postcondiciones

### En caso de éxito (durante el trial)

- El tenant tiene acceso completo a las funcionalidades del plan Starter trial.
- El tenant no recibe cobros automáticos durante los 14 días.
- Se muestra un banner en el dashboard indicando los días restantes de trial.
- Se envía recordatorio por email al día 7 y al día 12 del trial.

### En caso de éxito (conversión)

- Al confirmarse el primer pago exitoso, el estado cambia a `active` y el `trial_end_date` se marca como `converted_at`.
- Se genera el evento `trial_converted` en la tabla de métricas SaaS.

### En caso de falla (trial expirado sin pago)

- El estado cambia a `trial_expired`.
- El acceso a funcionalidades queda restringido a solo lectura (read-only) hasta que el tenant suscriba un plan.
- Se envía email de notificación de expiración.
- Se muestra un interstitial forcing upgrade antes de continuar usando la plataforma.

---

## Flujo Funcional

1. Al crearse el tenant (RF-029), el sistema asigna `trial_start_date = NOW()` y `trial_end_date = NOW() + 14 días`.
2. El sistema muestra en el dashboard del tenant_admin: "Tienes X días restantes de tu periodo de prueba".
3. Al día 7 del trial, el sistema envía email recordatorio: "Tu trial termina en 7 días. ¡Asegura tu lugar!"
4. Al día 12 del trial, el sistema envía email: "Quedan 2 días. Suscríbete ahora para no perder tus datos."
5. Al día 14 (fecha igual o mayor a `trial_end_date`):
   - Si existe un subscription activa: el flujo termina aquí (trial ya convertido).
   - Si no hay subscription y el tenant tiene usage > 0: el sistema cambia estado a `trial_expired`.
   - Si no hay subscription y usage == 0: el sistema cambia estado a `canceled` ( churn temprano).
6. Para `trial_expired`: el tenant puede seguir accediendo en modo solo lectura; el banner de upgrade es permanente hasta que se suscriba.
7. Para `canceled`: el sistema programado para eliminar datos retention policies comienza a correr (RF-036).

### Upgrade durante trial

1. El tenant decide suscribirse antes de que expire el trial.
2. El tenant selecciona plan (Starter, Professional, Enterprise).
3. El tenant ingresa información de pago ( Stripe Checkout o Stripe Customer Portal).
4. El sistema confirma el pago con Stripe webhook (`payment_succeeded`).
5. El sistema convierte el estado a `active`, registra `subscription_id`, `plan`, y `converted_at`.
6. El sistema genera el evento `trial_converted`.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tenant_id | UUID | Identificador del tenant | Sí |
| plan_seleccionado | string | Plan elegido al hacer upgrade (Starter, Professional, Enterprise) | Sí (al suscribir) |
| stripe_customer_id | string | ID del cliente en Stripe | Sí (al suscribir) |
| stripe_subscription_id | string | ID de la suscripción en Stripe | Sí (al suscribir) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| status | enum | `trial`, `trial_expired`, `active`, `canceled` |
| trial_start_date | datetime | Fecha de inicio del trial |
| trial_end_date | datetime | Fecha de fin del trial |
| trial_converted_at | datetime | Fecha de conversión (null si no se ha convertido) |
| days_remaining | integer | Días restantes del trial (calculado dinámicamente) |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Trial expira y el tenant tiene transacciones | Cambiar estado a `trial_expired`; permitir acceso read-only; forzar upgrade para volver a escribir. |
| Trial expira y el tenant no tiene transacciones | Marcar como `canceled`; comenzar retención de 90 días antes de eliminación. |
| Usuario intenta registrarse con email de un tenant trial_expired | Permitir registro; detectar que el email tiene historial; reutilizar datos? No — cada email es único. |
| Pago falla durante conversión de trial | Mantener estado `trial` hasta que se resuelva el pago; Stripe reintentos aplican (RF-042). |
| Tenant hace upgrade antes del día 14 | Conversion inmediata; no se cobra prorated del trial unused; se cobra el plan completo del ciclo. |

---

## Criterios de Aceptación

1. Todo tenant nuevo inicia con `status = trial` y `trial_end_date = fecha actual + 14 días`.
2. Ningún cargo se genera automáticamente durante los 14 días de trial.
3. El banner "X días restantes de tu trial" es visible en el dashboard del tenant_admin durante todo el periodo.
4. Se envía email recordatorio al día 7 y al día 12.
5. Al día 15, si no hay suscripción activa, el estado cambia a `trial_expired` o `canceled` según el uso.
6. En estado `trial_expired`, el tenant puede ver sus datos pero no puede registrar nuevas transacciones hasta suscribir.
7. La conversión a `active` ocurre exactamente cuando el primer pago se confirma, no antes.
8. El evento `trial_converted` se registra en la tabla de eventos SaaS con `tenant_id`, `plan` y `converted_at`.

## Endpoints API Involucrados

- `POST /api/v1/tenants` — Crear tenant con trial (interno)
- `POST /api/v1/subscriptions` — Crear suscripción (upgrade durante trial o después)
- `GET /api/v1/tenants/{tenant_id}/status` — Consultar estado y días restantes
- `POST /api/v1/webhooks/stripe` — Confirmar pago y disparar conversión

---

## Métricas de Éxito

- Tasa de conversión de trial a paid > 20% (benchmark SaaS típico)
- Tiempo promedio de conversión: 8-10 días
- Tasa de trial_expired vs canceled: > 80% trial_expired, < 20% canceled
- Email recordatorio abierto ( open rate) > 40%

---

## Notas

- El trial es siempre de 14 días, no hay opción de trial extendido manualmente. Solo el superadmin puede extender trial en casos especiales (no via UI de tenant).
- Si el tenant ya tuvo un trial previamente ( churn and re-signup ), no aplica segundo trial; se le muestra el mensaje "Ya usaste tu periodo de prueba. Suscríbete para continuar."
- El trial no requiere guardar método de pago por adelantado ( modelo "freemium" vs "trial with card"). El método de pago solo se pide al momento de suscribir.
- La fecha de trial se almacena en UTC en la base de datos; se convierte a la zona horaria del tenant al mostrarse en la UI.
