# RF-069 — Mapeo de Roles del IdP a Roles Internos del Tenant

## Metadata

- **Módulo**: SaaS Multi-Tenant — SSO / SAML (Enterprise)
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: auth-service, identity-provider, tenant-service

---

## User Story

**Como** administrador de un tenant Enterprise **quiero** que los grupos/roles que mis usuarios tienen en el IdP (Okta, Azure AD, Google Workspace) se traduzcan a los roles internos de ParkCore **para** que los permisos de acceso sean consistentes con lo que ya está definido en mi directorio corporativo.

---

## Objetivo

El sistema debe permitir que el tenant_admin configure un mapeo entre los grupos/roles recibidos en el SAML assertion (del IdP) y los roles internos del tenant en ParkCore. Cuando un usuario hace login via SSO, sus roles se determinan ejecutando este mapeo. La configuración es por IdP y se gestiona desde el admin panel del tenant.

---

## Actores

- **Actor principal**: Administrador del tenant (configura el mapeo)
- **Actor secundario**: Sistema (aplica el mapeo en cada login SSO)

---

## Precondiciones

- El tenant tiene SSO configurado con al menos un IdP (RF-067).
- Los grupos/roles existen en el IdP y son enviados como atributos en el SAML assertion.
- El mapeo aún no existe o necesita actualizarse.

---

## Postcondiciones

### En caso de éxito

- El mapeo se persiste en la tabla `idp_role_mappings`.
- En el siguiente login SSO de cualquier usuario de ese IdP, los roles se asignan según el nuevo mapeo.
- Los cambios en roles son inmediatos (no requieren re-login si ya hay sesión, pero sí en el próximo login).

### En caso de falla

- Si el rol destino no existe en el sistema, se rechaza la configuración y se muestra error.
- Si el grupo origen está vacío o tiene formato inválido, se muestra error de validación.

---

## Flujo Funcional

### Configuración del Mapeo (Admin)

1. El tenant_admin accede a `Settings → SSO → Role Mapping`.
2. Se muestra la lista de grupos/roles que el IdP ha enviado en assertions anteriores (para referencia).
3. El admin puede agregar una nueva regla de mapeo:
   - `idp_group`: nombre del grupo/rol en el IdP (ej: "parking-admins", "Finance Team").
   - `parkcore_role`: rol interno de ParkCore a asignar (`tenant_admin`, `operator`, `viewer`, etc.).
   - `tenant_id`: ID del tenant.
   - `priority`: orden de evaluación (para cuando un usuario pertenece a múltiples grupos).
4. El admin guarda la configuración.
5. El sistema valida que el `parkcore_role` exista en la tabla `roles`.
6. Se persiste en `idp_role_mappings`.

### Aplicación del Mapeo (En Login SSO)

1. El usuario completa el flujo SSO y el auth-service obtiene el SAML assertion con grupos/roles (RF-067).
2. El auth-service extrae la lista de `groups` del assertion.
3. Para cada grupo en el assertion (en orden de prioridad):
   a. Buscar en `idp_role_mappings` donde `idp_group = grupo` y `tenant_id` = tenant actual.
   b. Si hay match, asignar el `parkcore_role` correspondiente.
4. Si ningún grupo tiene mapeo, asignar el rol por defecto (`operator`).
5. Guardar los roles en el JWT de sesión.

### Mapeo por Defecto

Si un usuario tiene grupos del IdP pero ninguno tiene mapeo configurado:
- Asignar `operator` (rol más bajo).
- Generar una notificación al tenant_admin para que configure el mapeo.

---

## Datos de Entrada

### Al guardar mapeo

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| idp_group | string | Nombre del grupo/rol como viene en el assertion SAML | Sí |
| parkcore_role | string | Rol interno de ParkCore a asignar | Sí |
| priority | integer | Orden de prioridad (1 = más alta) | No (default: 99) |
| enabled | boolean | Si el mapeo está activo | Sí (default: true) |

### Grupos recibidos en SAML assertion

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| groups | array[string] | Lista de nombres de grupo/rol del IdP | No (puede venir vacío) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| mapped_roles | array | Lista de roles de ParkCore asignados al usuario |
| default_role_assigned | boolean | true si se usó el rol por defecto (ningún grupo tenía mapeo) |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Grupo del IdP no tiene mapeo configurado | Asignar rol por defecto (`operator`); notificar al admin. |
| Usuario pertenece a múltiples grupos con diferentes mapeos | Asignar todos los roles resultantes (acumulativo). |
| `parkcore_role` especificado no existe | Rechazar el guardado del mapeo; mostrar error de validación. |
| Mapeo deshabilitado (`enabled = false`) | Ignorar ese mapeo; no asignar rol. |
| IdP no envía ningún grupo en el assertion | Asignar rol por defecto; no notificar (es comportamiento normal). |
| Mapeo con `idp_group` duplicado para el mismo IdP | Rechazar; solo un mapeo por grupo por IdP. |

---

## Criterios de Aceptación

1. El tenant_admin puede crear, editar y eliminar reglas de mapeo desde el admin panel.
2. Los grupos recibidos en el SAML assertion pueden incluir nombres con espacios, mayúsculas y caracteres especiales; el matching es case-insensitive y se normaliza.
3. Un usuario puede recibir múltiples roles si pertenece a múltiples grupos con mapeo.
4. Los cambios en el mapeo se aplican en el siguiente login SSO (no requieren logout inmediato).
5. El rol por defecto (`operator`) se asigna cuando ningún grupo tiene mapeo.
6. Si el `parkcore_role` no existe en el sistema, la configuración del mapeo se rechaza con error claro.
7. La configuración del mapeo es específica por IdP: el mismo grupo de Azure AD puede mapear a roles diferentes que el mismo grupo en Okta.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/sso/role-mappings` — Lista de mapeos configurados
- `POST /api/v1/tenants/{tenant_id}/sso/role-mappings` — Crear nuevo mapeo
- `PUT /api/v1/tenants/{tenant_id}/sso/role-mappings/{mapping_id}` — Actualizar mapeo
- `DELETE /api/v1/tenants/{tenant_id}/sso/role-mappings/{mapping_id}` — Eliminar mapeo

---

## Métricas de Éxito

- 100% de logins SSO aplican el mapeo de roles correctamente.
- Tiempo de evaluación de mapeo en login: < 50ms.
- Cero excepciones por rol destino inexistente en producción.

---

## Notas

- El rol asignado por mapeo SSO puede ser modificado por el tenant_admin manualmente después (RF-071). La próxima vez que el usuario haga login, el mapeo SSO se vuelve a aplicar y podría sobreescribir la asignación manual si el admin no ha configurado protección contra esto. Se recomienda documentar este comportamiento.
- La normalización de nombres de grupo incluye: trim de espacios, conversión a lowercase, eliminación de acentos (para comparar "Gerência Financeira" con "gerencia financiera").
- Si el IdP soporta anidación de grupos (grupos dentro de grupos), se flattened y se incluyen todos los grupos ancestros en el assertion.
- El campo `priority` permite que si un usuario tiene "Admin" (prioridad 1) y "Operator" (prioridad 2), se le asigne el rol de Admin si el mapeo de Admin tiene prioridad más alta. Sin embargo, el comportamiento recomendado es acumular roles, no reemplazar.
