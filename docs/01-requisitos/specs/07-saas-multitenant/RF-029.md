# RF-029 — Creación de Cuenta Tenant

## Metadata

- **Módulo**: SaaS Multi-Tenant — Gestión de Tenants
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: auth-service, tenant-service, billing-service

---

## User Story

**Como** fundador o administrador de un parqueadero **quiero** registrar una nueva cuenta en la plataforma **para** comenzar a gestionar mi negocio con una пробная period.

---

## Objetivo

El sistema debe permitir la creación de una nueva cuenta tenant con datos básicos de la empresa, generando automáticamente un ambiente aislado y comenzando el periodo de prueba de 14 días sin cobro.

---

## Actores

- **Actor principal**: Usuario fundador (registrador inicial del tenant)
- **Actores secundarios**: Sistema (asignación de slug, generación de tenant_id)

---

## Precondiciones

- El usuario no tiene una cuenta existente en la plataforma.
- El slug propuesto no existe previamente en el sistema.
- El email proporcionado no está registrado en ningún otro tenant.

---

## Postcondiciones

### En caso de éxito

- Se crea un registro de tenant en la base de datos con estado `trial`.
- Se genera un slug único accesible en `/app/{slug}`.
- Se crean los espacios iniciales de configuración (sede, zonas, espacios) vacíos.
- El usuario fundador recibe un rol `tenant_admin` dentro de su organización.
- Se registra el evento `tenant_created` en el timeline de métricas SaaS.
- Se envía email de bienvenida al usuario fundador.

### En caso de falla

- Se muestra mensaje de error indicando el campo que falló.
- No se crea ningún registro parcial en la base de datos.
- Los recursos temporales creados durante el flujo (si los hay) se eliminan.

---

## Flujo Funcional

1. El usuario accede a la página de registro (`/register`) e ingresa: nombre de la empresa, nombre del fundador, email, contraseña, NIT de la empresa.
2. El sistema valida que el email no esté en uso por otro tenant.
3. El sistema valida que el NIT no esté en uso por otro tenant.
4. El sistema genera un slug a partir del nombre de la empresa (primera palabra + random de 4 caracteres alfanuméricos). Si el slug ya existe, se genera uno alternativo con sufijos numéricos (`-2`, `-3`).
5. El sistema crea el registro del tenant con: `tenant_id` (UUID), `name`, `slug`, `founder_email`, `nit`, `status = 'trial'`, `trial_start_date = NOW()`, `trial_end_date = NOW() + 14 días`.
6. El sistema crea el usuario fundador con email verificado = false, rol `tenant_admin`, `tenant_id` asignado.
7. El sistema encripta la contraseña con bcrypt (cost 12) y la almacena.
8. El sistema envía email de bienvenida con enlace de verificación de email.
9. El sistema genera la sesión JWT y redirige al onboarding wizard.
10. El sistema registra el evento `tenant_created` con `tenant_id`, timestamp y plan inicial estimado (`trial`).

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| empresa_nombre | string(100) | Nombre legal de la empresa | Sí |
| fundador_nombre | string(100) | Nombre completo del representante | Sí |
| email | string(255) | Email corporativo o personal del fundador | Sí |
| password | string(72) | Contraseña (mínimo 8 caracteres, 1 mayúscula, 1 número) | Sí |
| nit | string(20) | Número de identificación tributaria | Sí |
| telefono | string(20) | Teléfono de contacto | No |
| origen_referral | string(50) | Canal de adquisición (Google Ads, referído, etc.) | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| tenant_id | UUID | Identificador único del tenant |
| slug | string | Slug único para acceso (`/app/{slug}`) |
| status | enum | Estado inicial del tenant (`trial`) |
| trial_end_date | datetime | Fecha de fin del periodo de prueba |
| usuario_fundador_id | UUID | ID del usuario creado con rol tenant_admin |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Email ya existe en otro tenant | Retornar error 409 con mensaje "Este email ya está registrado. ¿Ya tienes cuenta? Inicia sesión." |
| NIT ya existe en otro tenant | Retornar error 409 con mensaje "Este NIT ya está registrado. Contacta a soporte si crees que es un error." |
| Slug generado ya existe | Intentar hasta 5 variaciones automáticas; si todas existen, pedir al usuario que elija un slug manualmente. |
| Password no cumple requisitos | Retornar error 400 listando los requisitos faltantes. |
| Campos requeridos vacíos | Retornar error 400 indicando cuáles campos faltan. |
| Error de base de datos al crear tenant | Revertir transacción, retornar error 500 con mensaje genérico. No exponer detalles del DB error. |

---

## Criterios de Aceptación

1. Un nuevo tenant se crea en la base de datos en menos de 3 segundos.
2. El slug generado es único y accesible desde `/app/{slug}`.
3. El tenant inicia con estado `trial` y `trial_end_date` exactamente 14 días en el futuro.
4. El usuario fundador recibe rol `tenant_admin` sin necesidad de confirmación adicional.
5. El email de bienvenida se envía dentro de los 30 segundos posteriores a la creación.
6. No es posible crear un tenant con un email ya existente en el sistema.
7. La contraseña se almacena encriptada con bcrypt (cost 12); nunca se almacena en texto plano.
8. El evento `tenant_created` se registra en la tabla de eventos SaaS al finalizar la creación.

- `POST /api/v1/tenants` — Crear nuevo tenant

---

## Métricas de Éxito

- Tiempo de creación de tenant < 3 segundos (p95)
- Tasa de éxito de creación en primer intento > 99%
- Email de bienvenida enviado < 30 segundos post-creación
- Count de `tenant_created` events coincide con nuevos registros en tabla `tenants`

---

## Notas

- El slug se genera automáticamente pero se permite al usuario sobrescribirlo antes de confirmar el registro.
- Los primeros 5 slugs alternativos se generan automáticamente; si todos existen, se muestra un input para que el usuario elija manualmente.
- El periodo trial de 14 días comienza desde `NOW()` (fecha del registro), no desde la verificación del email.
- La tabla `tenants` incluye campos de dirección, ciudad y país que pueden ser completados durante el onboarding.
