# RF-049 — Custom Domain para Tenants Enterprise+

## Metadata

- **Módulo**: SaaS Multi-Tenant — Branding y Personalización
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: dns-service, tenant-service, cdn-service, ssl-service

---

## User Story

**Como** administrador de un tenant Enterprise **quiero** que mi equipo acceda al dashboard desde `parking.empresa.com` en lugar de `empresa.parkcore.co` **para** dar una experiencia más profesional y reforzar mi marca.

---

## Objetivo

El sistema debe permitir que los tenants de plan Enterprise o superior configuren un dominio personalizado (custom domain) para acceder a su dashboard. El sistema se encargará del provisionamiento automático de certificados SSL, la configuración DNS y el routing correcto al tenant.

---

## Actores

- **Actor principal**: Tenant admin (configura el custom domain)
- **Actor secundario**: Sistema (provisiona SSL, configura routing), DNS del cliente

---

## Precondiciones

- El tenant tiene plan `Enterprise` o `Custom`.
- El tenant admin tiene acceso a la configuración DNS de su dominio.
- El dominio personalizado no está en uso por otro tenant.
- El dominio es un subdomain válido (no dominio raíz).

---

## Postcondiciones

### En caso de éxito

- El tenant puede acceder a su dashboard desde `https://{custom_domain}`.
- El certificado SSL se genera y renueva automáticamente (Let's Encrypt o similar).
- El branding del tenant se muestra automáticamente en el custom domain.
- El custom domain se usa como `from_address` en emails (RF-048).

### En caso de falla

- Si la validación DNS falla, el sistema muestra los registros CNAME/ TXT necesarios para configurar.
- Si el dominio ya está en uso, se muestra error indicando que debe contactar a soporte.
- El subdomain default `https://{slug}.parkcore.co` sigue funcionando como fallback.

---

## Flujo Funcional

### Configuración del custom domain

1. Tenant admin accede a Settings > Branding > \"Dominio personalizado\".
2. Sistema muestra input para ingresar el subdomain (ej: `parking.empresa.com`).
3. Tenant admin ingresa el dominio y hace clic en \"Guardar\".
4. Sistema valida que el plan del tenant permite custom domain.
5. Sistema verifica que el dominio no está en uso por otro tenant.
6. Sistema genera los registros DNS necesarios:
   - `CNAME`: `parking.empresa.com` → `cname.parkcore.co`
   - `TXT`: `_acme-challenge.parking.empresa.com` → `{challenge_token}` (para SSL)
7. Sistema muestra al tenant los registros que debe agregar en su proveedor DNS.
8. Tenant admin agrega los registros DNS en su panel.
9. Sistema poll-ea el DNS cada 5 minutos para verificar que los registros están propagateados.
10. Una vez propagateados, sistema genera el certificado SSL via ACME (Let's Encrypt).
11. Sistema configura el CDN/proxy para routing: `parking.empresa.com` → aplicación del tenant.
12. Sistema actualiza `tenant_domains` con `domain`, `ssl_status = active`, `verified_at`.
13. Sistema genera evento `custom_domain_configured`.
14. El tenant recibe email de confirmación: \"Tu dominio personalizado está activo\".

### Routing de requests

1. Request llega a `https://parking.empresa.com/dashboard`.
2. CDN/Proxy recibe el request; extrae el Host header.
3. CDN resuelve el CNAME y enruta al servicio de ParkCore.
4. El servicio identifica el tenant desde el dominio (`tenant_domains.domain = parking.empresa.com`).
5. Se aplica el branding del tenant (RF-046, RF-047) y se sirve el dashboard.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| custom_domain | string | Dominio personalizado (subdomain.empresa.com) | Sí |
| ssl_enabled | boolean | Habilitar SSL automático (default: true) | Sí |
| tenant_id | UUID | Identificador del tenant | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| domain_id | UUID | Identificador del dominio configurado |
| custom_domain | string | Dominio configurado |
| cname_target | string | Destino CNAME que el cliente debe configurar |
| dns_records | array | Lista de registros DNS a configurar |
| ssl_status | enum | `pending`, `provisioning`, `active`, `failed` |
| verified_at | datetime | Fecha de verificación DNS |
| expires_at | datetime | Fecha de expiración del certificado SSL |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Plan del tenant no permite custom domain | Mostrar error \"Esta feature requiere plan Enterprise o superior. Haz upgrade para activar.\" |
| Dominio ya en uso por otro tenant | Mostrar error 409 \"Este dominio ya está configurado. Contacta a soporte si crees que es un error.\" |
| Registro DNS no propagado después de 24h | Enviar recordatorio al tenant; permitir revalidación manual. |
| SSL renewal falla | Alertar al tenant; el equipo de ParkCore resuelve manualmente; el certificado anterior sigue válido hasta renovación. |
| Cliente elimina el registro DNS | El sistema detecta que el dominio no resuelve; marca `ssl_status = degraded` y alerta al tenant. |
| Custom domain solicitado con dominio raíz (empresa.com) | Rechazar; solo se permiten subdomains para evitar problemas de seguridad. |
| Tenant hace downgrade a plan inferior | El custom domain sigue activo hasta fin del ciclo; al renovar en plan inferior se desactiva y redirige al subdomain default. |

---

## Criterios de Aceptación

1. Un tenant Enterprise+ puede configurar hasta 3 custom domains.
2. El SSL se genera automáticamente en menos de 10 minutos después de que los DNS están propagateados.
3. El dashboard del tenant es accesible desde `https://{custom_domain}` sin warnings de certificado.
4. El custom domain usa el branding del tenant automáticamente (no requiere configuración adicional).
5. El from address de emails usa el custom domain (RF-048).
6. Si el SSL expira, el sistema renueva automáticamente sin intervención del tenant.
7. El subdomain default sigue funcionando si el custom domain tiene problemas.

## Endpoints API Involucrados

- `POST /api/v1/tenants/{tenant_id}/domains` — Configurar custom domain
- `GET /api/v1/tenants/{tenant_id}/domains` — Listar dominios configurados
- `DELETE /api/v1/tenants/{tenant_id}/domains/{domain_id}` — Eliminar custom domain
- `POST /api/v1/domains/validate` — Validar configuración DNS

## Métricas de Éxito

- Custom domains activos: 100% de los tenants Enterprise tienen al menos uno.
- Tiempo de provisioning SSL post-DNS: < 10 minutos.
- Uptime del custom domain: 99.9% (mismo SLA que subdomain default).
- Renovaciones SSL automáticas: 100%.

---

## Notas

- Solo se permiten subdomains (no dominio raíz) para evitar conflictos y problemas de seguridad con certificados wildcard.
- El sistema usa Let's Encrypt para certificados gratuitos con renovación automática 30 días antes de expirar.
- El CNAME target de ParkCore es `cname.parkcore.co` y nunca cambia; el cliente solo debe apuntar su subdomain a ese target.
- El tracking de dominio se hace via `dns.google.com` (Google DNS) para verificar propagación.
- Para planes Custom, se permite dominio raíz también (ej: `empresa.com`) usando un registro A con IP dedicada de ParkCore.
