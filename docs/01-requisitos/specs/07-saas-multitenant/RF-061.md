# RF-061 — Verificación de Email del Usuario mediante OTP

## Metadata

- **Módulo**: SaaS Multi-Tenant — Onboarding de Nuevo Tenant
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: auth-service, notification-service, tenant-service

---

## User Story

**Como** nuevo usuario de ParkCore **quiero** verificar mi email para activar mi cuenta **para** confirmar que soy yo y poder comenzar a usar la plataforma de forma segura.

---

## Objetivo

El sistema debe enviar un código OTP (One-Time Password) de 6 dígitos al email del usuario al registrarse, y validar ese código antes de permitir que el usuario acceda a la plataforma. El OTP expira después de 10 minutos y solo puede usarse una vez.

---

## Actores

- **Actor principal**: Nuevo usuario (recibe y escribe el OTP)
- **Actor secundario**: Sistema (envía el OTP y valida)

---

## Precondiciones

- El usuario existe en la base de datos con `status = pending_verification`.
- El usuario tiene un email válido.
- El usuario no ha verificado su email previamente.

---

## Postcondiciones

### En caso de éxito

- El email del usuario se marca como verificado (`email_verified_at = NOW()`, `status = active`).
- El usuario puede iniciar sesión y completar el onboarding.
- Se genera el evento `email_verified`.

### En caso de falla

- Si el OTP es incorrecto: se muestra error y se permite reintentar (máx 3 intentos).
- Si el OTP expiró: se debe solicitar uno nuevo.
- Si se exceeden los 3 intentos: el código expira y se debe solicitar reenvío.

---

## Flujo Funcional

### Envío del OTP

1. El usuario se registra (RF-029) o el sistema detecta `status = pending_verification`.
2. El sistema genera un OTP de 6 dígitos: `000000` - `999999`.
3. El sistema almacena en Redis o DB: `{user_id}_email_otp`, `{hash(otp)}`, `expires_at = NOW() + 10 min`, `attempts = 0`.
4. El sistema envía email con el OTP:
   - **Asunto**: "Tu código de verificación de ParkCore"
   - **Cuerpo**: "Tu código de verificación es: **XXXXXX**. Expira en 10 minutos. No compartas este código con nadie."
5. El sistema genera evento `verification_otp_sent`.

### Validación del OTP

1. El usuario ingresa el código de 6 dígitos en la interfaz de verificación.
2. El sistema consulta `{user_id}_email_otp` en Redis.
3. Se verifica:
   a. Si el OTP existe: continuar.
   b. Si expiró: retornar error "El código expiró. Solicita uno nuevo."
   c. Si los intentos >= 3: marcar como expirado, retornar error.
4. Se compara el OTP ingresado con el hash almacenado:
   - **Si coinciden**: marcar email como verificado, invalidar el OTP.
   - **Si no coinciden**: `attempts += 1`; si attempts >= 3, marcar como expirado.
5. En éxito: actualizar `users.email_verified_at = NOW()`, `status = active`.
6. Generar evento `email_verified`.
7. Redirigir al usuario al wizard de onboarding (RF-059).

### Reenvío de OTP

1. Si el usuario no recibe el email, puede hacer clic en "Reenviar código".
2. El sistema verifica que el OTP anterior haya expirado o que hayan pasado al menos 60 segundos.
3. Se genera un nuevo OTP y se envía.
4. El OTP anterior se invalida.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| user_id | UUID | Identificador del usuario | Sí |
| otp_code | string | Código de 6 dígitos | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| verified | boolean | Si la verificación fue exitosa |
| user_id | UUID | Usuario verificado |
| email_verified_at | datetime | Timestamp de verificación |
| attempts_remaining | integer | Intentos restantes (0-3) |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| OTP incorrecto | Mostrar error "Código incorrecto. Te quedan X intentos." |
| OTP expiró | Mostrar "El código expiró. Solicita uno nuevo." |
| 3 intentos fallidos | El OTP expira; el usuario debe pedir reenvío. |
| Email ya verificado | El usuario no ve la pantalla de verificación; va directo al dashboard. |
| User no existe | Retornar 404. |
| Redis caído al validar | Fail-closed: denegar verificación; el usuario debe reintentarlo. |
| El usuario cierra la página y vuelve | El sistema recuerda que está pendiente de verificación; no puede hacer login sin verificar. |

---

## Criterios de Aceptación

1. El OTP se envía al email del usuario dentro de los 30 segundos posteriores al registro.
2. El OTP expira después de 10 minutos.
3. El usuario tiene máximo 3 intentos para ingresa el código correcto.
4. Si el OTP expira o se exceeden los intentos, el usuario puede solicitar un nuevo código.
5. La verificación exitosa marca `email_verified_at` y cambia `status` a `active`.
6. El usuario no puede hacer login hasta verificar su email (excepto el link mágico en el email de invitación de equipo).
7. El evento `email_verified` se genera al completar la verificación.

## Endpoints API Involucrados

- `POST /api/v1/auth/verify-email` — Validar OTP de verificación
- `POST /api/v1/auth/resend-otp` — Reenviar OTP
- `GET /api/v1/auth/verification-status` — Consultar si el email ya está verificado

---

## Métricas de Éxito

- Emails de verificación entregados: > 95%.
- Tasa de verificación en primer intento: > 60%.
- Tiempo promedio de verificación: < 5 minutos después de recibir el email.
- Tasa de usuarios que no verifican (abandono): < 10%.

---

## Notas

- El OTP se almacena como hash (bcrypt o SHA-256) por seguridad; nunca se almacena en texto plano.
- El tiempo de expiración de 10 minutos es un balance entre seguridad y usabilidad.
- Los intentos se cuentan por OTP, no por sesión; si el usuario pide reenvío, los intentos se resetean.
- El email de bienvenida (RF-060) puede incluir el OTP directamente para evitar un paso adicional, pero el flujo actual usa verificación separada para ser más robusto.
