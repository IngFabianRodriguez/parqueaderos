# RF-070 — Creación de Usuarios dentro de un Tenant con Roles Específicos

## Metadata

- **Módulo**: SaaS Multi-Tenant — Users y Roles por Tenant
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: user-service, tenant-service, auth-service

---

## User Story

**Como** administrador de un tenant **quiero** crear usuarios dentro de mi organización con roles específicos **para** permitir que mis empleados accedan al sistema según sus responsabilidades.

---

## Objetivo

El sistema debe permitir que el `tenant_admin` cree usuarios dentro de su tenant, asignándoles un rol específico (`tenant_admin`, `operator`, `viewer`). Cada usuario creado pertenece exclusivamente al tenant donde se crea y sus permisos están limitados por el rol asignado.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Superadmin (puede crear usuarios en cualquier tenant)

---

## Precondiciones

- El actor tiene rol `tenant_admin` o `superadmin`.
- El email del nuevo usuario no está registrado en ningún tenant del sistema (o está registrado pero inactivo).
- El rol a asignar existe en la tabla `roles`.

---

## Postcondiciones

### En caso de éxito

- El usuario se crea en la tabla `users` con el `tenant_id` correspondiente.
- Se le asigna el rol indicado en `user_roles`.
- El usuario recibe un email de invitación con un link para configurar su contraseña.
- El usuario queda con estado `pending_invitation` hasta que acepte la invitación.

### En caso de falla

- Si el email ya está en uso por un usuario activo en otro tenant: error indicando conflicto.
- Si el rol no existe: error de validación.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Settings → Users → Add User`.
2. Completa el formulario:
   - `email`: dirección de correo del nuevo usuario.
   - `name`: nombre completo del usuario.
   - `role`: selección del rol (`tenant_admin`, `operator`, `viewer`).
   - `sites`: lista de sedes a las que tendrá acceso (solo para `operator` y `viewer`, opcional según configuración del tenant).
3. El sistema valida:
   - El email no esté en uso por un usuario activo en otro tenant.
   - El rol seleccionado exista.
   - Las sedes seleccionadas pertenezcan al tenant del admin.
4. Se crea el usuario en la tabla `users`:
   - `id`: UUID generado.
   - `tenant_id`: del admin que crea.
   - `email`, `name`: del formulario.
   - `auth_method`: `email_password`.
   - `status`: `pending_invitation`.
   - `created_at`: timestamp actual.
   - `created_by`: `user_id` del admin que crea.
5. Se inserta el rol en `user_roles`: `user_id`, `role`, `tenant_id`.
6. Si se seleccionaron sedes, se insertan en `user_site_access`: `user_id`, `site_id`.
7. Se genera un `invitation_token` (JWT de corta duración, 7 días) y se envía por email al usuario.
8. El usuario hace clic en el link del email, lo lleva a `GET /auth/invitation/{token}`.
9. El sistema valida el token y presenta un formulario para crear contraseña.
10. El usuario ingresa y confirma la contraseña.
11. El sistema actualiza: `password_hash`, `status = active`, `invitation_token = null`.
12. El usuario puede hacer login.

### Casos de creación por Superadmin

- El superadmin puede crear usuarios en cualquier tenant via `POST /api/v1/admin/tenants/{tenant_id}/users`.
- El flujo es el mismo, pero no se requiere ser `tenant_admin` del tenant destino.
- El superadmin también puede especificar el `tenant_id` destino.

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| email | string | Correo electrónico del usuario | Sí |
| name | string | Nombre completo | Sí |
| role | string | Nombre del rol a asignar | Sí |
| site_ids | array[uuid] | Lista de IDs de sedes con acceso (opcional) | No |
| tenant_id | uuid | ID del tenant (solo para superadmin) | No (se infiere del contexto para tenant_admin) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| user_id | uuid | ID del usuario creado |
| email | string | Email del usuario |
| name | string | Nombre del usuario |
| role | string | Rol asignado |
| status | string | Estado inicial: `pending_invitation` |
| invitation_sent_at | timestamp | Timestamp del envío del email |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Email ya existe en el mismo tenant (usuario activo) | Error: "Ya existe un usuario con este email en tu organización." |
| Email ya existe en otro tenant (usuario activo) | Error: "Este email ya está registrado en otro tenant. No se puede crear." |
| Email ya existe pero el usuario está `inactive` | Reactivar el usuario; mantener el email; no crear nuevo. |
| Rol seleccionado no existe | Error de validación; no crear usuario. |
| Sitio seleccionado no pertenece al tenant | Error de validación; no crear usuario. |
| Límite de usuarios del plan alcanzado | Error indicando que debe hacer upgrade del plan. |
| El email de invitación no puede enviarse (SMTP down) | Crear el usuario igual; mostrar warning; permitir reenvío manual. |

---

## Criterios de Aceptación

1. El `tenant_admin` puede crear usuarios con roles `operator` y `viewer` sin límite (sujeto al límite del plan).
2. El `tenant_admin` puede crear otros `tenant_admin` (mismo nivel), pero el sistema puede restringuirse por configuración del plan.
3. El usuario creado recibe un email de invitación antes de poder hacer login.
4. El usuario queda con estado `pending_invitation` hasta que complete el registro de contraseña.
5. La asignación de sites es opcional y solo aplica para roles `operator` y `viewer`.
6. El `tenant_admin` solo ve y puede crear usuarios de su propio tenant.
7. El superadmin puede crear usuarios en cualquier tenant.
8. El email de cada usuario es único en todo el sistema.

---

## Endpoints API Involucrados

- `POST /api/v1/tenants/{tenant_id}/users` — Crear usuario
- `GET /api/v1/tenants/{tenant_id}/users` — Listar usuarios del tenant
- `GET /api/v1/tenants/{tenant_id}/users/{user_id}` — Detalle de usuario
- `PUT /api/v1/tenants/{tenant_id}/users/{user_id}` — Actualizar usuario
- `DELETE /api/v1/tenants/{tenant_id}/users/{user_id}` — Eliminar usuario
- `POST /api/v1/users/{user_id}/resend-invitation` — Reenviar invitación
- `GET /auth/invitation/{token}` — Página de aceptación de invitación

---

## Métricas de Éxito

- Tiempo de creación de usuario: < 1 segundo (excluyendo envío de email).
- Email de invitación enviado en < 30 segundos después de la creación.
- Tasa de usuarios que completan el registro: > 80% (benchmar configurable).

---

## Notas

- Un usuario pertenece a un solo tenant. No existe un usuario "global" excepto el superadmin.
- La invitación expira en 7 días; después de eso, el admin puede reenviar una nueva invitación.
- El password del usuario debe cumplir con la política de passwords del sistema (min 8 caracteres, al menos una mayúscula, un número).
- Si el tenant tiene SSO habilitado, los usuarios pueden ser creados también via JIT (RF-068); los usuarios creados manualmente tienen `auth_method = email_password`.
