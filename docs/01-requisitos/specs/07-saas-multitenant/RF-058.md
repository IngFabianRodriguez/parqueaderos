# RF-058 — Activación de Features Mediante Upgrade de Plan

## Metadata

- **Módulo**: SaaS Multi-Tenant — Feature Flags por Plan
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, feature-flag-service, tenant-service, notification-service

---

## User Story

**Como** administrador de un tenant **quiero** que al hacer upgrade de mi plan las nuevas features se activen automáticamente **para** no tener que configurar nada adicional después de pagar.

---

## Objetivo

El sistema debe activar inmediatamente las features del nuevo plan cuando el upgrade se complete (después de confirmar el pago vía Stripe), sin necesidad de intervención manual. Las features del plan anterior se mantienen accesibles; las nuevas se habilitan.

---

## Actores

- **Actor principal**: Sistema (detecta upgrade completado y activa features)
- **Actor secundario**: Tenant admin (recibe notificación de nuevas features)

---

## Precondiciones

- El tenant tiene un plan activo (no trial).
- El upgrade se solicitó y el pago se completó exitosamente.
- El webhook `invoice.payment_succeeded` o `customer.subscription.updated` llegó con el nuevo plan.

---

## Postcondiciones

### En caso de éxito

- Las features del nuevo plan están activas inmediatamente.
- Los feature flags del tenant se actualizan en la base de datos.
- Los módulos correspondientes aparecen en el dashboard sin necesidad de recarga.
- Se envía email de confirmación indicando las nuevas features disponibles.

---

## Flujo Funcional

1. El tenant admin inicia upgrade desde Settings > Suscripción > \"Hacer upgrade\" (RF-040).
2. El sistema procesa el pago vía Stripe.
3. Stripe envía webhook `customer.subscription.updated` o `invoice.payment_succeeded`.
4. El billing-service recibe el webhook e identifica el nuevo plan.
5. El billing-service actualiza `tenants.plan_id` al nuevo plan.
6. El billing-service llama a `feature-flag-service.activatePlanFeatures(tenant_id, new_plan_id)`.
7. El servicio de features:
   a. Lee el mapa de `feature_flags` para el nuevo plan.
   b. Actualiza `tenant_features` con las nuevas features activas.
   c. Genera evento `features_activated` con lista de nuevas features.
8. El sistema envía email al tenant admin: "¡Felicitaciones! Tu plan ahora es [Plan]. Nuevas features disponibles: [lista]."
9. El dashboard del tenant muestra los nuevos módulos sin necesidad de logout/login (el cambio se detecta al hacer polling de features).

---

## Features Activadas por Upgrade

| Upgrade | Features nuevas activadas |
|---------|---------------------------|
| Starter → Professional | Multi-sede (3), BI Reportes, API Access, Overage billing |
| Professional → Enterprise | Multi-sede (ilimitado), SSO/SAML, Custom Domain, Flotas B2B, 50 usuarios, RPM 1,000 |
| Enterprise → Custom | White-label app, ilimitado todo, API RPM 10,000 |
| Starter → Enterprise | Todas las de Professional + Enterprise |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Pago de upgrade falla | No se activa el nuevo plan; el tenant permanece en el plan actual. |
| Upgrade parcial (prorrateo no pagado) | El sistema espera confirmación de Stripe antes de activar. |
| El tenant está suspended | El upgrade no se procesa hasta que el tenant se reactive ( RF-035). |
| Nuevo plan tiene menos features que el anterior | Esto es downgrade, no upgrade; no aplica a este RF (ver RF-041). |
| El webhook llega duplicado | Si el plan ya es el nuevo, no se hace nada; idempotencia. |

---

## Criterios de Aceptación

1. Al completar un upgrade, las features del nuevo plan están activas en menos de 10 segundos después del webhook.
2. Las features nuevas aparecen en el dashboard sin necesidad de recargar la página ( polling o WebSocket).
3. El email de confirmación incluye la lista de nuevas features desbloqueadas.
4. El evento `features_activated` se genera con la lista de features añadidas.
5. Si el upgrade fue Starter → Professional, las features específicas de Professional están disponibles y las de Enterprise siguen bloquedas.
6. Las features del plan anterior siguen funcionando; no se pierden acceso a nada.

## Endpoints API Involucrados

- `POST /api/v1/webhooks/stripe` — Confirmación de pago de upgrade

---

## Métricas de Éxito

- Tiempo de activación post-upgrade: < 10 segundos.
- Features activadas correctamente: 100%.
- Email de nuevas features enviado: < 60 segundos post-upgrade.
- Tasa de upgrade completados vs intentados: > 90%.

---

## Notas

- Las features no se \"desbloquean\" una por una; el mapa completo del plan se activa atomicamente.
- Los feature flags se cachean en Redis por 5 minutos; el cache se invalida al hacer upgrade para que los cambios se reflejen inmediatamente.
- Las features activadas por upgrade son inmediatas; el downgrade (RF-041) aplica al final del ciclo y desactiva features en ese momento.
