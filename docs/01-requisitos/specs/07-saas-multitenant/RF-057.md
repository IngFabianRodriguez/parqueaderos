# RF-057 — Error 403 al Acceder Feature No Incluida en el Plan

## Metadata

- **Módulo**: SaaS Multi-Tenant — Feature Flags por Plan
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: api-gateway, feature-flag-service, tenant-service

---

## User Story

**Como** operador de un tenant con plan Starter **quiero** ver un mensaje claro cuando intento acceder a una feature no incluida en mi plan **para** entender qué necesito hacer para desbloquearla y no pensar que es un error del sistema.

---

## Objetivo

El sistema debe retornar un error HTTP 403 (Forbidden) con cuerpo JSON estructurado cuando un tenant intente acceder a una feature, módulo o endpoint que no esté incluido en su plan. El mensaje debe ser claro, orientador y proporcionar un camino de upgrade.

---

## Actores

- **Actor principal**: Sistema (detecta y responde con error de feature no autorizada)
- **Actor secundario**: Operador/Tenant admin (recibe el error y decide qué hacer)

---

## Precondiciones

- El tenant tiene un plan que no incluye la feature solicitada.
- El usuario intenta acceder a la feature (vía UI o API directa).

---

## Postcondiciones

### En caso de intento de feature no incluida

- El sistema retorna HTTP 403 con cuerpo JSON detallado.
- El frontend muestra un modal o banner indicando que la feature requiere upgrade.
- El evento `feature_access_denied` se genera para métricas.

---

## Flujo Funcional

1. El usuario hace clic en una opción del menú o hace request directo a un endpoint de una feature no incluida en su plan.
2. El API Gateway o el servicio de la feature consulta `GET /api/v1/tenants/{tenant_id}/features`.
3. Se verifica que la feature solicitada no está en la lista de features activas.
4. El sistema retorna:

```
HTTP 403 Forbidden
Content-Type: application/json

{
  "error": "feature_not_included",
  "code": "PLAN_FEATURE_REQUIRED",
  "message": "Esta funcionalidad requiere un plan superior.",
  "feature": "bi_reportes",
  "feature_name": "Reportes Avanzados de BI",
  "current_plan": "Starter",
  "required_plan": "Professional",
  "upgrade_url": "/settings/subscription/upgrade",
  "can_trial": false,
  "contact_sales_url": "/contact-sales"
}
```

5. El frontend recibe el 403 y muestra un modal:
   - Título: "Feature no disponible en tu plan"
   - Descripción: "Los Reportes Avanzados de BI están disponibles en el plan Professional o superior."
   - Botón principal: "Hacer upgrade a Professional"
   - Botón secundario: "Ver todos los planes"
6. Se genera evento `feature_access_denied` en `events_saas` para tracking de demanda.

---

## Datos de Salida (error 403)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| error | string | Código de error: `feature_not_included` |
| code | string | Código interno: `PLAN_FEATURE_REQUIRED` |
| message | string | Mensaje legible para el usuario |
| feature | string | Identificador de la feature |
| feature_name | string | Nombre legible de la feature |
| current_plan | string | Plan actual del tenant |
| required_plan | string | Plan mínimo que incluye la feature |
| upgrade_url | string | URL para hacer upgrade |
| can_trial | boolean | Si la feature tiene trial disponible |
| contact_sales_url | string | URL para contactar a ventas (Enterprise+) |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Request directo al API (sin UI) | Retorna el JSON de error 403; el cliente debe manejarlo. |
| Feature no existe en la base de datos | Retorna 404, no 403; la feature debe estar definida. |
| Tenant trial intentando feature de paid | Mismo 403; el trial tiene acceso solo a lo del plan selected. |
| Feature de Enterprise+ sin plan Enterprise | El `upgrade_url` apunta al checkout del plan Enterprise; `contact_sales_url` también presente. |
| Feature deshabilitada por mantenimiento | Retorna 503 con mensaje de mantenimiento, no 403. |

---

## Criterios de Aceptación

1. Cada endpoint o feature no incluida en el plan retorna HTTP 403 con cuerpo JSON estructurado.
2. El mensaje de error incluye: nombre de la feature, plan requerido, y URL de upgrade.
3. El frontend muestra un modal informativo y no solo un error genérico.
4. El evento `feature_access_denied` se genera para métricas de demanda de features.
5. El error es rápido (< 50ms) porque feature flags se cachean.
6. La respuesta incluye links a upgrade y contacto con ventas según corresponda.
7. Si la feature requiere plan Custom, se incluye `contact_sales_url` además del `upgrade_url`.

## Endpoints API Involucrados

- Todos los endpoints que correspondan a features con feature flags.
- `GET /api/v1/tenants/{tenant_id}/features` — Consulta de features activas.

---

## Métricas de Éxito

- Tiempo de respuesta del error 403: < 50ms.
- Feature access denials por mes: tracking para identificar demanda.
- Conversion de denials a upgrades: meta > 5% de denials resultan en upgrade.
- Mensajes de error claros: 100% de denials tienen upgrade_url y feature_name.

---

## Notas

- El error 403 solo aplica para features basadas en plan; no para permisos de usuario (RBAC) donde el error sería 403 pero con mensaje diferente "No tienes permiso".
- Los feature flags se evalúan en el API Gateway, antes de que el request llegue al servicio.
- El tracking de `feature_access_denied` sirve para priorización de roadmap: las features más pedidas se priorizan.
