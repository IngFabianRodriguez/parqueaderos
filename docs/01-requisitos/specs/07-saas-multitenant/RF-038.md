# RF-038 — Cobro Automático del Plan Mensual

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, stripe-integration, notification-service

---

## User Story

**Como** administrador de ParkCore **quiero** que el cobro de la suscripción de cada tenant se realice automáticamente el día de renovación **para** que el servicio no se interrumpa y no haya trabajo manual de cobros.

---

## Objetivo

El sistema debe cobrar automáticamente el plan de suscripción de cada tenant activo el día de renovación (`current_period_end`), usando Stripe como motor de cobros recurrentes, con manejo de fallos y reintentos integrados.

---

## Actores

- **Actor principal**: Sistema (automático, basado en Stripe)
- **Actor secundario**: Tenant admin (recibe notificaciones)

---

## Precondiciones

- El tenant tiene estado `active` y una suscripción activa en Stripe.
- La suscripción tiene `current_period_end` en la fecha actual o pasada.
- El método de pago (tarjeta) está vigente y tiene fondos disponibles.

---

## Postcondiciones

### En caso de éxito

- El pago se procesa correctamente; Stripe emite una invoice.
- Se actualiza: `current_period_start` y `current_period_end` (+1 mes).
- Se genera el evento `subscription_renewed` en la tabla de eventos SaaS.
- Se envía email de confirmación de renovación al tenant_admin.
- El MRR del mes se confirma como recolectado.

### En caso de falla

- El pago falla; Stripe reintenta según las reglas configuradas (RF-042).
- Se genera el evento `payment_failed` cada vez que se intenta y falla.
- Se envía notificación al tenant sobre el pago fallido.

---

## Flujo Funcional

1. Cada día a las 00:00 UTC, el job de billing ejecuta la verificación de renovaciones.
2. El job consulta Stripe para subscriptions con `current_period_end = hoy`.
3. Para cada suscripción:
   a. Stripe intenta cobrar automáticamente usando el payment method guardado.
   b. Si el cobro es exitoso: Stripe genera `invoice.paid`, envía webhook `invoice.payment_succeeded`.
   c. El sistema recibe el webhook; verifica el `subscription_id`; actualiza `current_period_start` y `current_period_end`.
   d. Sistema genera evento `subscription_renewed`.
   e. Sistema envía email de confirmación de renovación.
   f. Sistema actualiza `mrr_history` confirmando el MRR del mes.
4. Si el cobro falla: Stripe intenta reintentos automáticos (24h, 48h, 72h).
5. Después de 3 intentos fallidos: Stripe envía webhook `invoice.payment_failed`.
6. Sistema recibe el webhook; verifica el tenant; inicia flujo de suspensión (RF-035).

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| stripe_subscription_id | string | ID de la suscripción en Stripe | Sí |
| tenant_id | UUID | Tenant asociado | Sí |
| amount | decimal | Monto a cobrar | Calculado |
| currency | string | Moneda (COP) | Fijo |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| invoice_id | string | ID de la invoice en Stripe |
| amount_paid | decimal | Monto cobrado exitosamente |
| current_period_start | datetime | Nuevo inicio del periodo |
| current_period_end | datetime | Nueva fecha de fin del periodo |
| status | enum | `active`, `past_due`, `canceled` |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Tarjeta vencida | Stripe rechaza el pago; reintentos fallidos llevan a suspensión (RF-035). |
| Fondos insuficientes | Igual al anterior; se intenta en los reintentos. |
| Método de pago no existe | Stripe no puede cobrar; se marca la suscripción como `uncollectible`; se notifica al tenant para actualizar método de pago. |
| Moneda no soportada | No aplica — se usa siempre COP para tenants colombianos. |
| Cobro duplicate (error de Stripe) | Stripe ya tiene deduplicación por `invoice_id`; no procesar la misma invoice dos veces. |
| Suspensión ocurrió durante el día de renovación | El job no debe intentar cobrar suscripciones suspendidas. |

---

## Criterios de Aceptación

1. Todo tenant con suscripción activa se cobra automáticamente en su fecha de renovación.
2. El cobro exitoso actualiza `current_period_end` sumando exactamente 1 mes (o 1 año si es annual).
3. La invoice de Stripe se genera y está disponible para descarga por el tenant admin.
4. El email de confirmación de renovación se envía dentro de los 5 minutos posteriores al cobro exitoso.
5. Si el pago falla, el proceso de reintentos de Stripe aplica automáticamente (3 intentos en 7 días).
6. Después de 3 reintentos fallidos, el sistema inicia la suspensión del tenant (RF-035).
7. El evento `subscription_renewed` se registra cada vez que se confirma una renovación exitosa.

## Endpoints API Involucrados

- `POST /api/v1/webhooks/stripe` — Recibir `invoice.payment_succeeded` y `invoice.payment_failed`
- `GET /api/v1/invoices` — Listar invoices del tenant
- `GET /api/v1/invoices/{invoice_id}` — Descargar invoice PDF

---

## Métricas de Éxito

- Tasa de renovación exitosa > 95% (del total de suscripciones activas)
- Tiempo de cobro: instante (Stripe procesa en segundos)
- MRR recolectado = suma de MRR de tenants con `status = active` y renovación este mes
- failed_renewals rate < 5%

---

## Notas

- Stripe maneja el scheduling y cobro automático; el sistema solo necesita procesar los webhooks de confirmación.
- Para planes annual, el cobro es único al inicio del periodo; no hay renovación automática sino al final del año.
- El job diario de billing es un backup por si los webhooks no llegan; compara lo que Stripe dice vs lo que el sistema tiene y alerta si hay discrepancia.
