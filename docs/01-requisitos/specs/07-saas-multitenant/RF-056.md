# RF-056 — Feature Flags por Plan del Tenant

## Metadata

- **Módulo**: SaaS Multi-Tenant — Feature Flags por Plan
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: tenant-service, feature-flag-service, auth-service

---

## User Story

**Como** administrador de un tenant **quiero** que las features exclusivas de mi plan estén disponibles y las de planes superiores estén ocultas o bloquadas **para** que mi equipo use solo lo que está incluido en mi suscripción.

---

## Objetivo

El sistema debe definir un mapa de feature flags asociadas a cada plan (Starter, Professional, Enterprise, Custom), activando automáticamente las features incluidas en el plan del tenant y desactivando las que no corresponden. El acceso a una feature no autorizada debe retornar error 403.

---

## Actores

- **Actor principal**: Sistema (evalúa y aplica feature flags)
- **Actor secundario**: Tenant admin, operadores del tenant

---

## Precondiciones

- El tenant tiene un plan asignado (`starter`, `professional`, `enterprise`, `custom`).
- Las features flags están definidas en la tabla `feature_flags`.

---

## Postcondiciones

### En caso de éxito

- Las features del plan están activas para el tenant.
- El dashboard muestra solo los módulos y opciones incluidos en el plan.
- Los endpoints de API de features no incluidas retornan 403.

### En caso de fallback

- Si el plan no tiene definido un feature flag, se asume `false` (no incluido) por seguridad.

## Flujo Funcional

1. Al cargar el dashboard o hacer login, el sistema consulta `GET /api/v1/tenants/{tenant_id}/features`.
2. El servicio retorna la lista de features activas para el plan del tenant.
3. El frontend muestra/oculta módulos según la lista:
   - Si `multi_sede = false`: no mostrar el selector de sede adicional.
   - Si `bi_reportes = false`: no mostrar el módulo de BI en el menú.
   - Si `sso = false`: no mostrar opción de login SSO.
4. Al intentar acceder a una feature no incluida:
   a. El frontend puede deshabilitar el botón desde el inicio ( UX friendly).
   b. Si se hace request directo al endpoint, el API Gateway verifica el feature flag:
      - Si la key tiene scope pero la feature no está en el plan → 403.
      - Retorna `{"error": "feature_not_included", "plan": "Starter", "upgrade_url": "/settings/subscription"}`.
5. Los feature flags se cachean en el tenant-service por 5 minutos para performance.

| Campo | Tipo | Descripción |
|-------|------|-------------|
| features | object | Mapa de feature flags booleanas |

| upgrade_available | boolean | Si hay upgrade disponible |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Feature flag no existe para un plan | Asumir `false`; loguear warning para corregir en config. |
| Plan del tenant es null | Asumir Starter; no permitir acceso a ninguna feature premium. |
| Feature flag en DB pero código no la reconoce | La feature se muestra como disabled; loguear error de implementación. |
| Tenant hace downgrade | Las features del plan anterior se desactivan inmediatamente alapplied downgrade. |
| Feature de trial vs paid | Los tenants en trial tienen acceso a todas las features de su plan selected pero no pueden hacer upgrades hasta convertir. |

---

## Criterios de Aceptación

1. Cada plan tiene definidas todas sus features en `feature_flags`; no hay features indefinidas.
2. Las features del plan se cargan al iniciar sesión y se cachean por 5 minutos.
3. Los endpoints de features no incluidas retornan 403 con mensaje claro y link a upgrade.
4. El downgrade de plan desactiva las features que ya no corresponden inmediatamente.
5. El trial tiene acceso a las features del plan seleccionado, no acceso limitado a menos features.
6. La UI muestra un candado o \"Upgrade para desbloquear\" en features no disponibles.

## Métricas de Éxito

- Feature flags evaluados correctamente: 100%.
- Tiempo de evaluación de feature flags: < 10ms por request.
- Features bloqueadas con mensaje claro: 100% de intentos con 403 возвращают upgrade_url.
- Downgrade aplicado correctamente: 100% de los downgrades reflejan nueva lista de features en < 1 hora.

---

## Notas

- Las feature flags se evalúan en el API Gateway como middleware, antes de llegar al servicio.
- Las features también se evalúan en el frontend para mostrar/ocultar opciones de menú y deshabilitar botones.
- Las features no son permissions de usuario (RBAC), sino limitaciones del plan del tenant.
- Una feature deshabilitada a nivel de plan no se puede habilitar por usuario individual; el downgrade de plan es la única forma de perder acceso a una feature.
