# RF-041 — Downgrade de Plan al Final del Ciclo de Facturación

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, stripe-integration, tenant-service

---

## User Story

**Como** administrador de un tenant **quiero** hacer downgrade de mi plan al final de mi ciclo de facturación actual **para** reducir mi costo mensual si ya no necesito las features del plan actual.

---

## Objetivo

El sistema debe permitir que un tenant solicite un downgrade de plan, programándolo para que se aplique al final del ciclo de facturación actual, sin afectar el servicio durante el periodo restante. El downgrade生效 en la próxima fecha de renovación.

---

## Actores

- **Actor principal**: Tenant admin
- **Actor secundario**: Sistema (aplica downgrade en fecha de renovación)

---

## Precondiciones

- El tenant tiene estado `active` con una suscripción activa.
- El plan destino es de menor precio que el plan actual (downgrade).
- El tenant no ha solicitado un downgrade para el ciclo actual (solo uno por ciclo).

---

## Postcondiciones

### En caso de éxito (solicitud)

- Se registra el downgrade programado para `current_period_end`.
- Se muestra confirmación con la fecha de activación del nuevo plan.
- Se envía email de confirmación con la fecha de生效 y lo que implica (pérdida de features).
- Se genera el evento `downgrade_scheduled`.

### En caso de aplicarse (en renovación)

- El plan cambia al plan solicitado en la fecha de `current_period_end`.
- Los feature flags del plan anterior se desactivan al inicio del nuevo periodo.
- Se genera el evento `plan_downgraded`.
- Se envía email notificando que el downgrade se aplicó.

---

## Flujo Funcional

### Solicitud de downgrade

1. Tenant admin accede a Settings > Suscripción > "Hacer downgrade".
2. Sistema muestra los planes disponibles y destaca las features que se perderán con el downgrade.
3. Tenant admin selecciona el plan destino y confirma.
4. Sistema verifica que el plan destino es más económico que el actual.
5. Sistema muestra pantalla de confirmación: "Tu plan cambiará a [plan] el [current_period_end]. Perderás acceso a: [lista de features]. ¿Confirmas?"
6. Tenant admin confirma.
7. Sistema crea registro en `scheduled_plan_changes`: `tenant_id`, `new_plan_id`, `effective_date = current_period_end`, `status = pending`.
8. Sistema genera evento `downgrade_scheduled` con: `tenant_id`, `current_plan`, `new_plan`, `effective_date`.
9. Sistema envía email de confirmación indicando la fecha de生效.
10. El dashboard del tenant muestra un banner: "Downgrade programado para el [fecha]. Nuevo plan: [plan]."

### Aplicación del downgrade en renovación

1. Job de billing se ejecuta en la noche de `current_period_end`.
2. Para cada `scheduled_plan_changes` con `effective_date = hoy` y `status = pending`:
   a. Actualizar `subscriptions.plan_id` al `new_plan_id`.
   b. Actualizar `current_period_start = hoy`, `current_period_end = hoy + 1 mes`.
   c. Desactivar feature flags del plan anterior; activar los del nuevo plan (RF-056).
   d. Generar evento `plan_downgraded`.
   e. Enviar email de confirmación del downgrade aplicado.
   f. Marcar `scheduled_plan_changes.status = applied`.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tenant_id | UUID | Identificador del tenant | Sí |
| new_plan | enum | Plan destino (debe ser más económico que el actual) | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| current_plan | string | Plan actual (hasta la fecha de生效) |
| new_plan | string | Plan que se activará |
| effective_date | datetime | Fecha de生效 del downgrade |
| features_lost | array | Lista de features que se perderán |
| features_gained | array | Lista de features del nuevo plan (normalmente menor) |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Tenant ya tiene downgrade programado | Mostrar error "Ya tienes un downgrade programado para este ciclo. Cancélalo si deseas cambiarlo." |
| El plan destino tiene límites menores que el uso actual | Permitir downgrade; si el uso excede el límite del nuevo plan, se facturará como overage (RF-039). |
| Tenant cancela el downgrade antes de la fecha | Puede cancelar desde Settings > Suscripción; se marca `scheduled_plan_changes.status = canceled`. |
| El downgrade se aplica pero el tenant tenía transacciones pendientes | Las transacciones no se borran; el plan nuevo aplica para el siguiente ciclo, no retroactively. |

---

## Criterios de Aceptación

1. Un downgrade solicitado se aplica exactamente en `current_period_end`, no antes.
2. El tenant mantiene acceso a todas las features de su plan actual hasta la fecha de生效.
3. El evento `downgrade_scheduled` se genera al solicitar, y `plan_downgraded` al aplicarse.
4. Se notifica al tenant por email en ambos momentos (solicitud y aplicación).
5. Los feature flags del plan nuevo se activan al inicio del nuevo periodo.
6. Si el tenant tiene overage de recursos (más sedes que el límite del nuevo plan), se facturará al cierre del ciclo.
7. Solo un downgrade programado por ciclo; intentos adicionales retornan error.

## Endpoints API Involucrados

- `POST /api/v1/subscriptions/downgrade` — Programar downgrade
- `DELETE /api/v1/subscriptions/downgrade` — Cancelar downgrade programado
- `GET /api/v1/tenants/{tenant_id}/scheduled-change` — Consultar cambio programado

---

## Métricas de Éxito

- Downgrades completados en la fecha programada > 99%
- Cancelaciones de downgrade programados < 20%
- Tiempo entre solicitud y aplicación = diferencia entre fecha de solicitud y fin del ciclo ( hasta ~30 días)

---

## Notas

- A diferencia del upgrade, el downgrade no tiene costo de prorrateo enStripe porque el plan nuevo es más económico — no haycargo adicional, solo se paga menos en la próxima renovación.
- El downgrade no elimina datos; solo restringe features para el siguiente ciclo.
- Si el nuevo plan tiene límite de sedes menor al actual, el tenant puede seguir operando pero las sedes excedentes se marcan como "inactivas" hasta que haga upgrade de nuevo.
