# RF-036 — Churn (Cancelación) y Retención de Datos por 90 Días

## Metadata

- **Módulo**: SaaS Multi-Tenant — Gestión de Tenants
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, tenant-service, notification-service, data-retention-service

---

## User Story

**Como** administrador de ParkCore **quiero** que cuando un tenant cancele su suscripción sus datos se retengan durante 90 días **para** dar oportunidad de recuperación y cumplir con obligaciones legales de conservación de datos.

---

## Objetivo

El sistema debe manejar la cancelación (churn) de un tenant de forma que los datos se retengan por 90 días en modo solo lectura, pasado ese periodo los datos se eliminen de forma permanente siguiendo la política de retención. Durante el periodo de retención el tenant puede reactivarse.

---

## Actores

- **Actor principal**: Tenant admin (inicia cancelación), Sistema (dispara churn automático)
- **Actor secundario**: superadmin (puede cancelar manualmente)

---

## Precondiciones

- El tenant tiene estado `active`, `suspended` o `trial_expired`.
- El tenant ha decidido cancelar su suscripción o ha estado suspendido por más de 30 días.

---

## Postcondiciones

### En caso de churn (cancelación)

- El estado cambia a `canceled`.
- La suscripción en Stripe se cancela (`subscription.status = canceled`).
- Todos los usuarios del tenant pierden acceso al login, pero sus datos persisten.
- El acceso es solo lectura (GET de datos existentes) para el tenant_admin.
- El `canceled_at` timestamp se registra.
- El periodo de retención de 90 días comienza a correr.
- Se genera el evento `tenant_churned` en la tabla de eventos SaaS.
- El MRR del tenant se marca como `churned` a partir del siguiente mes.
- Se envía email de confirmación de cancelación con fecha de eliminación de datos.

### Durante el periodo de retención (90 días)

- El tenant_admin puede iniciar sesión y ver sus datos en modo solo lectura.
- No puede crear nuevas transacciones ni modificar configuraciones.
- Puede exportar sus datos (clientes, transacciones históricas) antes de la eliminación.
- Puede reaccionarivate suscripción en cualquier momento (sin perder datos).

### Después de los 90 días

- Los datos del tenant se eliminan permanentemente de la base de datos.
- El proceso de eliminación se ejecuta programadamente cada noche.
- Se genera el evento `tenant_data_deleted` para auditoría.
- El slug queda disponible para ser reutilizado por otro tenant.

---

## Flujo Funcional

### Cancelación iniciada por el tenant

1. Tenant admin accede a Settings > Suscripción > "Cancelar suscripción".
2. Sistema muestra pantalla de confirmación con: razón de cancelación (dropdown), advertencia de pérdida de datos, botón "Entiendo, cancelar".
3. Tenant admin confirma la cancelación; selecciona razón: "Muy caro", "No lo uso", "Me cambio a competencia", "Falta de features", "Otro".
4. Sistema actualiza: `status = canceled`, `canceled_at = NOW()`, `cancellation_reason = reason`.
5. Sistema cancela la suscripción en Stripe: `stripe.subscriptions.update(subscription_id, { cancel_at period end: false })`.
6. Sistema configura el job programado para eliminación en `NOW() + 90 días`.
7. Sistema genera evento `tenant_churned` con `tenant_id`, `reason`, `canceled_at`, `data_deletion_date`.
8. Sistema envía email de confirmación de cancelación con: fecha de baja, fecha de eliminación de datos, link para exportar datos, link para reactivar.
9. Sistema actualiza `mrr_history`: MRR del tenant pasa a `churned` para el mes siguiente.

### Churn automático por suspensión prolongada

1. Sistema ejecuta job diario que verifica tenants con `status = suspended` y `suspended_at < NOW() - 30 días`.
2. Para cada tenant en esa condición: ejecutar flujo de cancelación (mismo flujo descrito arriba, pero sin opción de reacciónтива).
3. El `cancellation_reason` se marca como `payment_failed_30_days`.

### Reactivación durante periodo de retención

1. Tenant admin accede al link del email de cancelación o intenta hacer login.
2. Sistema detecta que el estado es `canceled` y que `canceled_at + 90 días > NOW()`.
3. Sistema muestra pantalla: "Tu cuenta fue cancelada. ¿Quieres reactivate?Tus datos están seguros por X días más."
4. Si el tenant decide reaccionarivate: se le redirige al flujo de suscripción.
5. Al confirmar el pago, el sistema restaura `status = active` y limpia los timestamps de churn.
6. Sistema genera evento `tenant_reactivated_from_churn`.

### Eliminación al cumplir 90 días

1. Job nocturno (daily, a las 2:00 AM UTC) ejecuta `DELETE` de tenants con `status = canceled` y `canceled_at + 90 días < NOW()`.
2. El proceso elimina en orden: transacciones → clientes → zonas → sedes → usuarios → tenant.
3. Se genera evento `tenant_data_deleted` con `tenant_id` y `deleted_at`.
4. El slug se libera para ser reutilizado.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Tenant cancela con suscripción annual activa | Se cancela la suscripción de inmediato; los días restantes no se prorratean ni reembolsan ( según T&C). |
| Durante los 90 días el admin intenta crear transacción | Retornar 403 con mensaje: "Tu cuenta está cancelada. Reactivа tu suscripción para continuar." |
| El tenant se reacciona después de 90 días | No es posible; los datos ya fueron eliminados. El usuario debe registrarse de nuevo. |
| Job de eliminación falla (error de DB) | Reintentar la siguiente noche; alertar al equipo de operaciones; nunca dejar datos sin eliminar si ya pasaron los 90 días. |
| Stripe webhook llega después de la eliminación del tenant | Ignorar el webhook; no hay tenant al que asociar el evento. |

---

## Criterios de Aceptación

1. Un tenant puede iniciar cancelación desde el dashboard de Settings.
2. La cancelación es inmediata; la suscripción de Stripe se cancela el mismo día.
3. Los datos del tenant se retienen durante 90 días en modo solo lectura.
4. Durante los 90 días, el tenant_admin puede exportar sus datos.
5. La reacciónтива durante los 90 días restaura el acceso completo sin pérdida de datos.
6. Después de 90 días, los datos se eliminan permanentemente de la base de datos.
7. El slug queda disponible para ser reutilizado después de la eliminación.
8. El evento `tenant_churned` se registra con `tenant_id`, `cancellation_reason`, `canceled_at`.
9. El MRR del tenant se marca como `churned` en `mrr_history` a partir del mes siguiente al churn.
10. La eliminación de datos después de 90 días es irreversible y completa ( PII, transactions, users, logs).

## Métricas de Éxito

- Count de `tenant_churned` events = tenants con `status = canceled`
- Churn Rate mensual = (tenants_churned_en_mes / total_tenants_inicio_mes) * 100
- Tasa de reactivación durante retención > 10% ( benchmark: se recupera ~10-15% de churned customers)
- Tiempo promedio entre cancelación y eliminación: exactamente 90 días
- Eliminación completada sin errores en 100% de los casos

---

## Notas

- La política de 90 días de retención responde a: (a) obligación legal de conservación de datos financieros ( 5 años para facturas en Colombia), (b) oportunidad de recuperación del cliente. Para facturas específicamente, se retienen 5 años aunque el tenant se elimine; se exportan a un bucket de archival antes de eliminar el tenant.
- El churn se considera "voluntario" cuando lo inicia el tenant_admin, y "involuntario" cuando es automático por suspensión prolongada. Ambos se trackean por igual en métricas.
- Los tenants en plan annual que cancelan antes del fin del año: no hay reembolso ( según T&C aceptadas al contratar), pero la suscripción se cancela de inmediato y el acceso termina al final del periodo pagado.
