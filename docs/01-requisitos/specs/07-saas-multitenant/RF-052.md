# RF-052 — Scopes Granulares por API Key

## Metadata

- **Módulo**: SaaS Multi-Tenant — API Keys y Acceso Externo
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: api-gateway, auth-service

---

## User Story

**Como** administrador de un tenant **quiero** asignar scopes específicos a cada API key **para** limitar qué operaciones puede realizar cada integración sin afectar otras.

---

## Objetivo

El sistema debe definir y aplicar scopes granulares sobre cada API key, de modo que una key solo pueda acceder a los endpoints y acciones específicos para los que fue autorizada. Cada endpoint del API declarará qué scope requiere, y el API Gateway validará que la key tenga ese scope antes de procesar el request.

---

## Actores

- **Actor principal**: Tenant admin (asigna scopes al crear/editar keys)
- **Actor secundario**: Sistema (valida scopes en cada request)

---

## Precondiciones

- La API key existe y tiene scopes asignados (RF-051).
- El endpoint solicitado declara el scope requerido.

---

## Postcondiciones

### En caso de éxito

- El request pasa la validación de scopes; se procesa normalmente.
- El scope usado se registra en los logs de la API key.

### En caso de falla

- Si la key no tiene el scope requerido: retorna 403 con mensaje "Scope `{scope}` requerido para este endpoint."
- Si la key está revocada o vencida: retorna 401.

---

## Flujo Funcional

1. API Gateway recibe request con `Authorization: Bearer pk_live_xxx`.
2. API Gateway extrae la key y busca el hash en `api_keys`.
3. Se valida: key activa, no revocada, no vencida, IP autorizada.
4. Se extraen los `scopes` asociados a la key.
5. Se determina el scope requerido para el endpoint solicitado (mapeo endpoint→scope en config).
6. Se verifica si `scope_requerido ∈ scopes_de_la_key`:
   - **Si sí**: request se procesa; se registra en `api_key_logs`.
   - **Si no**: retorna 403 Forbidden con cuerpo `{"error": "insufficient_scope", "required": "reservas:write", "message": "Esta API key no tiene permiso para crear reservas."}`.

## Mapa de Scopes por Endpoint

| Método | Endpoint | Scope requerido | Descripción |
|--------|----------|-----------------|-------------|
| GET | `/api/v1/disponibilidad` | `disponibilidad:read` | Consultar disponibilidad |
| GET | `/api/v1/sedes` | `disponibilidad:read` | Listar sedes |
| GET | `/api/v1/reservas` | `reservas:read` | Leer reservas |
| POST | `/api/v1/reservas` | `reservas:write` | Crear reserva |
| PUT | `/api/v1/reservas/{id}` | `reservas:write` | Modificar reserva |
| DELETE | `/api/v1/reservas/{id}` | `reservas:write` | Cancelar reserva |
| GET | `/api/v1/transacciones` | `transacciones:read` | Leer transacciones |
| POST | `/api/v1/transacciones/ingreso` | `transacciones:write` | Registrar ingreso manual |
| POST | `/api/v1/transacciones/salida` | `transacciones:write` | Registrar salida manual |
| GET | `/api/v1/clientes` | `clientes:read` | Leer clientes |
| POST | `/api/v1/clientes` | `clientes:write` | Crear cliente |
| PUT | `/api/v1/clientes/{id}` | `clientes:write` | Editar cliente |

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| scope | string | Identifier del scope (ej: `reservas:write`) | Sí |

---

## Datos de Salida (error 403)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| error | string | Código de error: `insufficient_scope` |
| required_scope | string | Scope que el endpoint requiere |
| message | string | Descripción legible |
| key_scopes | array | Lista de scopes que tiene la key |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Key tiene scope `*` (todas) | Se expansiona a todos los scopes disponibles; acceso total. |
| Endpoint sin scope definido | Por defecto requiere `admin:read` o se rejecciona; debe definirse siempre. |
| Scope desconocido en la key | Se ignora; no causa error. |
| Nuevo endpoint sin scope mapeado | El API Gateway retorna 500 hasta que se defina; se alerta al equipo. |

---

## Criterios de Aceptación

1. Cada endpoint del API tiene declarado el scope requerido en la configuración del API Gateway.
2. Un request con una key que tiene el scope correcto pasa; la respuesta es normal.
3. Un request con una key que falta el scope retorna 403 con el error de `insufficient_scope`.
4. Los scopes se asignan a la key al momento de creación (RF-051) y se pueden editar al rotar.
5. El sistema de logs registra qué scope se usó en cada request (RF-055).
6. Los scopes de una key no se pueden cambiar sin crear una nueva key (rotación).

---

## Endpoints API Involucrados

- Todos los endpoints del API de ParkCore declaran su scope en el router del API Gateway.

---

## Métricas de Éxito

- Requests rechazados por scope insuficiente: < 1% del total.
- Time de validación de scopes: < 5ms por request.
- Cobertura de mapeo de scopes por endpoint: 100%.

---

## Notas

- Los scopes siguen el patrón `recurso:acción` (recurso en plural, acción en inglés).
- La key con scope `*` solo se asigna a keys de uso interno del equipo de ParkCore; nunca a tenants.
- Los scopes son additive: una key puede tener `disponibilidad:read` y `reservas:read` sin `reservas:write`.
- Al editar una integración (cambiar scopes), se debe rotar la key para que los cambios apliquen.
