# RF-045 — Escucha de Webhooks de Stripe para Pagos, Cambios de Plan y Cancelaciones

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, webhook-service, tenant-service

---

## User Story

**Como** sistema de ParkCore **quiero** escuchar los webhooks de Stripe para enterarme en tiempo real de pagos, cambios de plan y cancelaciones **para** mantener el estado de los tenants sincronizado con Stripe.

---

## Objetivo

El sistema debe exponer un endpoint único de webhooks que Stripe llamará cada vez que ocurra un evento relevante (pago exitoso, pago fallido, cambio de plan, cancelación). El endpoint debe verificar la firma de cada request, procesar el evento idempotentemente, y actualizar el estado del tenant correspondientemente.

---

## Actores

- **Actor principal**: Stripe (envía los webhooks)
- **Actor secundario**: Sistema (procesa los webhooks)

---

## Precondiciones

- El endpoint `/api/v1/webhooks/stripe` está registrado en el dashboard de Stripe con los eventos suscritos.
- La clave pública de Stripe (`STRIPE_WEBHOOK_SECRET`) está configurada en el servidor.

---

## Postcondiciones

### En caso de éxito

- El sistema procesa el evento y actualiza el estado del tenant.
- Se genera el evento correspondiente en la tabla `events_saas`.
- Se retorna HTTP 200 a Stripe (confirmación de recibo).

### En caso de falla

- El sistema retorna HTTP 400 o 500 según el tipo de error.
- Stripe reintenta el webhook según su política de reintentos.

---

## Flujo Funcional

1. Stripe envía `POST /api/v1/webhooks/stripe` con cuerpo JSON y headers (`stripe-signature`).
2. El middleware de webhooks:
   a. Extrae el body raw (como texto, no parseado todavía).
   b. Extrae el header `stripe-signature`.
   c. Verifica la firma usando `stripe.webhooks.constructEvent()` con el secret `STRIPE_WEBHOOK_SECRET`.
   d. Si la firma es inválida: retorna 401 Unauthorized.
3. El cuerpo se parsea como JSON; se extrae `type` (nombre del evento) y `data.object` (el objeto afectado).
4. Se busca el handler correspondiente al tipo de evento:

### Handlers por tipo de evento

#### `checkout.session.completed`
- Tenant hace checkout por primera vez.
- Crear/finalizar suscripción del tenant (RF-032).
- Generar evento `subscription_created` o `trial_converted`.

#### `invoice.payment_succeeded`
- Cobro exitoso de renovación.
- Marcar invoice como `paid` (RF-043).
- Actualizar `current_period_end` en `subscriptions`.
- Generar evento `subscription_renewed`.

#### `invoice.payment_failed`
- Primer, segundo o tercer intento de cobro fallido.
- Registrar intento en `payment_attempts` (RF-042).
- Si `attempt_count = 3`: iniciar suspensión (RF-035).
- Generar evento `payment_failed` o `payment_failed_final`.

#### `customer.subscription.updated`
- Cambio de plan (upgrade/downgrade).
- Actualizar `plan_id` en `subscriptions`.
- Generar evento `plan_upgraded` o `plan_downgraded`.

#### `customer.subscription.deleted`
- Cancelación de suscripción ( desde Stripe Customer Portal).
- Marcar tenant como `canceled` (RF-036).
- Programar eliminación de datos en 90 días.

#### `invoice.created`
- Stripe crea una invoice (para tracking).
- No es necesario actualizar estado del tenant; solo registrar.

#### `payment_method.attached`
- Nuevo método de pago agregado por el tenant.
- Actualizar `default_payment_method_id` en la tabla `tenants`.

5. Después de procesar, retornar HTTP 200 con `{ received: true }`.
6. Guardar el `event_id` de Stripe en la tabla `webhook_events` para deduplicación.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| stripe-signature | string | Firma de verificación del webhook | Sí |
| event_type | string | Tipo de evento de Stripe | Sí |
| event_id | string | ID único del evento (para idempotencia) | Sí |
| customer_id | string | Stripe Customer ID | Sí |
| subscription_id | string | Stripe Subscription ID | Sí |
| invoice_id | string | Stripe Invoice ID | Sí (invoice events) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| processed | boolean | Si el evento fue procesado exitosamente |
| event_id | string | ID del evento de Stripe (para logging) |
| action_taken | string | Descripción de la acción tomada |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Firma de webhook inválida | Retornar 401; no procesar; loguear como intento de fraude. |
| Webhook duplicado ( mismo event_id dos veces) | Si ya se procesó, retornar 200 sin hacer nada (idempotencia). |
| Evento de un tenant que ya no existe en DB | Ignorar el evento; loguear warning. |
| Tipo de evento no reconocido | Loguear como `unknown_event_type`; retornar 200 ( no bloquear a Stripe). |
| El handler lanza exception | Retornar 500; Stripe reintenta hasta 24h. |
| La DB está caída al recibir el webhook | Retornar 503; Stripe reintenta con backoff. |

---

## Criterios de Aceptación

1. El endpoint de webhooks procesa todos los eventos de Stripe listados en el flujo funcional.
2. La verificación de firma ocurre antes de cualquier procesamiento; eventos con firma inválida se rechazan con 401.
3. Cada evento se procesa idempotentemente (el mismo evento enviado dos veces produce el mismo resultado).
4. El `event_id` de cada webhook recibido se registra en `webhook_events` para evitar duplicados.
5. Los eventos desconocidos se loguean pero no fallan el endpoint (retorna 200).
6. El tiempo de procesamiento de cada webhook < 2 segundos.
7. Todos los eventos relevantes disparan actualizaciones de estado en el tenant y generan eventos SaaS correspondientes.

- `POST /api/v1/webhooks/stripe` — Endpoint único de webhooks de Stripe

---

## Métricas de Éxito

- Webhooks procesados sin errores: 99.9%
- Tiempo promedio de procesamiento < 500ms
- Cero webhooks con firma inválida procesados (rechazados en la puerta)
- Deduplicación efectiva: si el mismo event_id llega dos veces, se procesa solo una vez

---

## Notas

- La firma de Stripe se verifica con HMAC SHA256 usando el `STRIPE_WEBHOOK_SECRET`. No usar el API key para esto.
- Stripe reintenta los webhooks hasta 24 horas con exponential backoff si retorna errores 4xx o 5xx. Los códigos 2xx indican receipt confirmado.
- Para testing, usar Stripe CLI para reenviar eventos al endpoint local.
- El orden de los handlers importa: `invoice.payment_succeeded` puede llegar antes o después de `checkout.session.completed`; el sistema debe manejar ambos escenarios sin generar estados inconsistentes.
