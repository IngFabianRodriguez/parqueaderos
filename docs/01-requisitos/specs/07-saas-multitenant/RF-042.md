# RF-042 — Reintentos de Pago Fallido (3 Intentos en 7 Días)

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, stripe-integration, notification-service

---

## User Story

**Como** administrador de ParkCore **quiero** que los pagos fallidos se reintenten automáticamente antes de suspender al tenant **para** dar oportunidad de resolver problemas de pago sin interrumpir el servicio prematuramente.

---

## Objetivo

El sistema debe confiar en los reintentos automáticos de Stripe para pagos fallidos de renovación de suscripción, con un máximo de 3 intentos en un periodo de 7 días, antes de marcar la suscripción como `past_due` y proceder con la suspensión (RF-035).

---

## Actores

- **Actor principal**: Sistema (Stripe maneja los reintentos automáticos)
- **Actor secundario**: Tenant admin (recibe notificaciones), Billing service

---

## Precondiciones

- El tenant tiene estado `active` con suscripción activa.
- El cobro automático de renovación falló (primer intento).
- El método de pago en Stripe está configurado para permitir reintentos.

---

## Postcondiciones

### Entre reintentos

- Entre cada intento, Stripe envía webhook `invoice.payment_failed`.
- El sistema registra cada intento fallido y envía email de notificación al tenant.
- Se adjunta en el email un enlace directo al Stripe Customer Portal para actualizar el método de pago.

### Después del tercer intento fallido

- Stripe marca la invoice como no pagada y la suscripción como `unpaid`.
- El sistema recibe el webhook final de pago fallido.
- Se inicia el flujo de suspensión del tenant (RF-035).

---

## Flujo Funcional

1. Día 0: Primer intento de cobro ( renovación). Stripe intenta cobrar con el payment method guardado.
2. Si falla: Stripe programa reintento automático a las 24h (Día 1).
3. Día 1: Segundo intento. Si falla:
   - Stripe envía webhook `invoice.payment_failed` (segundo intento).
   - Sistema registra el evento `payment_failed`.
   - Sistema envía email: "Tu pago no pudo procesarse. Actualiza tu método de pago."
   - Enlace directo al Stripe Customer Portal para actualizar tarjeta.
   - Se programa el siguiente reintento en 24h (Día 2).
4. Día 2: Tercer intento. Si falla:
   - Stripe envía webhook `invoice.payment_failed` (tercer intento).
   - Sistema envía email más urgente: "Último intento en 24h. Si no se concreta, tu cuenta será suspendida."
   - Se programa el último reintento en 48h (Día 4) — para un total de 7 días.
5. Día 4: Último intento. Si falla:
   - Stripe marca la suscripción como `unpaid` y deja de reintentar.
   - Stripe envía webhook `invoice.payment_failed` con `attempt_count = 3`.
   - Sistema recibe el webhook; detecta que es el intento final.
   - Sistema inicia flujo de suspensión (RF-035): `status = suspended`.
   - Sistema envía email: "Tu cuenta ha sido suspendida. Actualiza tu método de pago para reactivarla."
   - Se genera evento `payment_failed_final`.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| stripe_invoice_id | string | ID de la invoice de Stripe | Sí |
| tenant_id | UUID | Tenant asociado | Sí |
| attempt_count | integer | Número de intento (1, 2, 3) | Sí |
| failure_code | string | Código de error de Stripe (card_declined, insufficient_funds, etc.) | Sí |
| failure_message | string | Mensaje descriptivo del error | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| subscription_status | enum | `active`, `past_due`, `unpaid` |
| attempts_made | integer | Número de intentos realizados |
| next_attempt_at | datetime | Fecha del próximo intento (si aplica) |
| days_until_suspension | integer | Días restantes hasta suspensión (7 - intentos * días) |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Tenant actualiza método de pago antes del tercer intento | El siguiente cobro de Stripe usa el nuevo método; no se dispara suspensión. |
| El método de pago es removido completamente de Stripe | Stripe no puede reintentar; marca como `uncollectible`; se suspensión inmediatamente. |
| Error específico de Stripe: "Tu tarjeta fue reportada como perdida o robada" | Stripe no reintenta; suspensión inmediata; email de contacto con soporte. |
| El email de notificación cae en spam | Los intentos de contacto son 3 en 7 días; la probabilidad de que al menos uno sea leído es alta. |

---

## Criterios de Aceptación

1. Los reintentos se ejecutan en los días 1, 2 y 4 (24h, 48h, 96h desde el intento inicial) — total 7 días.
2. Cada intento fallido genera un email de notificación al tenant admin.
3. Cada intento fallido genera un evento `payment_failed` en la tabla de eventos SaaS.
4. El email incluye siempre el enlace directo al Stripe Customer Portal para actualizar el método de pago.
5. Si después de 3 intentos no hay pago exitoso, el sistema inicia la suspensión del tenant.
6. El evento final es `payment_failed_final` (no solo `payment_failed`).
7. Si el tenant actualiza el método de pago entre intentos, Stripe usa el nuevo método en el siguiente intento.

---

## Métricas de Éxito

- Tasa de recuperación entre intento 1 y 3: benchmark ~30% de tenants resuelven el pago antes de suspensión.
- Tiempo promedio de resolución: 2-3 días.
- De los que llegan al tercer intento, ~70% se suspenden.

---

## Notas

- Stripe es el que maneja el scheduling de reintentos; el sistema solo necesita procesar los webhooks.
- El evento `invoice.payment_failed` de Stripe incluye `attempt_count`; usar ese campo para saber qué intento es.
- Los días de reintento son configurables en Stripe (default: 24h, 24h, 72h); coincidir con el estándar de 7 días.
- Si Stripe tiene configuración de reintentos diferente, prevalece la configuración de Stripe como fuente de verdad.
