# RF-035 — Suspensión de Tenant por Falta de Pago

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, tenant-service, notification-service, api-gateway

---

## User Story

**Como** administrador de ParkCore **quiero** que los tenants que no paguen su suscripción pasen a modo solo lectura **para** proteger los ingresos de la plataforma y motivar el pago sin perder al cliente definitivamente.

---

## Objetivo

El sistema debe cambiar el estado de un tenant a `suspended` cuando el pago de su suscripción falle después de agotar los reintentos (RF-042), restringiendo su acceso a operaciones de escritura mientras mantiene sus datos disponibles para lectura. La suspensión debe ser reversible al confirmar el pago.

---

## Actores

- **Actor principal**: Sistema (dispara suspensión automáticamente)
- **Actor secundario**: Tenant admin (recibe notificaciones), superadmin (puede revertir suspensión manualmente)

---

## Precondiciones

- El tenant tiene estado `active` y una suscripción vigente.
- El último intento de cobro automático (RF-042) falló después de 3 reintentos en 7 días.
- El webhook de Stripe `invoice.payment_failed` fue recibido y procesado.

---

## Postcondiciones

### En caso de éxito (suspensión)

- El estado del tenant cambia a `suspended`.
- El acceso a operaciones de escritura se bloquea en la API (POST, PUT, DELETE de transacciones, clientes, etc.).
- Las lecturas (GET) siguen funcionando para que el tenant pueda ver sus datos.
- El dashboard muestra un banner de suspensión indicando el monto pendiente y el botón de "Pagar ahora".
- Se envía email de notificación de suspensión al tenant_admin.
- Se genera el evento `tenant_suspended` en la tabla de eventos SaaS.
- El MRR del tenant se marca como `churned` para el mes en curso.

### En caso de éxito (reactivación)

- Al recibir `invoice.payment_succeeded` para la suscripción del tenant suspendido, el estado vuelve a `active`.
- El acceso completo se restituye inmediatamente.
- Se envía email de confirmación de reactivación.
- Se genera el evento `tenant_reactivated`.
- El MRR del mes se actualiza para reflejar el pago recibido.

---

## Flujo Funcional

### Suspensión automática

1. Stripe intenta cobrar la renovación mensual (día `current_period_end - 1`).
2. Pago falla (tarjeta rechazada, fondos insuficientes, etc.).
3. Stripe reintenta a las 24h, 48h y 72h (3 intentos totales).
4. Después del tercer intento fallido, Stripe envía webhook `invoice.payment_failed`.
5. El sistema recibe el webhook; verifica que el `customer_id` corresponde a un tenant.
6. El sistema verifica que el tenant tiene estado `active`.
7. El sistema actualiza: `status = suspended`, `suspended_at = NOW()`, `suspension_reason = 'payment_failed'`.
8. El sistema crea entrada en `suspension_history` con: `tenant_id`, `reason`, `amount_due`, `stripe_invoice_id`.
9. El sistema envía email de notificación de suspensión con botón de pago directo a Stripe Customer Portal.
10. El sistema genera evento `tenant_suspended`.
11. El sistema actualiza `mrr_history` del mes: el MRR pasa a estado `pending` o `lost` según política contable.

### Reactivación por pago

1. Tenant admin hace clic en el banner de suspensión y es redirigido a Stripe Customer Portal.
2. Tenant actualiza método de pago y paga la factura pendiente.
3. Stripe envía webhook `invoice.payment_succeeded`.
4. Sistema recibe webhook; verifica que corresponde a un tenant `suspended`.
5. Sistema actualiza: `status = active`, `reactivated_at = NOW()`.
6. Sistema genera evento `tenant_reactivated`.
7. Sistema envía email de confirmación de reactivación.
8. Sistema actualiza `mrr_history`: se marca el MRR como `recovered` para el mes.

### Suspensión manual por superadmin

1. Superadmin accede al panel de admin de ParkCore.
2. Busca el tenant por nombre o slug.
3. Selecciona "Suspender manualmente" e ingresa un motivo.
4. Sistema actualiza `status = suspended`, `suspension_reason = 'manual'`, `suspended_by = superadmin_id`.
5. Se envía email al tenant_admin notificando la suspensión manual.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Tenant suspendido intenta registrar entrada | Retornar 403 con mensaje: "Tu cuenta está suspendida. Paga tu suscripción para continuar." |
| Tenant suspendido accede a dashboard | Los datos se muestran en modo lectura; banner de suspensión prominente. |
| Pago realizado pero webhook no llega (15 min) | Sistema hace polling a Stripe API cada 5 min hasta confirmar; mientras tanto, muestra "Procesando pago...". |
| Suspensión manual y el tenant tiene叛逆duda real | Superadmin puede revertir; el tenant recibe email de disculpa. |
| Tenant suspendido por > 30 días | Se considera churn (RF-036); el estado pasa a `canceled` programáticamente. |

---

## Criterios de Aceptación

1. Un tenant sin pago después de 3 reintentos en 7 días pasa a estado `suspended` automáticamente.
2. En estado `suspended`, las operaciones POST/PUT/DELETE de transacciones, clientes y configuraciones retornan 403.
3. Las operaciones GET (lectura) siguen funcionando para que el tenant pueda ver sus datos.
4. El banner de suspensión es visible en todas las páginas del dashboard hasta que se reactive.
5. El botón "Pagar ahora" lleva directamente al Stripe Customer Portal del tenant.
6. La reactivación ocurre dentro de los 5 segundos posteriores al webhook de pago exitoso.
7. El evento `tenant_suspended` se registra con `tenant_id`, `reason`, `amount_due`, `suspended_at`.
8. Un tenant puede estar suspendido un máximo de 30 días antes de ser marcado como churned (RF-036).

## Dependencias

**RFs relacionados**:
- RF-038 — Cobro automático mensual (el que dispara los intentos de pago)
- RF-042 — Reintentos de pago fallido (3 reintentos antes de suspender)
- RF-036 — Churn (30 días suspendido lleva a churn)
- RF-045 — Webhooks de Stripe ( `invoice.payment_failed` y `invoice.payment_succeeded`)

**Tablas de base de datos afectadas**:
- `tenants` — columnas `status`, `suspended_at`, `suspension_reason`
- `suspension_history` — registro de eventos de suspensión/reactivación
- `events_saas` — eventos `tenant_suspended`, `tenant_reactivated`
- `mrr_history` — MRR marcado como `lost` o `pending` durante suspensión

---

## Endpoints API Involucrados

- `POST /api/v1/webhooks/stripe` — Recibir `invoice.payment_failed` (dispara suspensión)
- `POST /api/v1/webhooks/stripe` — Recibir `invoice.payment_succeeded` (dispara reactivación)
- `GET /api/v1/tenants/{tenant_id}/status` — Consultar estado ( usado por el frontend)

---

## Métricas de Éxito

- Tiempo entre último intento fallido y suspensión completa < 1 minuto
- Tasa de reactivación dentro de los 7 días posteriores a suspensión > 40%
- Tasa de conversión de suspensión a churn (30 días) < 25%
- Ingresos recuperados por reactivación计入 el NRR del mes

---

## Notas

- La suspensión es un estado de "grisa" — el tenant no se pierde pero tampoco paga. El objetivo es motivar el pago, no castigar.
- Si el tenant tiene un plan annual y falla un pago mensual, la suspensión aplica solo al mes en curso; los meses restantes no se ven afectados si ya se cobraron por adelantado.
- El Stripe Customer Portal permite al tenant actualizar su método de pago sin necesidad de asistencia del equipo de ParkCore.
- El MRR se considera "perdido" contablemente durante la suspensión, pero se restaura como "recuperado" si hay reactivación antes del cierre del mes.
