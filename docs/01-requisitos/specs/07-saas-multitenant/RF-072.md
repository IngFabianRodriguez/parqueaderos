# RF-072 — Autenticación de Dos Factores (MFA)

## Metadata

- **Módulo**: SaaS Multi-Tenant — Users y Roles por Tenant
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: auth-service, user-service

---

## User Story

**Como** administrador de un tenant **quiero** que todos los usuarios de mi organización puedan usar autenticación de dos factores (MFA) **para** agregar una capa extra de seguridad a sus cuentas y proteger contra accesos no autorizados.

---

## Objetivo

El sistema debe soportar MFA (Multi-Factor Authentication) para todos los usuarios del sistema, independientemente del método de autenticación (email/password o SSO). El método por defecto es TOTP (Time-based One-Time Password) vía aplicaciones como Google Authenticator o Authy. EI `tenant_admin` puede habilitar el requisito obligatorio de MFA para todo su equipo desde el admin panel.

---

## Actores

- **Actor principal**: Usuario del sistema (activa y usa MFA)
- **Actor secundario**: Administrador del tenant (habilita MFA obligatorio)

---

## Precondiciones

- El usuario tiene una cuenta activa en el sistema.
- El usuario tiene un dispositivo móvil con una aplicación TOTP compatible (Google Authenticator, Authy, Microsoft Authenticator).

---

## Postcondiciones

### En caso de éxito (activación)

- Se genera un secreto TOTP para el usuario y se guarda en `users.mfa_secret` (cifrado).
- Se presenta un QR code para que el usuario lo escanee con su app de autenticación.
- El usuario debe verificar un código TOTP válido para confirmar la activación.
- `users.mfa_enabled` cambia a `true`.

### En caso de éxito (login con MFA)

- El usuario ingresa email y password.
- El sistema verifica el password.
- El sistema solicita el código TOTP de 6 dígitos.
- El usuario ingresa el código; el sistema lo valida (ventana de ±1 timestep de 30 segundos).
- Si es válido, se establece la sesión.

### En caso de falla

- Si el código TOTP es incorrecto: error "Código inválido"; el usuario puede reintentar.
- Si el usuario tiene MFA obligatorio y no lo ha activado: se le redirige a la pantalla de activación de MFA antes de poder hacer login.

---

## Flujo Funcional

### Activación de MFA (usuario)

1. El usuario autenticado accede a `Settings → Security → Enable MFA`.
2. El sistema genera un secreto TOTP aleatorio (base32, 20 bytes) y lo almacena cifrado en `users.mfa_secret`.
3. El sistema genera el QR code con la estructura:
   - `otpauth://totp/ParkCore:{email}?secret={secret}&issuer=ParkCore&algorithm=SHA1&digits=6&period=30`
4. El usuario escanea el QR code con su app TOTP.
5. El sistema solicita un código de verificación: "Ingresa el código de 6 dígitos de tu app".
6. El usuario ingresa el código; el sistema valida:
   - `TOTP = HMAC-SHA1(key, counter)` con el secreto guardado.
   - Acepta códigos en la ventana de ±1 timestep (30 segundos antes y después).
7. Si el código es correcto, `mfa_enabled = true`; se muestra confirmación.
8. Se muestran los códigos de backup (8 códigos de un solo uso) para que el usuario los guarde en un lugar seguro.

### Login con MFA obligatorio (flujo completo)

1. Usuario ingresa `email` y `password`.
2. Sistema verifica `password_hash`. Si es incorrecto, retorna error (RF-073).
3. Sistema verifica si `mfa_enabled = true` para este usuario:
   - Si `false` y MFA es obligatorio para el tenant: redirigir a `GET /auth/mfa/setup`.
   - Si `false` y MFA no es obligatorio: establecer sesión normalmente.
   - Si `true`: continuar al siguiente paso.
4. Sistema presenta pantalla: "Ingresa tu código de autenticación".
5. Usuario ingresa código de 6 dígitos.
6. Sistema valida código contra el secreto TOTP del usuario.
7. Si es válido: establecer sesión. Si no: mostrar error; permitir reintentos (máximo 3 antes de bloqueo por 5 minutos).

### Deshabilitación de MFA

1. El usuario accede a `Settings → Security → Disable MFA`.
2. El sistema solicita password actual + código TOTP actual como confirmación.
3. Si ambos son válidos: `mfa_enabled = false`; `mfa_secret` se elimina.

|-------|------|-------------|-----------|

| Campo | Tipo | Descripción |
|-------|------|-------------|

| Campo | Tipo | Descripción |
|-------|------|-------------|
| success | boolean | Si la verificación fue exitosa |
| remaining_attempts | integer | Intentos restantes antes de bloqueo temporal |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Código TOTP incorrecto | Error; el usuario tiene 3 intentos adicionales; después se bloquea 5 minutos. |
| Reloj del usuario desincronizado con el servidor | El servidor acepta códigos en ventana de ±1 timestep (±30 segundos). Si sigue fallando, indicar al usuario que sincronice su reloj. |
| Usuario pierde acceso a su app de autenticación | Usar uno de los 8 códigos de backup; después de usar un backup code, se invalida. |
| Todos los backup codes usados | El admin del tenant puede resetear el MFA del usuario desde el admin panel. |
| MFA habilitado por el tenant pero usuario no lo ha activado | Redirigir a pantalla de activación obligatoria; el usuario no puede acceder hasta activar MFA. |
| Ataque de replay (mismo código usado dos veces) | Cada código TOTP se marca como usado en la tabla `used_totp_codes`; no puede reutilizarse. |

---

## Criterios de Aceptación

1. Todo usuario puede activar MFA en su cuenta usando cualquier app TOTP estándar (Google Authenticator, Authy, Microsoft Authenticator).
2. El QR code se genera en formato estándar `otpauth://` y es scaneable por cualquier app TOTP.
3. Los códigos TOTP son válidos solo por 30 segundos; se acepta ±1 paso (±30 segundos) para compensar skew de reloj.
4. Cada código TOTP puede usarse solo una vez (protección contra replay).
5. El `tenant_admin` puede hacer obligatorio el uso de MFA para todos los usuarios de su tenant.
6. Si el MFA es obligatorio, usuarios que no lo han activado no pueden hacer login hasta configurarlo.
7. El admin del tenant puede resetear el MFA de cualquier usuario de su organización.
8. Se generan 8 códigos de backup cuando se activa MFA; cada uno puede usarse una sola vez.
9. El sistema no almacena el secreto TOTP en texto plano; se cifra antes de guardar.

## Dependencias

**RFs relacionados**:
- RF-070 — Creación de usuarios (el MFA se configura después de crear la cuenta)
- RF-073 — Bloqueo después de 5 intentos fallidos (interactúa con MFA: después de 3 errores de MFA, se bloquea 5 minutos)
- RF-067 — SSO (si el IdP ya tiene MFA, el usuario no necesita MFA adicional de ParkCore; se detecta por el atributo `amr` del assertion)

**Tablas de base de datos afectadas**:
- `users` — `mfa_secret` (cifrado), `mfa_enabled`, `backup_codes`
- `used_totp_codes` — tracking de códigos usados (TTL: 24 horas)
- `tenant_settings` — `mfa_mandatory` flag

---

## Endpoints API Involucrados

- `POST /api/v1/users/me/mfa/setup` — Iniciar activación de MFA
- `POST /api/v1/users/me/mfa/verify` — Verificar código TOTP para confirmar activación
- `POST /api/v1/users/me/mfa/disable` — Deshabilitar MFA (requiere password + código actual)
- `GET /auth/mfa/setup` — Página de configuración obligatoria de MFA
- `POST /auth/mfa/verify` — Verificación de código TOTP en login
- `POST /api/v1/admin/users/{user_id}/mfa/reset` — Admin resetea MFA de un usuario

---

## Métricas de Éxito

- Tasa de usuarios con MFA habilitado en tenants con MFA obligatorio: 100%
- Tiempo promedio de activación de MFA: < 2 minutos
- Cero códigos TOTP reutilizados exitosamente (replay attack controlada)
- Tiempo de verificación de código TOTP: < 50ms

---

## Notas

- El secreto TOTP se genera usando un CSPRNG (cryptographically secure pseudo-random number generator).
- El secreto se cifra con AES-256-GCM antes de almacenarse en la base de datos; la clave de cifrado vive en el secrets manager.
- Los backup codes se hashean (como passwords) antes de almacenarse.
- La ventana de ±1 timestep significa que el servidor acepta códigos generados hasta 30 segundos antes o después del tiempo actual del servidor.
- Para usuarios SSO (RF-067), si el IdP ya realiza MFA (lo cual se marca en el atributo SAML `amr` como `mfa`), el sistema de ParkCore no solicita un segundo factor.
