# RF-054 — Rotación y Revocación de API Keys

## Metadata

- **Módulo**: SaaS Multi-Tenant — API Keys y Acceso Externo
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: api-gateway, auth-service, tenant-service

---

## User Story

**Como** administrador de un tenant **quiero** poder rotar y revoke mis API keys **para** mantener la seguridad de mis integraciones si una key se expone o si necesito cambiar los permisos.

---

## Objetivo

El sistema debe permitir que el tenant admin revoque una API key existente (inutilizándola inmediatamente) y rote una key (creando una nueva con los mismos scopes mientras la anterior se invalida). Ambos procesos deben ser inmediatos y generar eventos de auditoría.

---

## Actores

- **Actor principal**: Tenant admin (ejecuta la rotación o revocación)
- **Actor secundario**: Sistema (invalida la key y genera la nueva)

---

## Precondiciones

- La API key existe y pertenece al tenant del admin.
- La key no está ya revocata.

---

## Postcondiciones

### En caso de revocación

- La key se marca como `revoked` en la base de datos.
- Cualquier request con esa key retorna 401 inmediatamente.
- Se genera el evento `api_key_revoked`.

### En caso de rotación

- Se genera una nueva API key con los mismos scopes que la anterior.
- La key anterior se marca como `revoked`.
- La nueva key se muestra una sola vez al admin.
- Se genera el evento `api_key_rotated`.

---

## Flujo Funcional

### Revocación

1. Tenant admin accede a Settings > API Keys.
2. Identifica la key a revocar; hace clic en \"Revocar\" (icono de papelera).
3. Sistema muestra modal de confirmación: \"¿Estás seguro de revocar esta key? Cualquier integración que la use dejará de funcionar.\"
4. Tenant admin confirma.
5. Sistema marca `api_keys.status = revoked`, `revoked_at = NOW()`.
6. Sistema genera evento `api_key_revoked` con `key_id`, `tenant_id`, `revoked_by`, `reason`.
7. Sistema invalidate cualquier cachée de la key (si estaba cached).
8. El admin ve la key en la lista con estado `Revocada` y fecha de revocación.

### Rotación

1. Tenant admin accede a Settings > API Keys.
2. Identifica la key a rotar; hace clic en \"Rotar\" (icono de flecha circular).
3. Sistema muestra modal:
   - Nombre de la key que se rotará (display only).
   - Los scopes que tendrá la nueva key (los mismos).
   - Checkbox: \"Confirmo que la key anterior será revocada inmediatamente.\"
4. Tenant admin confirma.
5. Sistema genera nueva API key (mismo proceso que creación en RF-051).
6. Sistema marca la key anterior como `revoked`.
7. Sistema muestra la nueva key completa **una sola vez**.
8. Sistema genera evento `api_key_rotated` con `old_key_id`, `new_key_id`, `tenant_id`.
9. El admin copia la nueva key y la usa en su integración; la anterior ya no funciona.

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| key_id | UUID | Identificador de la key a revocar/rotar | Sí |
| reason | string | Motivo de revocación (audit) | No |
| confirmation | boolean | Confirmación explícita del admin | Sí |

---

## Datos de Salida (revocación)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| key_id | UUID | Key que fue revocada |
| status | enum | `revoked` |
| revoked_at | datetime | Timestamp de revocación |
| revoked_by | UUID | ID del admin que revocó |

---

## Datos de Salida (rotación)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| new_api_key | string | Nueva key completa (solo se muestra una vez) |
| new_key_id | UUID | ID de la nueva key |
| old_key_id | UUID | ID de la key anterior (ahora revocada) |
| scopes | array | Scopes de la nueva key (mismos que la anterior) |
| created_at | datetime | Timestamp de creación |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Revocar key que ya está en uso por integración | La integración dejará de funcionar inmediatamente; el admin debe haber previsto la rotación. |
| Rotar key sin confirmar checkbox | El modal no se envía; se pide confirmación explícita. |
| Revocar key que ya fue revocada | Mostrar error \"Esta key ya está revocada.\" |
| La última key activa del tenant | Se permite revocar; el tenant puede crear una nueva con RF-051. |
| Rate limit de la integración afectado | Los contadores de rate limit se resetean con la nueva key; la anterior no tiene más acceso. |

---

## Criterios de Aceptación

1. La revocación de una key hace que cualquier request con esa key retorne 401 en menos de 1 segundo.
2. La rotación crea una nueva key funcional con los mismos scopes que la anterior.
3. La key anterior se invalida en el mismo momento que la nueva se genera.
4. Los eventos `api_key_revoked` y `api_key_rotated` se generan con usuario, timestamp y key IDs.
5. Las keys revocadas se mantienen en la base de datos para auditoría pero no se pueden usar.
6. El admin puede ver el historial de revocaciones y rotaciones en Settings > API Keys.
7. No hay límite en el número de rotaciones que se pueden hacer.

---

## Endpoints API Involucrados

- `DELETE /api/v1/api-keys/{key_id}` — Revocar key
- `POST /api/v1/api-keys/{key_id}/rotate` — Rotar key
- `GET /api/v1/api-keys` — Listar keys (incluye revocadas)

---

## Métricas de Éxito

- Revocación efectiva: key deja de funcionar < 1 segundo después de confirmar.
- Rotación sin pérdida de acceso: la nueva key funciona inmediatamente al ser copiada.
- Historial de auditoría completo para cada key (creación, uso, rotación, revocación).
- Count de rotaciones vs creación nueva: meta que 60% de nuevas keys sean por rotación.

---

## Notas

- Se recomienda que el admin actualice su integración con la nueva key ANTES derotar, para evitar downtime. Sin embargo, si ambos procesos son inmediatos, el downtime sería mínimo.
- Las keys revocadas se retienen en la base de datos por 2 años para auditoría regulatoria.
- El prefijo de la nueva key es el mismo ambiente que la anterior (`pk_live_` → `pk_live_`).
