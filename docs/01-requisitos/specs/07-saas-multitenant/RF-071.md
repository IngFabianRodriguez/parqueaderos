# RF-071 — Asignación de Roles a Usuarios por el Tenant Admin

## Metadata

- **Módulo**: SaaS Multi-Tenant — Users y Roles por Tenant
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: user-service, tenant-service

---

## User Story

**Como** administrador de un tenant **quiero** asignar y cambiar los roles de los usuarios de mi organización **para** ajustar los permisos de acceso según las necesidades del negocio sin tener que eliminar y recrear usuarios.

---

## Objetivo

El sistema debe permitir que el `tenant_admin` asigne, modifique y elimine roles de los usuarios dentro de su tenant. Cada usuario puede tener uno o múltiples roles; cada rol tiene un conjunto de permisos definidos. Los cambios de rol son inmediatos y no requieren que el usuario vuelva a hacer login para que el nuevo rol tome efecto (se actualiza en el siguiente refresh de token).

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Superadmin (puede modificar roles en cualquier tenant)

---

## Precondiciones

- El actor tiene rol `tenant_admin` o `superadmin`.
- El usuario target pertenece al mismo tenant del actor (o el actor es superadmin).
- El rol a asignar existe en la tabla `roles`.

---

## Postcondiciones

### En caso de éxito

- La asignación de rol se inserta/actualiza en `user_roles`.
- Si el usuario tiene una sesión activa, el próximo request con su JWT reflejará los nuevos permisos.
- Se registra la acción en el `audit_log` indicando quién cambió qué rol a quién.

### En caso de falla

- Si el usuario no pertenece al tenant del admin: error de permisos.
- Si el rol no existe: error de validación.

---

## Flujo Funcional

### Asignar un rol a un usuario

1. El `tenant_admin` accede a `Settings → Users → [Usuario] → Roles`.
2. Ve la lista de roles disponibles en el sistema y los roles actualmente asignados al usuario.
3. Selecciona uno o más roles para asignar.
4. Confirma la acción.
5. El sistema verifica que el usuario target pertenezca al tenant del admin.
6. El sistema verifica que los roles seleccionados existan.
7. Se insertan las nuevas asignaciones en `user_roles` (sin eliminar las existentes a menos que sea un reemplazo).
8. Se registra en `audit_log`: `action = role_assigned`, `target_user_id`, `role`, `performed_by`.
9. El usuario recibe una notificación por email indicando que sus permisos cambiaron (opcional, configurable por el tenant).

### Cambiar el rol principal

1. Similar al flujo anterior, pero si el usuario tiene un rol principal y se asigna uno nuevo como principal, el anterior pasa a ser secundario.
2. El campo `is_primary` en `user_roles` indica cuál es el rol principal.

### Eliminar un rol de un usuario

1. El `tenant_admin` selecciona un rol asignado al usuario y hace clic en "Eliminar".
2. El sistema elimina la fila correspondiente de `user_roles`.
3. Si el usuario no tiene ningún rol restante, se le asigna `viewer` por defecto (para evitar usuarios sin acceso).
4. Se registra en `audit_log`.

### Restricción: No puede asignarse a sí mismo como `tenant_admin`

- Un `tenant_admin` no puede asignarse roles más altos que el suyo (incluyendo promocionarse a sí mismo).
- El superadmin puede asignar cualquier rol a cualquier usuario.

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| user_id | uuid | ID del usuario target | Sí |
| roles | array[string] | Lista de nombres de rol a asignar | Sí |
| is_primary | boolean | Si el primer rol de la lista debe ser el principal | No (default: true para el primer rol) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| user_id | uuid | ID del usuario modificado |
| assigned_roles | array | Lista de roles actualmente asignados |
| updated_at | timestamp | Timestamp del cambio |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Usuario no pertenece al tenant del admin | Error 403 Forbidden. |
| Rol a asignar no existe | Error 400 Bad Request. |
| Intentando asignar `superadmin` a un usuario | Error: solo el superadmin puede asignar este rol. |
| Admin intentando modificarse sus propios roles | Error: no puedes modificar tus propios roles. |
| Usuario sin roles después de una eliminación | Asignar `viewer` por defecto automáticamente. |
| Mismo rol asignado dos veces | Ignorar; no duplicar. |
| Asignar rol a usuario `inactive` | Permitir; el rol se asigna pero el usuario no puede login hasta estar activo. |

---

## Criterios de Aceptación

1. El `tenant_admin` puede agregar roles a cualquier usuario de su tenant.
2. El `tenant_admin` puede eliminar roles de cualquier usuario de su tenant.
3. Un usuario puede tener múltiples roles simultáneamente.
4. Los cambios de rol son inmediatos en el sistema de permisos; el usuario puede necesitar re-autenticarse para que el JWT refleje los nuevos permisos (en la práctica, los permisos se reevalúan en cada request o en el refresh del token).
5. Los cambios de rol quedan registrados en el `audit_log` con `performed_by`, `timestamp` y detalle del cambio.
6. No se permite asignar roles más allá de los que el admin mismo tiene (principio de menor privilegio).
7. El superadmin puede modificar roles de cualquier usuario en cualquier tenant.

## Dependencias

**RFs relacionados**:
- RF-070 — Creación de usuarios (el usuario debe existir para asignarle roles)
- RF-072 — MFA (usuarios con MFA tienen una capa adicional de seguridad)
- RF-073 — Bloqueo de usuarios (un usuario bloqueado puede seguir teniendo roles, pero no puede hacer login)
- RF-074 — Restricción de acceso a sedes específicas (complementa los roles)

**Tablas de base de datos afectadas**:
- `user_roles` — inserción/eliminación de asignaciones de rol
- `users` — consulta de `tenant_id` para validación de permisos
- `roles` — validación de que los roles existan
- `audit_log` — registro de cambios de rol

---

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/users/{user_id}/roles` — Listar roles del usuario
- `POST /api/v1/tenants/{tenant_id}/users/{user_id}/roles` — Asignar rol(es)
- `DELETE /api/v1/tenants/{tenant_id}/users/{user_id}/roles/{role}` — Eliminar rol
- `PUT /api/v1/tenants/{tenant_id}/users/{user_id}/roles` — Reemplazar todos los roles (full replace)

---

## Métricas de Éxito

- Tiempo de asignación de rol: < 200ms
- Cero asignaciones de roles inexistentes en producción
- 100% de los cambios de rol quedan registrados en audit_log

---

## Notas

- Los roles disponibles para asignación dependen del plan del tenant. Planes menores no tienen acceso a roles avanzados como `superadmin` o roles de BI.
- El rol `tenant_admin` es único y solo puede haber uno por tenant (aunque esto puede configurarse para permitir múltiples).
- El orden de los roles en `user_roles` no importa para los permisos; la evaluación de permisos es por membresía, no por orden.
- Si un usuario pertenece a un grupo del IdP que también tiene asignación manual de roles del admin, el sistema debe definir cuál tiene prioridad. Por defecto: la unión de ambos (acumulativo).
