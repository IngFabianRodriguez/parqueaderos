# RF-064 — Cálculo de Churn Rate Mensual (Tenants Perdidos / Total al Inicio del Mes)

## Metadata

- **Módulo**: SaaS Multi-Tenant — Métricas SaaS
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, metrics-service, tenant-service

---

## User Story

**Como** administrador de ParkCore **quiero** conocer el Churn Rate mensual **para** saber cuántos clientes perdimos en el mes y tomar acciones para reducir la fuga.

---

## Objetivo

El sistema debe calcular el Churn Rate mensual como el porcentaje de tenants que cambiaron a estado `churned` o `canceled` durante el mes respecto al total de tenants activos al inicio del mes. El resultado se desglosa por plan, segmento y motivo de churn.

---

## Actores

- **Actor principal**: Sistema (calcula el churn rate)
- **Actor secundario**: Equipo de ParkCore (consume la métrica)

---

## Precondiciones

- Existe data histórica de tenants con estados en cada mes.

---

## Postcondiciones

### En caso de cálculo exitoso

- El Churn Rate del mes se calcula y almacena en `churn_metrics`.
- Se genera el evento `churn_calculated` con los detalles.
- Si el Churn Rate excede un threshold (ej: > 5%), se envía alerta al equipo.

---

## Fórmula del Churn Rate

```
Churn Rate (%) = (Tenants perdidos en el mes / Total tenants al inicio del mes) * 100

Dónde:
- "Tenants perdidos" = tenants que cambiaron a estado `churned` o `canceled` durante el mes
- "Total al inicio del mes" = tenants con estado `active` el día 1 del mes a las 00:00 UTC
```

**Variantes**:
- **Logo Churn**: Solo tenants que cancelaron completamente.
- **Net Churn**: Logo Churn menos upgrades/contractions (involuntary churn).
- **Gross Churn**: 100% - Net Revenue Retention (NRR).

---

## Flujo Funcional

### Cálculo mensual (ejecuta el día 1 de cada mes)

1. El job `churn-calculation` se ejecuta el día 1 de cada mes a las 00:00 UTC.
2. Para el mes que cerró (mes anterior):
   a. Contar tenants con `status = active` al inicio del mes anterior (snapshot day 1).
   b. Contar tenants que cambiaron a `churned` o `canceled` durante ese mes.
   c. Calcular: `churn_rate = (churned / total_start) * 100`.
   d. Almacenar en `churn_metrics`: `month`, `total_tenants_start`, `churned_tenants`, `churn_rate`, `by_plan`, `by_segment`, `reason_breakdown`.
3. Se genera evento `churn_calculated`.
4. Si `churn_rate > 5%`: enviar alerta al equipo con detalle.

### Cálculo en tiempo real (para dashboard)

1. Cada día se recalcula el churn rate del mes en curso de forma estimada.
2. Se usa el mismo numerator/denominator pero con datos parciales del mes.
3. Esto permite ver la tendencia de churn incluso cuando el mes no ha terminado.

---

## Desglose del Churn

| Dimensión | Descripción |
|-----------|-------------|
| by_plan | Churn rate desglosado por: Starter, Professional, Enterprise, Custom |
| by_segment | Churn rate por: sector, tamaño de tenant, antigüedad |
| by_reason | Motivo de churn: falta de pago, cancelación voluntaria, merger, migró a competencia |
| revenue_churn | MRR perdido por churn (en pesos) vs MRR total al inicio del mes |

---

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| month | string | Mes a calcular (YYYY-MM) | Sí |
| snapshot_date | date | Fecha del snapshot de tenants activos | Sí |

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| month | string | Mes calculado |
| total_tenants_start | integer | Tenants activos al inicio del mes |
| churned_tenants | integer | Tenants perdidos durante el mes |
| churn_rate | decimal | Porcentaje de churn (0.05 = 5%) |
| revenue_churn_rate | decimal | Porcentaje de MRR perdido |
| by_plan | object | Churn rate por plan |
| by_reason | object | Desglose por motivo de churn |
| calculated_at | datetime | Timestamp del cálculo |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| No hay churn en el mes | churn_rate = 0%; se almacena igual para tracking histórico. |
| Un solo tenant al inicio del mes | churn_rate puede ser 0% o 100% según perdió o no; cálculo correcto. |
| Tenant reactivado después de churn | No se cuenta como churn (ya se contabilizó al perderlo). |
| Churn rate > 50% (anomalía) | Se genera alerta crítica al equipo; posibles datos corruptos. |
| Tentant migró de plan y luego hizo churn | Se cuenta como churn; el downgrade previo no importa. |

---

## Criterios de Aceptación

1. El Churn Rate se calcula mensualmente y se almacena en `churn_metrics`.
2. El cálculo usa: tenants activos al inicio del mes / tenants que hicieron churn durante el mes.
3. Se desglosa por plan, segmento y motivo de churn.
4. El MRR perdido por churn se calcula y almacena como `revenue_churn_rate`.
5. Se genera alerta cuando el churn rate mensual excede el threshold configurable (default 5%).
6. Los datos históricos de churn rate están disponibles para comparar mes a mes.
7. El evento `churn_calculated` se genera después de cada cálculo.

---

## Métricas de Éxito

- Churn rate benchmark para SaaS B2B: < 5% mensual es bueno, < 2% es excelente.
- Alertas de churn anomaly: cuando el churn rate sube > 2% vs mes anterior.
- Revenue churn rate vs logo churn rate: normalmente el revenue churn es menor porque los tenants que hacen churn suelen ser los más pequeños.

---

## Notas

- El churn rate es una de las métricas más importantes del negocio SaaS; se monitorea semanalmente en las reviews de leadership.
- Un churn rate alto puede indicar problemas con el producto, pricing, o servicio al cliente.
- El "churn preventable" (cancelaciones por falta de pago donde se podría haber retenido) se mide para evaluar efectividad de estrategias de retención.
