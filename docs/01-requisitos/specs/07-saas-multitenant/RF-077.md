# RF-077 — Forecasting de Ocupación Basado en Patrones Históricos

## Metadata

- **Módulo**: SaaS Multi-Tenant — BI Avanzado (solo Enterprise+)
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: analytics-service, forecasting-service

---

## User Story

**Como** administrador de un tenant Enterprise **quiero** que el sistema prediga la demanda de ocupación basándose en patrones históricos **para** poder planificar el personal, ajustar precios y optimizar la operación anticipadamente.

---

## Objetivo

El sistema debe generar forecasts de ocupación (predicción de demanda) basándose en los datos históricos de ocupación y factores como día de la semana, hora del día, estacionalidad, eventos especiales y holidays. El forecast debe incluir proyecciones horarias para los próximos 7 días y diarias para los próximos 30 días.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Usuario con rol `bi_viewer`

---

## Precondiciones

- El tenant tiene plan Enterprise o superior.
- Existe al menos 90 días de datos históricos de ocupación para la sede.
- El actor tiene permisos de visualización de BI.

---

## Postcondiciones

### En caso de éxito

- El forecast se genera y se muestra como gráfico de línea con banda de confianza.
- El forecast se actualiza diariamente con los nuevos datos.

### En caso de falla

- Si no hay suficientes datos históricos (< 90 días): mensaje indicando que el forecast no está disponible aún.

---

## Flujo Funcional

1. El usuario accede a `GET /dashboard/bi/forecasting` o selecciona el tab "Forecast" dentro del dashboard BI.
2. El sistema verifica que exista suficiente historial (≥ 90 días). Si no, muestra mensaje de "datos insuficientes".
3. El sistema consulta los datos históricos de ocupación:
   - Series temporales de ocupación por hora para los últimos 90+ días.
   - Eventos especiales registrados (holidays, eventos locales).
4. El modelo de forecasting (baseline: ARIMA o Prophet) genera predicciones:
   - Para las próximas 24 horas: predicción horaria.
   - Para los próximos 7 días: predicción diaria por hora.
   - Para los próximos 30 días: predicción diaria agregada.
5. El sistema retorna los datos de forecast incluyendo:
   - Punto estimado (media).
   - Banda de confianza (percentil 10 y 90).
6. El usuario puede overlayer el forecast con datos reales en tiempo real (si están disponibles para el período actual).

### Factores considerados en el modelo

- **Día de la semana**: Los patrones de ocupación varían entre días laborables y fines de semana.
- **Hora del día**: Patrones внутри-día (peak hours, off-peak).
- **Estacionalidad semanal y anual**: Patterns que se repiten cada semana y cada año.
- **Holidays y eventos especiales**: Días où la ocupación varía significativamente (Navidad, eventos locales).
- **Trend de largo plazo**: Crecimiento o decrecimiento gradual de la demanda.

### Configuración de eventos especiales

- El `tenant_admin` puede marcar días como "evento especial" desde el admin panel.
- Ejemplo: "Mar 15 - Evento en el centro comercial — ожидается peak +30%"
- Estos días se usan como feature adicional en el modelo.

---

## Datos de Salida

### Forecast hourly (próximas 24 horas)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| timestamp | datetime | Hora predicha |
| predicted_occupancy_rate | float | Porcentaje de ocupación predicho |
| confidence_low | float | Percentil 10 (mínimo esperado) |
| confidence_high | float | Percentil 90 (máximo esperado) |
| is_peak_expected | boolean | true si se espera que sea hora pico |

### Forecast daily (próximos 30 días)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| date | date | Día predicho |
| predicted_occupancy_avg | float | Ocupación promedio predicha para el día |
| predicted_transactions | integer | Transacciones predichas |
| confidence_low | float | Banda inferior |
| confidence_high | float | Banda superior |
| day_type | string | `weekday`, `weekend`, `holiday`, `special_event` |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Menos de 90 días de historial | Mostrar mensaje: "Se necesitan al menos 90 días de datos para generar el forecast." |
| Forecast para fecha donde hay evento especial no registrado | El modelo no puede predecir exactamente; la banda de confianza será más ancha. |
| Todos los datos históricos son 0 (sede nueva sin uso) | No generar forecast; mensaje de "Sin datos de ocupación." |
| Modelo devuelve predicción > 100% | Limitar a 100% como máximo. |
| Fallo del servicio de forecasting | Retornar 503; mostrar en UI "Pronóstico temporalmente no disponible." |

---

## Criterios de Aceptación

1. El forecast genera predicciones horarias para los próximos 7 días y diarias para los próximos 30 días.
2. El forecast incluye bandas de confianza (percentil 10-90).
3. El modelo considera día de la semana, hora del día, holidays y estacionalidad.
4. El usuario puede marcar días como "evento especial" y el forecast los considera.
5. El forecast se actualiza automáticamente cada 24 horas con los nuevos datos históricos.
6. Se indica claramente la precisión esperada del modelo (ej: "±8%" para predicciones a 24 horas).
7. Si hay menos de 90 días de datos, se muestra un mensaje informativo en lugar de datos incorrectos.
8. El acceso está restringido a tenants Enterprise+ con rol adecuado.

## Endpoints API Involucrados

- `GET /api/v1/analytics/forecast/hourly` — Forecast hourly para los próximos 7 días
- `GET /api/v1/analytics/forecast/daily` — Forecast daily para los próximos 30 días
- `POST /api/v1/admin/special-events` — Crear evento especial (para会影响 el forecast)
- `GET /api/v1/admin/special-events` — Listar eventos especiales configurados

---

## Métricas de Éxito

- Precisión del forecast a 24 horas: MAPE < 10% (mean absolute percentage error).
- Precisión del forecast a 7 días: MAPE < 15%.
- Tiempo de generación del forecast: < 30 segundos.
- Actualización automática: diario.

---

## Notas

- La implementación del modelo de forecasting puede usar Facebook Prophet, ARIMA, o un modelo LSTM según la madurez del equipo. Prophet es un buen punto de partida por su robustez con datos faltantes y seasonalidad.
- Las bandas de confianza más amplias indican mayor incertidumbre en la predicción.
- El forecast puede intégrarse con el sistema de pricing dinámico: si se predice alta ocupación, el sistema puede sugerir subir precios.
- Se debe guardar el historial de forecasts generados para poder comparar predicciones vs reales posteriormente.
