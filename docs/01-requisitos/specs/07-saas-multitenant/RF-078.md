# RF-078 — Exposición de Datos vía API GraphQL o REST para BI

## Metadata

- **Módulo**: SaaS Multi-Tenant — BI Avanzado (solo Enterprise+)
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: bi-service, api-gateway, analytics-service

---

## User Story

**Como** administrador de un tenant Enterprise **quiero** que los datos de analytics estén disponibles via API REST o GraphQL **para** poder conectar herramientas de BI externas (Tableau, Power BI, Looker) y construir dashboards personalizados con los datos de ParkCore.

---

## Objetivo

El sistema debe exponer un conjunto de endpoints REST y/o un schema GraphQL que permita a herramientas externas de BI consultar los datos de ocupación, ingresos y transacciones. La API debe soportar filtros por sede, rango de fechas y granularidad. Los datos se sirven en formato JSON. La autenticación es via API key válida para el tenant.

---

## Actores

- **Actor principal**: Herramienta de BI externa (Tableau, Power BI, Looker, custom)
- **Actor secundario**: Administrador del tenant (gestiona las API keys)

---

## Precondiciones

- El tenant tiene plan Enterprise o superior.
- Se ha generado una API key para el tenant con permisos de lectura de analytics.
- La herramienta externa tiene la API key y conoce los endpoints disponibles.

---

## Postcondiciones

### En caso de éxito

- La herramienta de BI recibe los datos en JSON con la estructura y filtros solicitados.
- Se registra el uso de la API (para billing/limites).

### En caso de falla

- Si la API key es inválida: 401 Unauthorized.
- Si el plan no es Enterprise+: 403 Forbidden.
- Si el rango de fechas es inválido: 400 Bad Request.

---

## Flujo Funcional

### Generación de API Key

1. El `tenant_admin` accede a `Settings → API Access → Generate New Key`.
2. Ingresa un nombre para la key (ej: "Tableau Production").
3. Selecciona los permisos: `read:analytics` (mínimo), `read:transactions`, `read:occupancy`.
4. El sistema genera una API key (256-bit random string) y la muestra una sola vez.
5. El admin copia la key y la configura en su herramienta de BI.

### Consulta de datos (REST)

1. La herramienta de BI hace `GET /api/v1/analytics/data` con:
   - Header: `Authorization: Bearer <api_key>`
   - Query params: `site_id`, `start_date`, `end_date`, `granularity` (hour/day/week/month).
2. El api-gateway valida la API key y los permisos asociados.
3. El analytics-service consulta los datos y retorna JSON.

### Consulta de datos (GraphQL)

1. La herramienta de BI hace `POST /api/v1/graphql` con query GraphQL.
2. El endpoint GraphQL acepta queries como:

```graphql
query AnalyticsData($siteId: UUID!, $startDate: Date!, $endDate: Date!) {
  analytics(siteId: $siteId, startDate: $startDate, endDate: $endDate) {
    occupancyData {
      timestamp
      occupancyRate
    }
    revenueData {
      timestamp
      revenue
      transactionCount
    }
  }
}
```

3. El sistema resuelve la query, consulta la base de datos y retorna el resultado en formato GraphQL (JSON).

### Estructura del Schema GraphQL

```graphql
type Query {
  analytics(
    siteId: UUID
    startDate: Date!
    endDate: Date!
    granularity: Granularity!
    zoneId: UUID
  ): AnalyticsPayload!

  occupancyHistory(
    siteId: UUID!
    startDate: Date!
    endDate: Date!
    granularity: Granularity!
  ): [OccupancyDataPoint!]!

  revenueHistory(
    siteId: UUID!
    startDate: Date!
    endDate: Date!
    granularity: Granularity!
  ): [RevenueDataPoint!]!
}

type AnalyticsPayload {
  occupancy: [OccupancyDataPoint!]!
  revenue: [RevenueDataPoint!]!
  transactions: [TransactionDataPoint!]!
}

type OccupancyDataPoint {
  timestamp: DateTime!
  occupancyRate: Float!
  spacesTotal: Int!
  spacesOccupied: Int!
}

type RevenueDataPoint {
  timestamp: DateTime!
  revenue: Decimal!
  transactionCount: Int!
  avgTicket: Decimal!
}
```

## Datos de Entrada

### REST API

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| site_id | uuid | Filtrar por sede | No (si no se incluye, todas las sedes) |
| start_date | date | Inicio del rango | Sí |
| end_date | date | Fin del rango | Sí |
| granularity | enum | `hour`, `day`, `week`, `month` | Sí |
| zone_id | uuid | Filtrar por zona | No |

### GraphQL

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| siteId | UUID | Filtrar por sede | No |
| startDate | Date | Inicio del rango | Sí |
| endDate | Date | Fin del rango | Sí |
| granularity | Granularity | hour/day/week/month | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| data | array | Array de data points según la granularidad solicitada |
| meta | object | Metadata: tenant_id, site_names, currency, generated_at |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| API key inválida o expirada | Retornar 401 con mensaje "API key inválida o expirada." |
| API key sin permisos suficientes | Retornar 403 con mensaje "Permiso insuficiente para este recurso." |
| Rango de fechas > 1 año | Retornar 400 con mensaje "Rango de fechas no puede exceder 1 año." |
| `site_id` no pertenece al tenant | Retornar 400 con mensaje "Sede no encontrada." |
| API rate limit excedido | Retornar 429 con headers de rate limit (X-RateLimit-Remaining). |
| Servicio de analytics no disponible | Retornar 503 con retry-after header. |

---

## Criterios de Aceptación

1. Los endpoints REST y GraphQL están disponibles solo para tenants Enterprise+.
2. La autenticación es via API key en el header `Authorization: Bearer <key>`.
3. Los endpoints soportan filtros por sede, zona, rango de fechas y granularidad.
4. Los datos se retornan en JSON con timestamps en UTC.
5. Los datos monetarios incluyen la moneda del tenant.
6. El rate limit es de 100 requests/minuto por API key (configurable).
7. Todas las llamadas se registran para auditoría y billing de uso.
8. La API es estável (versionada) para no romper integraciones existentes.

## Endpoints API Involucrados

- `GET /api/v1/analytics/data` — Consulta REST de datos de analytics
- `GET /api/v1/analytics/occupancy` — Ocupación histórica
- `GET /api/v1/analytics/revenue` — Ingresos históricos
- `POST /api/v1/graphql` — Endpoint GraphQL
- `POST /api/v1/admin/api-keys` — Generar nueva API key
- `GET /api/v1/admin/api-keys` — Listar API keys existentes
- `DELETE /api/v1/admin/api-keys/{key_id}` — Revocar API key

---

## Métricas de Éxito

- Tiempo de respuesta al percentile 95: < 2 segundos para queries de hasta 1 año de datos.
- Disponibilidad: 99.9%
- Rate limit: 100 requests/minuto por key, con burst de 20.
- Cero data leaks entre tenants.

---

## Notas

- La API GraphQL es preferible para herramientas de BI ya que permite pedir exactamente los campos necesarios (evita over-fetching).
- Se recomienda implementar response caching a nivel de API gateway para queries frecuentes con los mismos parámetros.
- Para herramientas que no soportan GraphQL, la API REST es suficiente.
- La API key debe almacenarse de forma segura por el cliente; ParkCore no almacena la key en texto plano, solo el hash.
- Se puede implementar un dry-run mode para que el tenant pruebe la API sin consumir datos reales.
