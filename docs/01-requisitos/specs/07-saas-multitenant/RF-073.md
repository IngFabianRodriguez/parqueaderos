# RF-073 — Bloqueo de Usuarios tras 5 Intentos Fallidos de Login

## Metadata

- **Módulo**: SaaS Multi-Tenant — Users y Roles por Tenant
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: auth-service, user-service

---

## User Story

**Como** administrador de un tenant **quiero** que después de 5 intentos fallidos de login la cuenta del usuario se bloquee temporalmente **para** proteger contra ataques de fuerza bruta y credential stuffing.

---

## Objetivo

El sistema debe bloquear una cuenta de usuario después de 5 intentos fallidos consecutivos de login (password incorrecto). El bloqueo dura 15 minutos, después de los cuales la cuenta se desbloquea automáticamente. Cada intento de login exitoso reinicia el contador. El `tenant_admin` puede desbloquear una cuenta manualmente antes de que expire el tiempo de bloqueo.

---

## Actores

- **Actor principal**: Sistema (gestiona el bloqueo automáticamente)
- **Actor secundario**: Administrador del tenant (puede desbloquear manualmente)

---

## Precondiciones

- El usuario existe y tiene estado `active`.
- El usuario no está previamente bloqueado.

---

## Postcondiciones

### En caso de bloqueo

- El campo `failed_login_attempts` del usuario se incrementa.
- Cuando llega a 5, `locked_until` se establece a `now() + 15 minutes`.
- El usuario recibe el mensaje "Cuenta bloqueada. Intenta de nuevo en X minutos o contacta al administrador."

### En caso de desbloqueo (automático)

- Cuando `locked_until` expira, `failed_login_attempts` se reinicia a 0.
- El usuario puede intentar hacer login de nuevo.

### En caso de desbloqueo (manual por admin)

- `locked_until` se establece a `null`.
- `failed_login_attempts` se reinicia a 0.

---

## Flujo Funcional

### Intento de Login

1. El usuario ingresa email y password.
2. El sistema busca el usuario por email.
3. Si el usuario tiene `status = blocked`:
   - Verificar si `locked_until` ha expirado.
   - Si expiró: desbloquear (reset counters) y continuar.
   - Si no expiró: retornar error 423 Locked con mensaje indicando tiempo restante.
4. Si el usuario tiene `status = inactive` o `pending_invitation`: retornar error 403.
5. El sistema verifica el password:
   - Si es incorrecto: `failed_login_attempts += 1`.
   - Si `failed_login_attempts >= 5`: establecer `locked_until = now() + 15 min`.
   - Retornar error de autenticación fallida.
6. Si el password es correcto:
   - `failed_login_attempts = 0`.
   - `locked_until = null` (por si estaba bloqueado).
   - Continuar con verificación de MFA si está habilitado (RF-072).

### Registro de intento fallido

- Cada intento fallido se registra también en `login_attempts` con:
  - `user_id`
  - `ip_address`
  - `user_agent`
  - `timestamp`
  - `reason` (invalid_password, account_locked, etc.)

### Desbloqueo manual por Admin

1. El `tenant_admin` accede a `Settings → Users → [Usuario] → Unlock Account`.
2. El admin confirma la acción.
3. El sistema establece `locked_until = null` y `failed_login_attempts = 0`.
4. Se registra en el `audit_log`.

### Notificación al Admin

- Cuando una cuenta se bloquea, se puede enviar una notificación al `tenant_admin` indicando: usuario afectado, IP origen, timestamp.
- Configurable por el tenant (on/off).

## Datos de Entrada

### Login request

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| email | string | Email del usuario | Sí |
| password | string | Password del usuario | Sí |
| ip_address | string | IP del cliente (enviada por el servidor) | Sí |
| user_agent | string | User agent del cliente | No |

---

## Datos de Salida

### Respuesta de login bloqueado

| Campo | Tipo | Descripción |
|-------|------|-------------|
| error | string | "account_locked" |
| message | string | Mensaje amigable con tiempo restante |
| locked_until | timestamp | Hora en que se desbloquea automáticamente |
| attempts_made | integer | Intentos fallidos realizados |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Usuario bloqeado pero el admin lo desbloquea manualmente antes de los 15 min | `locked_until = null`; `failed_login_attempts = 0`; el usuario puede login inmediatamente. |
| Ataque distribuido (diferentes IPs) | El contador de intentos es por usuario, no por IP. Si el ataque viene de muchas IPs, el usuario se bloqueará igualmente. Se puede implementar lock por IP como capa adicional. |
| El usuario cambia su password mientras está bloqueado | El contador no se reinicia automáticamente solo con cambio de password; el admin debe desbloquear manualmente o esperar expiry. |
| Usuario con MFA obligatorio se bloquea | La protección de 5 intentos aplica al primer factor (password). Con MFA habilitado, el atacante necesitaría el segundo factor también. |
| Reset de password mientras está bloqueado | Al resetear password, el admin debería también desbloquear la cuenta; el flujo de reset de password debe desbloquear automáticamente. |

---

## Criterios de Aceptación

1. Después de 5 intentos fallidos de password, la cuenta del usuario se bloquea por 15 minutos.
2. Los intentos fallidos se cuentan por usuario, no por sesión ni por IP.
3. Un login exitoso reinicia el contador de intentos fallidos a 0.
4. El usuario recibe un mensaje claro indicando cuánto tiempo falta para desbloquearse.
5. El `tenant_admin` puede desbloquear manualmente cualquier usuario de su tenant.
6. Los intentos fallidos se registran para auditoría (IP, timestamp, user agent).
7. Si el tenant tiene MFA obligatorio, el atacante necesitaría ambos factores; el contador de intentos se aplica al primer factor.
8. El sistema permite hasta 3 reintentos de código TOTP (RF-072) después del password, que también cuentan para el total de 5.

## Dependencias

**RFs relacionados**:
- RF-072 — MFA (la verificación de MFA viene después del password; el contador de intentos se reinicia cuando se pasa la verificación de password)
- RF-070 — Creación de usuarios (el estado de bloqueo aplica a usuarios activos)
- RF-074 — Restricción de acceso a sedes (el bloqueo previene acceso incluso si el usuario tiene acceso a sedes)

**Tablas de base de datos afectadas**:
- `users` — `failed_login_attempts`, `locked_until`
- `login_attempts` — registro histórico de intentos (para auditoría y detección de anomalías)
- `audit_log` — cuando un admin desbloquea manualmente

---

## Endpoints API Involucrados

- `POST /auth/login` — Login (retorna 423 si está bloqueado)
- `POST /api/v1/admin/users/{user_id}/unblock` — Admin desbloquea usuario
- `GET /api/v1/admin/users/{user_id}/login-history` — Admin consulta historial de login

---

## Métricas de Éxito

- Tiempo de verificación de bloqueo en login: < 10ms
- Bloqueo efectivo: 100% de las cuentas con 5 intentos fallidos se bloquean
- Cero usuarios desbloqueados manualmente sin registro en audit_log

---

## Notas

- Se recomienda también implementar detección de anomalías: si más de N intentos fallidos vienen de la misma IP en un período corto, esa IP puede ser blokeda temporalmente a nivel de WAF/API gateway.
- El registro de `login_attempts` tiene un TTL de 90 días para cumplir con regulaciones de retención de logs.
- Si el sistema recibe más de 100 login requests por minuto desde una misma IP, se puede activar un circuit breaker temporario.
- La funcionalidad de bloqueo aplica tanto para login con email/password como para SSO; sin embargo, el bloqueo de SSO lo gestiona el IdP, no ParkCore (ParkCore solo blokearía si el usuario hace login por email/password).
