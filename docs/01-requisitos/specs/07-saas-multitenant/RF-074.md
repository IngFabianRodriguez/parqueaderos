# RF-074 — Restricción de Acceso de Usuarios a Sedes Específicas

## Metadata

- **Módulo**: SaaS Multi-Tenant — Users y Roles por Tenant
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: user-service, site-service, tenant-service

---

## User Story

**Como** administrador de un tenant con múltiples sedes **quiero** restringir qué usuarios pueden acceder a qué sedes **para** que cada operador solo vea y gestione la información de las sedes que le corresponden, facilitando el control operativo y la confidencialidad de datos.

---

## Objetivo

El sistema debe permitir que el `tenant_admin` configure qué sedes puede acceder cada usuario dentro de su tenant. Un usuario con rol `operator` o `viewer` puede tener acceso a todas las sedes o solo a un subconjunto específico. Cuando el usuario hace operaciones, el sistema filtra automáticamente los datos para mostrar solo las sedes autorizadas.

---

## Actores

- **Actor principal**: Administrador del tenant (configura los permisos por sede)
- **Actor secundario**: Usuario del sistema (accede solo a las sedes permitidas)

---

## Precondiciones

- El usuario existe y tiene rol `operator` o `viewer`.
- Las sedes que se quieren asignar pertenecen al tenant del admin.
- El usuario tiene un `tenant_id` asignado.

---

## Postcondiciones

### En caso de éxito

- Las asignaciones de sede se insertan en `user_site_access`.
- El usuario solo puede ver/operar en las sedes asignadas.
- Si el usuario intenta acceder a una sede no asignada, el sistema retorna 403 Forbidden.

### En caso de falla

- Si el admin intenta asignar una sede que no pertenece a su tenant: error de validación.

---

## Flujo Funcional

### Asignar sedes a un usuario (Admin)

1. El `tenant_admin` accede a `Settings → Users → [Usuario] → Site Access`.
2. Ve la lista de todas las sedes de su tenant con checkboxes.
3. Selecciona las sedes a las que el usuario debe tener acceso.
4. Guarda la configuración.
5. El sistema elimina las asignaciones existentes en `user_site_access` para ese usuario y reinserta las nuevas (replace completo).
6. Se registra en el `audit_log`: `action = site_access_updated`, `user_id`, `sites_assigned`, `performed_by`.

### Asignación de "Todas las sedes"

- Si el admin marca la opción "Acceso a todas las sedes", se deja `user_site_access` vacío (NULL) para ese usuario, lo que significa acceso total.
- La consulta de "todas las sedes" es: `WHERE tenant_id = user.tenant_id`.

### Validación de acceso en tiempo de query

1. El usuario hace una request a `GET /api/v1/sites` o `GET /api/v1/parking-lots`.
2. El sistema añade automáticamente la condición `site_id IN (SELECT site_id FROM user_site_access WHERE user_id = ?)` O, si `user_site_access` está vacío para el usuario, omite el filtro (acceso total).
3. Los datos de otras sedes no son retornados ni accesibles.

### Validación de acceso en operaciones

- Cuando un operador registra un ingreso/salida (RF-005, RF-006), el sistema verifica que la `site_id` de la operación esté en las sedes autorizadas del usuario.
- Si intenta hacer una operación en una sede no autorizada: 403 Forbidden.

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| user_id | uuid | ID del usuario | Sí |
| site_ids | array[uuid] | Lista de IDs de sedes con acceso | Sí (puede estar vacía para "todas") |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| user_id | uuid | ID del usuario |
| site_count | integer | Número de sedes asignadas |
| all_sites_access | boolean | true si tiene acceso a todas las sedes |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Usuario con rol `tenant_admin` | Siempre tiene acceso a todas las sedes; `user_site_access` no aplica (se ignora). |
|Usuario con rol `viewer` sin sedes asignadas | El usuario ve 0 sedes; esto es válido. |
| Admin intenta asignar sede de otro tenant | Error 400; no se permite cross-tenant. |
| Sede eliminada después de estar asignada al usuario | El registro huerfano se limpia en un job nocturno; el usuario simplemente no tendrá acceso a esa sede. |
| Usuario promueve a `tenant_admin` | La restricción de sedes se deja de aplicar. |
| Lista vacía de site_ids = "todas las sedes" | Permitir como opción para que el admin dé acceso total a un operador. |

---

## Criterios de Aceptación

1. El `tenant_admin` puede asignar 0, 1 o múltiples sedes a cualquier usuario con rol `operator` o `viewer`.
2. Los usuarios con rol `tenant_admin` tienen acceso implícito a todas las sedes; no necesitan asignación.
3. Un usuario sin asignación de sedes (NULL) tiene acceso a todas las sedes de su tenant.
4. Un usuario con lista vacía de sitios asignados ve 0 sitios; no hay acceso implícito.
5. Toda operación de lectura y escritura filtra automáticamente por las sedes asignadas; no depende del rol del usuario.
6. El intento de acceso a una sede no autorizada retorna 403 Forbidden.
7. Los cambios de asignación de sedes se registran en el `audit_log`.
8. El sistema soporta hasta 500 sedes por tenant sin degradación de performance.

- `user_site_access` — tabla de asignaciones usuario-sede

- `users` — para determinar el rol y si se aplica el filtro
- `audit_log` — registro de cambios

---

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/users/{user_id}/site-access` — Ver asignaciones de sede
- `PUT /api/v1/tenants/{tenant_id}/users/{user_id}/site-access` — Actualizar asignaciones
- `DELETE /api/v1/tenants/{tenant_id}/users/{user_id}/site-access` — Eliminar todas las asignaciones (acceso a todas las sedes)

---

## Métricas de Éxito

- Tiempo de verificación de acceso: < 5ms por query (usando índice en `user_site_access.site_id` + `user_id`).
- Cobertura de filtro: 100% de las queries de sitios/lotes/zonas preguntan al sistema de autorización.
- Cero fugas de datos de sedes no autorizadas.

---

## Notas

- La asignación de sitios es por usuario, no por rol. Esto permite que dos operadores del mismo rol tengan acceso a conjuntos de sedes diferentes.
- Si un usuario pertenece a múltiples sitios, el sistema de búsqueda debe soportar `OR` entre las sedes: `site_id IN (site_1, site_2, site_3)`.
- Para performance, se recomienda cachear la lista de sedes autorizadas por usuario en Redis con TTL de 5 minutos, invalidate on update.
- El rol `superadmin` tiene acceso a todas las sedes de todos los tenants; el filtro no aplica.
- Esta funcionalidad es especialmente relevante para operadores de estacionamiento que trabajan en sedes específicas y no deben ver información de otras sedes.
