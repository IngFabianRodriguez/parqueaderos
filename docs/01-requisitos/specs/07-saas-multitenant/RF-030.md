# RF-030 — Asignación de Slug Único por Tenant

## Metadata

- **Módulo**: SaaS Multi-Tenant — Gestión de Tenants
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: tenant-service, api-gateway, auth-service

---

## User Story

**Como** administrador de un parqueadero **quiero** que mi cuenta tenga un slug único en la URL **para** que mis operadores y clientes accedan a través de `/app/{mi-parqueadero}`.

---

## Objetivo

El sistema debe generar, asignar y validar un slug único para cada tenant, asegurando que sea accesible en la ruta `/app/{slug}`, navegable desde cualquier navegador y utilizado como identificador público del tenant en la plataforma.

---

## Actores

- **Actor principal**: Sistema (generación automática)
- **Actor secundario**: Usuario fundador (puede proponer o sobrescribir slug sugerido)

---

## Precondiciones

- El tenant está siendo creado (RF-029) o el tenant existente solicita cambio de slug.
- El slug propuesto no existe en el sistema ni está reservado.
- El tenant tiene al menos un `tenant_id` generado.

---

## Postcondiciones

### En caso de éxito

- El slug queda reservado y asociado al `tenant_id` correspondiente.
- La ruta `/app/{slug}` responde con el dashboard del tenant (redirección al frontend).
- El slug es inmutable para tenants con suscripción activa ( Starter+ ); solo modificable en trial.
- Se registra el evento `slug_assigned` o `slug_changed` según corresponda.

### En caso de falla

- El slug no se reserva; se retorna error al usuario con opciones sugeridas.
- Si el slug ya estaba reservado temporalmente (en proceso de registro que falló), se libera tras 30 minutos.

---

## Flujo Funcional

### Asignación automática durante creación de tenant

1. Sistema recibe `empresa_nombre` del formulario de registro.
2. Sistema normaliza el nombre: convierte a lowercase, elimina acentos, reemplaza espacios por guiones, elimina caracteres especiales.
3. Sistema genera slug base: `primera-palabra` + `-` + 4 caracteres alfanuméricos aleatorios (ej: `park` + random).
4. Sistema verifica disponibilidad del slug en tabla `slugs`.
5. Si está disponible: reservar slug, asociar a `tenant_id`, continuar flujo.
6. Si no disponible: intentar hasta 5 variantes con sufijos numéricos (`-2`, `-3`, ... `-6`).
7. Si todas las variantes están tomadas: solicitar al usuario que ingrese un slug manualmente.
8. Slug manual: validar formato (lowercase, guiones, números, letras, 3-50 caracteres), verificar disponibilidad, reservar.

### Cambio de slug (solo para tenants en trial)

1. Tenant admin solicita cambio de slug desde Settings > Informações da Conta.
2. Sistema muestra input con slug actual como sugerencia.
3. Sistema valida formato y disponibilidad del nuevo slug.
4. Sistema registra evento `slug_changed` con slug anterior y nuevo.
5. Sistema actualiza registro en tabla `slugs`.
6. Sistema redirige la sesión del admin a la nueva URL.

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| empresa_nombre | string | Nombre de la empresa para generar slug base | Sí (auto) |
| slug_personalizado | string | Slug elegido manualmente por el usuario | No |
| tenant_id | UUID | Identificador del tenant al que se asocia | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| slug | string | Slug único asignado (ej: `mi-parqueadero-a7b2`) |
| tenant_id | UUID | Tenant asociado |
| url_acceso | string | URL completa de acceso (`https://parkcore.app/app/mi-parqueadero-a7b2`) |
| disponible | boolean | Indica si el slug estaba disponible antes de asignar |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Slug generado ya existe | Probar hasta 5 variantes; si todas existen, pedir input manual. |
| Slug manual con formato inválido | Mostrar error "El slug solo puede contener letras minúsculas, números y guiones. Longitud: 3-50 caracteres." |
| Slug manualmente propuesto ya existe | Mostrar error "Este slug ya está en uso. Prueba otro." |
| Tenant con suscripción activa intenta cambiar slug | Mostrar error 403 "No puedes cambiar tu slug mientras tengas una suscripción activa. Cancela y vuelve a trial si necesitas cambiarlo." |
| Slug contiene palabras reservadas | Bloquear: `admin`, `app`, `api`, `www`, `mail`, `support`, `billing`, `staging`, `parkcore`, `localhost` |
| Slug con nombre de marca famosa | Permitir pero marcar para revisión manual (flag `slug_needs_review = true`) |
| Registro iniciado pero no completado (caso orphan) | Slug temporal reservado 30 minutos; si no completa registro, se libera automáticamente. |

---

## Criterios de Aceptación

1. Todo tenant nuevo tiene exactamente un slug asignado en el momento de su creación.
2. El slug es único a nivel global en el sistema; no existen dos tenants con el mismo slug.
3. La ruta `/app/{slug}` responde con el dashboard del tenant correspondiente en menos de 500ms.
4. Un tenant en estado `trial` puede cambiar su slug una vez durante los 14 días.
5. Un tenant con estado `active` (suscrito) no puede cambiar su slug.
6. Los slugs no pueden contener espacios, caracteres especiales ni mayúsculas.
7. Los slugs deben tener entre 3 y 50 caracteres.
8. El sistema bloquea slugs que coincidan con palabras reservadas listadas.
9. Cada cambio de slug genera un evento `slug_changed` en la tabla de eventos.

---

## Endpoints API Involucrados

- `POST /api/v1/tenants` — Crear tenant con slug automático
- `POST /api/v1/tenants/{tenant_id}/slug` — Actualizar slug (solo trial)
- `GET /api/v1/slugs/check?slug={slug}` — Verificar disponibilidad de slug

---

## Métricas de Éxito

- Slugs únicos generados sin colisiones en primer intento > 85%
- Tiempo de verificación de disponibilidad < 50ms
- Tasa de slugs reservados por usuarios manualmente < 5% del total
- URL `/app/{slug}` responde < 500ms (p95)

---

## Notas

- El slug funciona como identificador público del tenant. No debe confundirse con el `tenant_id` (UUID interno).
- La normalización del nombre sigue rules: `Parqueadero Los Pinos` → `parqueadero-los-pinos-a7b2`.
- Para tenants Enterprise con custom domain, el slug sigue existiendo pero el acceso primario es vía el dominio propio (ej: `miempresa.com` → slug `miempresa` reservado pero acceso por dominio).
- El endpoint de verificación de disponibilidad debe ser público (no requiere auth) para permitir que el frontend valide en tiempo real mientras el usuario escribe.
