# RF-032 — Conversión de Trial a Suscripción Activa

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, tenant-service, notification-service

---

## User Story

**Como** usuario de un tenant en trial **quiero** que mi cuenta se convierta automáticamente a suscripción activa cuando realice mi primer pago **para** que mi acceso a la plataforma sea permanente y no haya interrupciones.

---

## Objetivo

El sistema debe detectar el primer pago exitoso de un tenant en estado trial y convertir su estado a `active`, registrando la suscripción en Stripe, actualizando el plan asociado y generando el evento de métricas correspondiente.

---

## Actores

- **Actor principal**: Tenant (a través de Stripe Checkout o Customer Portal)
- **Actor secundario**: Sistema (escucha de webhooks), Billing service

---

## Precondiciones

- El tenant tiene estado `trial` con `trial_end_date` vigente o ya expirado.
- El tenant ha seleccionado un plan (Starter, Professional, Enterprise, Custom).
- El tenant ha proporcionado información de pago válida a Stripe.
- Existe un `stripe_customer_id` asociado al tenant.

---

## Postcondiciones

### En caso de éxito

- El estado del tenant cambia de `trial` a `active`.
- Se crea un registro de `subscription` en la base de datos con el `stripe_subscription_id`.
- El `trial_end_date` se marca como `converted_at` con la fecha del pago.
- Se genera el evento `trial_converted` en la tabla de eventos SaaS.
- El MRR del tenant se actualiza con el valor del plan adoptado.
- Se envía email de confirmación de suscripción al tenant_admin.
- Todas las funcionalidades restringidas para trial se desbloquean según el plan.

### En caso de falla

- El pago se marca como fallido en Stripe.
- El tenant permanece en estado `trial` hasta el siguiente intento de pago.
- Se genera el evento `payment_failed` con detalle del error.
- Se muestra mensaje al usuario según el tipo de error de Stripe.

---

## Flujo Funcional

1. El tenant inicia el flujo de upgrade desde el dashboard (botón "Suscribirme").
2. El sistema crea o reutiliza un `Stripe Customer` para el tenant.
3. El sistema crea una `Stripe Subscription` con el plan seleccionado y `trial_end_date` como fecha de inicio (si el trial aún no ha vencido) o `NOW()` (si ya venció).
4. El sistema redirige al usuario a Stripe Checkout o usa el Customer Portal.
5. El usuario completa el pago en la interfaz de Stripe.
6. Stripe envía webhook `invoice.payment_succeeded` al sistema.
7. El sistema verifica que el `customer_id` y `subscription_id` corresponden a un tenant en la base de datos.
8. El sistema verifica que el tenant tiene estado `trial`.
9. El sistema actualiza el estado del tenant a `active`.
10. El sistema crea/actualiza el registro de `subscription` en la base de datos con: `stripe_subscription_id`, `plan`, `status = active`, `current_period_start`, `current_period_end`.
11. El sistema genera el evento `trial_converted` con `tenant_id`, `plan`, `converted_at`, `mrr`.
12. El sistema actualiza la tabla `mrr_history` con el nuevo MRR del tenant.
13. El sistema envía email de bienvenida a suscripción o email de confirmación de upgrade.
14. El sistema redirige al usuario de vuelta al dashboard con mensaje de éxito.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Pago exitoso pero webhook llega duplicado | El sistema verifica si ya se hizo la conversión; si el tenant ya está `active`, no hacer nada (idempotencia). |
| Pago exitoso pero tenant ya está `active` | Ignorar conversión; la suscripción se actualiza como renovación normal. |
| Pago fallido (tarjeta rechazada) | Mantener en trial; Stripe maneja reintentos (RF-042); mostrar error al usuario con mensaje de Stripe. |
| Trial ya expiró y se convierte | El `trial_end_date` queda registrado como `converted_at`; no hay prorated de dias no usados. |
| Pago exitoso pero el plan seleccionado cambió de precio | Guardar el precio del plan al momento del pago (snapshot); no recalcular con precios nuevos. |
| Suscripción de prueba (trial con cardinality en Stripe) | Stripe ya maneja trial en la suscripción; el webhook de `trial_end` confirma el momento de conversión real. |

---

## Criterios de Aceptación

1. Un pago exitoso en Stripe para un tenant en `trial` convierte automáticamente el tenant a `active`.
2. La conversión ocurre en menos de 5 segundos después de recibir el webhook `invoice.payment_succeeded`.
3. El registro de `subscription` contiene el `stripe_subscription_id` vinculado al tenant.
4. El evento `trial_converted` se genera con `tenant_id`, `plan`, `converted_at`, `mrr`.
5. El MRR del tenant se actualiza en la tabla `mrr_history` el mismo día de la conversión.
6. El email de confirmación de suscripción se envía dentro de los 60 segundos posteriores a la conversión.
7. Un tenant en `active` no puede volver a `trial` aunque su suscripción se cancele (churn).
8. Si el pago es exitoso pero el tenant no está en trial (suscripción renovada), el evento no se marca como `trial_converted`.

## Métricas de Éxito

- Tiempo de conversión post-pago < 5 segundos
- Tasa de webhooks duplicados manejados correctamente (idempotencia) > 99.9%
- MRR registrado coincide con el precio del plan en menos de 1 hora después de la conversión
- Count de `trial_converted` events = count de tenants con `status = active` y `converted_at` no null

---

## Notas

- La conversión de trial a paid es el momento más crítico en el funnel SaaS. El email de confirmación debe ser cálido y orientador, no solo transaccional.
- Si el tenant seleccionó un plan annual, el MRR se calcula como `precio_anual / 12` para consistencia en métricas mensuales.
- El Stripe Subscription incluye el trial de Stripe mismo si el plan lo tiene configurado. En ese caso, la conversión real se da cuando Stripe envía `customer.subscription.updated` con `status = active` y sin trial.
- Para planes Custom (Enterprise), el billing es manual (no Stripe checkout); la conversión se dispara desde el admin de ParkCore al confirmar el contrato firmado.
