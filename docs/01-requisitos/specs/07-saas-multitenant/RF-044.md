# RF-044 — Integración con Stripe como Motor de Suscripciones y Billing

## Metadata

- **Módulo**: SaaS Multi-Tenant — Suscripciones y Facturación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, stripe-integration, tenant-service

---

## User Story

**Como** administrador de ParkCore **quiero** usar Stripe como el motor de cobros y suscripciones **para** no tener que construir y mantener un sistema de pagos complejo y garantizar seguridad en el manejo de datos financieros.

---

## Objetivo

El sistema debe integrar Stripe como el único motor de procesamiento de pagos, suscripciones recurrentes, creación de customers, y manejo de métodos de pago. Toda la información sensible de tarjetas se almacena en Stripe (PCI compliance), nunca en la base de datos de ParkCore.

---

## Actores

- **Actor principal**: Sistema (integración programática con Stripe API)
- **Actor secundario**: Tenant admin ( Stripe Customer Portal)

---

## Precondiciones

- El tenant existe en el sistema con `tenant_id`.
- El tenant ha seleccionado un plan y está listo para ingresar información de pago.

---

## Postcondiciones

### En caso de éxito

- Se crea un Stripe Customer asociado al tenant.
- Se crea una Stripe Subscription con el plan seleccionado.
- Los pagos se procesan a través de Stripe.
- Las renovaciones se manejan automáticamente por Stripe.
- Los webhooks de Stripe notifican al sistema de todos los eventos relevantes.

---

## Flujo Funcional

### Creación del customer y suscripción

1. Cuando un tenant inicia el flujo de suscripción (RF-037), el sistema crea un Stripe Customer.
2. Los datos del customer: `email`, `name` (empresa), `metadata: { tenant_id }`.
3. Se guarda el `stripe_customer_id` en la tabla `tenants`.
4. El sistema crea una Stripe Subscription con:
   - `customer`: `stripe_customer_id`
   - `items`: `[ { price: plan_price_id } ]`
   - `billing_cycle_anchor`: fecha actual
   - `trial_end`: null (o `trial_end_date` si es durante trial)
5. Se guarda el `stripe_subscription_id` en la tabla `subscriptions`.

### Checkout para nuevos tenants

1. El sistema crea un Stripe Checkout Session: `mode: 'subscription'`, `customer`: `stripe_customer_id`, `line_items`: plan seleccionado.
2. El tenant es redirigido a la página de checkout de Stripe.
3. Al confirmar el pago, Stripe redirige al tenant a la URL de éxito de ParkCore.
4. Stripe envía webhook `checkout.session.completed` al sistema.
5. El sistema procesa el webhook; el tenant pasa a estado `active`.

### Customer Portal para tenants existentes

1. El tenant accede a Settings > Suscripción > "Gestionar pago".
2. El sistema crea un Stripe Customer Portal Session para el `stripe_customer_id`.
3. El tenant es redirigido al Stripe Customer Portal.
4. Desde ahí puede: actualizar método de pago, ver invoices, cancelar suscripción.
5. Al finalizar, Stripe redirige de vuelta a ParkCore.

### Manejo de métodos de pago

1. Los métodos de pago se almacenan en Stripe como PaymentMethods.
2. El sistema nunca almacena números de tarjeta en la base de datos.
3. Para cobrar, el sistema usa `stripe.invoices.create()` y `stripe.invoices.pay()` o Stripe automaticamente cobra en la fecha de renovación.
4. Los webhooks notifican el resultado de cada cobro.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tenant_id | UUID | Identificador del tenant | Sí |
| email | string | Email del tenant (para Stripe Customer) | Sí |
| empresa_nombre | string | Nombre de la empresa | Sí |
| plan_price_id | string | ID del precio del plan en Stripe | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| stripe_customer_id | string | ID del customer en Stripe |
| stripe_subscription_id | string | ID de la suscripción en Stripe |
| stripe_payment_method_id | string | ID del método de pago por defecto |
| billing_portal_url | string | URL del Stripe Customer Portal |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Stripe no responde ( API timeout) | Retry con exponential backoff (3 intentos); si falla, mostrar error al usuario y sugerir esperar. |
| Tarjeta rechazada en checkout | Stripe muestra mensaje de error en su propia UI; el usuario puede reintentar sin volver a ParkCore. |
| El customer ya existe en Stripe al crear tenant | Verificar por `stripe_customer_id` en tabla `tenants`; si existe, reutilizarlo. |
| El precio del plan cambió en Stripe | El precio se toma del `price_id` en la tabla `plans`; si se actualiza, el sistema usa el nuevo en el siguiente ciclo. |
| El tenant intenta usar un método de pago de otro customer | Stripe lo rechaza; no se puede agregar un payment method de otro customer. |

---

## Criterios de Aceptación

1. Todo tenant con suscripción activa tiene un `stripe_customer_id` en la tabla `tenants`.
2. La creación de customer y suscripción en Stripe ocurre sin errores en > 99.5% de los casos.
3. Los pagos se procesan sin que los datos de tarjeta pasen por los servidores de ParkCore.
4. Las facturas generadas por Stripe son las mismas que se muestran en el invoice interno (mapeo 1:1).
5. El Stripe Customer Portal permite al tenant gestionar su suscripción sin asistencia.
6. Si Stripe no responde, el sistema hace retry con backoff y no deja al usuario en estado inconsistente.
7. El `stripe_subscription_id` se sincroniza correctamente con la tabla `subscriptions`.

**RFs relacionados**:

- RF-042 — Reintentos de pago fallido ( Stripe los maneja)

- `subscriptions` — columnas `stripe_subscription_id`, `stripe_price_id`

---

## Endpoints API Involucrados

- `POST /api/v1/subscriptions/create-checkout` — Crear Stripe Checkout Session
- `POST /api/v1/subscriptions/portal` — Crear Stripe Customer Portal Session
- `POST /api/v1/webhooks/stripe` — Endpoint único para todos los webhooks de Stripe

---

## Métricas de Éxito

- Stripe API uptime > 99.9%
- Tiempo de respuesta de Stripe API < 500ms (p95)
- Cero datos de tarjeta almacenados en DB de ParkCore (PCI DSS compliance)
- Revenue reconocido correctamente: suma de `invoice.payment_succeeded` = MRR reportado

---

## Notas

- La integración con Stripe debe seguir las mejores prácticas de Stripe: usar Stripe Checkout para nuevos customers, Customer Portal para gestión, y webhooks para todos los eventos asíncronos.
- El endpoint de webhooks debe verificar la firma de Stripe (`stripe-signature` header) antes de procesar cualquier evento.
- Los `price_id` de cada plan se crean manualmente en el dashboard de Stripe y se almacenan en la tabla `plans` como `stripe_price_id`.
- Para planes Custom (Enterprise), el flujo de checkout no aplica; se crea la suscripción manualmente desde el admin de ParkCore y se define el precio como `custom`.
