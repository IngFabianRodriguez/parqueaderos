# RF-025 — Reportes Consolidados Multi-Sede para Operadores con Múltiples Ubicaciones

## Metadata

- **Módulo**: BI y Reportes
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: report-service, multi-tenant-service, billing-service

---

## User Story

**Como** operador con múltiples sedes **quiero** consultar reportes consolidados que agreguen datos de todas mis ubicaciones en una sola vista **para** comparar rendimiento entre sedes, identificar oportunidades de optimización y tener visibilidad global del negocio.

---

## Objetivo

El sistema debe permitir a operadores que gestionan múltiples parqueaderos (multi-sede) consultar reportes consolidados que muestren KPIs agregados y desglosados por sede, incluyendo ingresos totales, ocupación consolidada, tiempo promedio de estadía por ubicación, y comparativas de rendimiento con métricas de desviación y ranking.

---

## Actores

- **Actor principal**: Superadmin o admin_multisede (usuario con permisos en ≥ 2 sedes)
- **Actores secundarios**: Analista de negocio, sistema de BI (consumo programático)

---

## Precondiciones

- El usuario tiene permisos de lectura en al menos 2 sedes.
- Las sedes del usuario están activas y no suspendidas.
- Existen datos en al menos una de las sedes para el período consultado.

---

## Postcondiciones

### En caso de éxito

- El reporte consolidado se genera mostrando datos de todas las sedes autorizadas.
- El sistema incluye rankings y métricas de desviación.
- El usuario puede hacer drill-down a una sede específica (expandiendo el reporte individual).

### En caso de falla

- El sistema retorna error con mensaje indicando cuál sede tuvo problemas.
- El reporte parcial (sedes que no fallaron) se muestra si al menos una succeeded.
- Se loguea el error con idempotency key para retry.

---

## Flujo Funcional

### Flujo Principal: Generación de Reporte Consolidado

1. El usuario accede al módulo "Reportes Consolidados" desde el panel multi-sede.
2. El sistema detecta las sedes autorizadas para el usuario y las muestra como selector múltiple.
3. El usuario selecciona una o más sedes (o "Todas") y define rango de fechas y tipo de reporte.
4. El usuario hace clic en "Generar Reporte Consolidado".
5. El sistema verifica que el usuario tenga permisos en todas las sedes seleccionadas.
6. El sistema lanza queries en paralelo a cada sede (o replica a base de datos central si existe).
7. El sistema agrega los resultados y calcula métricas consolidadas:
   - Ingreso total = Σ ingresos por sede
   - Ocupación ponderada = Σ (ocupación_sede × espacios_sede) / Σ espacios
   - Tiempo promedio ponderado = Σ (tiempo_promedio_sede × tickets_sede) / Σ tickets
8. El sistema calcula ranking de sedes por cada KPI (rank 1 = mejor).
9. El sistema calcula desviación de cada sede vs. promedio del grupo.
10. El sistema renderiza tabla consolidada con drill-down colapsable por sede.
11. El sistema renderiza gráficos comparativos (barras agrupadas, heatmap por día).

### Flujo Alternativo: Drill-down a Sede Individual

1. Desde el reporte consolidado, el usuario hace clic en el nombre de una sede.
2. El sistema expande la fila mostrando el reporte completo de esa sede (como RF-023 individual).
3. El usuario puede colapsar nuevamente para volver a la vista consolidada.

### Fluzo Alternativo: Comparativa de Períodos

1. El usuario activa el toggle "Comparar con período anterior".
2. El sistema solicita un segundo rango de fechas (anterior) o lo infiere automáticamente (mes anterior, semana anterior).
3. El sistema genera columnas adicionales: valor período actual, valor período anterior, variación %.

### Flujo Alternativo: Exportación Multi-Sede

1. El usuario hace clic en "Exportar Consolidado".
2. El sistema genera archivo con una hoja por sede + una hoja de resumen consolidado (ver RF-024).

### Gráficos Generados

- **Barras agrupadas**: Ingresos por sede por día/semana/mes
- **Heatmap**: Ocupación horaria promediada por día de la semana por sede
- **Líneas**: Tendencia consolidada (ingresos totales en el tiempo)
- **Boxplot**: Distribución de tiempo promedio por sede

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Usuario con acceso a solo 1 sede | Sistema muestra error: "Requiere acceso a mínimo 2 sedes para reporte consolidado" |
| Una sede no tiene datos en el período | Fila muestra valores en cero, columna "sin_datos: true" y se excluye del ranking |
| Una sede tiene error de conexión | Se muestra advertencia en la sede con error; las demás se muestran completas; se ofrece retry parcial |
| Todas las sedes seleccionadas son inactivas | Sistema retorna 400: "Todas las sedes seleccionadas están inactivas" |
| Ranking con empates | Empates comparten posición; el siguiente rank se salta (ej: 1,1,3) |
| Más de 20 sedes seleccionadas | Sistema limita a 20 y muestra advertencia: "Se incluyen las primeras 20 sedes por límite de visualización" |
| Comparativa con período sin datos en alguna sede | Columnas de variación muestran "N/A" para esa sede |
| sesiicróncrono: timeout en una sede | Con timeout global de 30s; si una sede no responde, se toma el último resultado cacheado o se marca como error |

## Criterios de Aceptación

1. **CA-025-01**: Un usuario con acceso a 5 sedes puede generar un reporte consolidado que muestra ingresos, ocupación y tiempo promedio para las 5 sedes en una sola consulta con datos consistentes.
2. **CA-025-02**: El reporte consolidado calcula correctamente el ingreso total como la suma de los ingresos de cada sede individual.
3. **CA-025-03**: El ranking de sedes se muestra correctamente ordenando por el KPI seleccionado, sin duplicados de posición excepto en empates.
4. **CA-025-04**: Las métricas de desviación muestran el porcentaje de variación de cada sede respecto al promedio del grupo, con precisión de 2 decimales.
5. **CA-025-05**: El drill-down desde una sede en el reporte consolidado abre el reporte individual (RF-023) sin pérdida de contexto (mismos filtros de fecha).
6. **CA-025-06**: La exportacion a XLSX genera una hoja de resumen consolidado y una hoja por cada sede con sus datos.
7. **CA-025-07**: El sistema valida que el usuario tenga permisos en todas las sedes seleccionadas antes de ejecutar el reporte.
8. **CA-025-08**: Si una sede tiene meta de ingresos configurada, el reporte muestra % de cumplimiento de meta.
9. **CA-025-09**: Con 20 sedes y 90 días de datos, el reporte consolidado se genera en ≤ 15 segundos (P95).

## Métricas de Éxito

- Tiempo de generación de reporte consolidado ≤ 15s para 20 sedes (P95)
- Tasa de error en reportes consolidados ≤ 2%
- Porcentaje de operadores multi-sede que usan reportes consolidados ≥ 70%
- Satisfacción del operador multi-sede ≥ 4.5/5
- Cantidad de drill-downs por sesión promedio ≥ 3

---

## Notas

- Se recomienda implementar una tabla agregada/ materialized view `daily_sites_aggregation` que se actualice cada noche con métricas por sede por día, para acelerar reportes consolidados de rangos grandes.
- El ranking se calcula en memoria o con ventana SQL; no requiere query adicional si se estructura bien.
- La desviación estándar y otras métricas estadísticas se calculan solo si el número de sedes ≥ 3; si no, se muestra N/A.
- Para el heatmap de ocupación horaria, se promedia la ocupación de cada hora en el rango seleccionado y se muestra por día de la semana.
- Se debe validar que el operador no intente consolidar sedes de diferentes inquilinos (tenant_id diferente) — esto es un requisito de seguridad, no solo de funcionalidad.
- El campo `meta_ingresos` por sede se configura en RF-091 (Configuración de Metas) y se lee de `sites_targets`.
