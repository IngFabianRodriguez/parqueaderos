# RF-023 — Generación de Reportes de Ingresos, Ocupación y Tiempo Promedio de Estadía

## Metadata

- **Módulo**: BI y Reportes
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: report-service, billing-service, occupancy-service

---

## User Story

**Como** operador de parqueo **quiero** consultar reportes de ingresos, ocupación y tiempo promedio de estadía **para** tomar decisiones informadas sobre precios, capacidad y operación diaria.

---

## Objetivo

El sistema debe permitir generar reportes operativos en formato tabular y gráfico que muestren: (1) ingresos por período, (2) tasa de ocupación por sede y por hora, y (3) tiempo promedio de estadía por tipo de vehículo. Los reportes deben ser consultables en línea y exportables, con datos actualizados en tiempo real o con delta máximo de 15 minutos.

---

## Actores

- **Actor principal**: Operador de parqueo (admin_sede, operador)
- **Actores secundarios**: Superadmin (consulta cross-sede), cliente (solo consulta de receipts vía API pública)

---

## Precondiciones

- El usuario autenticado tiene rol admin_sede, operador o superadmin.
- Existen registros de transacciones de ingreso y egreso en el sistema.
- La sede tiene espacios definidos y categorías configuradas.
- Si es cross-sede: el usuario tiene permisos de lectura en las sedes solicitadas.

---

## Postcondiciones

### En caso de éxito

- El reporte se genera y muestra en pantalla con datos coherentes.
- El usuario puede exportar el reporte en los formatos disponibles.
- El sistema registra la generación del reporte en logs de auditoría.

### En caso de falla

- El sistema muestra un mensaje de error claro indicando la causa.
- No se genera reporte parcial; se muestra estado consistente.
- El error se loguea con id de correlación para trazabilidad.

---

## Flujo Funcional

### Flujo Principal: Generación de Reporte de Ingresos

1. El operador accede al módulo "Reportes" desde el panel de administración.
2. El sistema presenta formulario con filtros: sede, tipo de reporte (ingresos/ocupación/tiempo), rango de fechas, granularidad (día/hora/minuto).
3. El operador selecciona los filtros y hace clic en "Generar Reporte".
4. El sistema valida que el rango de fechas no supere 90 días corridos.
5. El sistema consulta agregaciones en base de datos según filtros.
6. El sistema renderiza tabla de resultados con totales y subtotales por dimensión.
7. El sistema renderiza gráfico de líneas/barras según tipo de reporte.
8. El operador puede exportar el reporte (ver RF-024).
9. El sistema registra la generación en auditoría.

### Flujo Alternativo: Reporte de Ocupación en Tiempo Real

1. El operador selecciona tipo de reporte "Ocupación" y marca checkbox "Tiempo Real".
2. El sistema muestra ocupación actual con刷新 automático cada 60 segundos.
3. Si el operador desmarca "Tiempo Real", se muestra ocupación del último día cerrado.

### Flujo Alternativo: Tiempo Promedio de Estadía

1. El operador selecciona tipo de reporte "Tiempo Promedio de Estadía".
2. El sistema presenta filtro adicional: tipo de vehículo (moto/carro/camioneta/otro).
3. El sistema calcula duración entre ingreso y egreso para tickets cerrados.
4. Se muestra distribución en histograma y estadísticas (media, mediana, percentil 90).

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Rango de fechas > 90 días | Sistema rechaza con mensaje: "El rango máximo es 90 días" |
| Sede sin transacciones en el período | Reporte muestra todos los campos en cero con leyenda "Sin datos" |
| Usuario sin permisos en la sede | Sistema retorna 403 Forbidden con mensaje descriptivo |
| Base de datos no disponible | Sistema retorna 503 con retry-after de 30s y registra alerta |
| Ticket sin egreso (abierto) | Se excluye del cálculo de tiempo promedio; se incluye en ocupación |
| Moneda no configurada | Usa moneda por defecto de la sede o 'USD' como fallback |
| Granularidad incompatible con rango | Sistema ajusta automáticamente al mínimo válido |
| concurrencia masiva (>100 solicitudes/min) | Cola de procesamiento con backlog máximo de 60s; 429 si se excede |

---

## Criterios de Aceptación

1. **CA-023-01**: Un operador puede generar un reporte de ingresos para una sede específica en un rango de hasta 90 días y el reporte se despliega en menos de 5 segundos para rangos ≤ 30 días.
2. **CA-023-02**: El reporte de ingresos muestra correctamente la suma de montos de transacciones completadas, diferenciando por estado (pagado, pendiente, reembolsado).
3. **CA-023-03**: El reporte de ocupación muestra datos actualizados con latencia máxima de 15 minutos respecto al estado real de espacios.
4. **CA-023-04**: El reporte de tiempo promedio de estadía se calcula solo con tickets cerrados (con egreso registrado) y excluye outliers > 24 horas.
5. **CA-023-05**: Todos los reportes incluyen breakdown por tipo de vehículo cuando el filtro está activo.
6. **CA-023-06**: El sistema registra en log de auditoría cada generación de reporte con usuario, timestamp, filtros aplicados y tamaño del resultado.
7. **CA-023-07**: Un usuario sin rol adecuado recibe 403 al intentar acceder al módulo de reportes.

## Endpoints API Involucrados

- `GET /api/v1/reports/ingresos?sede_id={uuid}&fecha_inicio={date}&fecha_fin={date}` — Genera reporte de ingresos
- `GET /api/v1/reports/ocupacion?sede_id={uuid}&fecha={date}&realtime={bool}` — Genera reporte de ocupación
- `GET /api/v1/reports/tiempo-promedio?sede_id={uuid}&fecha_inicio={date}&fecha_fin={date}&tipo_vehiculo={string}` — Genera reporte de tiempo promedio
- `GET /api/v1/reports/metadata/sedes/{sede_id}/campos` — Describe los campos disponibles para filtros (para UI)
- `POST /api/v1/reports/async` — Cola generación asíncrona de reportes grandes (> 30 días)

---

## Métricas de Éxito

- Tiempo de generación de reportes ≤ 5s para rangos ≤ 30 días (P95)
- Tasa de error en generación de reportes ≤ 1%
- Cobertura de logs de auditoría: 100% de las generaciones registradas
- Satisfacción del operador con módulo de reportes ≥ 4/5 (survey quarterly)
- Porcentaje de reportes exportados (vs. solo visualizados) ≥ 30%

---

## Notas

- Se asume que la zona horaria de la sede es la misma que la del servidor para evitar desfases en granularidad horaria. Si no, se implementa offset configurable por sede.
- Los outliers de tiempo de estadía (> 24h) se excluyen del cálculo de promedio para evitar distorsión por vehículos estacionados indebidamente.
- Para ocupación en tiempo real se usa cache en memoria (Redis) con TTL de 60s para evitar sobrecarga en la base de datos.
- Los gráficos se generan client-side con librería liviana (Chart.js) para no depender de servicios externos.
