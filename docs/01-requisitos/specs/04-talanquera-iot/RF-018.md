# RF-018 — Apertura de talanquera de salida con pago confirmado

## Metadata

- **Módulo**: Talanquera IoT
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: iot-gateway, parking-service, billing-service

---

## User Story

**Como** Sistema de Control de Talanquera IoT **quiero** abrir la talanquera de salida únicamente cuando el pago del vehículo esté confirmado **para** garantizar que no salga ningún vehículo sin pagar y evitar pérdidas económicas para el operador del parqueadero.

---

## Objetivo

El sistema debe enviar un comando de apertura a la talanquera de salida únicamente cuando la transacción de pago asociada al registro de salida tenga estado "confirmado" (pagado, exonerado o cubierto por prepago). Cualquier estado diferente (pendiente, fallido, en disputa) debe mantener la talanquera cerrada. La talanquera debe permanecer bloqueada por defecto y solo abrirse mediante comando explícito tras confirmación de pago.

---

## Actores

- Sistema IoT Gateway (actuador de talanquera)
- Parking-service (orquestador del flujo de salida)
- Billing-service (confirmación de estado de pago)
- Operador (puede intervenir en casos excepcionales)

---

## Precondiciones

- Existe un registro de salida activo para el vehículo en la sede
- El sistema conoce el estado de pago de la transacción asociada
- La talanquera está en estado "cerrada" o "bloqueada" antes del comando
- El dispositivo IoT de la talanquera está operativo y conectado

---

## Postcondiciones

### En caso de éxito

- La talanquera se abre y permite el paso del vehículo
- Se registra timestamp de apertura y usuario/sistema que originó el comando
- El vehículo sale del parqueadero y el espacio se libera
- Se emite evento de salida por websocket para actualizar paneles y contadores

### En caso de falla

- La talanquera permanece cerrada si el pago no está confirmado
- Se genera log de intento de apertura rechazado con motivo
- Se notifica al conductor (display o operador) el motivo del rechazo
- Si la talanquera no responde al comando (fallo de hardware), se genera alerta (RF-022)

---

## Flujo Funcional

1. Sistema ANPR de salida reconoce la placa del vehículo (o ingreso manual por operador)
2. Parking-service busca el registro de entrada activo y genera cálculo de tarifa (RF-006)
3. Si hay saldo prepago suficiente, billing-service confirma pago directamente (wallet)
4. Si no hay wallet o saldo insuficiente, se presenta opción de pago (efectivo/PSE/tarjeta/Nequi/Daviplata)
5. Usuario realiza el pago a través del medio elegido
6. Payment-gateway confirma la transacción y actualiza estado a "confirmado"
7. Parking-service recibe evento/respuesta de pago confirmado
8. Parking-service valida que el estado de la transacción sea "pagado", "exonerado" o "prepago_aprobado"
9. Parking-service envía comando `OPEN_GATE` a IoT-gateway con entrada_id, salida_id, y motivo "pago_confirmado"
10. IoT-gateway valida que el comando proviene de parking-service autenticado
11. IoT-gateway envía señal de apertura al dispositivo de la talanquera (relé/RS485/HTTP)
12. Dispositivo ejecuta apertura física (brazo de talanquera se levanta)
13. Sensor de flujo detecta paso del vehículo
14. Sensor de estado confirma que la talanquera está abierta
15. Transcurrido el tiempo de seguridad (configurable, default 5s), IoT-gateway envía comando `CLOSE_GATE`
16. Talanquera vuelve al estado cerrado
17. Parking-service actualiza el registro de salida con timestamp de salida efectiva

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Pago en estado "pendiente" (autorización pendiente del gateway) | Se rechaza apertura, se muestra mensaje "Espere confirmación de pago", se reconsulta estado cada 5s máximo 3 intentos |
| Pago en estado "fallido" | Se rechaza apertura, se muestra mensaje "Pago fallido — acercarse al punto de pago", operador es alertado |
| Pago en estado "reversado" | Se rechaza apertura, se muestra mensaje "Pago revertido — comuníquese con atención", se bloquea salida |
| Placa no tiene registro de entrada (intento de fraude) | Se rechaza apertura, se genera alerta de seguridad, operador interviene |
| Saldo prepago exacto al valor exacto (sin decimales) | Se marca como pagado y se abre talanquera |
| Saldo prepago mayor al valor (sobresaldo) | Se descuenta exacto, se abre talanquera, excedente permanece en wallet |
| Cliente con exoneración válida (empleado, visitante VIP) | Se confirma exonerado y se abre talanquera sin cobro |
| Pago confirmado pero dispositivo IoT no responde | Se genera alerta de falla (RF-022), operador puede realizar apertura manual (RF-019) |
| Múltiples vehículos en cola (sensor detecta más de un vehículo) | Se abre talanquera solo para el vehículo que acaba de pagar; siguiente debe pagar para pasar |
| Intento de apertura sin solicitud de salida registrada | Se rechaza comando, se loguea como intento no autorizado |

---

## Criterios de Aceptación

1. La talanquera de salida NO se abre si el estado de pago es diferente a "confirmado" (pagado/exonerado/prepago_aprobado)
2. Tiempo desde confirmación de pago hasta apertura de talanquera: < 1.5 segundos
3. El comando de apertura incluye firma/metadata que permite auditar quién autorizó la apertura
4. Si la talanquera no responde al comando en 3 segundos, se genera alerta de falla (RF-022)
5. La talanquera se cierra automáticamente después del paso del vehículo (timer configurable)
6. Si el vehículo no pasa en 30 segundos (sensor de flujo no detecta paso), se cierra la talanquera y se genera alerta
7. El log de auditoría registra cada comando de apertura con timestamp, usuario, motivo y resultado
8. El sistema rechaza comandos de apertura duplicados mientras la talanquera está en proceso de abrirse

## Endpoints API Involucrados

- `POST /api/v1/salidas` — Registra salida del vehículo y genera cobro (RF-006)
- `POST /api/v1/pagos/confirmar` — Gateway de pago confirma transacción
- `POST /api/v1/gate/open` — IoT-gateway recibe comando de apertura
- `GET /api/v1/gate/status/{gate_id}` — Consulta estado actual de talanquera (RF-020)
- `POST /api/v1/gate/manual-open` — Apertura manual por operador (RF-019)

---

## Métricas de Éxito

- Tasa de apertura exitosa con pago confirmado: 100% (siempre que pago confirmado)
- Tiempo promedio apertura: < 1.5s desde confirmación de pago
- Tasa de rechazo indebido (abre sin pago): 0%
- Tiempo de cierre automático tras paso: configurable (default 5s)
- Trazabilidad: 100% de las aperturas tienen log de auditoría

---

## Notas

- El flujo asume que la talanquera está normalmente cerrada (fail-secure). Solo se abre con comando explícito.
- El sistema NO debe abrir la talanquera por simple presencia de vehículo — requiere flujo completo de pago.
- En caso de fallo del sistema IoT, el operador tiene la capacidad de apertura manual (RF-019) pero cada apertura manual debe quedar registrada.
- La comunicación entre parking-service e IoT-gateway debe ser sobre canal autenticado (JWT o mTLS).
- Se recomienda implementar circuit-breaker en el IoT-gateway para no enviar comandos a dispositivos que no responden.
