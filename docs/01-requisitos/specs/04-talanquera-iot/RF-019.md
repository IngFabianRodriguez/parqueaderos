# RF-019 — Apertura manual de talanquera por operador en emergencia

## Metadata

- **Módulo**: Talanquera IoT
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: iot-gateway, parking-service, audit-log

---

## User Story

**Como** Operador de parqueadero **quiero** poder abrir manualmente la talanquera en situaciones de emergencia (falla del sistema, vehículo con problemas técnicos, falla de pago, etc.) **para** resolver situaciones excepcionales rápidamente sin bloquear la salida de vehículos y causar congestión.

---

## Objetivo

El sistema debe permitir que un operador con rol apropiado (admin_sede, operador) ejecute una apertura manual de la talanquera en cualquier momento, independientemente del estado de pago. Esta apertura debe estar protegida por autenticación, debe registrar el motivo obligatorio proporcionado por el operador, y debe generar trazabilidad completa en el log de auditoría. La apertura manual es un mecanismo de fallback que no debería usarse en operación normal.

---

## Actores

- Operador (nivel básico — puede abrir con motivo y supervisión)
- Admin de sede (puede abrir sin límite)
- Superadmin (puede abrir en cualquier sede)
- Sistema IoT Gateway (ejecuta el comando físico)

---

## Precondiciones

- El operador tiene sesión activa en el sistema con rol permitido (operador, admin_sede, superadmin)
- La talanquera está operativa (no está en estado de falla que impida físicamente abrirse)
- El dispositivo IoT de la talanquera está conectado y responde a comandos
- El operador conoce o proporciona un motivo para la apertura manual

---

## Postcondiciones

### En caso de éxito

- La talanquera se abre inmediatamente tras la autorización
- Se registra en log de auditoría: operador, timestamp, sede, talanquera, motivo, resultado
- El evento queda disponible para revisión por parte del admin
- Si la apertura es por falla de pago, el sistema permite luego conciliar la transacción

### En caso de falla

- Si el dispositivo no responde, se muestra error al operador y se sugiere contacto con soporte
- Si el operador no tiene permisos, se muestra mensaje de "Acceso denegado"
- Si la talanquera está bloqueada por protocolo de seguridad (ej: falla detectada), se muestra mensaje de bloqueo y se sugiere esperar

---

## Flujo Funcional

1. Operador identifica situación que requiere apertura manual (vehículo varado, falla de sistema, pago no procesado, etc.)
2. Operador ingresa al panel de control de talanqueras en la interfaz del sistema
3. Sistema solicita autenticación del operador (si no hay sesión activa)
4. Operador identifica la talanquera a abrir (entrada o salida, por sede)
5. Operador selecciona opción "Apertura Manual de Emergencia"
6. Sistema presenta formulario con campos obligatorios:
   - Motivo de apertura (lista predefinida + campo de descripción adicional)
   - Placa del vehículo (si aplica y se conoce)
   - Confirmación de que comprende que esta acción queda registrada
7. Operador llena el formulario y confirma la acción
8. Sistema valida que el operador tiene rol con permisos de apertura manual
9. Sistema registra la solicitud en log de auditoría antes de ejecutar
10. Sistema envía comando `MANUAL_OPEN` a IoT-gateway con:
    - gate_id
    - operador_id
    - timestamp_solicitud
    - motivo (código + descripción)
11. IoT-gateway recibe comando y lo ejecuta en el dispositivo
12. Dispositivo de talanquera ejecuta apertura física
13. IoT-gateway reporta resultado (éxito/fracaso) a parking-service
14. Sistema actualiza log de auditoría con resultado y timestamp de ejecución
15. Si apertura fue exitosa, se registra timestamp de apertura y se inicia timer de cierre automático
16. Si apertura falló (dispositivo no responde), se genera alerta de falla (RF-022)

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| gate_id | UUID | Identificador de la talanquera a abrir | Sí |
| operador_id | UUID | ID del operador que ejecuta la acción | Sí (autenticado) |
| sede_id | UUID | Identificador de la sede | Sí |
| motivo_codigo | VARCHAR(50) | Código predefinido del motivo | Sí |
| motivo_descripcion | TEXT | Descripción adicional del motivo | Sí |
| placa_vehiculo | VARCHAR(20) | Placa del vehículo implicado (si aplica) | No |
| confirmacion_operador | BOOLEAN | Operador confirma que la acción queda registrada | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| command_id | UUID | Identificador del comando ejecutado |
| gate_status | VARCHAR | Estado resultante: opened, error, timeout |
| timestamp_apertura | TIMESTAMP | Momento de apertura efectiva |
| timestamp_cierre_programado | TIMESTAMP | Momento en que se cerrará automáticamente |
| auditoria_id | UUID | Referencia al registro en log de auditoría |
| mensaje_error | VARCHAR | Descripción de error si falla |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Operador sin rol adecuado intenta apertura manual | Se rechaza acción, mensaje "No tiene permisos para apertura manual" |
| Dispositivo IoT no responde al comando | Se muestra error "Talanquera no responde", se sugiere contactar soporte, se genera alerta RF-022 |
| Operador cierra el formulario sin confirmar | No se ejecuta acción, no se genera log |
| Se intenta abrir una talanquera que ya está abierta | Se muestra mensaje "La talanquera ya está abierta" y no se reenvía comando |
| Múltiples operadores intentan abrir la misma talanquera simultáneamente | Se procesa la primera solicitud válida, las demás reciben mensaje de "Operación en curso" |
| Apertura manual durante falla de red | IoT-gateway opera en modo offline (RF-008) y registra la acción para sincronización posterior |
| Operador selecciona motivo "Falla de pago" pero no hay registro de salida | Sistema permite la apertura pero genera alerta de "Salida sin registro previo" para revisión |
| Placa no existe en el sistema y operador la proporciona | Se crea registro de salida improvisado con placa proporcionada para trazabilidad |

---

## Criterios de Aceptación

1. Solo operadores con rol admin_sede, operador o superior pueden ejecutar apertura manual
2. El campo "motivo" es obligatorio y no puede ser vacío
3. Toda apertura manual queda registrada en log de auditoría con operador, timestamp, motivo y resultado
4. La apertura manual se completa en < 2 segundos si el dispositivo responde
5. El sistema genera alerta cuando más de 3 aperturas manuales ocurren en la misma sede en 1 hora
6. Si el dispositivo no responde, el operador recibe mensaje de error claro
7. Los logs de apertura manual son consultables por el admin para auditoría
8. La apertura manual no afecta la integridad de los registros de entrada/salida — se genera registro complementario

## Métricas de Éxito

- Tiempo de respuesta de apertura manual: < 2s
- Tasa de ejecución exitosa: ≥ 99% (considerando dispositivos operativos)
- 100% de las aperturas manuales tienen registro trazable
- Alertas generadas cuando frecuencia > 3 por hora por sede
- Cantidad de aperturas manuales por operador/mes consultable

---

## Notas

- La apertura manual es un mecanismo de contingencia y no debe ser la vía normal de salida de vehículos.
- Se recomienda limitar la cantidad de aperturas manuales por operador para prevenir abusos.
- El campo "motivo" utiliza códigos predefinidos para facilitar reportes y análisis:
  - `FALLA_SISTEMA` — Falla del sistema de pago o reconocimiento
  - `FALLA_PAGO` — Pago no procesado pero vehículo necesita salir
  - `VEHICULO_VARADO` — Vehículo con problema técnico
  - `EMERGENCIA_MEDICA` — Emergencia médica del conductor
  - `OPERADOR_ERROR` — Corrección de error del operador en registro previo
  - `OTRO` — Otro motivo (requiere descripción)
- El sistema puede configurarse para禁止 aperturas manuales durante ventanas de mantenimiento (parámetro configurable por sede).
- La apertura manual debe poder ejecutarse incluso si el billing-service está caído (fallback).
