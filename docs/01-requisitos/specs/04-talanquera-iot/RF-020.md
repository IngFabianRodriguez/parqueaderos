# RF-020 — Monitoreo de estado de talanquera en tiempo real

## Metadata

- **Módulo**: Talanquera IoT
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: iot-gateway, parking-service, websocket-service

---

## User Story

**Como** Administrador de sede, Operador de control **quiero** conocer el estado actual de cada talanquera (abierta, cerrada, en movimiento, error, offline) en tiempo real **para** tomar decisiones operativas inmediatas, detectar fallas y mantener la operación fluida del parqueadero.

---

## Objetivo

El sistema debe monitorear y reportar el estado de cada talanquera en tiempo real, con actualización menor a 5 segundos. El estado debe incluir: posición física del brazo (arriba/abajo), estado de comunicación con el dispositivo, nivel de batería si aplica, última actividad registrada, y alertas activas. El operador debe visualizar esta información en un dashboard y recibir notificaciones ante cambios de estado relevantes.

---

## Actores

- Sistema IoT Gateway (reporta estado de dispositivos)
- Parking-service (consume y procesa estados)
- Operador de control (observa dashboard)
- Administrador de sede (recibe alertas)

---

## Precondiciones

- Los dispositivos IoT de las talanqueras están instalados y conectados
- IoT-gateway tiene comunicación activa con los dispositivos
- El servicio de websocket está operativo para transmitir actualizaciones en tiempo real

---

## Postcondiciones

### En caso de éxito

- El dashboard muestra estado actualizado de todas las talanqueras de la sede
- Cambios de estado se transmiten por websocket en < 5 segundos
- Alertas se generan y notifican cuando el estado es "error" o "offline"

### En caso de falla

- Si un dispositivo no reporta estado durante 30 segundos, se marca como "offline"
- Si el servicio de websocket cae, el sistema muestra estado "desconectado" y reintenta conexión

---

## Flujo Funcional

1. Dispositivo IoT de la talanquera envía heartbeat cada 10 segundos a IoT-gateway
2. Heartbeat incluye: estado del relé, posición del brazo (sensor binario), voltaje batería, temperatura, timestamp
3. IoT-gateway procesa heartbeat y actualiza estado en memoria local
4. IoT-gateway envía actualización de estado a parking-service via MQTT o HTTP POST
5. Parking-service actualiza base de datos con estado actual de la talanquera
6. Parking-service emite evento por websocket a todos los clientes conectados (dashboard)
7. Dashboard recibe evento y actualiza interfaz visual para el operador
8. Si el estado recibido indica "error" o "offline", se genera alerta inmediata
9. Alerta se envía por push/email/SMS al administrador de sede según configuración
10. Si no se recibe heartbeat de un dispositivo durante 30 segundos, parking-service marca dispositivo como "offline"
11. Dispositivo "offline" genera alerta de falla de comunicación

## Estados de Talanquera

| Estado | Descripción |
|--------|-------------|
| CLOSED | Brazo abajo, talanquera en posición de cierre, lista para funcionar |
| OPEN | Brazo arriba, vehículo puede pasar |
| OPENING | Brazo en movimiento hacia arriba |
| CLOSING | Brazo en movimiento hacia abajo |
| ERROR | Dispositivo reporta error interno o falla de hardware |
| OFFLINE | Dispositivo no responde a heartbeats (>30s sin comunicación) |
| BLOCKED | Talanquera bloqueada por protocolo de seguridad (ej: falla detectada, no permitir apertura) |

---

## Datos de Entrada (Heartbeat del dispositivo)

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| device_id | UUID | Identificador único del dispositivo IoT | Sí |
| gate_id | UUID | Identificador lógico de la talanquera | Sí |
| relay_state | BOOLEAN | Estado del relé (true = energizado/abierto) | Sí |
| arm_position | VARCHAR | Posición física: up, down, moving | Sí |
| battery_voltage | DECIMAL | Voltaje de batería (si aplica) | No |
| signal_strength | INTEGER | Intensidad de señal RSSI | No |
| temperature | DECIMAL | Temperatura del dispositivo (°C) | No |
| timestamp | TIMESTAMP | Timestamp del dispositivo | Sí |
| error_code | VARCHAR | Código de error si hay falla | No |

| Campo | Tipo | Descripción |
|-------|------|-------------|

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Dispositivo no envía heartbeat por 30s | Sistema marca como OFFLINE, genera alerta, reintenta conexión cada 10s |
| Dispositivo envía estado contradictorio (brazo arriba pero relé cerrado) | Se registra anomalía en log, se muestra advertencia en dashboard, se genera alerta de mantenimiento |
| Múltiples dispositivos de la misma sede offline simultáneamente | Se genera alerta crítica de "Falla de comunicación masiva" |
| Websocket desconectado (cliente cierra navegador) | Servidor detecta desconexión, limpia sesión, no intenta reenviar a ese cliente |
| Falla de base de datos al actualizar estado | IoT-gateway mantiene estado en caché y reintenta; no bloquea dispositivo |
| Heartbeat con timestamp muy antiguo (retraso de red) | Se procesa el heartbeat pero se detecta delay y se alerta si > 15s |
| Dispositivo reporta ERROR pero sigue respondiendo | Se muestra error en dashboard, se mantiene alerta activa hasta que operador confirme resolución |
| Nuevo dispositivo registrado (primera conexión) | Sistema crea registro en base de datos, lo marca como "commissioning", permite configuración inicial |

## Criterios de Aceptación

1. Estado de talanquera se actualiza en el dashboard en < 5 segundos desde que el dispositivo reporta cambio
2. El sistema detecta y marca como OFFLINE cualquier dispositivo sin heartbeat por > 30 segundos
3. El dashboard muestra todos los estados posibles: CLOSED, OPEN, OPENING, CLOSING, ERROR, OFFLINE, BLOCKED
4. Cada cambio de estado genera un evento en el log de auditoría
5. El operador recibe notificación push/email/SMS ante estado ERROR u OFFLINE
6. La información de batería y señal se muestra cuando el dispositivo lo soporta
7. El sistema puede manejar al menos 50 talanqueras por sede sin degradación de rendimiento
8. Los datos de estado histórico están disponibles para consulta (últimas 24h mínimo)

## Endpoints API Involucrados

- `GET /api/v1/gates/status` — Lista todos los estados de talanqueras de una sede
- `GET /api/v1/gate/status/{gate_id}` — Estado de una talanquera específica
- `GET /api/v1/gate/history/{gate_id}` — Historial de estados (últimas 24h)
- `WebSocket /ws/gates` — Canal de actualizaciones en tiempo real
- `POST /api/v1/devices/heartbeat` — Endpoint para recibir heartbeats de dispositivos IoT

## Métricas de Éxito

- Latencia de actualización de estado: < 5s desde el dispositivo hasta el dashboard
- Disponibilidad de dashboard de estados: ≥ 99.5%
- Tiempo de detección de dispositivo offline: ≤ 35s (30s timeout + 5s procesamiento)
- Precisión de estado: 100% de los estados reportados son reflejados correctamente
- Rate de heartbeats perdidos: < 0.1%

---

## Notas

- Se recomienda usar MQTT para comunicación dispositivo-gateway por eficiencia de recursos.
- Para sitios con conectividad limitada, el IoT-gateway puede funcionar en modo store-and-forward: guarda estados locally y sincroniza cuando hay conexión.
- La talanquera BLOCKED es un estado de seguridad que impide cualquier comando de apertura hasta que un admin lo limpie manualmente.
- El dashboard debe mostrar una representación visual (icono/color) por cada talanquera para facilitar monitoreo rápido.
- En el futuro se puede agregar métricas predictivas: "Este dispositivo ha tenido X errores en los últimos 30 días, considerar reemplazo".
