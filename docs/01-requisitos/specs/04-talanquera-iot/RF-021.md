# RF-021 — Registro de comando enviado a la talanquera

## Metadata

- **Módulo**: Talanquera IoT
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: iot-gateway, parking-service, audit-service

---

## User Story

**Como** Administrador de sede, Auditor de cumplimiento, Superadmin **quiero** que todo comando enviado a cualquier talanquera quede registrado de forma inmutable **para** poder auditar quién abrió cada talanquera, en qué momento, con qué motivo, y con qué resultado, facilitando la resolución de disputas y el cumplimiento de regulaciones.

---

## Objetivo

Cada vez que el sistema envía un comando a una talanquera (OPEN, CLOSE, MANUAL_OPEN, RESET), debe crear un registro en la tabla `comando_talanquera` con todos los detalles del comando: quién looriginó (usuario/sistema), desde qué dirección IP, timestamp exacto, tipo de comando, parámetros, y resultado. Estos registros son inmutables (no se pueden modificar ni eliminar) y constituyen la base de la auditoría operativa.

---

## Actores

- Sistema (orquestador — parking-service envía comandos)
- Operador (apertura manual)
- Admin de sede (override de seguridad)
- Superadmin (comandos de administración)
- Auditor (consulta logs)

---

## Precondiciones

- Existe un dispositivo de talanquera configurado en el sistema
- El usuario o sistema que envía el comando está autenticado
- IoT-gateway tiene comunicación con el dispositivo

---

## Postcondiciones

### En caso de éxito

- El comando se registra en la tabla de auditoría antes de ser enviado al dispositivo
- El registro permanece inmutable durante toda la vida del sistema
- El registro se puede consultar para generación de reportes

### En caso de falla

- Si el comando falla en el dispositivo, el registro se actualiza con el resultado de error
- El registro mantiene trazabilidad completa del intento fallido

---

## Flujo Funcional

1. Un actor (sistema u operador) solicita un comando de talanquera (apertura, cierre, reset)
2. Parking-service o el endpoint de API recibe la solicitud
3. Sistema crea registro PRELIMINAR en tabla `comando_talanquera` con:
   - comando_id (UUID generado)
   - tipo_comando (OPEN, CLOSE, MANUAL_OPEN, RESET)
   - gate_id
   - sede_id
   - originator_type (system, operator, admin)
   - originator_id (user_id o "SYSTEM")
   - originator_ip
   - timestamp_solicitud
   - motivo_id (referencia a catálogo de motivos)
   - motivo_descripcion
   - parametros_json (datos adicionales del contexto)
   - resultado (PENDING — estado inicial)
4. Una vez creado el registro, se envía el comando al IoT-gateway
5. IoT-gateway ejecuta el comando en el dispositivo físico
6. IoT-gateway reporta resultado (SUCCESS, FAILED, TIMEOUT) a parking-service
7. Parking-service actualiza el registro de comando con:
   - timestamp_resultado
   - resultado (SUCCESS/FAILED/TIMEOUT)
   - error_description (si falló)
   - device_response_time_ms
8. Si el comando fue MANUAL_OPEN, se crea registro adicional en log_auditoria
9. Si el comando fue open y la talanquera tiene sensores de paso, se puede cerrar el loop:
   - Sensor detecta paso de vehículo
   - Sensor de cierre confirma que brazo bajó
   - Se actualiza timestamp_cierre_effectivo en el registro
10. Reportes y dashboards consumen estos registros para métricas y auditoría

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Comando se envía pero dispositivo no responde (timeout 3s) | Registro se actualiza con resultado TIMEOUT, se genera alerta RF-022 |
| Comando genera error en el dispositivo | Registro se actualiza con resultado FAILED y error_description |
| Registro no se puede crear por falla de BD | Sistema no envía comando (fail-safe), retorna error al originador |
| Operador cierra navegador durante comando manual | Comando se completa si fue enviado, registro se actualiza cuando llega respuesta |
| Intento de modificar o eliminar registro de comando | Operación rechazada — logs son inmutables |
| Comando duplicado enviado (retry por timeout) | Se detecta duplicado por comando_id, se ignora segundo envío |
| Originador es "SYSTEM" (comando automático por pago) | Se registra con originator_type=system y originator_id="PARKING_SERVICE" |
| Comando MANUAL_OPEN sin motivo | Se rechaza creación del registro hasta que motivo sea proporcionado |

---

## Criterios de Aceptación

1. Todo comando enviado a cualquier talanquera genera un registro en la tabla comando_talanquera
2. El registro se crea ANTES de enviar el comando al dispositivo (fail-safe)
3. Los registros son inmutables: no se puede UPDATE ni DELETE, solo INSERT nuevos registros
4. Cada registro contiene: comando_id, tipo, gate_id, sede_id, originator, IP, timestamp, motivo, resultado
5. Registros de apertura manual incluyen referencia al operador que autorizó
6. El tiempo de registro (creación) es < 100ms para no ralentizar el comando
7. Los registros son consultables por rango de fechas, sede, gate_id, operador, tipo de comando
8. El log de auditoría diferenciará comandos automáticos (por pago) de manuales (por operador)

## Endpoints API Involucrados

- `POST /api/v1/gate/open` — Comando de apertura (genera registro)
- `POST /api/v1/gate/close` — Comando de cierre (genera registro)
- `POST /api/v1/gate/manual-open` — Apertura manual (genera registro + log_auditoria)
- `POST /api/v1/gate/reset` — Reset de talanquera en error (genera registro)
- `GET /api/v1/audit/gate-commands` — Consulta de comandos con filtros
- `GET /api/v1/gate/command/{command_id}` — Detalle de un comando específico

---

## Métricas de Éxito

- 100% de los comandos tienen registro en BD
- Tiempo de creación de registro: < 100ms
- Consultas de historial: < 500ms para rangos de hasta 1 mes
- Tasa de comandos exitosos: meta ≥ 99%
- Precisión de trazabilidad: 100% de aperturas tienen originador identificable

---

## Notas

- El patrón de registro antes de ejecutar (write-ahead log) asegura que nunca se pierde trazabilidad.
- Para compliance con regulaciones colombiana (SIC, DIAN), los logs deben mantenerse mínimo 5 años.
- Se recomienda hacer backup de tabla comando_talanquera con frecuencia (sharding por fecha si crece mucho).
- El campo `parametros_json` permite guardar contexto adicional sin modificar el esquema: entrada_id, salida_id, transaction_id, etc.
- Para queries de auditoría eficientes, crear índices en: gate_id, timestamp_solicitud, originator_id, tipo_comando.
- El sistema debe generar reportes automáticos de: comandos por día/sede, comandos fallidos, aperturas manuales por operador.
