# RF-001 — Mostrar espacios disponibles en tiempo real

## Metadata

- **Módulo**: Disponibilidad y Gestión de Sedes
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: `parking-service`, `api-gateway`

---

## User Story

**Como** Cliente (conductor) **quiero** ver cuántos espacios hay disponibles en una sede antes de llegar **para** decidir si voy a ese parqueadero o busco otro.

---

## Objetivo

El sistema debe consultar en tiempo real el conteo de espacios libres, ocupados, en mantenimiento y reservados por cada sede, y actualizar la información en máximo 5 segundos cada vez que ocurre un ingreso o salida de vehículo.

---

## Actores

- **Cliente**: Consulta disponibilidad desde app móvil o página web
- **Sistema**: Actualiza los conteos automáticamente cuando ocurre un registro de entrada o salida

---

## Precondiciones

- La sede existe y está activa en el sistema
- Al menos una zona y un espacio están definidos para la sede
- El modo de operación de la sede es `iot` (los espacios se actualizan vía eventos IoT) o `manual` (el operador registra cambios)

---

## Postcondiciones

### En caso de éxito

- El cliente recibe el conteo de espacios libres por zona
- El dato se actualiza automáticamente en todos los clientes conectados en menos de 5 segundos

### En caso de falla

- Si la sede no existe o está inactiva, se retorna error 404
- Si la sede está suspendida (tenant en churn), se retorna 403

---

## Flujo Funcional

1. Cliente abre la app y selecciona una sede del mapa o lista
2. Sistema recibe request `GET /api/v1/sedes/{sede_id}/disponibilidad`
3. Sistema consulta la tabla `espacio` contando por estado: `libre`, `ocupado`, `mantenimiento`, `reservado`
4. Sistema agrupa los resultados por `zona_id` y calcula totales
5. Sistema retorna JSON con: `{ sede_id, total, libres, ocupados, mantenimiento, reservados, zonas: [{zona_id, nombre, libres, capacidad}] }`
6. Si el cliente tiene websocket activo, se le envía la actualización en tiempo real

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| sede_id | UUID | Identificador de la sede consultada | Sí |
| incluye_zonas | BOOLEAN | Si true, incluye desglose por zona (default: true) | No |
| tipo_espacio | VARCHAR | Filtrar por tipo: cubierta, descubierta, VIP, moto | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| sede_id | UUID | Identificador de la sede |
| total | INTEGER | Capacidad total de la sede |
| libres | INTEGER | Espacios actualmente libres |
| ocupados | INTEGER | Espacios ocupados |
| mantenimiento | INTEGER | Espacios en mantenimiento |
| reservados | INTEGER | Espacios reservados |
| tasa_ocupacion | DECIMAL(5,2) | Porcentaje de ocupación (ocupados / total * 100) |
| ultima_actualizacion | TIMESTAMP | Timestamp de la última actualización |
| zonas | ARRAY | Array de objetos con disponibilidad por zona |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Sede no existe | Retorna 404 con mensaje "Sede no encontrada" |
| Sede inactiva | Retorna 404 (se trata como no existente para no exponer datos) |
| Tenant suspendido (churned) | Retorna 403 "Tenant suspendido, contacte a soporte" |
| Base de datos no disponible | Retorna 503 "Servicio no disponible, intente más tarde" |
| 0 espacios definidos para la sede | Retorna 200 con totales en 0 y mensaje de configuración pendiente |

---

## Criterios de Aceptación

1. El tiempo entre que ocurre un registro de entrada/salida y que el conteo se refleja en la consulta no excede **5 segundos**
2. La consulta de disponibilidad responde en **menos de 200ms** para sedes con hasta 500 espacios
3. El sistema soporta al menos **100 consultas concurrentes** de disponibilidad por sede sin degradación
4. La tasa de ocupación se calcula con **2 decimales de precisión**
5. Si una sede tiene múltiples zonas, el desglose por zona está disponible en la misma respuesta

**RFs relacionados**:

**Tablas de base de datos afectadas**:
- `espacio` — Se consulta el campo `estado` para contar por categoría
- `zona` — Se usa para agrupar y mostrar nombre de zona
- `sede` — Se valida que la sede exista y esté activa
- `configuracion_sede` — Se consulta el `modo_operacion` para saber si es iot o manual

---

## Endpoints API Involucrados

- `GET /api/v1/sedes/{sede_id}/disponibilidad` — Consulta de disponibilidad por sede
- `GET /api/v1/sedes/{sede_id}/zonas/{zona_id}/disponibilidad` — Consulta de disponibilidad por zona específica
- `WS /ws/sedes/{sede_id}/disponibilidad` — WebSocket para actualizaciones en tiempo real

---

## Métricas de Éxito

- Tiempo de actualización de disponibilidad: **< 5 segundos** desde el evento
- Latencia de respuesta P99: **< 200ms**
- Disponibilidad del endpoint: **99.9%**
- Numero de consultas concurrentes soportadas: **≥ 100 por sede**

---

## Notas

- La consulta usa un índice compuesto en `(sede_id, estado)` para garantizar velocidad
- En modo `manual`, la actualización de espacios la hace el operador explícitamente desde la app, por lo que el timestamp de última actualización es crucial para detectar datos obsoletos
- Se requiere un mecanismo de cache en memoria (Redis) para evitar sobrecargar la base de datos en sedes con alta frecuencia de consulta
