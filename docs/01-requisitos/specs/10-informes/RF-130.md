# RF-130 — Log de Reportes Programados Entregados

## Metadata

- **Módulo**: Informes
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: reports-service, scheduler-service, notification-service

---

## User Story

**Como** administrador de sede **quiero** consultar el historial de reportes programados que han sido entregados **para** auditar qué reportes se enviaron, cuándo, a quién y en qué estado.

---

## Objetivo

El sistema debe mantener un log detallado de cada ejecución de reporte programado, incluyendo: destinatario, fecha y hora de entrega, formato del archivo, tamaño, estado (entregado, fallido, reintentado) y mensaje de error si aplica. Este log debe ser consultable por el administrador y persistir durante al menos 90 días.

---

## Actores

- **Actor principal**: Administrador de sede (`tenant_admin`)
- **Actor secundario**: Superadmin, Auditor

---

## Precondiciones

- Existe al menos una entrega de reporte programado en el log.

---

## Postcondiciones

### En caso de éxito

- El log muestra el registro completo de la entrega con todos los campos descritos.
- El log puede filtrarse por programa, destinatario, estado y rango de fechas.

### En caso de falla

- Si no hay registros en el período seleccionado, se muestra mensaje "Sin registros para los filtros seleccionados".

---

## Flujo Funcional

1. Cada vez que se ejecuta un reporte programado (RF-129), el sistema registra un entry en el log de entregas.
2. El log se escribe tanto en éxito como en falla de la entrega.
3. El usuario accede a la sección "Log de Entregas" dentro del módulo de informes.
4. El usuario puede filtrar por:
   - Programa de reporte específico.
   - Destinatario (email).
   - Estado: ENTREGADO, FALLIDO, REINTENTADO, PENDIENTE.
   - Rango de fechas.
5. El sistema retorna la lista de registros paginados ordenados por fecha descendente.
6. Por cada registro se puede ver el detalle expandido incluyendo mensaje de error si falló.

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| programa_id | UUID | Filtrar por programa específico | No |
| destinatario | String | Filtrar por email de destinatario | No |
| estado | Enum | Filtrar por estado de entrega | No |
| fecha_inicio | DateTime | Inicio del rango de fechas | No |
| fecha_fin | DateTime | Fin del rango de fechas | No |
| pagina | Integer | Número de página para paginación | No (default: 1) |
| tamano_pagina | Integer | Registros por página | No (default: 25, max: 100) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| registro_id | UUID | Identificador único del log |
| programa_id | UUID | ID del programa que ejecutó |
| nombre_programa | String | Nombre del programa |
| nombre_reporte | String | Nombre del reporte base |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Sin registros en el período | Muestra mensaje "Sin entregas para el período seleccionado" |
| Reintento exitoso | El registro muestra estado FINAL y número de intentos = 2+; se puede expandir para ver intentos anteriores |
| Destinatario dio de baja su email | El log muestra el email original; no se actualiza a la nueva dirección |
| Log para programa eliminado | Se preserva el log; el nombre del programa se muestra como "[Programa eliminado]" |
| Más de 1000 registros en una página | Se paginan automáticamente; máximo 100 por página |
| Fecha de entrega posterior a la programada | Se muestra la diferencia como "Retraso de X minutos" en el detalle |
| Programa pausado y nunca ejecutado | No hay log de entrega para esa ventana; el estado del programa es PAUSADO |

---

## Criterios de Aceptación

1. El log registra cada ejecución de reporte programado, tanto exitoso como fallido.
2. Cada registro contiene: destinatario, fecha/hora, formato, tamaño, estado y mensaje de error si falló.
3. El log es consultable con filtros por programa, destinatario, estado y rango de fechas.
4. El log persiste durante al menos 90 días; después de 90 días se archiva o elimina.
5. Un administrador puede ver los logs de todos los programas de su sede.
6. Los registros de log no son editables ni eliminables por usuarios (son audit trail).
7. Los logs pueden exportarse a CSV para auditoría externa.

## Métricas de Éxito

- 100% de entregas registradas en el log (ninguna pérdida de registro).
- Log disponible para consulta en < 2 segundos para rangos de hasta 30 días.
- Retención de 90 días sinimpacto en performance del sistema.

---

## Notas

- El log es de solo adición; no se permiteupdate ni delete de registros existentes.
- Los registros de log incluyen un campo `ejecutor` que muestra quién creó el programa, no quién recibió el reporte.
- Para programas con múltiples destinatarios, se crea un registro por destinatario.
- El campo `fecha_entrega` es null cuando el estado es FALLIDO o REINTENTADO; solo se populate cuando el estado es ENTREGADO.
- Se recomienda implementar rotación de logs: архивировать registros mayores a 90 días a una tabla de histórico para compliance.
