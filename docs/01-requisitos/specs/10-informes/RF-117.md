# RF-117 — Filtrado de Reportes

## Metadata

- **Módulo**: Informes
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: reporting-service, transaction-service, customer-service

---

## User Story

**Como** tenant_admin o sede_admin **quiero** aplicar múltiples filtros a cualquier reporte **para** obtener datos específicos según la sede, operador, cliente, rango de fechas o tipo de vehículo que me interese analizar.

---

## Objetivo

El sistema debe permitir que todo reporte del módulo de Informes sea filtrable por uno o más de los siguientes criterios: sede, operador, cliente, rango de fechas y tipo de vehículo. Los filtros son combinables entre sí y se aplican antes de cualquier agregación o cálculo.

---

## Actores

- **tenant_admin**: Puede aplicar cualquier filtro sobre cualquier sede
- **sede_admin**: Puede aplicar filtros, pero仅限于 su propia sede (el filtro por sede se ignora si intenta ver otra)
- **reporting-service**: Aplica los filtros recibidos en la query a la consulta subyacente

---

## Precondiciones

- El usuario debe estar autenticado y tener permisos de lectura de reportes
- Los IDs de referencia (operador, cliente, sede, tipo de vehículo) deben existir en la base de datos

---

## Postcondiciones

### En caso de éxito

- El reporte retorna los datos filtrados según todos los criterios activos
- Los filtros activos se reflejan en la respuesta para transparencia

### En caso de falla

- Si un filtro referencia un ID inexistente, se ignora ese filtro y se muestra una advertencia en la respuesta
- Si el usuario no tiene acceso a la sede filtrada, se retorna error 403

---

## Flujo Funcional

1. El usuario construye la consulta de reporte añadiendo uno o más parámetros de filtro
2. El `reporting-service` valida cada filtro:
   a. Verifica que el `sede_id` exista y que el usuario tenga acceso
   b. Verifica que el `operador_id` exista y pertenezca a una sede accesible
   c. Verifica que el `cliente_id` exista
   d. Verifica que `tipo_vehiculo` sea un valor válido del catálogo
   e. Verifica que `fecha_inicio <= fecha_fin`
3. Se construye la consulta con los filtros activos aplicados en la capa de datos
4. Se ejecuta la consulta y se retorna el resultado
5. La respuesta incluye los filtros aplicados y advertencias si algunos fueron ignorados

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| sede_id | uuid | Filtrar por sede | No |
| operador_id | uuid | Filtrar por operador que cerró la transacción | No |
| cliente_id | uuid | Filtrar por cliente (vehículo o usuario) | No |
| fecha_inicio | date | Inicio del rango de fechas | Si |
| fecha_fin | date | Fin del rango de fechas | Si |
| tipo_vehiculo | string | Filtrar por tipo de vehículo (carro, moto, etc.) | No |
| estado_transaccion | enum | Filtrar por estado (open, closed, cancelled) | No (default: closed) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| filtros_aplicados | object | Objeto con cada filtro y su valor |
| advertencias | array | Lista de advertencias (ej: filtro ignorado por no existir) |
| datos | object | Resultado del reporte con los filtros aplicados |
| total_registros | integer | Cantidad de registros que cumplen los filtros |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Filtro por sede que no existe | Se ignora el filtro y se incluye advertencia en respuesta |
| Filtro por operador que no existe | Se ignora el filtro y se incluye advertencia |
| Rango de fechas vacío (inicio > fin) | Retorna error 400 con mensaje "fecha_inicio debe ser menor o igual a fecha_fin" |
| sede_admin filtrando por otra sede | Se ignora el filtro (se muestra solo su sede) y se incluye advertencia |
| Combinación de filtros que retorna 0 registros | Retorna reporte vacío (no es error) |
| Filtro por tipo_vehiculo inválido | Se ignora el filtro y se incluye advertencia |

---

## Criterios de Aceptación

1. Cualquier combinación de filtros válidos debe retornar el subconjunto de datos correspondiente
2. Los filtros no aplicados (nulos o vacíos) no deben afectar el resultado
3. Un filtro por operador que no existe no debe causar error, sino warn + ignore
4. El tiempo de respuesta con 5 filtros activos debe ser < 3 segundos para 1 año de datos
5. La respuesta siempre incluye `filtros_aplicados` mostrando qué filtros se usaron

## Métricas de Éxito

- Tiempo promedio de consulta con 5 filtros (target: < 3s)
- Porcentaje de consultas con filtros válidos que retornan datos (target: > 80%)
- Cantidad de advertencias generadas por filtros inválidos (monitorear para UX)

---

## Notas

- Este RF define la interfaz de filtrado que todos los demás reportes del módulo heredan
- Los filtros deben documentarse en el OpenAPI del servicio de reporting
- Se recomienda cachear resultados de filtros frecuentes para mejorar rendimiento
