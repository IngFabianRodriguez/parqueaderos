# RF-009 — Almacenar imágenes de captura con hash para auditoría

## Metadata

- **Módulo**: ANPR
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: anpr-service, storage-service

---

## User Story

**Como** Sistema (almacenamiento automático) **quiero** almacenar la imagen de cada captura de placa con su hash SHA-256 para auditoría **para** garantizar la integridad de las imágenes como evidencia en disputas de cobro, recursos de tránsito o investigaciones.

---

## Objetivo

Cada imagen capturada por el sistema ANPR en entrada o salida debe almacenarse de forma segura con: (1) la imagen original en object storage (S3 o compatible), (2) un hash SHA-256 calculado sobre la imagen original, (3) metadatos (timestamp, sede, placa, confianza). El hash permite verificar que la imagen no ha sido alterada. Las imágenes se retienen mínimo 90 días.

- Sistema (almacenamiento automático)

## Precondiciones

      - El servicio de storage está configurado y accesible

---

## Postcondiciones

### En caso de éxito

      - Imagen almacenada en object storage con URL pública/privada
      - Hash calculado y almacenado junto con el registro de entrada/salida

### En caso de falla

      - Storage no disponible → imagen se almacena temporalmente en base de datos como BLOB y se hace retry asíncrono
      - Hash no puede verificarse → se recalcula antes de cualquier operación de auditoría

---

## Flujo Funcional

1. ANPR-service captura imagen de placa en entrada o salida
1. ANPR-service calcula SHA-256 hash de la imagen cruda
1. ANPR-service envía imagen + hash + metadatos a storage-service
1. Storage-service sube imagen a object storage (S3/MinIO/Cloudflare R2)
1. Storage-service retorna URL de la imagen almacenada
1. ANPR-service actualiza el registro de entrada/salida con imagen_url y hash_sha256
1. Sistema programa tarea de limpieza para eliminar imágenes > 90 días

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Imagen corrupta o ilegible | Se almacena igualmente con hash del archivo recibido y se marca como 'low_quality' |
| Object storage no disponible | Se guarda en base de datos como BLOB temporal y se programa retry cada 5 minutos |
| Hash no coincide al verificar (imagen alterada) | Se genera alerta de seguridad y se marca el registro como 'sospechoso' |

---

## Criterios de Aceptación

- Cada imagen tiene un hash único y reproducible
- La verificación de hash debe poder hacerse sin acceso al object storage (solo con hash y archivo)
- Almacenamiento mínimo 90 días configurable por tenant
