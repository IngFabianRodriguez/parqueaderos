# RF-167 — Gestión de Incidencias y Tickets de Soporte

## Metadata

- **Módulo**: 13 — Admin Panel
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: support-service, admin-service

---

## User Story

**Como** superadmin **quiero** gestionar las incidencias reportadas por los tenants **para** dar soporte eficiente y mantener la calidad del servicio.

---

## Objetivo

El sistema permite registrar, clasificar, asignar y resolver incidencias reportadas por los administradores de tenant. Cada incidencia tiene historial completo de interacciones.

---

## Actores

- **Actor principal**: Superadmin
- **Actor secundario**: Admin de tenant (reporta la incidencia)

---

## Flujo Funcional

### Crear Incidencia (Admin de Tenant)

1. Admin accede a "Ayuda > Reportar Problema".
2. Describe el problema, selecciona categoría, adjunta screenshots.
3. Sistema crea incidencia con estado `ABIERTA`.
4. Admin recibe confirmación con número de ticket.

### Atender Incidencia (Superadmin)

1. Superadmin ve la incidencia en su cola.
2. La asigna a un agente de soporte.
3. Agente interactúa con el cliente (chat interno).
4. Cuando resuelto, marca como `CERRADA` con solución.

---

## Estados de la Incidencia

| Estado | Descripción |
|--------|-------------|
| ABIERTA | Recibida, sin asignar |
| EN_PROCESO | Asignada y siendo atendida |
| PENDIENTE_CLIENTE | Esperando respuesta del cliente |
| RESUELTA | Solucionada, esperando confirmación |
| CERRADA | Confirmada por cliente o auto-cerrada |

---

## Criterios de Aceptación

1. Cada incidencia tiene SLA según prioridad (Crítica: 1h, Alta: 4h, Media: 24h, Baja: 72h).
2. El cliente recibe notificaciones de cambio de estado.
3. Se registra tiempo total de resolución.
4. Las incidencias cerradas se mantienen en historial 90 días.

---

## Notas

- Las categorías de incidencia: BUG, MEJORA, CONSULTA, FACTURACION, INTEGRACION.
