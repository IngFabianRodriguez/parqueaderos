# RF-153 — Gestión de Tenants (Cuentas SaaS)

## Metadata

- **Módulo**: 13 — Admin Panel
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: tenant-service, billing-service, admin-service

---

## User Story

**Como** superadmin **quiero** crear, configurar y desactivar cuentas tenant **para** gestionar el ciclo de vida completo de los clientes del SaaS.

---

## Objetivo

El panel de administración permite gestionar completamente las cuentas tenant: crear nuevas cuentas con sus datos básicos, configurar plan y facturación, establecer límites de uso, y desactivar cuentas cuando sea necesario. Cada tenant opera de forma aislada con su propio slug, sedes, usuarios y datos.

---

## Actores

- **Actor principal**: Superadmin del SaaS
- **Actor secundario**: Admin del tenant (puede ver/editar solo su propia cuenta)

---

## Precondiciones

- El superadmin tiene sesión activa.
- El slug elegido no está en uso por otro tenant.

---

## Postcondiciones

### En caso de éxito

- El tenant queda creado con estado `ACTIVO` (o `TRIAL` si aplica).
- Se provisionan los servicios base del tenant.
- Se registra en auditoría la creación del tenant.

### En caso de falla

- No se crea el tenant.
- Se muestra error específico (slug duplicado, datos inválidos).

---

## Flujo Funcional

### Crear Nuevo Tenant

1. El superadmin accede a "Tenants" y hace clic en "Nuevo Tenant".
2. El sistema muestra formulario:
   - Nombre de la empresa
   - NIT
   - Email del admin principal
   - Teléfono
   - Ciudad
   - Plan seleccionado (TRIAL, BASIC, PROFESSIONAL, ENTERPRISE)
   - Slug (auto-generado desde el nombre, editable)
3. El superadmin llena los datos y confirma.
4. El sistema valida slug único, email válido, NIT no duplicado.
5. El sistema crea el tenant con estado `TRIAL` (14 días).
6. El sistema crea el usuario admin principal y envía email de bienvenida.
7. El sistema registra en auditoría: `TENANT_CREATED`.

### Configurar Plan y Límites

1. El superadmin abre la configuración del tenant.
2. En la pestaña "Plan y Facturación", ve el plan actual, precio mensual, próximo cobro.
3. Puede cambiar el plan (upgrade/downgrade inmediato).
4. Puede establecer límites: número máximo de sedes, usuarios, transacciones/mes.
5. Al guardar, el sistema actualiza y registra: `TENANT_PLAN_UPDATED`.

### Desactivar Tenant

1. El superadmin abre la configuración del tenant.
2. Hace clic en "Desactivar Tenant".
3. Sistema muestra confirmación: "Esto suspenderá el acceso a todos los usuarios del tenant. ¿Continuar?"
4. Superadmin confirma.
5. Sistema marca `estado = SUSPENDIDO`, invalida todos los tokens.
6. Sistema registra: `TENANT_SUSPENDED`.

### Reactivar Tenant

1. Superadmin abre tenant suspendido.
2. Hace clic en "Reactivar".
3. Sistema restaura `estado = ACTIVO` y envía email de notificación.
4. Sistema registra: `TENANT_REACTIVATED`.

## Datos de Entrada — Crear Tenant

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| nombre_empresa | String(100) | Razón social | Si |
| nit | String(20) | Número de identificación tributaria | Si |
| email_admin | String(255) | Email del admin principal | Si |
| telefono | String(20) | Teléfono de contacto | No |
| ciudad | String(50) | Ciudad/ubicación | No |
| plan | Enum | TRIAL, BASIC, PROFESSIONAL, ENTERPRISE | Si |
| slug | String(50) | URL única del tenant | Si (auto, editable) |

---

## Datos de Salida — Tenant

| Campo | Tipo | Descripción |
|-------|------|-------------|
| tenant_id | UUID | Identificador único |
| nombre_empresa | String | Razón social |
| nit | String | NIT |
| slug | String | Slug único para acceso |
| estado | Enum | TRIAL, ACTIVO, SUSPENDIDO, CANCELADO |
| plan | Enum | Plan actual |
| fecha_creacion | DateTime | Cuándo se creó |
| fecha_inicio_trial | DateTime | Inicio del periodo de prueba |
| fecha_fin_trial | DateTime | Fin del periodo de prueba |
| max_sedes | Integer | Límite de sedes |
| max_usuarios | Integer | Límite de usuarios |
| max_transacciones_mes | Integer | Límite de transacciones mensuales |
| admin_principal | UUID | Usuario admin principal |

---

## Estados del Tenant

| Estado | Descripción | Acceso de usuarios |
|--------|-------------|-------------------|
| TRIAL | Periodo de prueba (14 días) | Acceso completo, sin cobro |
| ACTIVO | Suscrito y pagando | Acceso completo |
| SUSPENDIDO | Pago pendiente o decisión admin | Sin acceso |
| CANCELADO | Dados de baja definitivamente | Sin acceso, datos retenidos 90 días |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Slug duplicado | Error: "Este slug ya está en uso. Elija otro." |
| NIT duplicado | Error: "Ya existe un tenant con este NIT." |
| Trial expirado sin pago | Se suspende automáticamente; email de notificación. |
| Intentar crear tenant sin plan | Error: "Seleccione un plan." |
| Desactivar tenant con transacciones pendientes | Warning: "Este tenant tiene transacciones sin conciliar." |
| Reactivar tenant cancelado | Solo superadmin puede; error si pasó más de 90 días. |

---

## Criterios de Aceptación

1. El superadmin puede crear un tenant y este queda operativo en < 30 segundos.
2. El slug es único y se genera automáticamente en minúsculas sin espacios.
3. El trial expira automáticamente a los 14 días y se envía recordatorio 3 días antes.
4. Un tenant suspendido no puede recibir login de ningún usuario.
5. Cada cambio de plan queda registrado en auditoría con el valor anterior y nuevo.
6. Los límites configurados se aplican en tiempo real (no permite crear sede 6 si límite es 5).
7. El sistema envía email al admin principal en cada cambio de estado.

## Endpoints API Involucrados

- `POST /api/v1/admin/tenants` — Crear tenant
- `GET /api/v1/admin/tenants` — Listar tenants (con filtros: estado, plan)
- `GET /api/v1/admin/tenants/{id}` — Detalle de tenant
- `PUT /api/v1/admin/tenants/{id}` — Actualizar tenant
- `PUT /api/v1/admin/tenants/{id}/suspend` — Suspender
- `PUT /api/v1/admin/tenants/{id}/reactivate` — Reactivar
- `PUT /api/v1/admin/tenants/{id}/plan` — Cambiar plan

---

## Métricas de Éxito

- Creación de tenant: < 30 segundos hasta acceso disponible.
- Suspensión/reactivación: < 5 segundos.
- 0 slugs duplicados en el sistema.
- 100% de cambios de estado auditados.

---

## Notas

- El slug debe ser alphanumeric con guiones, sin espacios ni caracteres especiales.
- Al crear un tenant en TRIAL, se debe crear automáticamente el admin con rol ADMIN_TENANT.
- Los límites deben poder excederse temporalmente (grace period de 5%) para evitar bloquear por picos.
- La facturación se maneja en el módulo de conciliación (RF-170+).
