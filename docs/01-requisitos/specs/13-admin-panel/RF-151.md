# RF-151 — Gestión de Usuarios y Operadores

## Metadata

- **Módulo**: 13 — Admin Panel
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: admin-service, user-service, tenant-service

---

## User Story

**Como** administrador del sistema **quiero** crear, editar y desactivar usuarios (operadores, supervisores, admins de sede) **para** mantener el control de acceso y la seguridad del sistema.

---

## Objetivo

El panel de administración permite gestionar completamente el ciclo de vida de los usuarios: crear cuentas con roles asignados, modificar sus datos y permisos, resetear contraseñas, y desactivar usuarios cuando dejan la organización. Todo con registro de auditoría de cada cambio.

---

## Actores

- **Actor principal**: Superadmin (dueño del SaaS)
- **Actor secundario**: Admin de tenant, supervisor de sede

---

## Precondiciones

- El usuario administrador tiene sesión activa con rol `superadmin` o `admin_tenant`.
- No existen usuarios con email duplicado en el sistema.

---

## Postcondiciones

### En caso de éxito

- El usuario queda creado/actualizado con los roles especificados.
- Se envía email de bienvenida con credenciales temporales (si es creación).
- Se registra en log de auditoría: acción, usuario que realizó el cambio, timestamp.

### En caso de falla

- No se modifica el estado del usuario.
- Se muestra mensaje de error específico (email duplicado, rol no permitido, etc.).

---

## Flujo Funcional

### Crear Usuario

1. El admin accede a la sección "Usuarios" del panel.
2. El admin hace clic en "Nuevo Usuario".
3. El sistema muestra formulario con campos: nombre, email, teléfono, rol, sede(s) asignadas.
4. El admin llena los campos y confirma.
5. El sistema valida que el email no exista, que el rol sea válido para el contexto.
6. El sistema crea el usuario con contraseña temporal generateda.
7. El sistema envía email de bienvenida con credenciales.
8. El sistema registra en auditoría: `USER_CREATED` con user_id, admin_id, timestamp.

### Editar Usuario

1. El admin busca el usuario (por nombre, email o documento).
2. El admin abre el perfil del usuario.
3. El admin modifica los campos permitidos (nombre, rol, sedes).
4. El admin guarda los cambios.
5. El sistema valida y actualiza.
6. El sistema registra en auditoría: `USER_UPDATED` con campos modificados.

### Desactivar Usuario

1. El admin busca el usuario.
2. El admin hace clic en "Desactivar".
3. El sistema muestra confirmación: "Este usuario perderá acceso. ¿Continuar?".
4. El admin confirma.
5. El sistema marca `activo = false`, sesiones activas son invalidadas.
6. El sistema registra en auditoría: `USER_DEACTIVATED`.

### Resetear Contraseña

1. El admin abre perfil del usuario.
2. El admin hace clic en "Resetear contraseña".
3. El sistema genera nueva contraseña temporal y envía email al usuario.
4. El sistema registra en auditoría: `PASSWORD_RESET`.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| nombre | String(100) | Nombre completo | Si |
| email | String(255) | Correo electrónico único | Si |
| telefono | String(20) | Teléfono de contacto | No |
| rol | Enum | ADMIN_TENANT, SUPERVISOR, OPERADOR | Si |
| sede_ids | Array[UUID] | Sedes donde tiene acceso | Si (para OPERADOR, SUPERVISOR) |
| documento | String(20) | Documento de identidad | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| user_id | UUID | Identificador único |
| nombre | String | Nombre completo |
| email | String | Correo electrónico |
| telefono | String | Teléfono |
| rol | Enum | Rol asignado |
| sede_ids | Array[UUID] | Sedes asignadas |
| activo | Boolean | Si está activo |
| fecha_creacion | DateTime | Cuándo fue creado |
| ultimo_acceso | DateTime | Último login |
| creado_por | UUID | Admin que lo creó |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Email ya existe | Error: "Este email ya está registrado. Use otro o busque el usuario existente." |
| Rol inválido para contexto | Error: "No tiene permisos para asignar este rol." |
| Intentar desactivar su propia cuenta | Error: "No puede desactivarse a sí mismo." |
| Intentar crear usuario sin sedes (operador) | Error: "Debe asignar al menos una sede." |
| Múltiples admins editando mismo usuario | El último save gana; se muestra warning si detecta conflicto. |
| Usuario desactivado intenta login | Rechazado con mensaje "Cuenta desactivada. Contacte al administrador." |

---

## Criterios de Aceptación

1. El admin puede crear un usuario con rol OPERADOR y asignarlo a 1 o más sedes.
2. El admin puede cambiar el rol de un usuario existente.
3. Un usuario desactivado no puede iniciar sesión ni recibir tokens activos.
4. Cada acción de gestión de usuarios queda registrada en auditoría.
5. El sistema envía email de bienvenida al crear un nuevo usuario.
6. La búsqueda de usuarios devuelve resultados en < 500ms.
7. El admin no puede asignarse roles superiores al suyo.
8. Los campos de auditoría incluyen: quién, qué, cuándo (user_id, acción, timestamp).

## Endpoints API Involucrados

- `POST /api/v1/admin/users` — Crear usuario
- `GET /api/v1/admin/users` — Listar usuarios (con filtros)
- `GET /api/v1/admin/users/{id}` — Detalle de usuario
- `PUT /api/v1/admin/users/{id}` — Actualizar usuario
- `DELETE /api/v1/admin/users/{id}` — Desactivar usuario
- `POST /api/v1/admin/users/{id}/reset-password` — Resetear contraseña

---

## Métricas de Éxito

- Tiempo de creación de usuario: < 3 segundos.
- Tiempo de búsqueda: < 500ms para textos válidos.
- 0 usuarios con email duplicado en el sistema.
- 100% de acciones auditadas.

---

## Notas

- Los roles disponibles dependen del nivel del admin: Superadmin ve todos los roles, Admin tenant solo ve ADMIN_TENANT, SUPERVISOR, OPERADOR.
- Los OPERADORES siempre deben tener al menos una sede asignada.
- La contraseña temporal expira en 24 horas; el usuario debe cambiarla al primer login.
