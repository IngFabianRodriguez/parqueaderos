# RF-152 — Gestión de Roles y Permisos (RBAC)

## Metadata

- **Módulo**: 13 — Admin Panel
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: admin-service, auth-service

---

## User Story

**Como** superadmin **quiero** definir roles con permisos específicos y asignarlos a usuarios **para** garantizar que cada persona tenga acceso solo a lo que necesita.

---

## Objetivo

El sistema implementa Role-Based Access Control (RBAC) donde los roles definen qué acciones puede realizar cada usuario. El superadmin puede crear roles personalizados, asignar permisos granulares, y los admin de tenant pueden asignar roles predefinidos a sus usuarios.

---

## Actores

- **Actor principal**: Superadmin del SaaS
- **Actor secundario**: Admin de tenant (solo ve roles disponibles para su contexto)

---

## Precondiciones

- El usuario administrador tiene rol `superadmin`.
- Existen permisos base definidos en el sistema.

---

## Postcondiciones

### En caso de éxito

- El rol queda creado con sus permisos asociados.
- El rol se puede asignar a usuarios.
- Se registra en auditoría la creación/modificación del rol.

### En caso de falla

- No se crea/modifica el rol.
- Se muestra mensaje de error (nombre duplicado, permisos inválidos).

---

## Flujo Funcional

### Crear Rol Personalizado

1. El superadmin accede a "Roles y Permisos".
2. El superadmin hace clic en "Nuevo Rol".
3. El sistema muestra formulario: nombre, descripción, lista de permisos agrupados por módulo.
4. El superadmin selecciona los permisos deseados (checkboxes por grupo).
5. El superadmin guarda.
6. El sistema valida que no exista otro rol con el mismo nombre.
7. El sistema crea el rol y sus relaciones permiso-rol.
8. El sistema registra en auditoría: `ROLE_CREATED`.

### Asignar Rol a Usuario

1. El admin abre perfil del usuario (RF-151).
2. El admin modifica el campo "Rol" y selecciona de la lista de roles disponibles.
3. El admin guarda.
4. El sistema actualiza el rol del usuario.
5. Si el usuario tiene sesión activa, se le invalidan los tokens actuales.

### Modificar Permisos de un Rol

1. El superadmin abre la configuración del rol.
2. El superadmin modifica los permisos (agregar/quitar).
3. El superadmin guarda.
4. El sistema actualiza y registra en auditoría: `ROLE_UPDATED`.
5. Todos los usuarios con este rol reciben sesión invalidada (forzar re-login).

### Eliminar Rol

1. El superadmin selecciona un rol.
2. El superadmin hace clic en "Eliminar".
3. El sistema verifica que no haya usuarios activos con este rol.
4. Si hay usuarios, error: "Este rol tiene usuarios activos. Reasigne antes de eliminar."
5. Si no hay usuarios, el sistema elimina lógicamente (soft delete) el rol.
6. El sistema registra en auditoría: `ROLE_DELETED`.

## Roles Predefinidos del Sistema

| Rol | Descripción | Permisos por defecto |
|-----|-------------|---------------------|
| SUPERADMIN | Acceso total al SaaS | Todos |
| ADMIN_TENANT | Admin de una cuenta tenant | usuarios.ver, usuarios.crear, usuarios.editar, sedes.ver, sedes.editar, reportes.ver, reportes.exportar |
| SUPERVISOR | Supervisa una o más sedes | sedes.ver, talanqueras.ver, talanqueras.abrir_manual, facturas.ver, reportes.ver |
| OPERADOR | Opera día a día | talanqueras.ver, talanqueras.abrir_manual, facturas.ver, facturas.crear |

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| nombre | String(50) | Nombre único del rol | Si |
| descripcion | String(255) | Descripción legible | No |
| permisos | Array[String] | Lista de permisos (codenames) | Si (al menos 1) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| rol_id | UUID | Identificador único |
| nombre | String | Nombre del rol |
| descripcion | String | Descripción |
| permisos | Array[String] | Lista de permisos |
| es_sistema | Boolean | Si es rol predefinido (no editable) |
| fecha_creacion | DateTime | Cuándo se creó |
| usuarios_count | Integer | Cuántos usuarios tienen este rol |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Nombre de rol duplicado | Error: "Ya existe un rol con este nombre." |
| Eliminar rol con usuarios activos | Error: "Reasigne los usuarios primero." |
| Intentar editar rol de sistema (SUPERADMIN) | Los permisos no son editables; solo descripción. |
| Asignar rol sin permisos | Error: "El rol debe tener al menos un permiso." |
| Permisos inválidos | Se ignoran los no reconocidos; solo se asignan los válidos. |

---

## Criterios de Aceptación

1. El superadmin puede crear roles con cualquier combinación de permisos.
2. Los roles predefinidos del sistema no pueden eliminarse ni modificar sus permisos.
3. Un usuario con rol OPERADOR no puede crear otros usuarios.
4. Los permisos se evalúan en cada request API (no solo en la UI).
5. El cambio de permisos de un rol fuerza re-autenticación de todos los usuarios con ese rol.
6. El log de auditoría registra creación, modificación y eliminación de roles.
7. Los admin de tenant solo ven roles que tienen permisos permitidos para su nivel.

## Endpoints API Involucrados

- `GET /api/v1/admin/roles` — Listar roles
- `POST /api/v1/admin/roles` — Crear rol
- `GET /api/v1/admin/roles/{id}` — Detalle de rol
- `PUT /api/v1/admin/roles/{id}` — Actualizar rol
- `DELETE /api/v1/admin/roles/{id}` — Eliminar rol
- `GET /api/v1/admin/permissions` — Listar permisos disponibles

---

## Métricas de Éxito

- Tiempo de creación de rol: < 1 segundo.
- Validación de permisos por request: < 10ms.
- 0 roles duplicados en el sistema.
- 100% de cambios en roles auditados.

---

## Notas

- Los permisos deben ser evaluados tanto en frontend como en backend (defensa en profundidad).
- Se recomienda implementar middleware de autorización que valide permisos en cada endpoint.
- Los logs de auditoría de permisos deben incluir: quién hizo el cambio, rol afectado, permisos antes y después.
