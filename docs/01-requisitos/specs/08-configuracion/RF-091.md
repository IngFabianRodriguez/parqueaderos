# RF-091 — Exportación e Importación Masiva de Configuración

## Metadata

- **Módulo**: Configuración — Herramientas de Administración
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: config-service, tenant-service, import-export-service

---

## User Story

**Como** administrador del tenant **quiero** exportar la configuración de mis sedes a un archivo y luego importarla para replicarla en otro tenant o restaurarla después de un incidente **para** reducir el tiempo de configuración en múltiples sedes y tener un backup recuperable.

---

## Objetivo

El sistema debe permitir al `tenant_admin` exportar la configuración completa de un tenant o de una sede específica a un archivo JSON estructurado. Este archivo incluye: horarios, tarifas, categorías de vehículos, métodos de pago, impuestos, descuentos, y plantillas de notificación. También debe permitir importar este archivo en otro tenant del mismo tipo para replicar la configuración, con soporte para mapeo de IDs (los IDs del archivo se reemplazan por IDs nuevos en el tenant destino).

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Super Admin (puede exportar/importar entre tenants)

---

## Precondiciones

- El actor tiene rol `tenant_admin` o es `super_admin`.
- Si es `tenant_admin`, solo puede exportar/importar en su propio tenant.

---

## Postcondiciones

### En caso de éxito

- Se genera un archivo JSON con toda la configuración exportada.
- La configuración importada se persiste en la base de datos.
- Se crea un registro en `import_export_logs` con el detalle de la operación.

### En caso de falla

- Si el archivo importado tiene estructura inválida: error 400 con detalle del problema.
- Si el archivo tiene una versión de schema incompatible: error 400 pidiendo actualizar.
- Si la importación modifica recursos en uso (ej: plan de tarifa activo en transacciones): se aplica la estrategia de merge configurada.

---

## Flujo Funcional

### Exportación

1. El `tenant_admin` accede a `Configuración → Herramientas → Exportar / Importar`.
2. Selecciona "Exportar Configuración".
3. Elige el alcance de la exportación:
   - `full_tenant`: Toda la configuración del tenant.
   - `single_venue`: Solo una sede específica.
   - `modules`: Selección manual de módulos (tarifas, horarios, categorías, etc.).
4. Opcionalmente, marca "Incluir datos de usuario" (para replicar en otro ambiente de desarrollo).
5. Hace clic en "Generar Exportación".
6. El sistema:
   - Consulta todos los datos de configuración según el alcance seleccionado.
   - Genera un JSON con la estructura definida en el schema v1.0.
   - Incluye metadata: `exported_at`, `tenant_id_origin`, `schema_version`, `modules_included`.
   - Comprime el JSON en un archivo `.zip` si incluye imágenes (logos, etc.).
7. El archivo se genera y se descarga en el navegador del admin.

### Importación

8. El admin selecciona "Importar Configuración" y elige el archivo `.json` o `.zip`.
9. El sistema valida el archivo:
   - Schema version compatible.
   - Estructura de JSON válida.
   - Módulos presentes y sin datos corruptos.
10. El sistema muestra un preview de lo que se importará:
    - Lista de módulos y cantidad de registros.
    - Conflictos detectados (recursos con mismo código pero datos diferentes).
11. El admin selecciona estrategia de merge para cada conflicto:
    - `skip`: No importar si ya existe.
    - `overwrite`: Reemplazar con los datos del archivo.
    - `keep_both`: Crear con un sufijo "_imported" (para evitar duplicados de código).
12. El admin confirma la importación.
13. El sistema importa módulo por módulo:
    - Valida cada registro.
    - Aplica la estrategia de merge seleccionada.
    - Genera nuevos IDs para todos los recursos.
    - Mantiene un mapeo `old_id -> new_id` para referencias cruzadas.
14. Se registra la operación en `import_export_logs`.
15. Se publica `CONFIG_IMPORTED` con `{tenant_id, modules_imported, record_count}`.

## Estructura del Archivo de Exportación (Schema v1.0)

```json
{
  "schema_version": "1.0",
  "exported_at": "2026-05-08T12:00:00Z",
  "tenant_id_origin": "uuid",
  "tenant_name": "Parqueadero Central",
  "modules": {
    "general_settings": { ... },
    "venue_schedules": [ ... ],
    "rate_plans": [ ... ],
    "vehicle_categories": [ ... ],
    "payment_methods": [ ... ],
    "taxes": [ ... ],
    "discounts": [ ... ],
    "notification_templates": [ ... ],
    "roles": [ ... ]
  }
}
```

## Datos de Entrada

### Exportación

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| scope | enum | `full_tenant`, `single_venue`, `modules` | Sí |
| venue_id | uuid | ID de sede (si scope = single_venue) | No |
| module_filter | array[string] | Lista de módulos a incluir (si scope = modules) | No |
| include_user_data | boolean | ¿Incluir datos de usuarios? | No (default: false) |

### Importación

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| file | file | Archivo JSON o ZIP de exportación | Sí |
| merge_strategy | enum | `skip`, `overwrite`, `keep_both` | Sí |
| module_merge_strategies | object | Estrategia por módulo (opcional, overriding global) | No |

---

## Datos de Salida

### Exportación

| Campo | Tipo | Descripción |
|-------|------|-------------|
| file_url | string | URL de descarga del archivo |
| file_name | string | Nombre del archivo (ej: `config_export_20260508_120000.zip`) |
| modules_included | array | Lista de módulos exportados |
| record_count | object | Conteo de registros por módulo |

### Importación

| Campo | Tipo | Descripción |
|-------|------|-------------|
| import_id | uuid | ID del proceso de importación |
| modules_imported | array | Módulos importados exitosamente |
| errors | array | Errors durante la importación |
| warnings | array | Warnings (ej: registros saltados) |
| record_count | object | Conteo de registros importados por módulo |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Archivo JSON mal formado | Error 400: "El archivo no es un JSON válido." |
| Schema version incompatible | Error 400: "Este archivo fue exportado con una versión incompatible (X). Actualice el sistema o exporte nuevamente desde la versión de origen." |
| `tenant_id` en archivo igual al actual | Warning: "Esta importación provine del mismo tenant. ¿Desea continuar?" |
| Recurso con `code` duplicado en el tenant destino | Se aplica la estrategia de merge elegida. |
| Categoría de vehículo referenced en una tarifa que no existe en la importación | El sistema importa la categoría primero, luego la tarifa. |
| Metadato de taxa/descuento con `old_id` que no tiene mapping | Se crea como nuevo recurso. |
| Archivo muy grande (> 50MB) | Error: "El archivo excede el límite de 50MB. Divida la exportación en módulos." |
| Importación de roles con permisos que no existen en el destino | Se crean los permisos faltantes automáticamente. |
| Intento de importar `tenant_admin` | Se ignora; los roles de sistema no se importan. |

---

## Criterios de Aceptación

1. El `tenant_admin` puede exportar toda la configuración de su tenant a un archivo JSON estructurado.
2. La exportación incluye un schema version para validación en importación.
3. La exportación puede limitarse a una sede específica o a módulos seleccionados.
4. El admin puede importar un archivo de exportación en su tenant.
5. La importación detecta conflictos y permite elegir estrategia de merge por módulo.
6. Los IDs de los recursos se reemplazan por nuevos IDs en el destino; las referencias cruzadas se actualizan.
7. Los roles predefinidos del sistema (`tenant_admin`, `operator`, `cashier`) no se importan.
8. La importación genera un log detallado en `import_export_logs`.
9. Se puede exportar desde un tenant y importar en otro tenant del mismo tipo (replicación).
10. El evento `CONFIG_IMPORTED` se publica tras una importación exitosa.

## Métricas de Éxito

- Exportación de tenant completo: < 30 segundos para hasta 10,000 registros.
- Importación sin errores: 100% de los registros importados según estrategia.
- Cero pérdida de datos en importación (todos los módulos validados antes de persistir).
- Archivo exportado puede importarse en un tenant vacío en < 2 minutos.

---

## Notas

- El archivo exportado NO incluye datos transaccionales (transacciones, vehículos registrados, usuarios), solo configuración.
- Las credenciales (API keys, secrets) se exportan como referencias, no como valores. El admin debe configurar los valores en el tenant destino.
- Para backup completo, se recomienda usar las herramientas de backup de base de datos además de esta exportación.
- La importación en un tenant con datos existentes usa merge strategies; para una importación limpia, usar `overwrite` en todos los módulos primero.
- Se recomienda comprimir la exportación en ZIP si incluye logos o imágenes de plantillas.
