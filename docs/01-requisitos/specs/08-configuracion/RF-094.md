# RF-094 — Descuentos y Promociones Configurables

## Metadata

- **Módulo**: Configuración — Reglas de Negocio
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, promotion-service, tenant-service

---

## User Story

**Como** administrador de tenant **quiero** configurar descuentos y promociones con reglas de validez y límites de uso **para** atraer clientes, fidelizar usuarios frecuentes y ofrecer ofertas especiales en fechas o condiciones específicas.

---

## Objetivo

El sistema debe permitir a los administradores de tenant crear, modificar y desactivar descuentos y promociones. Cada promoción debe definir: el tipo de descuento (porcentaje, monto fijo, o tiempo gratis), las condiciones de aplicación (monto mínimo, tipo de vehículo, días específicos), el período de validez, y un límite máximo de usos (global y por usuario).

---

## Actores

- **Actor principal**: Administrador de tenant (crea y gestiona promociones)
- **Actor secundario**: Sistema (aplica los descuentos al facturar)
- **Actor terciario**: Cliente/Usuario (redeems el descuento)

---

## Precondiciones

- El tenant tiene el módulo de facturación activo.
- El tenant tiene tarifas base configuradas (RF-093).

---

## Postcondiciones

### En caso de éxito

- La promoción queda creada y disponible para uso.
- Se genera un evento `promotion_created`.
- Si es un código específico, el sistema lo activa para redimir.

### En caso de falla

- La promoción no se crea y se retorna error de validación.
- El sistema continúa sin cambios.

---

## Tipos de Descuento

| Tipo | Descripción | Ejemplo |
|------|-------------|---------|
| `percentage` | Porcentaje de descuento sobre el total | 20% de descuento |
| `fixed_amount` | Monto fijo descontado | $5 USD de descuento |
| `free_time` | Tiempo de estacionamiento gratis | 30 minutos gratis |
| `free_exit` | Exoneración de pago completa | Salida gratis |

---

## Estructura de una Promoción

| Campo | Tipo | Descripción |
|-------|------|-------------|
| promotion_id | uuid | Identificador único |
| name | string | Nombre visible de la promoción |
| description | string | Descripción para el cliente |
| code | string | Código de redención (null = automática) |
| discount_type | enum | `percentage`, `fixed_amount`, `free_time`, `free_exit` |
| discount_value | decimal | Valor del descuento |
| min_purchase_amount | decimal | Monto mínimo de compra para aplicar |
| max_discount_amount | decimal | Monto máximo de descuento (para percentage) |
| applicable_vehicle_types | array | Tipos de vehículo aplicables |
| applicable_zones | array | Zonas donde aplica (null = todas) |
| valid_from | datetime | Fecha y hora de inicio |
| valid_until | datetime | Fecha y hora de fin |
| max_total_uses | integer | Límite total de usos (null = ilimitado) |
| max_uses_per_user | integer | Límite por usuario (null = ilimitado) |
| current_uses | integer | Contador de usos actuales |
| requires_vehicle_match | boolean | El vehículo debe estar pre-registrado |
| days_of_week | array | Días de la semana aplicables [0-6] |
| time_restriction | object | Restricción de horario (start_time, end_time) |
| stackable | boolean | ¿Se puede combinar con otras promociones? |
| priority | integer | Prioridad de aplicación (mayor = primero) |
| active | boolean | Si está activa |

## Flujo Funcional

### Creación de promoción por el administrador

1. El administrador accede a **Configuración → Promociones → Nueva Promoción**.
2. Ingresa nombre, descripción y código (si es manual).
3. Selecciona el tipo de descuento y su valor.
4. Define condiciones: monto mínimo, tipo de vehículo, zonas aplicables.
5. Configura validez: fechas de inicio y fin.
6. Establece límites de uso: máximo global y por usuario.
7. Si es combinable, define la prioridad.
8. Guarda la promoción.
9. El sistema valida que no haya conflicto con promociones activas del mismo período.
10. Se persiste en `promotions` y se activa.

### Aplicación de promoción en facturación

1. Un vehículo registra salida (RF-009).
2. El billing-service consulta las `promotions` aplicables.
3. Filtra por: zona, vehículo, monto mínimo, fechas, días de semana, horario.
4. Si hay código: busca coincidencia exacta en `promotion_codes`.
5. Selecciona la promoción de mayor prioridad (si stackable=false, solo una).
6. Aplica el descuento al subtotal de la factura.
7. Registra el uso en `promotion_redemptions` (incrementa `current_uses`).
8. Si se alcanzó el `max_total_uses`, marca la promoción como `exhausted`.

### Redención de código promocional (flujo de usuario)

1. El usuario presenta el código en la app o al operador.
2. El sistema busca la promoción por código.
3. Valida: fechas, límites, condiciones.
4. Si es válida, aplica el descuento a la factura.
5. Registra la redención con `user_id` (si está autenticado).

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Código de promoción no existe | Retornar error: "Código de promoción inválido" |
| Promoción expirada | Retornar error: "Esta promoción ha vencido" |
| Límite de usos global alcanzado | Retornar error: "Esta promoción ya no está disponible" |
| Límite por usuario alcanzado | Retornar error: "Ya usaste esta promoción el máximo de veces permitido" |
| Monto menor al mínimo requerido | No se aplica el descuento; se informa al usuario |
| Promoción no stackable con otra | Se aplica solo la de mayor prioridad |
| Descuento mayor que la factura | El descuento se limita al monto total de la factura (no puede ser negativo) |
| Cambios en promoción activa | Los cambios aplican para nuevos usos; los ya aplicados remain válidos |
| Eliminación de promoción activa | Soft-delete; los usos ya realizados permanecen, no se reversan automáticamente |

## Criterios de Aceptación

1. Un administrador puede crear promociones con: porcentaje, monto fijo, tiempo gratis o salida gratis.
2. Las promociones pueden tener código manual o ser automáticas (aplican sin código).
3. Se pueden definir condiciones: monto mínimo, tipo de vehículo, zona, días, horario.
4. Los límites de uso (global y por usuario) se aplican correctamente.
5. Las fechas de validez se respetan; promociones expiradas no se aplican.
6. El orden de aplicación cuando hay múltiples promociones (prioridad) es correcto.
7. Los usos de promociones se registran para auditoría y reporting.
8. Las promociones pueden ser combinables o excluyentes entre sí.
9. El sistema soporta al menos 1000 códigos de promoción únicos por tenant.

## Métricas de Éxito

- Número de promociones creadas por mes por tenant.
- Tasa de redención: (redemptions / promotions_seen) > 15%.
- Descuento promedio concedido como porcentaje del total facturado.
- Promedio de usos por promoción activa.
- Tiempo promedio de vigencia de una promoción.

---

## Notas

- Los códigos de promoción deben ser únicos por tenant (case-insensitive).
- Se recomienda no tener más de 3 promociones automáticas activas simultáneamente para evitar confusión.
- Los descuentos por porcentaje deben tener un `max_discount_amount` para evitar pérdidas.
- Las promociones con tiempo gratis (`free_time`) requieren que el vehículo aún esté en el estacionamiento.
- El sistema debe registrar el `user_id` del cliente que usó la promoción para estadísticas.
