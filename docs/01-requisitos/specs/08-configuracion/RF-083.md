# RF-083 — Configuración General del Tenant

## Metadata

- **Módulo**: Configuración — Parámetros Generales
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: tenant-service, config-service, notification-service

---

## User Story

**Como** administrador del tenant **quiero** configurar los parámetros generales de mi organización **para** que el sistema refleje mi razón social, timezone, moneda y preferencias operativas.

---

## Objetivo

El sistema debe permitir al `tenant_admin` consultar y actualizar la configuración general del tenant: nombre legal, identificadores fiscales, timezone, moneda predominante, formato de fecha/hora, y preferencias de visualización. Estos parámetros son de ámbito tenant y determinan el comportamiento global del sistema para esa organización.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Sistema (aplica timezone y moneda en todos los cálculos)

---

## Precondiciones

- El actor tiene rol `tenant_admin` sobre el tenant.
- El tenant existe y está en estado `active`.

---

## Postcondiciones

### En caso de éxito

- La configuración general del tenant se actualiza en la tabla `tenant_settings`.
- Todos los servicios que dependan de estos valores reciben el evento `CONFIG_UPDATED` y refrescan su caché local.
- Se registra la auditoría del cambio con timestamp y usuario que lo realizó.

### En caso de falla

- Si la validación de timezone falla: error indicando timezone inválido (código IANA).
- Si el `tax_id` ya pertenece a otro tenant: error de integridad.
- La configuración anterior permanece intacta.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Configuración → General`.
2. El sistema carga la configuración actual desde `tenant_settings` filtrado por `tenant_id`.
3. El admin modifica uno o más campos:
   - `tenant_name`: Nombre legal de la organización.
   - `trade_name`: Nombre comercial (opcional).
   - `tax_id`: Identificador fiscal (NIT/RUC/CNPJ según país).
   - `timezone`: Timezone IANA (ej: `America/Bogota`).
   - `currency_code`: Código ISO 4217 (ej: `COP`, `MXN`, `USD`).
   - `date_format`: Formato de fecha (`DD/MM/YYYY`, `MM/DD/YYYY`, `YYYY-MM-DD`).
   - `time_format`: Formato de hora (`12h`, `24h`).
   - `language_code`: Código ISO 639-1 (`es`, `en`, `pt`).
   - `address`: Dirección fiscal completa.
   - `phone`: Teléfono de contacto.
   - `email_support`: Email de soporte visible para usuarios.
4. El admin hace clic en "Guardar cambios".
5. El sistema valida cada campo:
   - `timezone` debe ser un timezone IANA válido.
   - `currency_code` debe existir en la tabla `currencies`.
   - `tax_id` no debe pertenecer a otro tenant.
   - `email_support` debe tener formato válido.
6. Se persiste la actualización en `tenant_settings`.
7. Se publica evento `CONFIG_TENANT_UPDATED` en el message broker.
8. Los servicios relevantes invalidan su caché de configuración y recargan.
9. Se registra en `audit_log`: `{user_id, tenant_id, action: 'UPDATE_TENANT_CONFIG', changes: {...}, timestamp}`.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tenant_name | string | Razón social de la organización | Sí |
| trade_name | string | Nombre comercial | No |
| tax_id | string | Identificador fiscal (NIT/RUC/CNPJ) | Sí |
| timezone | string | Timezone IANA válido (ej: `America/Bogota`) | Sí |
| currency_code | string | Código ISO 4217 (3 letras) | Sí |
| date_format | string | Formato de fecha (`DD/MM/YYYY`, `MM/DD/YYYY`, `YYYY-MM-DD`) | Sí |
| time_format | string | Formato de hora (`12h`, `24h`) | Sí |
| language_code | string | Código ISO 639-1 (2 letras) | Sí |
| address | string | Dirección fiscal completa | No |
| phone | string | Teléfono de contacto (formato internacional) | No |
| email_support | string | Email de soporte | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| tenant_id | uuid | ID del tenant |
| config | object | Objeto completo con la configuración guardada |
| updated_at | timestamp | Timestamp de última modificación |
| updated_by | uuid | ID del usuario que realizó el cambio |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Timezone enviado no es válido según IANA | Error 400: "El timezone proporcionado no es válido. Use formato IANA (ej: America/Bogota)." |
| `currency_code` no existe en tabla `currencies` | Error 400: "Código de moneda no soportado." |
| `tax_id` ya existe en otro tenant | Error 409: "Este identificador fiscal ya está registrado en otro tenant." |
| `email_support` no tiene formato válido | Error 400: "Formato de email inválido." |
| Se intenta cambiar `currency_code` con transacciones activas | Warning: "Hay transacciones activas. El cambio de moneda puede afectar los cálculos. ¿Desea continuar?" |
| El admin no tiene permisos de edición | Error 403: "No tiene permisos para editar la configuración general." |
| Conexión a base de datos falla durante el guardado | Error 500: "No se pudo guardar la configuración. Intente nuevamente." |

---

## Criterios de Aceptación

1. El `tenant_admin` puede ver y editar todos los campos de configuración general de su tenant.
2. La actualización de `timezone` se refleja inmediatamente en todos lostimestamps mostrados al usuario.
3. La actualización de `currency_code` se usa en la generación de reportes y facturas.
4. El campo `tax_id` es único entre todos los tenants del sistema.
5. Cada cambio de configuración genera un registro en `audit_log` con el detalle de los campos modificados.
6. Un evento `CONFIG_TENANT_UPDATED` se publica en el message broker tras cada guardado exitoso.
7. Los servicios invalidate su caché de configuración al recibir el evento.
8. Los campos `tenant_name`, `timezone`, `currency_code` y `email_support` son obligatorios.
9. El sistema valida el formato IANA del timezone antes de persistir.

- `GET /api/v1/tenants/{tenant_id}/settings` — Obtener configuración general
- `PUT /api/v1/tenants/{tenant_id}/settings` — Actualizar configuración general
- `PATCH /api/v1/tenants/{tenant_id}/settings` — Actualización parcial de configuración

---

## Métricas de Éxito

- Tiempo de guardado de configuración: < 500 ms (excluyendo propagación a servicios).
- Cero configuraciones con timezone inválido persisted.
- 100% de cambios de configuración registrados en `audit_log`.
- Cero duplicados de `tax_id` entre tenants.

---

## Notas

- Se usa caché a nivel de servicio con TTL de 5 minutos para la configuración del tenant. El evento `CONFIG_TENANT_UPDATED`打破了缓存链，确保所有服务立即看到新配置。
- El campo `timezone` afecta la interpretación de todas las marcas de tiempo ingresadas sin zona horaria explícita.
- La moneda definida aquí es la moneda predominante del tenant; las transacciones pueden usar otras monedas si están configuradas en `currency_rates`.
- No se permite eliminar la configuración general; solo actualizarla.
