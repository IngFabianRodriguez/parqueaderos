# RF-087 — Configuración de Métodos de Pago por Sede

## Metadata

- **Módulo**: Configuración — Medios de Pago
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: payment-service, billing-service, transaction-service, config-service

---

## User Story

**Como** administrador del tenant **quiero** configurar qué métodos de pago acepta cada sede **para** ofrecer a los clientes las opciones de pago correctas (efectivo, tarjeta, app, etc.).

---

## Objetivo

El sistema debe permitir al `tenant_admin` configurar, por sede, los métodos de pago disponibles para el cobro de estacionamiento (efectivo, tarjetas de crédito/débito, billeteras digitales, pago previo via app, etc.). Cada método de pago tiene configuraciones específicas (por ejemplo, comisión, número de terminal, cuenta destino) y puede estar activo o inactivo por sede.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Cajero (usa los métodos configurados al cobrar)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.
- La sede existe y pertenece al tenant.

---

## Postcondiciones

### En caso de éxito

- Los métodos de pago habilitados para la sede se guardan en `venue_payment_methods`.
- El sistema activa/desactiva los botones de pago correspondientes en el POS y la app.
- El evento `PAYMENT_METHODS_UPDATED` se publica.

### En caso de falla

- Si se intenta habilitar un método de pago sin tener la configuración completa (ej: terminal ID para tarjetas): error de validación.
- Si se intenta eliminar un método de pago usado en transacciones pendientes: error 409.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Configuración → Sedes → {nombre_sede} → Métodos de Pago`.
2. El sistema muestra los métodos de pago disponibles globalmente y su estado por sede.
3. El admin habilita/deshabilita métodos de pago por sede toggling el switch correspondiente.
4. Para cada método habilitado, el admin completa la configuración específica:

   **Efectivo**:
   - `cash_enabled`: true
   - `cash_box_amount`: Monto inicial en caja (opcional)
   - `change_amount`: Monto máximo de cambio disponible (opcional)

   **Tarjeta de Crédito/Débito (POS físico)**:
   - `card_enabled`: true
   - `terminal_id`: ID de la terminal bancaria
   - `acquirer_code`: Código del adquirente (ej: `PROSA`, `REDeban`)
   - `commission_rate`: Porcentaje de comisión (ej: 2.5%)

   **Pago con App (digital wallet)**:
   - `app_payment_enabled`: true
   - `app_provider`: Nombre del proveedor (ej: `MercadoPago`, `PayPal`, `Stripe`)
   - `app_merchant_id`: ID del comercio en el proveedor
   - `app_api_key`: Referencia a la clave API (no el valor directo, sino su ID en `secrets`)
   - `app_enabled_banks`: Lista de bancos válidos (opcional)

   **Pago QR**:
   - `qr_enabled`: true
   - `qr_provider`: Proveedor del QR (ej: `CoDi`, `SPEI`)
   - `qr_cvu`: CVU de la cuenta destino
   - `qr_reference_prefix`: Prefijo para la referencia de pago

   **Prepago (cobro anticipado via app)**:
   - `prepaid_enabled`: true
   - `prepaid_minutes`: Minutos mínimos prepagados
   - `prepaid_max_minutes`: Minutos máximos prepagados

5. Al guardar:
   - Se valida que los campos requeridos por cada método estén presentes.
   - Se verifica que los `terminal_id`, `app_merchant_id` no estén duplicados en otra sede (si aplica).
6. Se persiste en `venue_payment_methods` y `payment_method_config`.
7. Se publica `PAYMENT_METHODS_UPDATED` con `{tenant_id, venue_id}`.

## Métodos de Pago Soportados

| Método | Tipo | Descripción | Configuración Requerida |
|--------|------|-------------|--------------------------|
| Efectivo | cash | Pago en caja física | cash_box_amount (opcional) |
| Tarjeta Crédito | card_credit | POS físico, marca | terminal_id, acquirer_code |
| Tarjeta Débito | card_debit | POS físico, marca | terminal_id, acquirer_code |
| App Wallet | digital_wallet | Pago desde app móvil | app_provider, app_merchant_id |
| QR CoDi/SPEI | qr | Pago con código QR | qr_provider, qr_cvu |
| Prepago App | prepaid | Cobro anticipado | prepaid_minutes, prepaid_max_minutes |
| Transferencia | bank_transfer | Transferencia bancaria manual | bank_account (referencia) |

| venue_id | uuid | ID de la sede | Sí |

|-------|------|-------------|
| venue_id | uuid | ID de la sede |
| payment_methods | array | Lista de métodos con su configuración |
| updated_at | timestamp | Última modificación |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| `terminal_id` duplicado en otra sede | Error: "Este terminal ID ya está en uso en otra sede." |
| Método habilitado sin config completa | Error: "Complete todos los campos requeridos para el método." |
| Método usado en transacción pendiente | Error 409: "No se puede deshabilitar. Hay transacciones pendientes con este método." |
| POS fuera de línea | El sistema marca el método como `offline` y muestra alerta al admin. |
| Comisión > 10% | Warning: "La comisión configurada es unusually alta. ¿Está seguro?" |
| `app_api_key` no existe en `secrets` | Error: "Referencia de API key inválida. Configure la key en Configuración → Secrets." |
| Prepago sin minutos configurados | Error: "Defina los minutos mínimos y máximos para prepago." |
| Efectivo sin caja abierta | El sistema permite cobrar en efectivo; se solicita abrir caja al cajero. |

---

## Criterios de Aceptación

1. El `tenant_admin` puede habilitar/deshabilitar métodos de pago por sede.
2. Cada método de pago tiene su propia configuración específica.
3. Los métodos deshabilitados no aparecen en el POS ni en la app del cliente.
4. El sistema valida que la configuración esté completa antes de habilitar un método.
5. Los `terminal_id` son únicos entre todas las sedes del tenant.
6. El evento `PAYMENT_METHODS_UPDATED` se publica al cambiar la configuración.
7. Las transacciones pendientes no bloquean la deshabilitación de un método; el método permanece activo hasta que no haya transacciones pendientes.
8. El sistema muestra una alerta cuando un POS está fuera de línea.
9. La comisión por tarjeta se calcula sobre el monto total de la transacción.

## Dependencias

**RFs relacionados**:
- RF-085 — Planes de tarifa (métodos de pago pueden variar por plan)
- RF-090 — Configuración de roles y permisos (solo `tenant_admin` puede modificar)
- RF-092 — Configuración de cajas y cuadres de caja (efectivo requiere gestión de caja)
- RF-003 — Control de acceso y barreras (el pago se verifica antes de abrir la talanquera de salida)

**Tablas de base de datos afectadas**:
- `venue_payment_methods` — métodos habilitados por sede
- `payment_method_config` — configuración específica de cada método
- `payment_terminals` — registro de terminales POS
- `secrets` — almacenamiento seguro de API keys (referencia por ID)
- `transactions` — registra el método de pago usado

---

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/venues/{venue_id}/payment-methods` — Métodos de pago de una sede
- `PUT /api/v1/tenants/{tenant_id}/venues/{venue_id}/payment-methods` — Actualizar métodos de pago
- `PATCH /api/v1/tenants/{tenant_id}/venues/{venue_id}/payment-methods/{method_id}/toggle` — Habilitar/deshabilitar
- `POST /api/v1/tenants/{tenant_id}/payment-methods/validate-config` — Validar configuración de un método
- `GET /api/v1/tenants/{tenant_id}/payment-methods/terminals` — Listar terminales registradas

---

## Métricas de Éxito

- Cero transacciones intentadas con método de pago deshabilitado.
- 100% de los métodos habilitados tienen configuración completa.
- Terminales POS con uptime > 99% (medido por sync periódico).
- Tiempo de respuesta de consulta de métodos: < 30 ms con caché.

---

## Notas

- Las API keys y secretos de los proveedores de pago se almacenan en la tabla `secrets`, nunca en texto plano en `payment_method_config`.
- La configuración de pago en efectivo está relacionada con RF-092 (Gestión de caja y cuadres).
- Los métodos de pago definidos aquí se usan tanto en el POS de cobro como en la app de prepago.
- El sistema debe poder indicar al usuario cuál es el método de pago disponible más cercano (ej: "Pague en la caseta 2").
- Se recomienda implementar circuit breaker para integraciones con procesadores de pago externos (tarjetas, wallets). Si el servicio de pago está caído, se muestra "Pago en efectivo únicamente" y se marca la transacción como `payment_pending`.
