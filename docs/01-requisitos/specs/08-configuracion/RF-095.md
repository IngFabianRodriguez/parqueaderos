# RF-095 — Políticas de Bloqueo de Vehículos

## Metadata

- **Módulo**: Configuración — Reglas de Negocio
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: vehicle-service, billing-service, access-control-service, tenant-service

---

## User Story

**Como** administrador de tenant **quiero** configurar políticas de bloqueo de vehículos basadas en días de mora o monto mínimo pendiente **para** garantizar el pago de estacionamiento y proteger los ingresos del negocio.

---

## Objetivo

El sistema debe permitir a los administradores de tenant definir y gestionar políticas de bloqueo de vehículos. Un vehículo se bloquea cuando incumple las condiciones definidas (días en mora o monto mínimo pendiente). Las políticas definen umbrales de bloqueo (cuándo se bloquea) y pueden incluir condiciones de desbloqueo automático (pago parcial, tiempo).

---

## Actores

- **Actor principal**: Administrador de tenant (configura las políticas)
- **Actor secundario**: Sistema (evalúa y ejecuta bloqueos/desbloqueos automáticos)
- **Actor terciario**: Operador de estacionamiento (recibe alertas, puede realizar desbloqueo manual)

---

## Precondiciones

- El tenant tiene el módulo de facturación activo.
- Existen vehículos registrados en el sistema.

---

## Postcondiciones

### En caso de éxito

- La política se guarda y se activa para evaluación.
- Los vehículos que cumplan los criterios son bloqueados según la política.
- Se genera un evento `vehicle_blocked` o `vehicle_unblocked`.

### En caso de falla

- La política no se guarda y se retorna error de validación.
- El sistema continúa evaluando con las políticas anteriores.

## Estructura de Política de Bloqueo

| Campo | Tipo | Descripción |
|-------|------|-------------|
| policy_id | uuid | Identificador único |
| name | string | Nombre de la política |
| description | string | Descripción |
| block_trigger | enum | `days_overdue`, `amount_overdue`, `combined` |
| days_overdue_threshold | integer | Días de mora para bloquear |
| amount_overdue_threshold | decimal | Monto mínimo pendiente para bloquear |
| applies_to | enum | `all_vehicles`, `registered_only`, `unregistered_only` |
| vehicle_types | array | Tipos de vehículo afectados (null = todos) |
| zones | array | Zonas donde aplica (null = todas) |
| grace_period_hours | integer | Horas de gracia antes del bloqueo |
| auto_unlock_on_payment | boolean | ¿Se desbloquea automáticamente al pagar? |
| partial_payment_unlocks | boolean | ¿Un pago parcial desbloquea? |
| auto_unlock_after_days | integer | Días para desbloqueo automático (0 = nunca) |
| notification_before_block | boolean | ¿Enviar notificación antes de bloquear? |
| notification_hours_before | integer | Horas antes del bloqueo para notificar |
| active | boolean | Si la política está activa |
| priority | integer | Prioridad de la política (mayor = primero) |
| effective_from | datetime | Inicio de vigencia |
| effective_until | datetime | Fin de vigencia (null = indefinida) |

---

## Estados de un Vehículo

| Estado | Descripción |
|--------|-------------|
| `active` | Vehículo sin restricciones; acceso normal |
| `pending_payment` | Tiene saldo pendiente dentro del período de gracia |
| `blocked` | Bloqueado; no puede entrar/salir hasta regularizar |
| `temporarily_unblocked` | Desbloqueado temporalmente (pago parcial) |

## Flujo Funcional

### Configuración de política de bloqueo

1. El administrador accede a **Configuración → Bloqueos → Políticas de Bloqueo**.
2. Crea una nueva política o edita una existente.
3. Define el tipo de disparador: días, monto, o ambos combinados.
4. Configura los umbrales: días de mora y/o monto mínimo pendiente.
5. Establece el período de gracia y las condiciones de desbloqueo.
6. Configura si envía notificaciones previas al bloqueo.
7. Guarda la política.
8. El sistema valida y persiste en `block_policies`.
9. Se activa el job de evaluación de bloqueos con la nueva política.

### Evaluación y ejecución de bloqueos (job periódico)

1. El job `block-policy-evaluator` se ejecuta cada hora.
2. Consulta las políticas activas de `block_policies`.
3. Para cada vehículo registrado:
   a. Obtiene el saldo pendiente y días en mora.
   b. Evalúa cada política por prioridad.
   c. Si un vehículo cumple los criterios de bloqueo y no está bloqueado:
      - Envía notificación previa (si está configurada).
      - Espera el período de gracia.
      - Si sigue en condiciones de bloqueo: cambia estado a `blocked`.
      - Genera evento `vehicle_blocked`.
4. Para vehículos bloqueados:
   a. Verifica si cumple condiciones de desbloqueo automático.
   b. Si `auto_unlock_on_payment=true` y se detectó pago: desbloquea.
   c. Si `auto_unlock_after_days > 0` y superó ese período: desbloquea.
   d. Si se desbloquea: genera evento `vehicle_unblocked`.

### Pago que desbloquea vehículo

1. Un vehículo tiene estado `blocked` y realiza un pago (RF-010).
2. El billing-service procesa el pago.
3. Si `auto_unlock_on_payment=true`: se actualiza saldo pendiente.
4. Si el saldo pendiente queda en $0 (o bajo umbral):
   - El vehículo cambia a `active` o `temporarily_unblocked` (según `partial_payment_unlocks`).
   - Se genera evento `vehicle_unblocked`.
   - Se envía notificación al propietario.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Vehículo bloqueado hace pago parcial | Según `partial_payment_unlocks`; si false, sigue bloqueado |
| El sistema se reinicia durante evaluación | El job se reanuda; no se duplican bloqueos (verificación de estado previo) |
| Dos políticas aplican al mismo vehículo | Se aplica la de mayor prioridad; si hay empate, la más reciente |
| Política con umbral muy bajo (ej: $0.01) | Se activa el bloqueo casi inmediatamente; el sistema muestra warning |
| Desbloqueo automático falla (error de sistema) | Se marca como `unblock_pending`; se reintenta en el siguiente ciclo |
| Un vehículo tiene múltiples facturas pendientes | Se suma el total pendiente para evaluar el umbral de monto |
| El propietario paga antes del bloqueo efectivo | El vehículo no se bloquea; el pago se registra normalmente |
| Política deshabilitada con vehículos ya bloqueados | Los vehículos permanecen bloqueados hasta que otra política los desbloquee |

---

## Criterios de Aceptación

1. Un administrador puede crear, editar y eliminar políticas de bloqueo.
2. El sistema bloquea vehículos automáticamente cuando se cumplen los criterios de la política.
3. Los bloqueos se aplican según la prioridad de políticas cuando hay múltiples.
4. El período de gracia se respeta antes de ejecutar el bloqueo.
5. Las notificaciones previas al bloqueo se envían con la anticipación configurada.
6. El desbloqueo automático por pago funciona según la configuración.
7. Un vehículo bloqueado no puede registrar entradas ni salidas (el control de acceso lo impide).
8. Los eventos de bloqueo y desbloqueo se registran para auditoría.
9. El sistema soporta múltiples políticas con diferentes disparadores y umbrales.

## Métricas de Éxito

- Porcentaje de vehículos bloqueados sobre el total de vehículos con deuda: target < 5%.
- Tiempo promedio entre que se cumplen los criterios y el bloqueo efectivo: < grace_period + 1 hora.
- Tasa de desbloqueo automático por pago: > 80% de bloqueos se resuelven con pago.
- Número de bloqueos manuales por mes (indica si las políticas automáticas están funcionando).
- Tiempo promedio de permanencia en estado bloqueado: target < 48 horas.

---

## Notas

- Se recomienda tener al menos una notificación antes del bloqueo para dar chance al usuario de pagar.
- El `grace_period_hours` es importante para evitar bloqueos por retrasos menores de pago.
- Las políticas con `combined` (días + monto) requieren que se cumplan ambos umbrales.
- El desbloqueo automático por pago partial es útil para negocios con múltiples facturas.
- Se recomienda revisar mensualmente los vehículos bloqueados para identificar patrones.
