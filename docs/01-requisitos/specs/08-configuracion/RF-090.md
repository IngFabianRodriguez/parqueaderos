# RF-090 — Configuración de Roles y Permisos de Usuario

## Metadata

- **Módulo**: Configuración — Seguridad y Accesos
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: auth-service, user-service, config-service, tenant-service

---

## User Story

**Como** administrador del tenant **quiero** definir los roles y permisos que controlan qué acciones puede realizar cada usuario **para** garantizar la seguridad y limitar el acceso a funcionalidades según la responsabilidad de cada persona.

---

## Objetivo

El sistema debe permitir al `tenant_admin` configurar roles (ej: Operador de Caseta, Supervisor, Administrador, Cajero) y asignar permisos específicos a cada rol (ej: `open_gate`, `process_refund`, `edit_rates`, `view_reports`). Los roles son globales al tenant y se asignan a usuarios. Un usuario puede tener múltiples roles. El sistema usa este modelo de RBAC (Role-Based Access Control) para autorizar acciones en todos los endpoints y en la interfaz.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Sistema (evalúa permisos en cada acción del usuario)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.

---

## Postcondiciones

### En caso de éxito

- Los roles se guardan en `roles`.
- Los permisos se guardan en `permissions`.
- Las asignaciones rol-permiso se guardan en `role_permissions`.
- Los usuarios existentes con ese rol reciben los nuevos permisos al hacer login o al refrescar tokens.

### En caso de falla

- Si se intenta eliminar un rol que tiene usuarios asignados: error 409 (primero reasignar usuarios).
- Si se intenta revocar un permiso que está siendo usado activamente en una sesión: se revoca pero se permite la sesión actual.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Configuración → Seguridad → Roles y Permisos`.
2. El sistema muestra la lista de roles del tenant con su descripción y cantidad de usuarios.
3. El admin puede crear un nuevo rol:
   - `role_name`: Nombre del rol (ej: "Cajero Noche", "Supervisor de Turno").
   - `role_code`: Código único (ej: `CAJERO_NOCHE`, `SUPERVISOR`).
   - `description`: Descripción del rol.
   - `is_active`: booleano.
4. Luego, dentro del rol, el admin ve los permisos disponibles agrupados por módulo:
   - **Acceso**: `open_gate`, `close_gate`, `emergency_open`, `override_barrier`.
   - **Caja**: `process_payment`, `issue_refund`, `close_cashier_shift`, `view_cashier_reports`.
   - **Configuración**: `edit_rates`, `edit_schedule`, `edit_payment_methods`, `edit_taxes`.
   - **Usuarios**: `create_user`, `edit_user`, `deactivate_user`, `assign_role`.
   - **Reportes**: `view_reports`, `export_reports`, `view_financial_reports`.
   - **B2B**: `manage_companies`, `manage_fleet_vehicles`, `view_fleet_reports`.
   - **Admin**: `tenant_settings`, `billing_access`, `api_keys_management`.
5. El admin marca los permisos que desea asignar al rol.
6. Se pueden crear roles adicionales además de los predefinidos del sistema:
   - Los roles predefinidos (`tenant_admin`, `operator`, `cashier`) no pueden eliminarse, solo editarse.
7. Al guardar:
   - Se valida que `role_code` sea único dentro del tenant.
   - Se valida que al menos un permiso esté asignado.
8. Se persiste en `roles`, `permissions`, `role_permissions`.
9. Se publica `ROLES_UPDATED`.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| `role_code` ya existe en el tenant | Error 409: "Ya existe un rol con este código." |
| Rol tiene usuarios asignados y se intenta eliminar | Error 409: "Reasigne los usuarios antes de eliminar el rol." |
| Se desactiva un rol con usuarios activos | Permitir; los usuarios con ese rol pierden acceso en su próximo login. |
| Un permiso se revoca mientras está siendo usado en una sesión activa | El permiso se revoca inmediatamente; el usuario debe volver a login para actualizar sus claims. |
| Se intenta modificar el `role_code` de un rol predefinido | No permitido; los roles predefinidos tienen código inmutable. |
| Usuario con múltiples roles | El sistema une todos los permisos de todos sus roles (OR). |
| Permiso no existe | Error: "Uno o más permisos seleccionados no son válidos." |
| Rol sin permisos asignados | Error: "El rol debe tener al menos un permiso." |

---

## Criterios de Aceptación

1. El `tenant_admin` puede crear roles personalizados además de los predefinidos del sistema.
2. Cada rol tiene un `role_code` único dentro del tenant.
3. Los permisos se asignan a roles; los usuarios heredan permisos a través de sus roles.
4. Un usuario puede tener múltiples roles; los permisos se acumulan.
5. Los roles predefinidos (`tenant_admin`, `operator`, `cashier`) no pueden eliminarse.
6. El sistema verifica permisos en cada endpoint protected.
7. El evento `ROLES_UPDATED` se publica cuando se modifican roles o permisos.
8. Los cambios de permiso se reflejan en el próximo token refresh del usuario, no inmediatamente en sesiones activas (salvo que se use token con claim dinámicos).
9. El historial de cambios de roles se registra en `audit_log`.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/roles` — Listar roles
- `POST /api/v1/tenants/{tenant_id}/roles` — Crear rol
- `GET /api/v1/tenants/{tenant_id}/roles/{role_id}` — Detalle de rol con permisos
- `PUT /api/v1/tenants/{tenant_id}/roles/{role_id}` — Actualizar rol
- `PATCH /api/v1/tenants/{tenant_id}/roles/{role_id}/deactivate` — Desactivar rol

- `DELETE /api/v1/tenants/{tenant_id}/users/{user_id}/roles/{role_id}` — Revocar rol

---

## Métricas de Éxito

- Cero usuarios con acceso a funcionalidades no autorizadas (validación por equipo de security).
- Cero roles con `role_code` duplicado.
- Tiempo de evaluación de permiso en endpoint: < 5 ms (caché en memoria).
- 100% de los endpoints protected tienen verificación de permiso.

---

## Notas

- Se recomienda implementar permisos en memoria del servicio de API con refresh periódico (cada 5 min) o invalidate via evento `ROLES_UPDATED`.
- Los roles predefinidos del sistema se crean en el seed del tenant y tienen `is_system: true`.
- Para auditoria, cada evaluación de permiso denegado debe logged con `{user_id, permission_requested, resource, timestamp}`.
- La asignacion de permisos sigue el principio de minimum privilege: otorgar solo los permisos necesarios.
- Si un usuario tiene rol `tenant_admin` y cualquier otro rol, el rol `tenant_admin` domina (todos los permisos).
