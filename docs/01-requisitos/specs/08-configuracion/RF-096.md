# RF-096 — Configuración de Webhooks Salientes

## Metadata

- **Módulo**: Configuración — Integraciones y Webhooks
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: webhook-service, tenant-service, notification-service

---

## User Story

**Como** administrador de tenant **quiero** configurar webhooks salientes con URL, eventos suscritos y headers de autenticación **para** recibir notificaciones en tiempo real en mis propios sistemas cuando ocurran eventos en ParkCore (ingresos, pagos, bloqueos, etc.).

---

## Objetivo

El sistema debe permitir a los administradores de tenant configurar uno o más webhooks salientes. Cada webhook define: la URL de destino, los eventos a los que se suscribe, los headers de autenticación (API key, Bearer token, HMAC signature), y opciones de retry. Cuando ocurre un evento configurado, el sistema envía una petición HTTP POST al endpoint con los datos del evento.

---

## Actores

- **Actor principal**: Administrador de tenant (configura webhooks)
- **Actor secundario**: Sistema (envía las notificaciones HTTP a los endpoints)

---

## Precondiciones

- El tenant tiene acceso al módulo de integraciones.

---

## Postcondiciones

### En caso de éxito

- El webhook se registra y queda activo.
- Se genera un evento `webhook_configured`.
- Los eventos suscritos comienzan a enviarse al endpoint.

### En caso de falla

- El webhook no se guarda y se retorna error de validación.
- El sistema continúa sin cambios.

## Estructura de un Webhook

| Campo | Tipo | Descripción |
|-------|------|-------------|
| webhook_id | uuid | Identificador único |
| name | string | Nombre descriptivo del webhook |
| description | string | Descripción para el admin |
| url | string | URL del endpoint (https://...) |
| events | array | Lista de eventos suscritos |
| headers | object | Headers HTTP personalizados |
| auth_type | enum | `none`, `api_key`, `bearer`, `hmac_sha256` |
| auth_value | string | Valor de autenticación (encriptado en DB) |
| hmac_secret | string | Secreto para HMAC (encriptado) |
| active | boolean | Si está activo |
| content_type | enum | `application/json`, `application/x-www-form-urlencoded` |
| timeout_seconds | integer | Timeout de la petición (default: 30) |
| retry_policy | object | Configuración de reintentos |
| created_at | datetime | Fecha de creación |
| updated_at | datetime | Última modificación |

## Flujo Funcional

### Configuración de webhook

1. El administrador accede a **Configuración → Integraciones → Webhooks**.
2. Crea un nuevo webhook.
3. Ingresa el nombre y la URL del endpoint.
4. Selecciona los eventos a los que se suscribe.
5. Configura la autenticación (tipo y valor/secret).
6. Ajusta el timeout y la política de reintentos.
7. Guarda la configuración.
8. El sistema valida la URL con una petición OPTIONS o HEAD.
9. Se persiste en `webhooks` y se activa.

### Envío de webhook cuando ocurre un evento

1. Ocurre un evento en el sistema (ej: `vehicle.exit`).
2. El servicio de eventos consulta los webhooks activos suscritos a ese evento.
3. Para cada webhook matcheado:
   a. Construye el payload JSON con los datos del evento.
   b. Añade headers: Content-Type, Authorization, X-ParkCore-Signature.
   c. Si `auth_type = hmac_sha256`: genera firma HMAC-SHA256 del payload.
   d. Envía POST HTTP al endpoint dentro del `timeout_seconds`.
   e. Registra el intento en `webhook_deliveries`.

### Retry en caso de fallo

1. Si el endpoint retorna código de error (>= 400) o timeout:
   a. Se registra como `failed`.
   b. Según la `retry_policy`, se programa reintento.
2. Estrategia de retry por defecto: 3 intentos con backoff exponencial (1min, 5min, 30min).
3. Si después de todos los reintentos sigue fallando: marca como `dead` y notifica al admin.
4. Los webhooks en estado `dead` no se reintentan hasta que se reactiven manualmente.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| URL inválida o no reachable | Error de validación en creación; se sugiere hacer ping a la URL |
| Autenticación HMAC con secret incorrecto | El endpoint no reconocerá la firma; se puede hacer test (RF-097) |
| Endpoint retorna 200 pero con payload malformado | Se considera exitoso; el endpoint es responsable de su parsing |
| Endpoint retorna 410 (Gone) | Se marca como `dead` inmediatamente; no se reintenta |
| Payload muy grande (> 1MB) | Se trunca o se simplifica; se registra warning |
| Demasiados webhooks activos (> 20) | Se permite pero se muestra warning de rendimiento |
| SSL del endpoint inválido | Se reintenta sin SSL verification solo si configurado; warning en logs |
| Reintentos agotados | Se marca como `dead`, se notifica al admin, se pausa el envío |
| El servidor de ParkCore cae durante envío | El mensaje se encola; se procesa cuando el servicio se恢复 |

---

## Criterios de Aceptación

1. Un administrador puede crear, editar y eliminar webhooks.
2. Cada webhook puede suscribirse a uno o más eventos.
3. Se soportan 4 tipos de autenticación: none, API key, Bearer token, HMAC-SHA256.
4. Los payloads de los eventos incluyen todos los datos relevantes en JSON.
5. Las solicitudes incluyen header `X-ParkCore-Signature` para verificación de origen.
6. Los reintentos automáticos se ejecutan con backoff exponencial.
7. Cada intento de entrega se registra con: timestamp, status code, response body (si error).
8. Los webhooks se pueden pausar y reanudar sin eliminar la configuración.
9. El sistema notifica al admin cuando un webhook entra en estado `dead`.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/webhooks` — Listar webhooks
- `POST /api/v1/tenants/{tenant_id}/webhooks` — Crear webhook
- `PUT /api/v1/tenants/{tenant_id}/webhooks/{id}` — Actualizar webhook
- `DELETE /api/v1/tenants/{tenant_id}/webhooks/{id}` — Eliminar webhook
- `POST /api/v1/tenants/{tenant_id}/webhooks/{id}/pause` — Pausar webhook
- `POST /api/v1/tenants/{tenant_id}/webhooks/{id}/resume` — Reanudar webhook
- `GET /api/v1/tenants/{tenant_id}/webhooks/{id}/deliveries` — Ver historial de entregas

## Métricas de Éxito

- Tasa de entrega exitosa: > 98% (200 response en primer intento).
- Tiempo promedio de entrega: < 2 segundos desde el evento.
- Número de webhooks activos por tenant (benchmark: 3-10).
- Tasa de webhooks en estado `dead`: < 2%.

---

## Notas

- Los webhooks son síncronos: el evento no se considera "procesado" hasta que el delivery intenta enviarse.
- Para eventos de alto volumen (ej: `vehicle.entry`), el admin debe ser consciente del tráfico que generará.
- La firma HMAC-SHA256 permite al endpoint verificar que el request vino realmente de ParkCore.
- Se recomienda usar HTTPS siempre; HTTP es permitido solo para desarrollo local.
- El `X-ParkCore-Signature` header contiene: `sha256=<hmac_hex>` del body completo.
