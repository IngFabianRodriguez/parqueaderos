# RF-092 — Configuración de Horarios de Notificaciones Batch

## Metadata

- **Módulo**: Configuración — Notificaciones
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: notification-service, scheduler-service, tenant-service

---

## User Story

**Como** administrador de un tenant **quiero** configurar los horarios de envío de notificaciones batch (como resúmenes diarios) **para** que los recordatorios y alertas se envíen en horarios convenientes para los operadores y propietarios de vehículos.

---

## Objetivo

El sistema debe permitir a los administradores de tenant configurar uno o más horarios programados para el envío de notificaciones batch (resúmenes de actividad, recordatorios de pago, alertas de mora). Estos horarios deben ser configurables por tipo de notificación y deben soportar zonas horarias específicas del tenant.

---

## Actores

- **Actor principal**: Administrador de tenant (configura los horarios)
- **Actor secundario**: Sistema (ejecuta los jobs programados según la configuración)

---

## Precondiciones

- El tenant tiene el módulo de notificaciones activo.
- Existe al menos una configuración de canal de notificación configurada (RF-083).

---

## Postcondiciones

### En caso de éxito

- Los horarios de notificación batch quedan almacenados y activos.
- El scheduler toma en cuenta la zona horaria del tenant para la ejecución.
- Se genera un evento `notification_schedule_updated`.

### En caso de falla

- La configuración no se persiste y se retorna error de validación.
- El scheduler continúa usando la última configuración válida.

---

## Tipos de Notificación Batch Soportados

| Tipo | Descripción | Frecuencia por defecto |
|------|-------------|------------------------|
| `daily_summary` | Resumen diario de ingresos y ocupación | Diaria |
| `payment_reminder` | Recordatorio de pagos pendientes | Diaria |
| `overdue_alert` | Alerta de vehículos en mora | Cada 6 horas |
| `occupancy_report` | Reporte de ocupación del estacionamiento | Diaria |
| `custom` | Plantilla personalizada (RF-091) | Según configuración |

---

## Flujo Funcional

### Configuración de horario por el administrador

1. El administrador accede a **Configuración → Notificaciones → Horarios Batch**.
2. Selecciona el tipo de notificación batch a programar.
3. Define la hora local de envío (formato HH:MM, ej: `08:00`).
4. Selecciona los días de la semana aplicables (Lun-Dom o días específicos).
5. Opcionalmente define la zona horaria si difiere de la del tenant (por defecto: zona horaria del tenant).
6. Guarda la configuración.
7. El sistema valida que no haya conflicto con otro horario del mismo tipo.
8. Se persiste en `notification_schedules` y se activa el job en el scheduler.

### Ejecución del job programado

1. El scheduler consulta `notification_schedules` para obtener las tareas activas.
2. Para cada.schedule con hora y zona horaria local, calcula el próximo UTC execution time.
3. Cuando llega el momento UTC, el scheduler dispara el job `batch-notification-sender`.
4. El job obtiene los datos relevantes (resumen diario, alertas pendientes, etc.).
5. Se envían las notificaciones a los destinatarios correspondientes.
6. Se registra en `notification_logs` con `schedule_type` y `batch_id`.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| notification_type | enum | Tipo de notificación batch | Sí |
| time | string | Hora local de envío (HH:MM) | Sí |
| days_of_week | array | Días de la semana [0-6] (0=Domingo) | Sí |
| timezone | string | Zona horaria IANA (ej: America/Bogota) | No (default: tenant timezone) |
| enabled | boolean | Si el schedule está activo | Sí |
| metadata | object | Datos adicionales (plantilla, filtros) | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| schedule_id | uuid | Identificador único del schedule |
| notification_type | enum | Tipo configurado |
| time_local | string | Hora local configurada |
| timezone | string | Zona horaria aplicada |
| next_execution_utc | datetime | Próxima ejecución en UTC |
| days_of_week | array | Días activos |
| enabled | boolean | Estado activo/inactivo |
| created_at | datetime | Fecha de creación |
| updated_at | datetime | Última modificación |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Hora inválida (ej: 25:00) | Retornar error 400 con mensaje de validación |
| Conflicto de horario (mismo tipo, misma hora) | Actualizar el schedule existente en lugar de crear duplicado |
| Zona horaria inválida | Usar la zona horaria por defecto del tenant y notificar al admin |
| Schedule deshabilitado | El scheduler lo ignora; no se ejecutan jobs |
| El servidor se reinicia durante ejecución | El job se marca como `interrupted`; se reintenta en la siguiente ventana |
| Doble ejecución (clock drift) | Se usa lock distribuido para evitar ejecuciones duplicadas |
| Cambio de zona horaria del tenant | Los schedules existentes se recalculan automáticamente |

---

## Criterios de Aceptación

1. Un administrador puede crear, editar y eliminar horarios de notificación batch desde el panel de administración.
2. Los horarios se ejecutan en la zona horaria correcta del tenant.
3. Si un schedule tiene `enabled=false`, el scheduler no lo ejecuta.
4. El sistema soporta al menos 5 tipos de notificación batch configurables.
5. Se puede definir múltiples schedules para el mismo tipo (ej: recordatorio a las 8am y otro a las 6pm).
6. Los cambios en la configuración se reflejan en el scheduler en menos de 60 segundos.
7. Cada ejecución batch genera un registro en `notification_logs` con `batch_id` único.
8. El sistema alerta si un schedule no se ejecuta durante 3 ciclos consecutivos.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/notification-schedules` — Listar schedules de un tenant
- `POST /api/v1/tenants/{tenant_id}/notification-schedules` — Crear un schedule
- `PUT /api/v1/tenants/{tenant_id}/notification-schedules/{id}` — Actualizar schedule
- `DELETE /api/v1/tenants/{tenant_id}/notification-schedules/{id}` — Eliminar schedule
- `POST /api/v1/tenants/{tenant_id}/notification-schedules/{id}/test` — Enviar notificación de prueba

---

## Métricas de Éxito

- Porcentaje de schedules que se ejecutan dentro de una ventana de ±5 minutos de la hora programada.
- Tasa de entrega exitosa de notificaciones batch (> 95%).
- Número de notificaciones batch enviadas por día por tenant.
- Tiempo promedio de ejecución del job batch (target: < 30 segundos para 10,000 notificaciones).

---

## Notas

- La zona horaria es crítica: un tenant en Colombia (America/Bogota) que configura 8am recibe a las 8am hora local, no 8am UTC.
- Se recomienda no programar más de 3 schedules por tipo de notificación para evitar saturación.
- Los logs de ejecución batch incluyen `batch_id` para facilitar debugging.
- En caso de cambio de horario de verano (DST), el scheduler debe recalcular automáticamente las horas UTC afectadas.
