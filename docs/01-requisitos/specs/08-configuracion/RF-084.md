# RF-084 — Configuración de Horarios de Operación y Días Festivos

## Metadata

- **Módulo**: Configuración — Horarios y Festivos
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: config-service, rate-service, access-control-service

---

## User Story

**Como** administrador del tenant **quiero** definir los horarios de operación de mis sedes y los días festivos **para** que el sistema aplique las tarifas correctas y controle el acceso según el calendario operativo.

---

## Objetivo

El sistema debe permitir al `tenant_admin` configurar horarios de operación por sede (hora de apertura y cierre para cada día de la semana) y registrar días festivos con fecha, nombre y si la sede permanece abierta o cerrada ese día. Estos parámetros determinan cuándo una sede acepta ingresos/egresos y qué esquema tarifario aplicar.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Sistema (evalúa horarios en cada acceso vehicular)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.
- La sede (`venue`) existe y pertenece al tenant.

---

## Postcondiciones

### En caso de éxito

- Los horarios de operación se guardan en `venue_schedules`.
- Los días festivos se guardan en `venue_holidays`.
- El servicio de control de acceso recibe el evento `SCHEDULE_UPDATED` y actualiza sus reglas.

### En caso de falla

- Si la hora de cierre es anterior a la hora de apertura (sin overnight): error de validación.
- Si el día festivo ya existe para esa sede y fecha: error indicando duplicado.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Configuración → Sedes → {nombre_sede} → Horarios`.
2. El sistema carga los horarios actuales de la sede desde `venue_schedules`.
3. El admin configura para cada día de la semana (Lunes–Domingo):
   - `is_open`: booleano indicando si la sede opera ese día.
   - `open_time`: hora de apertura en formato `HH:MM` (24h).
   - `close_time`: hora de cierre en formato `HH:MM` (24h).
   - `is_overnight`: booleano indicando si la sede cierra al día siguiente.
4. Para días marcados como `is_open: false`, el sistema marca toda la jornada como cerrada.
5. El admin puede configurar horarios especiales (24h, horarios partido, etc.).
6. Luego, en la pestaña "Días Festivos", el admin puede:
   - Agregar un día festivo: `date` (YYYY-MM-DD), `name`, `is_closed` (booleano), `applies_to_all_venues` (booleano).
   - Si `applies_to_all_venues: true`, el festivo se replica a todas las sedes del tenant.
   - Si `is_closed: true`, la sede estará completamente cerrada ese día.
   - Si `is_closed: false`, se aplican horarios de un día domingo (configurable).
7. Al guardar:
   - Se validan que no existan rangos horarios vacíos para días marcados como `is_open`.
   - Se verifica que no exista un festivo duplicado para la misma sede y fecha.
8. Se persiste la información.
9. Se publica `SCHEDULE_UPDATED` con `{tenant_id, venue_id}`.

---

## Datos de Entrada

### Horarios por Día

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| day_of_week | integer | Día de la semana (1=Lunes, 7=Domingo) | Sí |
| is_open | boolean | ¿La sede opera este día? | Sí |
| open_time | string | Hora de apertura (formato HH:MM) | Sí si is_open=true |
| close_time | string | Hora de cierre (formato HH:MM) | Sí si is_open=true |
| is_overnight | boolean | ¿Cierra al día siguiente? | No (default: false) |

### Días Festivos

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| date | date | Fecha del festivo (YYYY-MM-DD) | Sí |
| name | string | Nombre o descripción del festivo | Sí |
| is_closed | boolean | ¿Sede cerrada este día? | Sí |
| applies_to_all_venues | boolean | ¿Aplica a todas las sedes del tenant? | Sí |
| venue_ids | array[uuid] | IDs de sedes específicas (si applies_to_all=false) | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| venue_id | uuid | ID de la sede |
| schedules | array | Lista de horarios por día |
| holidays | array | Lista de días festivos configurados |
| updated_at | timestamp | Última fecha de modificación |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| `close_time` < `open_time` sin `is_overnight` | Error: "La hora de cierre debe ser posterior a la hora de apertura, o marcar la sede como overnight." |
| Festivo agregado para fecha pasada | Permitir (los festivo históricos importan para reportes). |
| Día festivo duplicado para misma sede | Error 409: "Ya existe un festivo registrado para esta sede en esa fecha." |
| `applies_to_all_venues: true` con `venue_ids` no vacío | Se ignora `venue_ids` y se aplica a todas. |
| Horario configurado para una sede 24h | Se usa `open_time: 00:00`, `close_time: 23:59`, `is_overnight: false`. |
| Intento de acceso fuera del horario | El sistema rechaza el ingreso y registra `ACCESS_DENIED_SCHEDULE`. |
| Cambio de horario con vehículos dentro de la sede | Permitir el cambio; el vehículo podrá salir normalmente. |
| Festivo con fecha null o formato inválido | Error 400: "Fecha de festivo inválida." |

---

## Criterios de Aceptación

1. El `tenant_admin` puede configurar horarios diferentes para cada día de la semana por sede.
2. Los horarios se aplican tanto a ingresos como a egresos de vehículos.
3. Un vehículo que ingresa antes del cierre puede salir después (lógica overnight aplicada).
4. Los días festivos pueden configurarse globalmente (todas las sedes) o por sede individual.
5. El sistema genera `ACCESS_DENIED` cuando un vehículo intenta entrar o salir fuera del horario operativo.
6. Un evento `SCHEDULE_UPDATED` se publica cuando se modifican horarios o festivos.
7. Los festivo con `is_closed: false` aplican tarifas de día domingo.
8. El sistema permite consultar los horarios de una sede para una fecha dada, resolviendo festivo si aplica.
9. Los horarios se almacenan en timezone del tenant.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/venues/{venue_id}/schedules` — Obtener horarios de una sede
- `PUT /api/v1/tenants/{tenant_id}/venues/{venue_id}/schedules` — Actualizar horarios
- `GET /api/v1/tenants/{tenant_id}/holidays` — Listar días festivos del tenant
- `POST /api/v1/tenants/{tenant_id}/holidays` — Crear día festivo
- `DELETE /api/v1/tenants/{tenant_id}/holidays/{holiday_id}` — Eliminar día festivo
- `GET /api/v1/venues/{venue_id}/operational-hours?date=YYYY-MM-DD` — Consultar si la sede está abierta en una fecha

---

## Métricas de Éxito

- 100% de las sedes del tenant tienen al menos un horario configurado.
- Cero accesos denegados por horario cuando la sede debería estar abierta (falso positivo).
- Tiempo de respuesta de consulta de horarios: < 50 ms con caché.
- Cero festivo duplicados para la misma sede y fecha.

---

## Notas

- Se recomienda implementar caché Redis con TTL de 1 hora para `venue_schedules` e `venue_holidays`. El evento `SCHEDULE_UPDATED`打破了缓存，确保新鲜度。
- El servicio de control de acceso (talanqueras) debe tener la lógica de "cierre programado" y consultar horarios antes de cada transacción.
- En festivo con `is_closed: false`, se aplica el horario del domingo; si no hay horario dominical, se usa horario颗粒度 reducida (máximo 8h).
- Los festivo internacionales no aplican automáticamente; el admin debe configurarlos.
