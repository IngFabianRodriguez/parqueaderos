# RF-093 — Reglas de Tarifación Configurables

## Metadata

- **Módulo**: Configuración — Reglas de Negocio
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, pricing-service, tenant-service

---

## User Story

**Como** administrador de tenant **quiero** definir y modificar reglas de tarifación desde el panel de administración **para** controlar cómo se calcula el costo del estacionamiento, incluyendo fracciones de tiempo, topes máximos y tarifas especiales por fechas o condiciones específicas.

---

## Objetivo

El sistema debe permitir a los administradores de tenant configurar reglas de tarifación completas y flexibles. Estas reglas definen cómo se calcula el monto a cobrar según la duración del estacionamiento, con soporte para fracciones de tiempo (ej: cada 15 minutos), topes máximos diarios/mensuales, tarifas especiales (navidad, hora pico, fines de semana) y tarifas diferenciadas por zona del estacionamiento.

---

## Actores

- **Actor principal**: Administrador de tenant (crea y modifica reglas)
- **Actor secundario**: Sistema (aplica las reglas al calcular tarifas)

---

## Precondiciones

- El tenant tiene el módulo de facturación activo.
- El tenant tiene al menos una zona de estacionamiento configurada (RF-006).

---

## Postcondiciones

### En caso de éxito

- Las reglas de tarifación se almacenan y quedan activas para el cálculo de tarifas.
- Los cambios afectan inmediatamente a nuevos ingresos (no a los ya facturados).
- Se genera un evento `pricing_rules_updated`.

### En caso de falla

- La configuración no se persiste y se retorna error de validación.
- El sistema continúa usando las últimas reglas válidas.

## Estructura de Reglas de Tarifación

### Tarifa Base

| Campo | Tipo | Descripción |
|-------|------|-------------|
| rule_id | uuid | Identificador único |
| zone_id | uuid | Zona a la que aplica (null = todas) |
| vehicle_type | enum | `car`, `motorcycle`, `truck`, `all` |
| base_rate | decimal | Tarifa base (ej: 2.50 USD) |
| time_fraction | integer | Fracción de tiempo en minutos (ej: 15) |
| fraction_rate | decimal | Tarifa por fracción (ej: 0.50 USD) |
| minimum_charge | decimal | Costo mínimo por estadía |
| effective_from | datetime | Fecha de inicio de vigencia |
| effective_until | datetime | Fecha de fin de vigencia (null = indefinida) |

### Topes Máximos

| Campo | Tipo | Descripción |
|-------|------|-------------|
| rule_id | uuid | FK a tarifa base |
| cap_type | enum | `daily`, `monthly`, `fixed` |
| cap_amount | decimal | Monto máximo a cobrar |
| cap_duration_hours | integer | Horas máxima (ej: 8 horas) |

### Tarifas Especiales

| Campo | Tipo | Descripción |
|-------|------|-------------|
| rule_id | uuid | FK a tarifa base |
| special_type | enum | `holiday`, `peak_hour`, `weekend`, `seasonal` |
| multiplier | decimal | Factor multiplicador (ej: 1.5 para 50% más caro) |
| fixed_amount | decimal | Monto fijo替代 (si no es multiplicador) |
| date_pattern | string | Patrón de fechas (ej: "12-24 to 12-31" para navidad) |
| time_pattern | string | Patrón de horas (ej: "08:00-18:00" para hora pico) |
| day_of_week | array | Días de la semana aplicables [0-6] |

## Flujo Funcional

### Creación de regla de tarifación

1. El administrador accede a **Configuración → Tarifas → Reglas de Tarifación**.
2. Selecciona la zona y tipo de vehículo al que aplica.
3. Define la tarifa base (monto y fracción de tiempo).
4. Configura topes máximos (diarios, mensuales o fijo).
5. Opcionalmente, añade tarifas especiales (navidad, hora pico, etc.).
6. Define el período de vigencia (fechas de inicio y fin).
7. Guarda la regla.
8. El sistema valida que no haya conflicto con reglas existentes para el mismo período.
9. Se persiste en `pricing_rules` y se activa.
10. Se genera el evento `pricing_rules_updated`.

### Aplicación de reglas al calcular tarifa

1. Un vehículo registra salida (RF-009).
2. El billing-service consulta las `pricing_rules` activas para la zona y tipo de vehículo.
3. Se aplica la tarifa base según la duración registrada.
4. Se verifican y aplican topes máximos.
5. Se verifican tarifas especiales aplicables (fecha/hora actual).
6. Se calcula el monto final y se almacena en `invoices`.
7. Si la duración excede un tope, se aplica el monto máximo correspondiente.

### Evaluación de tarifas especiales

1. Se evalúa si la fecha/hora actual coincide con algún `date_pattern` o `time_pattern`.
2. Si hay múltiples tarifas especiales, se aplica la de mayor prioridad (configurable).
3. Las tarifas especiales pueden ser: multiplicador sobre la base o monto fijo.
4. Las tarifas especiales no aplican sobre el mínimo de estadía (si está configurado).

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Regla sin fracción de tiempo | Error de validación; la fracción es obligatoria |
| Tope máximo menor que tarifa base | El sistema permite pero muestra warning; puede resultar en cobros mínimos constantes |
| Dos reglas activas para el mismo período y zona | Error de conflicto; se debe desactivar una o definir prioridades |
| Tarifa especial con fecha pasada | La regla existe pero no se aplica; se marca como `expired` |
| Tipo de vehículo no definido en reglas | El sistema usa la regla con `vehicle_type = all` como fallback |
| Cambio de tarifa durante estadía activa | Se cobra según la tarifa vigente al momento del ingreso (locking) |
| Múltiples tarifas especiales coincidentes | Se aplica la de mayor multiplicador (o la primera según prioridad) |

---

## Criterios de Aceptación

1. Un administrador puede crear, editar, duplicar y eliminar reglas de tarifación.
2. Las reglas soporta fracciones de tiempo configurables (1, 5, 10, 15, 30, 60 minutos).
3. Los topes máximos (diarios/mensuales) se aplican correctamente al calcular la tarifa final.
4. Las tarifas especiales (navidad, hora pico, fines de semana) se aplican cuando corresponde.
5. Las reglas tienen período de vigencia y se auto-desactivan al expirar.
6. El sistema soporta múltiples zonas con diferentes tarifas en el mismo tenant.
7. Los cambios en reglas no afectan facturas ya emitidas ( historial inmutable).
8. Se puede configurar tarifa diferenciada por tipo de vehículo (carro, moto, camión).
9. La reglas se validan antes de guardar para evitar inconsistencias.

## Dependencias

**RFs relacionados**:
- RF-006 — Configuración de zonas de estacionamiento (necesaria para tarifas por zona)

- `pricing_rules` — Almacena las reglas de tarifación
- `pricing_special_rates` — Tarifas especiales asociadas a una regla
- `invoices` — Referencia la regla usada para el cálculo (historial)
- `zones` — Relación zona-tarifa

## Métricas de Éxito

- Tiempo promedio de cálculo de tarifa: < 50ms por transacción.
- Porcentaje de reglas de tarifación con al menos una tarifa especial activa: > 60%.
- Precisión en la aplicación de topes máximos: 100% de las facturas aplican el tope correcto.
- Número de reglas activas por tenant (benchmark: 5-20 reglas por zona).

---

## Notas

- La fracción de tiempo más común es 15 minutos para estacionamientos urbanos.
- Los topes mensuales son útiles para empleados con estacionamiento mensualizado.
- Las tarifas especiales de navidad (diciembre) y hora pico (8-10am, 5-8pm) son las más comunes.
- Se recomienda no tener más de 3 tarifas especiales por regla para mantener la legibilidad.
- El historial de reglas se mantiene para auditorías: nunca se borran, solo se marcan `inactive`.
