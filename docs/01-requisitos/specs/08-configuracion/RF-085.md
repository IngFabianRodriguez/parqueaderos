# RF-085 — Configuración de Tarifas y Planes de Precio

## Metadata

- **Módulo**: Configuración — Tarifas y Planes
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: rate-service, billing-service, transaction-service, access-control-service

---

## User Story

**Como** administrador del tenant **quiero** configurar las tarifas de estacionamiento por hora, fracciones y planes especiales **para** que el sistema cobre correctamente a los clientes según el tiempo de uso y tipo de vehículo.

---

## Objetivo

El sistema debe permitir al `tenant_admin` definir uno o más planes de tarifa por sede, donde cada plan especifica tarifa base, fracciones de cobro (minutos de gracia, duración de fracción), tarifas por exceso, y tarifas diferenciadas por categoría de vehículo. Un plan puede ser el default (aplica a todos los vehículos sin plan asignado) o asignarse a vehículos específicos, empresas con flota (B2B), o usuarios con suscripción.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Sistema (aplica tarifas en cada cierre de transacción)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.
- La sede existe y tiene horarios configurados (RF-084).

---

## Postcondiciones

### En caso de éxito

- El plan de tarifa se guarda en `rate_plans`.
- Las tarifas por categoría se guardan en `rate_plan_categories`.
- El evento `RATE_PLAN_UPDATED` se publica para que los servicios invaliden su caché.

### En caso de falla

- Si la fracción de cobro es 0 o negativa: error de validación.
- Si se intenta activar un plan sin tener al menos una tarifa definida: error.
- Si el plan a eliminar está en uso por transacciones activas: error 409.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Configuración → Tarifas → Planes de Precio`.
2. El sistema muestra los planes existentes para el tenant.
3. El admin puede crear un nuevo plan o editar uno existente:
   - `plan_name`: Nombre del plan (ej: "Tarifa Regular", "Tarifa Mensual", "Plan Corporativo").
   - `plan_type`: `standard`, `subscription`, `corporate`, `event` (determina lógica de cobro).
   - `venue_id`: Sede donde aplica (`null` = todas las sedes).
   - `is_active`: booleano.
   - `is_default`: booleano (solo un plan default por tenant).
   - `valid_from`: fecha de inicio de vigencia.
   - `valid_until`: fecha de fin de vigencia (null = indefinido).
4. Dentro del plan, el admin configura las fracciones de cobro:
   - `grace_minutes`: minutos de gracia antes de cobrar (ej: 5 min).
   - `fraction_minutes`: duración de cada fracción de cobro (ej: 15 min, 60 min).
   - `max_free_minutes`: máximo de minutos gratuitos (ej: 15 min = gratuito).
5. Para cada categoría de vehículo autorizada en el plan (RF-086):
   - `vehicle_category_id`: referencia a la categoría.
   - `base_rate`: tarifa base por la primera fracción.
   - `fraction_rate`: tarifa por cada fracción subsiguiente.
   - `hourly_cap`: tarifa máxima por hora (null = sin límite).
   - `daily_cap`: tarifa máxima por día (null = sin límite).
   - `overnight_rate`: tarifa fija si el vehículo pernocta (null = no aplica).
   - `sunday_rate`: tarifa especial para domingos (puede ser `null` = misma que base).
   - `holiday_rate`: tarifa especial para días festivos (puede ser `null` = sama que base).
6. Opcionalmente, se configuran reglas especiales:
   - `early_bird_rate`: tarifa fija si entra antes de `early_bird_cutoff` y sale antes de `early_bird_exit`.
   - `lost_ticket_fee`: tarifa cuando el usuario pierde el ticket.
   - `special_event_rate`: tarifa cuando la sede está en modo evento especial.
7. El admin guarda el plan. El sistema valida que:
   - `fraction_minutes` > 0.
   - `grace_minutes` >= 0.
   - Al menos una categoría tenga `base_rate` > 0.
   - No exista otro plan default activo si `is_default: true`.
8. Se persiste en `rate_plans` y `rate_plan_categories`.
9. Se publica `RATE_PLAN_UPDATED`.

## Tipos de Plan de Tarifa

| Plan | Descripción | Ejemplo de uso |
|------|-------------|----------------|
| `standard` | Cobro por tiempo real | Visitantes ocasionales |
| `subscription` | Tarifa fija mensual | Abonados mensuales |
| `corporate` | Tarifas B2B con límites | Empresas con flota |
| `event` | Tarifa plana por evento | Conciertos, estadios |

---

## Fracciones de Cobro — Ejemplos

| Fracción | grace_minutes | fraction_minutes | Uso |
|----------|---------------|------------------|-----|
| 15 min | 5 | 15 | Estacionamientos de alta rotación |
| 30 min | 5 | 30 | Centros comerciales |
| 1 hora | 10 | 60 | Oficinas, aeropuertos |
| 2 horas | 0 | 120 | Cines, restaurantes |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| `fraction_minutes` = 0 | Error: "La fracción de cobro debe ser mayor a 0." |
| Dos planes default activos para el mismo tenant | Error: "Solo puede haber un plan default activo. Desactive el actual primero." |
| Plan con `valid_until` < `valid_from` | Error: "La fecha de fin de vigencia debe ser posterior al inicio." |
| Plan en uso por transacciones activas | Error 409: "No se puede eliminar un plan con transacciones activas. Marque como inactivo." |
| Categoría sin `base_rate` definida | El sistema no cobra a esa categoría o muestra error según configuración. |
| Cambio de plan default con transacciones en curso | Las transacciones en curso mantienen la tarifa vigente al momento de ingreso. |
| `hourly_cap` < `fraction_rate` | Se ignora `hourly_cap` y se usa `fraction_rate` directamente. |
| Tarifa para día domingo no definida | Se usa la `base_rate` normal. |

---

## Criterios de Aceptación

1. El `tenant_admin` puede crear, editar, activar e inactivar planes de tarifa.
2. Cada plan puede tener tarifas diferentes por categoría de vehículo.
3. Solo existe un plan default activo por tenant en un momento dado.
4. El sistema aplica minutos de gracia antes de iniciar el cobro.
5. Las fracciones se cobran correctamente: primera fracción a `base_rate`, siguientes a `fraction_rate`.
6. Los `hourly_cap` y `daily_cap` limitan el cobro máximo por hora/día.
7. El evento `RATE_PLAN_UPDATED` se publica al crear, actualizar o eliminar un plan.
8. Los planes con fecha `valid_until` vencida se treatan como inactivos automáticamente.
9. Las transacciones abiertas heredan la tarifa del plan vigente al momento del ingreso, no al momento del egreso.

- `rate_plans` — cabecera del plan de tarifas
- `rate_plan_categories` — tarifas por categoría dentro de un plan

- `transactions` — las transacciones referencian el plan usado

## Métricas de Éxito

- Cero transacciones cobradas con tarifa incorrecta (validación post-cobro).
- Tiempo de cálculo de tarifa para transacción: < 10 ms.
- 100% de los planes con `is_default: true` tienen exactamente un plan activo por tenant.
- Cero planes activos con `valid_until` vencida.

---

## Notas

- El `rate-service` debe caching los planes activos con TTL de 5 min. El evento `RATE_PLAN_UPDATED`打破缓存。
- Para planes `subscription`, el sistema no cobra por tiempo sino que valida la suscripción activa (RF-XXX de Suscripciones).
- Para planes `corporate`, las tarifas pueden tener límites de gasto por vehículo (RF-080) o por empresa (RF-079).
- Las tarifas se almacenan en la moneda del tenant (RF-083). Si la transacción usa otra moneda, se convierte usando `currency_rates` del día.
- El sistema debe poder calcular una tariffa estimada antes del egreso, para mostrar al usuario en pantallas de confirmación.
