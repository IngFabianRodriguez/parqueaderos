# RF-089 — Configuración de Impuestos y Descuentos por Sede

## Metadata

- **Módulo**: Configuración — Impuestos y Descuentos
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: billing-service, rate-service, config-service, transaction-service

---

## User Story

**Como** administrador del tenant **quiero** configurar los impuestos aplicables y los descuentos promocionales por sede **para** que el sistema calcule el monto final correcto a cobrar al cliente.

---

## Objetivo

El sistema debe permitir al `tenant_admin` configurar los impuestos que se aplican sobre el subtotal de cada transacción (IVA, ISR, etc.) y los descuentos o promociones que pueden aplicarse (descuentos por tiempo, cupones, tarifas planas con descuento, etc.). Los impuestos y descuentos se configuran por sede, son acumulables, y tienen fechas de vigencia.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Sistema (aplica impuestos y descuentos en cada cálculo de total)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.
- La sede existe y pertenece al tenant.

---

## Postcondiciones

### En caso de éxito

- Los impuestos se guardan en `venue_taxes`.
- Los descuentos se guardan en `venue_discounts`.
- El evento `TAX_DISCOUNT_CONFIG_UPDATED` se publica.
- Las nuevas transacciones usan la configuración actualizada.

### En caso de falla

- Si el porcentaje de impuesto es < 0 o > 100: error de validación.
- Si se intenta eliminar un impuesto usado en transacciones facturadas: error 409.
- Si las fechas de vigencia del descuento están invertidas: error.

---

## Flujo Funcional

### Configuración de Impuestos

1. El `tenant_admin` accede a `Configuración → Sedes → {sede} → Impuestos`.
2. El sistema muestra los impuestos activos de la sede.
3. El admin puede agregar un impuesto:
   - `tax_name`: Nombre del impuesto (ej: "IVA 19%", "ISR 2%", "impuesto local").
   - `tax_type`: `percentage` (porcentaje) o `fixed` (monto fijo).
   - `tax_rate`: Porcentaje (ej: 19.0) o monto fijo según `tax_type`.
   - `applies_to`: `all` (todas las transacciones), `cash_only` (solo efectivo), `card_only` (solo tarjetas).
   - `is_default`: ¿Se aplica automáticamente a toda transacción? (si false, requiere activación manual).
   - `is_active`: booleano.
4. Se pueden agregar varios impuestos simultáneamente. El orden de aplicación es el orden en que se agregaron.
5. Al guardar: se valida que `tax_rate` sea válido según `tax_type`.

### Configuración de Descuentos

6. El admin accede a `Configuración → Sedes → {sede} → Descuentos`.
7. Puede crear un descuento:
   - `discount_name`: Nombre (ej: "15% OFF primeros 30 min", "Tarifa plana domingo").
   - `discount_type`: `percentage` (%), `fixed_amount` (monto fijo), `flat_rate` (tarifa plana).
   - `discount_value`: Valor (% o monto fijo según tipo).
   - `target`: `all_vehicles`, `specific_category` (una categoría), `subscribed_only`, `corporate_only`.
   - `target_category_id`: Si `target` = `specific_category`.
   - `discount_trigger`: `none` (siempre aplica), `time_based` (por horario), `duration_based` (si dura más de X min), `coupon_code` (requiere cupón).
   - `trigger_config`: JSON con parámetros del trigger (ej: `{"min_minutes": 60}` para `duration_based`).
   - `coupon_code`: Código de cupón (si `discount_trigger` = `coupon_code`).
   - `max_uses`: Número máximo de usos (null = ilimitado).
   - `max_uses_per_user`: Usos máximos por usuario (null = ilimitado).
   - `valid_from`, `valid_until`: Rango de fechas de vigencia.
   - `is_active`: booleano.
8. Al guardar: se valida que las fechas sean consistentes.
9. Se persiste en `venue_taxes` y `venue_discounts`.
10. Se publica `TAX_DISCOUNT_CONFIG_UPDATED`.

---

## Orden de Cálculo en Transacción

```
Subtotal (base_rate + fractions + extras)
  → Descuentos aplicados (en orden de prioridad)
  → Impuestos aplicados (en el orden configurado)
= Total a Cobrar
```

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| `tax_rate` > 100 (tipo percentage) | Error: "El porcentaje de impuesto no puede exceder 100%." |
| `tax_rate` < 0 | Error: "El porcentaje de impuesto no puede ser negativo." |
| Impuesto en uso en transacciones facturadas | Error 409: "El impuesto no puede eliminarse porque está en facturas emitidas." |
| Descuento con `valid_until` < `valid_from` | Error: "La fecha de fin debe ser posterior al inicio." |
| `coupon_code` ya existe | Error 409: "Este código de cupón ya está en uso." |
| Cupón alcanzado su `max_uses` | El sistema rechaza el uso y muestra "Cupón agotado." |
| Cupón vencido | El sistema rechaza el uso y muestra "Cupón vencido." |
| Descuento `flat_rate` > subtotal | El total se establece en 0 (mínimo a cobrar es 0). |
| Dos descuentos con mismo `coupon_code` | Error: "Código de cupón duplicado." |
| Descuento sin `valid_until` | Se treata como indefinido hasta que se desactive manualmente. |

## Criterios de Aceptación

1. El `tenant_admin` puede configurar múltiples impuestos por sede, cada uno con tipo y porcentaje/monto fijo.
2. Los impuestos pueden aplicar a todas las transacciones o solo a ciertos métodos de pago.
3. Los descuentos pueden ser百分比, monto fijo, o tarifa plana.
4. Los cupones de descuento tienen códigos únicos y controles de uso (max_uses, fecha de vigencia).
5. Los descuentos por duración (minutos) se aplican automáticamente si el vehículo supera el umbral.
6. Los impuestos se calculan sobre el subtotal después de aplicar descuentos.
7. El evento `TAX_DISCOUNT_CONFIG_UPDATED` se publica tras cada cambio.
8. Los impuestos y descuentos inactivos no se aplican pero permanecen en la BD para historial.
9. El sistema soporta impuestos incluidos (tax-included pricing) si se configura `tax_type: inclusive`.

## Métricas de Éxito

- Cero transacciones con impuestos calculados incorrectamente (validación post-cobro).
- Cero cupones usados más allá de su `max_uses`.
- Tiempo de cálculo de total con impuestos y descuentos: < 5 ms.
- 100% de los descuentos con cupón tienen código único.

---

## Notas

- Los impuestos siempre se calculan sobre el monto después de descuentos (no sobre el subtotal bruto).
- Si un impuesto es "included" (tax-included), el sistema back-calcula la base imponible: `base = total / (1 + rate)`.
- Para descuentos `flat_rate`, el monto final no puede ser negativo; si el descuento excede el subtotal, el total es 0.
- El campo `order` en impuestos define el orden de aplicación cuando hay múltiples impuestos acumulativos.
