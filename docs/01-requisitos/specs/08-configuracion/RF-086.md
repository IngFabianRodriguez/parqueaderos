# RF-086 — Configuración de Categorías de Vehículos y Tarifas Diferenciadas

## Metadata

- **Módulo**: Configuración — Categorías de Vehículos
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: vehicle-service, rate-service, access-control-service, transaction-service

---

## User Story

**Como** administrador del tenant **quiero** definir las categorías de vehículos que mis sedes aceptan y asociar tarifas diferenciadas a cada una **para** cobrar correctamente según el tipo de vehículo (moto, auto, camión, etc.).

---

## Objetivo

El sistema debe permitir al `tenant_admin` crear, editar e inactivar categorías de vehículos (ej: Motocicleta, Automóvil, Camioneta, Camión, Bus) y asignarles tarifas diferenciadas dentro de cada plan de precio. Cada categoría tiene una identificación única, descripción, y puede estar autorizada o no por sede. Las categorías son globales al tenant pero sus autorizadas varían por sede.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Sistema (clasifica vehículos en transacciones según su categoría)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.

---

## Postcondiciones

### En caso de éxito

- La categoría se crea en `vehicle_categories`.
- Las asociaciones sede-categoría se guardan en `venue_vehicle_categories`.
- Los planes de tarifa existentes pueden incluir esta categoría.

### En caso de falla

- Si se intenta eliminar una categoría en uso por vehículos registrados: error 409.
- Si la categoría ya existe con el mismo nombre en el tenant: error 409.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Configuración → Categorías de Vehículos`.
2. El sistema muestra las categorías existentes del tenant.
3. El admin puede crear una nueva categoría:
   - `category_name`: Nombre (ej: "Motocicleta", "Automóvil", "Camión").
   - `category_code`: Código único dentro del tenant (ej: `MOT`, `AUT`, `CAM`).
   - `description`: Descripción detallada.
   - `size_class`: `small`, `medium`, `large`, `extra_large` (para lógica de ocupación de espacios).
   - `default_rate_multiplier`: Multiplicador de tarifa por defecto (ej: 0.5 para motos, 1.5 para camiones).
   - `requires_special_access`: booleano (ej: para vehículos de carga que requieren autorización previa).
   - `is_active`: booleano.
4. Al guardar:
   - Se valida que `category_code` sea único dentro del tenant.
   - Se valida que `size_class` sea un valor válido del enum.
5. Luego, en la vista de la categoría, el admin puede marcar en qué sedes está autorizada:
   - Selecciona sedes donde la categoría tiene permitido el acceso.
   - Las sedes no seleccionadas heredan la configuración default del tenant.
6. Se persiste en `vehicle_categories` y `venue_vehicle_categories`.
7. Se publica `VEHICLE_CATEGORY_UPDATED`.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| `category_code` ya existe en el tenant | Error 409: "Ya existe una categoría con este código." |
| Categoría en uso por vehículos registrados | Error 409: "No se puede eliminar. Desactive la categoría en su lugar." |
| Categoría desactivada con transacciones abiertas | Permitir; las transacciones abiertas mantienen la categoría como estaba. |
| Sede no autorizada para una categoría | El sistema rechaza ingresos de vehículos de esa categoría en esa sede. |
| `default_rate_multiplier` = 0 | Error: "El multiplicador debe ser mayor a 0." |
| Cambio de `size_class` en categoría activa | Permitir; afecta solo nuevas transacciones. |
| Intentar crear categoría sin nombre | Error 400: "El nombre de la categoría es obligatorio." |

---

## Criterios de Aceptación

1. El `tenant_admin` puede crear, editar y desactivar categorías de vehículos.
2. Cada categoría tiene un `category_code` único dentro del tenant.
3. Las categorías pueden estar autorizadas o no por sede (relación N:M).
4. Los vehículos se clasifican obligatoriamente en una categoría al registrarse.
5. El `size_class` se usa para calcular ocupación de espacios en los reportes de ocupación.
6. El `default_rate_multiplier` es sugerido al crear tarifas, pero puede ser sobreescrito por plan.
7. Una categoría desactivada no aparece en los listados de selección, pero permanece en el historial.
8. El sistema rechaza el ingreso de un vehículo cuya categoría no está autorizada en la sede.
9. Los cambios de categoría se auditan en `audit_log`.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/vehicle-categories` — Listar categorías
- `POST /api/v1/tenants/{tenant_id}/vehicle-categories` — Crear categoría

- `PATCH /api/v1/tenants/{tenant_id}/vehicle-categories/{category_id}/deactivate` — Desactivar
- `GET /api/v1/venues/{venue_id}/vehicle-categories` — Categorías autorizadas por sede
- `PUT /api/v1/venues/{venue_id}/vehicle-categories` — Actualizar categorías autorizadas por sede

## Métricas de Éxito

- Cero categorías con `category_code` duplicado dentro del tenant.
- Cero vehículos intentando acceder a una sede que no autoriza su categoría (validación en frontera).
- Tiempo de carga de lista de categorías: < 100 ms.
- 100% de los vehículos registrados tienen una categoría válida.

---

## Notas

- La categoría "Automóvil" (`AUT`) se usa como referencia para el `default_rate_multiplier: 1.0`.
- Las categorías de vehículos son distintas de los tipos deabonados: una moto puede ser abonada o casual.
- El campo `requires_special_access` es para casos donde vehículos de carga o extragrandes requieren autorización del operador antes de ingresar.
- Se recomienda cachear las categorías autorizadas por sede con TTL de 5 min. El evento `VEHICLE_CATEGORY_UPDATED`打破缓存。
