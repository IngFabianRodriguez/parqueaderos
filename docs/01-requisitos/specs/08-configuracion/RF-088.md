# RF-088 — Configuración de Notificaciones y Plantillas de Comunicación

## Metadata

- **Módulo**: Configuración — Notificaciones
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: notification-service, config-service, tenant-service

---

## User Story

**Como** administrador del tenant **quiero** personalizar las plantillas de notificación que el sistema envía a mis clientes **para** mantener mi marca y comunicar información relevante de mi organización.

---

## Objetivo

El sistema debe permitir al `tenant_admin` personalizar las plantillas de correo electrónico, SMS y push notifications que se envían a los usuarios del tenant (clientes, operadores, administradores). Las plantillas usan un sistema de variables (placeholders) que se reemplazan en tiempo de ejecución. El admin puede activar/desactivar cada notificación y modificar el contenido mantendo las variables requeridas.

---

## Actores

- **Actor principal**: Administrador del tenant (`tenant_admin`)
- **Actor secundario**: Sistema (envía las notificaciones según configuración)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.

---

## Postcondiciones

### En caso de éxito

- Las plantillas se guardan en `notification_templates`.
- El evento `NOTIFICATION_TEMPLATES_UPDATED` se publica.
- Las notificaciones futuras usan las nuevas plantillas.

### En caso de falla

- Si una plantilla hace referencia a una variable inexistente: error de validación.
- Si el contenido del email/SMS excede el límite de caracteres: error.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Configuración → Notificaciones → Plantillas`.
2. El sistema muestra las categorías de notificación:
   - **Transaccionales**: ingreso, egreso, pago, factura.
   - **Administrativas**: alertas de caja, reportes programados.
   - **Marketing**: promociones, encuestas de satisfacción (si aplica).
3. El admin selecciona una plantilla para editar:
   - `template_name`: Identificador interno.
   - `channel`: `email`, `sms`, `push`, `in_app`.
   - `subject` (solo email): Asunto del mensaje.
   - `body`: Cuerpo del mensaje con soporte para placeholders `{{variable}}`.
   - `is_active`: booleano (habilita o deshabilita el envío de esta notificación).
   - `footer`: Pie de página (puede incluir dirección, teléfono — configurable por el admin).
4. Variables disponibles por tipo de notificación:
   - `{{customer_name}}` — Nombre del cliente.
   - `{{vehicle_plate}}` — Placa del vehículo.
   - `{{venue_name}}` — Nombre de la sede.
   - `{{entry_time}}` — Hora de ingreso.
   - `{{exit_time}}` — Hora de egreso.
   - `{{amount}}` — Monto cobrado.
   - `{{currency}}` — Moneda.
   - `{{ticket_number}}` — Número de ticket.
   - `{{company_name}}` — Nombre de la empresa (para B2B).
5. El admin guarda la plantilla. El sistema valida:
   - Todas las variables usadas en el body existen en el catálogo de variables.
   - El `body` no está vacío.
   - El `subject` (si aplica) tiene entre 3 y 100 caracteres.
6. Se persiste en `notification_templates`.
7. Se publica `NOTIFICATION_TEMPLATES_UPDATED`.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Variable `{{invalid_var}}` usada en body | Error: "La variable `invalid_var` no existe. Use una variable válida." |
| Plantilla de email sin `subject` | Error: "El asunto es obligatorio para notificaciones por email." |
| `body` excede 10,000 caracteres (email) | Error: "El cuerpo del mensaje excede el límite de 10,000 caracteres." |
| `body` excede 1,600 caracteres (SMS) | Error: "El mensaje SMS excede el límite de 1,600 caracteres." |
| Plantilla desactivada | El sistema no envía esa notificación; no genera error. |
| Variable requerida no presente en el contexto | El sistema usa `[N/A]` como valor por defecto y logged como warning. |
| Envío de notification falla | Retry 3 veces con backoff exponencial; si falla, se marca como `failed` y se puede reenviar manualmente. |
| язы код no soportado | Se usa la plantilla por defecto (`es`) como fallback. |

---

## Criterios de Aceptación

1. El `tenant_admin` puede ver y editar todas las plantillas de notificación de su tenant.
2. Cada plantilla tiene variables que se reemplazan en tiempo de ejecución con datos reales.
3. Las plantillas pueden estar activas o inactivas; las inactivas no se envían.
4. Las variables no reconocidas se detectan en la validación y se rechazan.
5. Los cambios en plantillas se reflejan en las siguientes notificaciones; las ya enviadas no se modifican.
6. Existe una plantilla por defecto en español (`es`) que se usa como fallback si no hay plantilla en el idioma del usuario.
7. El evento `NOTIFICATION_TEMPLATES_UPDATED` se publica tras cada guardado.
8. Los logs de notificación registran `template_id` usado y las variables substituidas.
9. Las notificaciones fallidas pueden reenviarse manualmente desde el panel de administración.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/notification-templates` — Listar plantillas
- `GET /api/v1/tenants/{tenant_id}/notification-templates/{template_id}` — Detalle
- `PUT /api/v1/tenants/{tenant_id}/notification-templates/{template_id}` — Actualizar plantilla
- `PATCH /api/v1/tenants/{tenant_id}/notification-templates/{template_id}/toggle` — Activar/desactivar
- `POST /api/v1/tenants/{tenant_id}/notification-templates/validate` — Validar placeholders
- `GET /api/v1/tenants/{tenant_id}/notification-logs` — Logs de notificaciones (filtrable por status)

---

## Métricas de Éxito

- Tasa de entrega de notificaciones: > 95%.
- Tasa de apertura de emails (si está configurado tracking): > 20%.
- Cero plantillas con variables inválidas en producción.
- Tiempo de procesamiento de notificación: < 2 segundos.

---

## Notas

- El sistema de notificación debe ser agnóstico del proveedor (SMTP, SMS gateway, Push service). Se delega a un `NotificationProvider` que implementa la interfaz.
- Se recomienda implementar unsubscribe logic para emails marketing (opt-out link en el footer).
- Los placeholders deben ser HTML-safe cuando se usan en emails.
- Para SMS, el cuerpo debe ser puro texto sin formateo.
- La tabla `notification_logs` es inmutable (insert-only) para auditoría.
