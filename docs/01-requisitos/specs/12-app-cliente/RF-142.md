# RF-142 — Ver Espacios Disponibles en Tiempo Real

## Metadata

- **Módulo**: App Móvil del Cliente
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: espacios_service, availability_api, app-cliente-backend

---

## User Story

**Como** cliente **quiero** ver la disponibilidad de espacios en tiempo real por sede y zona **para** saber con anticipación dónde hay lugar y planificar mi visita.

---

## Objetivo

El sistema debe mostrar al cliente, desde la app móvil, la disponibilidad en tiempo real de espacios de parqueo discriminada por sede y por zona dentro de cada sede. Esta información se actualiza automáticamente cada 30 segundos o mediante pull-to-refresh.

---

## Actores

- **Actor principal**: Cliente (usuario final autenticado)
- **Actores secundarios**: Sistema de inventario de espacios, servicios de geolocalización

---

## Precondiciones

- El cliente debe tener la app instalada y haber iniciado sesión.
- El tenant debe tener al menos una sede activa con al menos una zona configurada.
- Los dispositivos IoT (cámaras ANPR / sensores) de la sede deben estar operativos.

---

## Postcondiciones

### En caso de éxito

- El cliente visualiza un mapa o lista de zonas con el número de espacios libres, ocupados y totales.
- Los datos de disponibilidad se mantienen sincronizados con el estado real del parqueadero.

### En caso de falla

- Se muestra un mensaje de error amigable indicando que no se pudo cargar la disponibilidad.
- Se permite reintentar la consulta.

---

## Flujo Funcional

1. El cliente abre la vista de "Disponibilidad" en la app.
2. El sistema obtiene la ubicación actual del cliente (siolocation) para sugerencia de sede más cercana, o muestra todas las sedes del tenant.
3. El cliente selecciona una sede del listado.
4. El sistema consulta el servicio de espacios y devuelve la disponibilidad por zona (libres, ocupados, totales, en mantenimiento).
5. La app presenta la información como tarjetas por zona o un mapa interactivo con indicadores de color (verde=disponible, amarillo=pocos espacios, rojo=lleno).
6. El cliente puede pulsar sobre una zona para ver el detalle de espacios individuales.
7. La información se refresca automáticamente cada 30 segundos o mediante pull-to-refresh.

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| sede_id | UUID | Identificador de la sede a consultar | Sí |
| zona_id | UUID | Filtro opcional por zona específica | No |
| latitud | decimal | Latitud del cliente (para ordenamiento por cercanía) | No |
| longitud | decimal | Longitud del cliente (para ordenamiento por cercanía) | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| sede_id | UUID | Identificador de la sede |
| sede_nombre | string | Nombre de la sede |
| zonas | array | Lista de zonas con disponibilidad |
| zona_id | UUID | Identificador de la zona |
| zona_nombre | string | Nombre de la zona |
| espacios_totales | int | Total de espacios en la zona |
| espacios_libres | int | Espacios actualmente libres |
| espacios_ocupados | int | Espacios actualmente ocupados |
| espacios_mantenimiento | int | Espacios en mantenimiento |
| porcentaje_disponibilidad | decimal | Porcentaje de espacios libres |
| ultima_actualizacion | timestamp | Fecha/hora de la última actualización de estado |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| La sede no tiene zonas configuradas | Se muestra mensaje "Esta sede aún no tiene zonas disponibles" |
| Todos los espacios están ocupados | Se muestra la zona en rojo con mensaje "Sin espacios disponibles" y opción de recibir notificación cuando se libere |
| sensores IoT no están transmitiendo (datos stale > 5 min) | Se muestra advertencia "La información puede no estar actualizada" |
| Error de conexión con el servicio de espacios | Se muestra mensaje de error con botón de reintentar |
| Cliente sin conexión a internet | Se muestra mensaje offline y se intenta cargar datos en caché si existen |

---

## Criterios de Aceptación

1. El cliente puede ver la lista de sedes disponibles ordenadas por cercanía.
2. Al seleccionar una sede, se muestran todas sus zonas con conteo de espacios libres/ocupados.
3. La información de disponibilidad se actualiza automáticamente cada 30 segundos.
4. El cliente puede hacer pull-to-refresh para forzar una actualización inmediata.
5. Los indicadores de color reflejan el estado: verde (>20% libre), amarillo (1-20%), rojo (0%).
6. Los datos mostrados tienen un margen de edad máximo de 5 minutos; si los datos son más antiguos, se advierte al usuario.
7. La consulta de disponibilidad no toma más de 3 segundos en presentar resultado inicial.

## Dependencias

**RFs relacionados**:
- RF-159 — El admin puede crear/editar zonas y espacios (necesario para que existan zonas)
- RF-160 — El admin puede marcar espacios como libre/ocupado/mantenimiento
- RF-107 — Sistema de monitoreo de dispositivos IoT (valida que sensores estén operativos)

**Tablas de base de datos afectadas**:
- `zonas` — Almacena configuración de zonas por sede
- `espacios` — Almacena estado individual de cada espacio
- `sedes` — Almacena datos de la sede
- `espacios_disponibilidad` — Vista materializada o tabla de resumen de disponibilidad en tiempo real

---

## Endpoints API Involucrados

- `GET /api/v1/cliente/sedes` — Lista sedes disponibles para el cliente
- `GET /api/v1/cliente/sedes/{sede_id}/disponibilidad` — Obtiene disponibilidad por zonas de una sede
- `GET /api/v1/cliente/sedes/{sede_id}/zonas/{zona_id}/espacios` — Detalle de espacios individuales de una zona

---

## Métricas de Éxito

- Tiempo de respuesta promedio del endpoint de disponibilidad < 800 ms
- Tasa de datos stale (más de 5 min) < 2% del tiempo
- engagement: > 60% de clientes que consultan disponibilidad realizan un parqueo en las siguientes 24h

---

## Notas

- La disponibilidad se calcula a partir del último evento de entrada/salida registrado por los dispositivos ANPR o sensores de ocupación.
- Se debe implementar polling o WebSocket para actualizaciones en tiempo real desde el backend.
- En zonas con más de 100 espacios, se permite agrupar visualmente para mejorar rendimiento.
