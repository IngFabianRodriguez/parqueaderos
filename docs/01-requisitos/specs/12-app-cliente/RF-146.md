# RF-146 — Métodos de Pago del Cliente

## Metadata

- **Módulo**: App Móvil del Cliente
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: pago_service, app-cliente-backend

---

## User Story

**Como** cliente **quiero** añadir métodos de pago como tarjeta, Nequi o Daviplata y definir uno como predeterminado **para** poder pagar mis parqueos de forma rápida sin tener que ingresar los datos cada vez.

---

## Objetivo

El sistema debe permitir al cliente registrar y gestionar sus métodos de pago desde la app. El cliente puede registrar tarjetas de crédito/débito, cuentas Nequi y Daviplata. Debe poder tener múltiples métodos y designar uno como predeterminado para cobros automáticos.

---

## Actores

- **Actor principal**: Cliente (usuario final autenticado)
- **Actores secundarios**: Pasarela de pagos, servicios de tokenización de tarjetas

---

## Precondiciones

- El cliente debe haber iniciado sesión en la app.
- Para tarjetas: el cliente debe tener una tarjeta física para completar la validación 3D Secure.

---

## Postcondiciones

### En caso de éxito

- El método de pago queda asociado a la cuenta del cliente.
- Las renovaciones y prepagos futuros usan el método default automáticamente.
- El cliente puede ver sus métodos en la sección de pagos.

### En caso de falla

- El método no se guarda y se muestra error al cliente.

---

## Flujo Funcional

### Agregar Tarjeta de Crédito/Débito

1. El cliente accede a "Mi Cuenta" → "Métodos de Pago" → "Agregar".
2. El cliente selecciona "Tarjeta de crédito/débito".
3. El cliente ingresa: número de tarjeta, fecha de expiración, CVV, nombre en la tarjeta.
4. El sistema tokeniza la tarjeta con el proveedor de pagos (no almacena el número completo).
5. El sistema ejecuta validación 3D Secure con el banco emisor.
6. Si la validación es exitosa, se guarda el método y se muestra confirmación.
7. Si es la primera tarjeta, se marca automáticamente como default.

### Agregar Nequi

1. El cliente selecciona "Nequi" como tipo de método.
2. El cliente ingresa su número de teléfono asociado a Nequi.
3. El sistema envía un código de verificación SMS al número.
4. El cliente ingresa el código.
5. El sistema valida el código con la pasarela de Nequi.
6. Si es válido, se guarda el método.

### Agregar Daviplata

1. El cliente selecciona "Daviplata" como tipo de método.
2. El cliente ingresa su número de teléfono asociado a Daviplata.
3. El sistema envía un código de verificación SMS al número.
4. El cliente ingresa el código.
5. El sistema valida con la pasarela Daviplata.
6. Si es válido, se guarda el método.

### Definir Método como Default

1. El cliente accede a "Métodos de Pago".
2. El cliente pulsa sobre el menú de un método → "Marcar como predeterminado".
3. El sistema actualiza el campo default y desmarca el anterior.
4. Se muestra confirmación.

### Eliminar Método de Pago

1. El cliente pulsa "Eliminar" en un método.
2. El sistema verifica que no haya parqueos activos con cobros pendientes hacia ese método.
3. Si hay cobros pendientes, se muestra advertencia y se solicita confirmar.
4. El cliente confirma.
5. El método se marca como inactivo.

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| tipo | enum | 'tarjeta', 'nequi', 'daviplata' | Sí |
| numero | string | Número de tarjeta o teléfono | Sí |
| expira_mes | int | Mes de expiración (1-12) | Sí (tarjeta) |
| expira_anio | int | Año de expiración (4 dígitos) | Sí (tarjeta) |
| cvv | string | CVV de la tarjeta | Sí (tarjeta) |
| nombre_titular | string | Nombre en la tarjeta | Sí (tarjeta) |
| telefono | string | Número de teléfono | Sí (Nequi/Daviplata) |
| token | string | Token generado por la pasarela | Sí |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| metodo_pago_id | UUID | Identificador del método |
| tipo | string | 'tarjeta', 'nequi', 'daviplata' |
| nombre | string | Nombre para mostrar (ej: "Visa ...4242", "Nequi **1234") |
| ultimos_digitos | string | Últimos 4 dígitos (tarjeta) o últimos 4 del teléfono |
| esta_default | boolean | Si es el método predeterminado |
| marca | string | Marca de la tarjeta (Visa, Mastercard) |
| estado | string | 'activo' o 'inactivo' |
| fecha_agregado | timestamp | Fecha de registro |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Tarjeta rechazada por el banco | Se muestra mensaje de rechazo y se pide intentar con otra tarjeta |
| Teléfono de Nequi/Daviplata no existe o no está activado | Se muestra mensaje indicando que debe activar el servicio |
| Código de verificación expirado (> 5 min) | Se debe reenviar el código |
| Límite de métodos alcanzado (5 métodos) | Se muestra mensaje indicando que debe eliminar uno para agregar otro |
| Cliente intenta eliminar método con renewal de parqueo pendiente | Se advierte y se ofrece cambiar el default antes de eliminar |
| Tarjeta vencida | Se permite el registro pero se muestra advertencia "Esta tarjeta está vencida" |
| Validación 3D Secure falla | Se muestra error y se permite reintentar |

---

## Criterios de Aceptación

1. El cliente puede agregar hasta 5 métodos de pago.
2. El cliente puede eliminar un método de pago.
3. El cliente puede designar cualquier método activo como predeterminado.
4. Las tarjetas se tokenizan (no se almacena el número completo).
5. Los métodos Nequi y Daviplata se validan mediante código SMS.
6. Al agregar la primera tarjeta, esta se marca automáticamente como default.
7. El método default se usa automáticamente para renovaciones (RF-144) y prepagos (RF-143).
8. El cliente ve solo los últimos 4 dígitos del número de tarjeta/teléfono en la lista.

## Endpoints API Involucrados

- `GET /api/v1/cliente/metodos-pago` — Lista los métodos de pago del cliente
- `POST /api/v1/cliente/metodos-pago` — Agrega un nuevo método
- `DELETE /api/v1/cliente/metodos-pago/{metodo_pago_id}` — Elimina un método
- `PUT /api/v1/cliente/metodos-pago/{metodo_pago_id}/default` — Define el método default
- `POST /api/v1/pasarela/validar-tarjeta` — Validación y tokenización de tarjeta
- `POST /api/v1/pasarela/enviar-codigo` — Envía código de verificación Nequi/Daviplata
- `POST /api/v1/pasarela/verificar-codigo` — Verifica el código recibido

---

## Métricas de Éxito

- Tasa de registro exitoso de tarjetas > 90% (después de intentos)
- Promedio de métodos de pago por cliente activo > 1.8
- Tasa de uso del default en renovaciones > 85%

---

## Notas

- El número de tarjeta completo nunca se almacena en nuestra base de datos; solo el token devuelto por la pasarela.
- Las transacciones futuras se cobran usando el token sin necesidad de volver a ingresar datos de tarjeta.
- Para Nequi/Daviplata, el número de teléfono actúa como identificador de la cuenta.
