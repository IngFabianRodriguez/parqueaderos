# RF-014 — Enviar notificaciones push/SMS/email al cliente

## Metadata

- **Módulo**: CRM y Pagos
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: notification-service, crm-service

---

## User Story

**Como** Sistema **quiero** notificar al cliente en eventos clave de su sesión de parqueo (entrada, salida, recordatorio, pago) **para** mejorar su experiencia, reducir consultas al operador y mantenerlo informado de su actividad.

---

## Objetivo

Enviar notificaciones al cliente a través de múltiples canales (push notification, SMS, email) activadas por eventos del sistema. Las notificaciones son configurables por el tenant_admin: se puede activar/desactivar cada tipo, definir templates personalizados, y elegir el canal preferido por cliente. El sistema debe manejar la complejidad de deliverar (offline, retry, unsubscribe) y trackear la apertura de cada notificación para métricas.

---

## Actores

- Sistema (automático según eventos)
- Cliente (recibe, puede configurar preferencias)
- Operador (puede enviar notificación manual)
- [Actores secundarios]Proveedor Push (Firebase/Apple), SMS gateway (Nexmo, Twilio), SMTP provider

---

## Precondiciones

- Cliente registrado con al menos un canal de contacto (email o teléfono)
- Canal de notificación activo para el tenant
- Cliente tiene preferencias de notificación configuradas (o usa defaults del tenant)

---

## Postcondiciones

### En caso de éxito

- Notificación enviada por el canal indicado
- Registro en tabla notificacion con estado `enviada`
- Si canal es push: dispositivo marcado como `received`
- Métricas de apertura actualizadas si aplica

### En caso de falla

- Push no recibido en 5 min: se intenta SMS fallback si configurado
- SMS falla: se intenta email como último recurso
- Todos fallan: se marca como `fallida`, no se reintenta automáticamente (solo manual)

---

## Flujo Funcional

**Flujo 1 — Notificación por entrada:**
1. Sistema detecta vehículo entra (RF-005)
1. Notification-service recibe evento `vehiculo.ingresado`
1. Sistema obtiene cliente asociado a la placa
1. Sistema consulta preferencias del cliente (canales activos)
1. Sistema renderiza template de notificación de entrada con datos dinámicos (sede, hora, plaza)
1. Sistema envía por el canal preferido (push → SMS → email en fallback)
1. Registro de envío guardado

**Flujo 2 — Notificación por salida (pago confirmado):**
1. Sistema detecta sesión pagada (RF-011)
1. Notification-service recibe evento `sesion.pagada`
1. Genera notificación con: sede, monto pagado, tiempo de estadía
1. Incluye link a factura (RF-012) y opción de dejar opinión
1. Envío similar al flujo 1

**Flujo 3 — Recordatorio de tiempo por vencer:**
1. Job programado evalúa sesiones activas con prepago (RF-017)
1. Si quedan 15 min del tiempo comprado: envía recordatorio push
1. Si no se renueva y quedan 5 min: segundo recordatorio
1. Si expira: notificación de "tiempo expirado, debe pagar excedente"

**Flujo 4 — Admin envía notificación manual:**
1. Admin/operador selecciona cliente o grupo de clientes
1. Escribe mensaje personalizado o selecciona template
1. Envía
1. Sistema procesa batch si son muchos destinatarios

**Flujo 5 — Cliente se desuscribe:**
1. Cliente abre link de unsubscribe en email o desde app
1. Sistema actualiza preferencias del cliente
1. Canal desactivado para ese cliente

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Cliente sin email ni teléfono | Notificación no enviada. Sesión de parqueo continúa normal. Se loguea como "no contactable" |
| Push no enviado (Firebase down) | Sistema detecta vía callback de entrega. Reintenta por SMS si disponible |
| SMS gateway returns error | Retry 3 veces con backoff 30s, luego fallback email |
| Template con variable faltante | Renderiza el campo vacío, no falla. Loguea warning |
| Cliente con Opt-out explícito | Sistema no envía por ese canal. No se cuenta como falla |
| Rate limit del proveedor alcanzado | Cola de espera, reintento en 1 minuto. Alerta si se acumula |

---

## Criterios de Aceptación

1. Todas las notificaciones se registran en tabla `notificacion` con estado
2. Push notifications deliveradas < 5s después del evento en condiciones normales
3. SMS entregados < 30s después del evento
4. Email enviado < 60s después del evento
5. Un solo envío por canal activo por notificación (no duplicar si cliente tiene múltiples dispositivos)
6. Cliente puede desuscribirse de cualquier canal sin afectar otros
7. Rate limits de proveedores respetados (no saturar canal)
8. Plantillas de notificación editables desde admin (RF-091)

## Endpoints API Involucrados

- `POST /api/v1/notificaciones` — Enviar notificación (manual o desde evento)
- `GET /api/v1/notificaciones/cliente/{cliente_id}` — Historial de notificaciones del cliente
- `PUT /api/v1/notificaciones/preferencias` — Actualizar preferencias de un cliente

---

## Métricas de Éxito

- Tasa de delivery > 98% para push, > 95% para SMS
- Tiempo promedio de delivery < 5s para push
- Cero notificaciones enviadas a clientes con Opt-out
- Tasa de apertura trackeable (mínimo para push y email)

---

## Notas

- Colombian context: SMS still widely used, especially in areas with poor data connectivity. Push notifications are primary for smartphone users. Email is used for receipts and formal communication.
- Se usará Firebase Cloud Messaging (FCM) para push en Android, y APNs para iOS.
- SMS gateway: configurable, se integra con proveedores como Twilio, Nexmo, o proveedores locales comoizzi.
- Todas las notificaciones transactional-no comerciales están permitidas bajo normativa SIC.
- El unsubscribe debe estar en cada email comercial. Para notificaciones de sesión (transaccional), no aplica opt-out.
