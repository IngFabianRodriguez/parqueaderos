# RF-027 — Logueo de Acciones Sensibles en Auditoría Inmutable

## Metadata

- **Módulo**: Seguridad y Acceso
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: audit-service, gateway-service, auth-service, todos los servicios que generen eventos sensibles

---

## User Story

**Como** superadmin **quiero** que todas las acciones sensibles del sistema queden registradas en un log inmutable **para** poder investigar incidentes, demostrar cumplimiento regulatorio y auditar el comportamiento de usuarios y operadores.

---

## Objetivo

El sistema debe registrar en un log de auditoría todas las acciones que impliquen modificación de datos sensibles, errores de seguridad, o cambios en la configuración del sistema. Este log debe ser inmutable (no modificable ni deletable por usuarios del aplicativo) y solo debe permitir consultas de lectura con permisos elevados.

---

## Actores

- **Actor principal**: Superadmin (consulta y exportación de logs de auditoría)
- **Actores secundarios**: admin_sede (consulta logs de su sede), sistemas automatizados (escritura de eventos), regulador externo (consulta read-only mediante export cifrado)

---

## Precondiciones

- El actor tiene permisos de lectura sobre el módulo de auditoría (superadmin tiene acceso global; admin_sede solo a logs de su sede).
- El sistema está operativo y la base de datos de auditoría accesible.

---

## Postcondiciones

### En caso de éxito

- Cada acción sensible genera un registro en la tabla `audit_logs` de forma síncrona o asíncrona según criticidad.
- El registro contiene: timestamp (UTC con precisión de milisegundos), actor_id, actor_rol, acción, recurso, recurso_id, tenant_id, sede_id, IP, user_agent, payload (JSON con detalles), correlation_id, resultado (SUCCESS/FAILURE).

### En caso de falla

- El sistema intenta escribir el log de auditoría en una cola de fallback si la base de datos primaria falla.
- Si la cola también falla, el evento se escribe en el filesystem local como último recurso (emergency log) y se genera alerta de pager.
- Bajo ninguna circunstancia se permite que una acción sensible se ejecute sin al menos un intento de registro de auditoría.

---

## Flujo Funcional

### Flujo Principal: Escritura de Evento de Auditoría

1. Un servicio detecta que una acción sensible está por ejecutarse o ya se ejecutó.
2. El servicio construye el objeto `AuditEvent` con todos los campos requeridos.
3. El servicio incluye un `correlation_id` (generado al inicio de la request HTTP o evento) para trazabilidad distributed.
4. El servicio envía el evento al `audit-service` vía gRPC o mensaje Kafka.
5. El `audit-service` persiste el evento en la tabla `audit_logs` con fecha de ingestión.
6. Para acciones CRITICAL (login, logout, cambio de permisos, eliminación de datos): el evento se replica en un storage separado inmutable (WORM-style) después de 24h.

### Flujo Alternativo: Consulta de Logs (Superadmin)

1. El superadmin accede al módulo "Auditoría" desde el panel de administración.
2. El sistema presenta formulario con filtros: rango de fechas, actor, acción, recurso, tenant, sede.
3. El superadmin aplica filtros y hace clic en "Buscar".
4. El sistema valida que el rango no supere 90 días (protección contra queries pesadas).
5. El sistema retorna resultados paginados (50 por página) ordenados por timestamp DESC.
6. El superadmin puede exportar los resultados a CSV/JSON para análisis offline.

### Flujo Alternativo: Consulta de Logs (Admin de Sede)

1. El admin_sede accede al módulo "Auditoría" desde su panel.
2. El sistema filtra automáticamente los logs a la sede(s) asignadas al admin.
3. El admin no puede consultar logs de otras sedes ni del tenant padre.
4. El sistema solo muestra acciones que el admin tiene permisos de ver según configuración de visibilidad.

### Flujo Alternativo: Evento CRITICAL - Login Exitoso

1. Un usuario inicia sesión exitosamente.
2. El sistema genera evento AUDIT_AUTH_SUCCESS con payload: user_id, email, rol, IP, user_agent, MFA_used (bool).
3. El evento se persiste síncronamente antes de retornar el token al usuario.
4. Si la persistencia falla, el login se rechaza (FAIL-Closed principle para eventos de auth).

### Flujo Alternativo: Evento CRITICAL - Cambio de Permisos

1. Un superadmin modifica el rol de un usuario.
2. El sistema genera evento AUDIT_PERMISSION_CHANGE con payload: admin_id, target_user_id, old_rol, new_rol, reason.
3. El evento se persiste y además se replica a storage inmutable en 24h.

### Flujo Alternativo: Overflow de Cola de Auditoría

1. El sistema detecta que la cola de auditoría tiene > 10,000 eventos pendientes (lag > 30s).
2. El sistema activa modo "degraded": solo eventos CRITICAL se procesan síncronamente; los demás se batching.
3. Se genera alerta de pager para el equipo de operaciones.
4. Cuando el lag se reduce a < 1,000 eventos, se restauran las escrituras normales.

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Base de datos de auditoría no disponible | Se escribe a cola fallback (Kafka); si también falla, se escribe a filesystem local (emergency.log) |
| Query con rango > 90 días | Sistema rechaza con mensaje: "El rango máximo es 90 días. Use exportación para rangos mayores." |
| Admin de sede intenta ver logs de otra sede | Sistema retorna 403 Forbidden |
| Payload > 64KB | Se trunca a 64KB y se marca campo `payload_truncated: true` |
| IP del cliente es IPv6 | Se registra completa, sin cambios |
| Actor es un sistema (service account) | El campo actor_email identifica al servicio (ej: "gateway-service@system") |
| Log de acción敏感 (eliminación masiva) | Se registra con payload enriquecido: lista de IDs afectados y causa |
| Ataque de flooding en auditoría | Rate limiting: máximo 10,000 eventos por segundo por tenant; excedente se dropea con alerta |
| Correlación con logs de otros servicios | El correlation_id permite JOIN en sistemas de logging distributed (Jaeger, etc.) |
| Intento de borrado de log por usuario | El sistema NO permite DELETE/UPDATE en tabla audit_logs; cualquier intento se registra como evento AUDIT_TAMPERING_ATTEMPT |

## Criterios de Aceptación

1. **CA-027-01**: Toda acción sensible listada en el catálogo genera un registro en `audit_logs` con todos los campos requeridos.
2. **CA-027-02**: Los logs de auditoría son inmutables: no existe endpoint DELETE ni UPDATE para la tabla `audit_logs`.
3. **CA-027-03**: Un superadmin puede consultar logs de cualquier sede del tenant con filtros por fecha, actor, acción y recurso.
4. **CA-027-04**: Un admin_sede solo puede consultar logs de las sedes asignadas a su usuario.
5. **CA-027-05**: Los eventos de autenticación (login, logout, MFA) se persisten síncronamente (FAIL-Closed) antes de retornar respuesta al cliente.
6. **CA-027-06**: Un intento de tampering (UPDATE/DELETE en audit_logs) genera un evento de alerta y se rechaza.
7. **CA-027-07**: Los logs se pueden exportar a CSV/JSON con filtros aplicados; el export está sujeto a rate limiting (máx 10/minuto).
8. **CA-027-08**: El sistema mantiene los logs de auditoría por mínimo 5 años (compliance).
9. **CA-027-09**: En caso de falla de la base de datos de auditoría, el sistema NO bloquea la operación principal; la escritura es asíncrona con retry con exponential backoff.

## Dependencias

**RFs relacionados**:
- RF-026 — Autenticación JWT (el actor_id se extrae del JWT en cada request)
- RF-028 — RBAC (los permisos sobre el módulo de auditoría se definen aquí)

**Tablas de base de datos afectadas**:
- `audit_logs` — escritura de todos los eventos
- `users` — referencia para actor_email
- `tenants` — filtro de tenant en consultas
- `sedes` — filtro de sede en consultas

**Infraestructura adicional**:
- Kafka topic `audit-events` para buffer asíncrono
- Redis para caching de consultas frecuentes de auditoría
- Object storage (S3 compatible) para logs archivados > 90 días

---

## Endpoints API Involucrados

- `GET /api/v1/audit/logs` — Consulta paginada de logs (filtros: fecha_inicio, fecha_fin, actor_id, accion, recurso, recurso_id, sede_id, page, per_page)
- `GET /api/v1/audit/logs/{id}` — Detalle de un log específico (solo CRITICAL actions)
- `GET /api/v1/audit/logs/export` — Exportación a CSV/JSON (máx 90 días, async)
- `GET /api/v1/audit/stats` — Estadísticas agregadas (conteo por acción, actor, sede) para dashboards
- `POST /api/v1/audit/events` — Interno: recibe eventos de otros servicios para persistencia

---

## Métricas de Éxito

- Cobertura de auditoría: 100% de acciones sensibles registradas
- Latencia de escritura de auditoría: ≤ 50ms (P99) para eventos síncronos
- Disponibilidad del servicio de auditoría: ≥ 99.9%
- Tiempo de retención: 5 años sin pérdida de datos
- Tiempo de consulta (logs de 30 días): ≤ 3s (P95)
- Rate de eventos perdidos por falla de sistema: 0% (principio FAIL-Closed para CRITICAL)

---

## Notas

- La inmutabilidad se garantiza a nivel de base de datos: la tabla `audit_logs` solo tiene permisos SELECT e INSERT para el service account de los microservicios; no hay UPDATE ni DELETE.
- Para consultas de logs por rango > 90 días se debe usar el sistema de archivado (S3) que se alimenta diariamente de los logs menos recientes.
- El campo `correlation_id` se genera en el API Gateway al recibir la request y se propaga a todos los servicios llamados, permitiendo reconstruir la traza completa de una operación.
- Las service accounts (sistemas) tienen actor_rol = "SYSTEM" y actor_email con formato `service-name@system`.
- Se considera "acción sensible" cualquier operación que: (a) modifique estado financiero, (b) cambie permisos de usuarios, (c) acceda a datos personales de otros usuarios, (d) tenga potencial de abuso o fraude, o (e)sea irreversible.
