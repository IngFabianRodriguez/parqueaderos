# RF-026 — Autenticación de Usuarios con JWT y Roles Diferenciados

## Metadata

- **Módulo**: Seguridad y Acceso
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: auth-service, user-service, gateway-service

---

## User Story

**Como** administrador del sistema **quiero** que cada usuario sea autenticado con credenciales seguras y tenga un rol definido **para** garantizar que solo personas autorizadas accedan a funciones apropiadas según su responsabilidad.

---

## Objetivo

El sistema debe implementar un mecanismo de autenticación basado en JSON Web Tokens (JWT) que permita identificar y autorizar a los usuarios según cuatro roles diferenciados: superadmin, admin_sede, operador y cliente. El sistema debe soportar flujo de login/logout, refresh de tokens, y MFA opcional para roles sensibles.

---

## Actores

- **Actor principal**: Todo usuario del sistema (sin distinción de rol)
- **Actores secundarios**: Superadmin (gestión de usuarios y roles), admin_sede (gestión de operadores de su sede), sistema (validación automática de tokens en cada request)

---

## Precondiciones

- El usuario existe en la base de datos con credenciales válidas.
- El tenant asociado al usuario está activo (no suspendido ni eliminado).
- Si el rol requiere MFA: el usuario tiene un método MFA configurado (TOTP o SMS).

---

## Postcondiciones

### En caso de éxito

- El usuario recibe un access token JWT válido (TTL 15 min) y un refresh token (TTL 7 días).
- El JWT contiene claims: user_id, tenant_id, rol, sede_ids (array), permisos, exp, iat.
- Los logs de acceso registran el evento con IP y user agent.
- El refresh token se almacena en la base de datos con estado ACTIVE.

### En caso de falla

- El usuario recibe mensaje de error descriptivo sin exponer datos sensibles.
- El intento fallido se registra en logs de auditoría con timestamp e IP.
- Después de 5 intentos fallidos consecutivos, la cuenta se bloquea por 15 minutos.

---

## Flujo Funcional

### Flujo Principal: Login con JWT

1. El usuario navega a la página de login (`/login`).
2. El sistema presenta formulario con campos: email, contraseña.
3. El usuario ingresa sus credenciales y hace clic en "Iniciar Sesión".
4. El sistema valida que el email exista y la contraseña coincida (usando bcrypt con cost factor 12).
5. Si el rol es superadmin o admin_sede y MFA está habilitado: el sistema envía código TOTP al dispositivo configurado o SMS como fallback.
6. El usuario ingresa el código MFA de 6 dígitos en un plazo de 60 segundos.
7. El sistema valida el código MFA (window de 1 paso atrás y 1 adelante para drift de tiempo).
8. El sistema genera un JWT con los claims del usuario y TTL de 15 minutos.
9. El sistema genera un refresh token con TTL de 7 días y lo almacena en la tabla `refresh_tokens`.
10. El sistema retorna `{ access_token, refresh_token, expires_in, token_type: "Bearer" }` al cliente.
11. El cliente almacena ambos tokens y envía el access token en el header `Authorization: Bearer <token>` en todas las requests subsecuentes.
12. El sistema registra el evento de login exitoso en `audit_logs`.

### Flujo Alternativo: Refresh de Access Token

1. El cliente detecta que el access token está por expirar (falta < 2 min) o recibe 401 Unauthorized.
2. El cliente envía el refresh token a `POST /api/v1/auth/refresh`.
3. El sistema valida que el refresh token no haya sido revocado, usado más de 5 veces, ni haya expirado.
4. El sistema emite un nuevo access token con los mismos claims actualizados (nuevo exp).
5. El sistema incrementa el contador de uso del refresh token.
6. El sistema retorna el nuevo access token.

### Flujo Alternativo: Logout

1. El usuario hace clic en "Cerrar Sesión" desde el panel.
2. El cliente envía `POST /api/v1/auth/logout` con el refresh token.
3. El sistema marca el refresh token como REVOKED en la base de datos.
4. El sistema registra el logout en `audit_logs`.
5. El cliente elimina los tokens almacenados localmente.

### Flujo Alternativo: Login con Credenciales Inválidas

1. El usuario ingresa credenciales incorrectas.
2. El sistema incrementa el contador de intentos fallidos del usuario.
3. Si intentos >= 5 en ventana de 10 minutos: el sistema bloquea la cuenta por 15 min y envía notificación al email del usuario.
4. El sistema retorna 401 Unauthorized con mensaje: "Credenciales inválidas. Intentos restantes: X" (sin revelar si el email existe).

### Flujo Alternativo: MFA - Código Incorrecto

1. El usuario ingresa un código MFA incorrecto.
2. El sistema incrementa contador de intentos MFA fallidos.
3. Si intentos >= 3 en ventana de 5 minutos: el código MFA se reinvalidá y debe solicitarse uno nuevo.
4. El sistema retorna 401 con mensaje: "Código MFA inválido. Solicitar nuevo código."

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| email | string | Correo electrónico del usuario (formato válido) | Sí |
| password | string | Contraseña del usuario (mín 8 caracteres) | Sí |
| mfa_code | string | Código TOTP de 6 dígitos | Condicional (si MFA habilitado) |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| access_token | string | JWT con claims del usuario |
| refresh_token | string | Token opaco para refresh |
| expires_in | integer | TTL del access token en segundos (900 = 15 min) |
| token_type | string | Siempre "Bearer" |
| user | object | Datos públicos del usuario (id, nombre, email, rol) |

### Estructura del JWT Claims

```json
{
  "sub": "user_uuid",
  "tenant_id": "tenant_uuid",
  "rol": "admin_sede",
  "sede_ids": ["sede_uuid_1", "sede_uuid_2"],
  "permisos": ["reports:read", "gate:operate"],
  "iat": 1715250000,
  "exp": 1715250900
}
```

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Refresh token expirado | Retorna 401 y obliga al usuario a re-autenticarse |
| Refresh token ya usado > 5 veces | Invalida toda la familia de tokens, requiere re-login (sospecha de token theft) |
| Refresh token revoked | Retorna 401, el cliente debe redirigir a login |
| Usuario desactivado/suspendido | Retorna 403 con mensaje: "Cuenta suspendida. Contacte al administrador." |
| Tenant inactivo | Retorna 403 con mensaje: "Tenant inactivo. Contacte soporte." |
| MFA configurado pero no enviado | El sistema NO permite login sin MFA; retorna 428 cuando el factor es requerido |
| Dos usuarios mismo email en tenants diferentes | Cada tenant tiene su namespace de email; email + tenant_id es único |
| Bloqueo por fuerza bruta | Después de 5 intentos fallidos, cuenta bloqueada 15 min; se envía email de alerta |
| Clock skew > 30s en验证 MFA | Se permite window de ±1 step (30s adelante, 30s atrás) |
| Intento de acceso con token manipulado | El sistema detecta firma inválida, retorna 401, y registra alerta de seguridad |

## Criterios de Aceptación

1. **CA-026-01**: Un usuario con credenciales válidas recibe un JWT válido y puede hacer requests autenticadas durante 15 minutos sin re-autenticación.
2. **CA-026-02**: El JWT contiene todos los claims necesarios (user_id, tenant_id, rol, sede_ids, permisos) para que los servicios downstream puedan hacer autorización sin consultar la base de datos.
3. **CA-026-03**: Un superadmin tiene acceso a todas las sedes de su tenant; admin_sede solo a las asignadas; operador solo a las asignadas; cliente solo a sus propios datos.
4. **CA-026-04**: Cuando el access token expira, el cliente puede usar el refresh token para obtener uno nuevo sin re-ingresar credenciales.
5. **CA-026-05**: Un intento de acceso con credenciales inválidas no revela si el email existe o no (mensaje genérico).
6. **CA-026-06**: Después de 5 intentos fallidos, la cuenta se bloquea 15 minutos y se notifica al usuario por email.
7. **CA-026-07**: El login con MFA para superadmin requiere código TOTP válido además de credenciales.
8. **CA-026-08**: El logout revoca el refresh token y el sistema rechaza intentos de usar tokens revocados.
9. **CA-026-09**: El sistema soporta múltiples sesiones simultáneas por usuario (ej: celular + desktop) sin límite artificial.

---

## Endpoints API Involucrados

- `POST /api/v1/auth/login` — Inicio de sesión (retorna access + refresh token)
- `POST /api/v1/auth/refresh` — Refresca el access token usando refresh token
- `POST /api/v1/auth/logout` — Revoca el refresh token
- `POST /api/v1/auth/mfa/setup` — Inicia configuración de MFA (genera TOTP secret)
- `POST /api/v1/auth/mfa/verify` — Verifica código MFA durante login
- `GET /api/v1/auth/me` — Retorna información del usuario autenticado (del JWT)
- `POST /api/v1/auth/password/reset` — Solicita reset de contraseña (envía email con link)

---

## Métricas de Éxito

- Tasa de login exitosos ≥ 95% del total de intentos
- Tiempo de respuesta de login ≤ 500ms (P95)
- Tasa de tokens refresh exitosos ≥ 99%
- Sla de incidentes de seguridad por autenticación ≤ 2 por mes
- Cobertura de logs de auditoría: 100% de eventos de auth

---

## Notas

- La contraseña se hashea con bcrypt (cost factor 12) para resistir ataques de fuerza bruta.
- El JWT se firma con RS256 (asimétrico) para permitir que microservicios validen sin conocer la clave secreta del emisor.
- Los refresh tokens son opacos (random UUID v4) y se almacenan en la base de datos para poder revocarlos masivamente si se sospecha compromiso.
- El límite de 5 usos por refresh token mitiga el riesgo de token theft: si un atacante roba un token usado, la víctima lo usó primero y el atacante queda bloqueado.
- El TTL de 15 minutos para access token balancea seguridad (ventana de exposición) con UX (frecuencia de refresh).
- Se usa Redis para almacenar refresh tokens invalidated para permitir logout global (single logout) y blacklist de tokens comprometidos.
