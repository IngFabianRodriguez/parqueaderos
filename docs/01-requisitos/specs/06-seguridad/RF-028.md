# RF-028 — RBAC: Control de Acceso Basado en Roles

## Metadata

- **Módulo**: Seguridad y Acceso
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: auth-service, user-service, gateway-service, todos los servicios que requieren autorización

---

## User Story

**Como** superadmin **quiero** definir qué acciones puede realizar cada rol sobre cada recurso del sistema **para** garantizar el principio de menor privilegio y evitar que usuarios accedan a funciones fuera de su responsabilidad.

---

## Objetivo

El sistema debe implementar un sistema de control de acceso basado en roles (RBAC) donde cada rol (superadmin, admin_sede, operador, cliente) tiene un conjunto de permisos específicos sobre recursos definidos. Los permisos se verifican en cada request y se incluyen como claims en el JWT para permitir autorización sin consulta a la base de datos en el camino crítico.

---

## Actores

- **Actor principal**: Superadmin (define y modifica roles y permisos)
- **Actores secundarios**: admin_sede (administra operadores de su sede), operador (ejerce permisos asignados), cliente (accede solo a sus datos propios)

---

## Precondiciones

- El usuario tiene una sesión JWT válida con el rol y permisos embebidos.
- El superadmin tiene permisos de ADMIN_ROLES para modificar definiciones de roles.
- Los recursos y acciones del sistema están correctamente registrados en la tabla de permisos.

---

## Postcondiciones

### En caso de éxito

- El sistema permite o denied el acceso al recurso/acción según la matriz de permisos.
- El resultado de la verificación se incluye en los logs de auditoría (si es acción sensible).
- Si el acceso es denegado, el sistema retorna 403 Forbidden con mensaje descriptivo.

### En caso de falla

- Error de sistema: se retorna 500 y se loguea el error.
- Recurso no encontrado: se retorna 404 según tipo de recurso (no se revela si existe o no para evitar enumeración).

---

## Flujo Funcional

### Flujo Principal: Verificación de Permiso en Request

1. El API Gateway recibe una request con header `Authorization: Bearer <jwt>`.
2. El Gateway decodifica el JWT y extrae los claims: user_id, rol, permisos, sede_ids.
3. El Gateway consulta la ruta del endpoint y el método HTTP para mapearlos a un recurso y acción.
4. El Gateway verifica si el rol del usuario tiene el permiso requerido para ese recurso.
5. Si el permiso existe: la request se reenvía al servicio de backend.
6. Si el permiso no existe: se retorna 403 Forbidden con payload: `{ "error": "access_denied", "required_permission": "reports:write", "your_permissions": ["reports:read"] }`.
7. Para endpoints cross-sede: se verifica que la sede_id del request esté en el array sede_ids del JWT.

### Flujo Alternativo: Superadmin Crea Nuevo Rol

1. El superadmin accede al módulo "Roles y Permisos" desde el panel.
2. El sistema muestra lista de roles existentes con sus permisos.
3. El superadmin hace clic en "Crear Rol".
4. El sistema presenta formulario: nombre del rol, descripción, permisos disponibles (agrupados por módulo).
5. El superadmin selecciona los permisos deseados y hace clic en "Guardar".
6. El sistema valida que el nombre sea único y que no existan conflictos.
7. El sistema crea el rol en la tabla `roles` y los permisos en `role_permissions`.
8. El sistema registra la creación en auditoría (RF-027).

### Flujo Alternativo: Superadmin Asigna Rol a Usuario

1. El superadmin accede a la gestión de usuarios.
2. El superadmin selecciona un usuario y hace clic en "Editar Rol".
3. El sistema muestra dropdown con roles disponibles.
4. El superadmin selecciona el nuevo rol y hace clic en "Guardar".
5. El sistema actualiza el rol del usuario en `users`.
6. El sistema genera nuevos JWT para el usuario (el próximo login) con los permisos del nuevo rol.
7. El sistema registra el cambio en auditoría (RF-027).

### Flujo Alternativo: Admin de Sede Crea Operador

1. El admin_sede accede a la gestión de usuarios de su sede.
2. El admin_sede hace clic en "Crear Operador".
3. El sistema presenta formulario con campos: nombre, email, contraseña temporal.
4. El admin_sede llena los datos y asigna los permisos (subconjunto de los propios del admin).
5. El sistema crea el usuario con rol "operador" y sede_ids limitados a la sede del admin.
6. El sistema notifica al operador por email con credenciales temporales.

### Flujo Alternativo: Permiso Dinámico por Recurso

1. Un operador intenta acceder a un reporte de la sede XYZ.
2. El sistema verifica que sede XYZ esté en el array sede_ids del JWT.
3. El operador tiene permiso `reports:read` por su rol.
4. El acceso es permitido solo para el recurso de la sede XYZ.
5. Si el operador intentara acceder a reporte de otra sede, recibiría 403.

### Flujo Alternativo: Acceso a Datos Propios (Cliente)

1. Un cliente hace request a `GET /api/v1/clientes/me/tickets`.
2. El sistema extrae el client_id del JWT (claim sub).
3. El sistema filtra los tickets donde client_id = sub del JWT.
4. El cliente recibe solo sus propios tickets aunque su rol sea "cliente".

## Modelo de Permisos

### Estructura: recurso:acción

Cada permiso sigue el formato `recurso:acción` donde:
- **recurso**: nombre del módulo o entidad (ej: tickets, reports, sedes, users, gates, payments)
- **acción**: operación permitida (read, write, delete, execute, admin)

### Catálogo de Roles y Permisos

#### Superadmin

| Recurso | Permisos |
|---------|----------|
| tenants | admin, read, write |
| sedes | admin, read, write, delete |
| users | admin, read, write, delete |
| roles | admin, read, write |
| reports | admin, read, write, export |
| gates | admin, read, write, execute |
| tickets | admin, read, write, delete |
| payments | admin, read, write |
| audit | read, export |
| config | admin, read, write |

#### Admin de Sede

| Recurso | Permisos |
|---------|----------|
| sedes | read (propia) |
| users | read, write (operadores de su sede) |
| reports | read, export (propia) |
| gates | read, execute |
| tickets | read, write (propios) |
| payments | read |
| audit | read (propia sede) |

#### Operador

| Recurso | Permisos |
|---------|----------|
| gates | execute |
| tickets | read, write (apertura/cierre) |
| reports | read (propia sede) |
| payments | read |
| vehicles | read (búsqueda por placa) |

#### Cliente

| Recurso | Permisos |
|---------|----------|
| tickets | read (propios) |
| payments | read (propios) |
| vehicles | write (registro propio) |
| profile | read, write (propio) |

## Datos de Entrada

### Creación de Rol

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| nombre | string | Nombre único del rol (ej: "operador_sede_B") | Sí |
| descripcion | string | Descripción del propósito del rol | No |
| permisos | array | Lista de permisos en formato "recurso:accion" | Sí |

### Asignación de Rol a Usuario

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| usuario_id | UUID | ID del usuario a modificar | Sí |
| rol_id | UUID | ID del rol a asignar | Sí |
| sede_ids | array | IDs de sedes a las que tiene acceso (null = todas del tenant) | Condicional |
| motivo | string | Razón del cambio (para auditoría) | Sí |

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Rol con nombre duplicado | Sistema rechaza con 409 Conflict y mensaje: "Ya existe un rol con ese nombre" |
| Permiso inválido (no existe recurso:accion) | Sistema rechaza con 400 y lista de permisos válidos |
| Usuario con rol eliminado | Sistema invalidates JWT del usuario en el siguiente refresh; requiere re-login |
| Admin intenta asignar rol superior al suyo | Sistema rechaza con 403: "No puedes asignar un rol con permisos superiores a los tuyos" |
| Intento de modificar permisos del propio rol | Sistema rechaza con 403: "No puedes modificar tu propio rol" |
| Acceso a sede no asignada | Sistema retorna 403 con mensaje: "No tienes acceso a la sede solicitada" |
| JWT manipulado (firma inválida) | Gateway retorna 401 y registra alerta de seguridad |
| Permiso no existe en JWT | El sistema verifica contra la lista de permisos del rol (no solo el JWT) para evitar bypass |
| Múltiples roles para un usuario (future) | La implementación inicial usa un rol único; multi-rol se considera en versión 2.0 |
| Recurso eliminado pero aún referenciado en permisos | Sistema不许 حذفالمورد si tiene permisos asociados; debe primero limpiar permisos |

## Criterios de Aceptación

1. **CA-028-01**: Un usuario con rol "operador" NO puede ejecutar comandos sobre talanqueras de sedes diferentes a las asignadas en su JWT.
2. **CA-028-02**: Un admin_sede NO puede crear usuarios con permisos superiores a los suyos (principio de menor privilegio).
3. **CA-028-03**: El sistema retorna 403 Forbidden con mensaje claro cuando un usuario intenta acceder a un recurso sin permisos.
4. **CA-028-04**: Los permisos se verifican en el API Gateway antes de reenviar la request al servicio de backend (defense in depth).
5. **CA-028-05**: Un superadmin puede crear nuevos roles, asignar permisos arbitrarios, y asignar roles a usuarios.
6. **CA-028-06**: Los permisos en el JWT son consistentes con los permisos actuales en la base de datos en el momento del login; después del login, los permisos se refriescan solo en el siguiente access token (no hay propagation instantánea de cambios de permisos).
7. **CA-028-07**: El sistema permite que un admin_sede gestione operadores de múltiples sedes (todas las sedes asignadas a su usuario).
8. **CA-028-08**: El intento de acceso sin autenticación (sin JWT) retorna 401 Unauthorized.
9. **CA-028-09**: Los endpoints públicos (ej: login, salud del sistema) no requieren autenticación y están excluidos de RBAC.

## Dependencias

**RFs relacionados**:
- RF-026 — Autenticación JWT (el JWT contiene los permisos como claim)
- RF-027 — Auditoría (todos los cambios de roles y permisos se auditan)

**Tablas de base de datos afectadas**:
- `roles` — catálogo de roles del sistema
- `role_permissions` — relación many-to-many entre roles y permisos
- `users` — referencia al rol asignado
- `user_sedes` — relación entre usuarios y sedes (para permisos por sede)

**Configuración**:
- La matriz de permisos se puede poblar vía seed data en migración.

---

## Endpoints API Involucrados

- `GET /api/v1/roles` — Lista todos los roles disponibles (solo superadmin)
- `POST /api/v1/roles` — Crea un nuevo rol (solo superadmin)
- `GET /api/v1/roles/{id}` — Detalle de un rol con sus permisos (solo superadmin)
- `PUT /api/v1/roles/{id}` — Actualiza un rol (solo superadmin)
- `DELETE /api/v1/roles/{id}` — Elimina un rol (solo superadmin, no se puede eliminar si hay usuarios asignados)
- `GET /api/v1/permissions` — Lista todos los permisos disponibles con descripción
- `GET /api/v1/users/{id}/role` — Consulta el rol actual de un usuario
- `PUT /api/v1/users/{id}/role` — Asigna un rol a un usuario (con validación de jerarquía)
- `GET /api/v1/permissions/check?resource={r}&action={a}` — Verifica si el usuario actual tiene un permiso específico (útil para UI show/hide)

---

## Métricas de Éxito

- Tiempo de verificación de permiso en Gateway: ≤ 5ms (P99)
- Tasa de denegación por falta de permisos: monitoreo como indicador de seguridad
- Cobertura de matriz de permisos: 100% de endpoints cubiertos
- Tiempo de propagación de cambio de rol: ≤ 15 min (hasta el próximo login del usuario)
- Número de roles creados: ≥ 4 (los 4 roles base) + roles custom por tenant

---

## Notas

- La verificación de permisos en el Gateway usa un cache local (in-memory) con TTL de 60s para no consultar Redis en cada request.
- La estructura de permisos `recurso:accion` permite extensibilidad: se pueden agregar nuevos recursos y acciones sin modificar código de autorización.
- La jerarquía de roles es: superadmin > admin_sede > operador > cliente. Un rol superior hereda todos los permisos del inferior automáticamente.
- Para permisos de sede específicos (admin_sede y operador), el JWT incluye un array `sede_ids` que limita el acceso a solo esas sedes.
- El claim `permisos` en el JWT es un array flat de strings, no un objeto anidado, para facilitar la verificación con `includes()`.
- En la versión 2.0 se contempla multi-rol (un usuario con operador + viewer) y permisos condicionales (ej: solo en horario laboral).
