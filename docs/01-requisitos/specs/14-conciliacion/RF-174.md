# RF-174 — Notificación al Admin cuando un Operador hace Cierre de Turno

## Metadata

- **Módulo**: Cierre de Turno
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: [notificacion-service, turno-service]

---

## User Story

**Como** administrador de sede **quiero** recibir una notificación cuando un operador hace cierre de turno **para** estar informado de la actividad en mi sede y poder supervisar los cierres en tiempo real.

---

## Objetivo

El sistema debe enviar una notificación inmediata al administrador de sede cuando un operador completa el cierre de turno. La notificación debe incluir los datos clave del cierre (operador, turno, resumen) y un enlace directo al reporte PDF generado.

---

## Actores

- Sistema (actor automatizado, genera la notificación)
- Administrador de sede (principal, recibe la notificación)
- Operador de caja (secundario, su cierre dispara la notificación)

---

## Precondiciones

- Un operador ha completado exitosamente el cierre de turno (RF-172)
- Se ha generado el reporte de cierre (RF-173)
- El administrador de sede tiene preferencias de notificación configuradas

---

## Postcondiciones

### En caso de éxito

- La notificación se envía al administrador de sede por los canales configurados
- La notificación incluye enlace al PDF del cierre
- Se registra el envío en audit log

### En caso de falla

- Si falla el envío, se reintenta automáticamente hasta 3 veces
- Si todos los intentos fallan, se registra la falla pero no se afecta el cierre del turno

---

## Flujo Funcional

1. Sistema detecta cierre de turno completado exitosamente
2. Sistema identifica al administrador(es) de sede asociados a la sede del operador
3. Sistema recolecta datos para la notificación:
   - Nombre del operador
   - Hora de inicio y cierre del turno
   - Total de ingresos del turno
   - Resultado de conciliación (conciliado / discrepancy)
   - Enlace al PDF del reporte
4. Sistema determina canales de notificación según preferencias del admin:
   - Email (si configurado)
   - Push notification (si tiene app móvil)
   - Slack (si configurado, ver RF-166)
   - SMS (si configurado, ver RF-166)
5. Sistema envía notificación por cada canal habilitado
6. Sistema registra en audit log:
   - Operador que cerró
   - Admin que recibió notificación
   - Canal utilizado
   - Timestamp
   - Contenido de la notificación

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|

| Campo | Tipo | Descripción |
|-------|------|-------------|
| notificacion_id | UUID | Identificador de la notificación |

| estado_envio | Enum | "enviado" / "fallido" / "pendiente_reintento" |
| timestamp_envio | Timestamp | Fecha y hora del envío |
| admin_receptor | UUID | ID del admin que recibió |

---

## Contenido de la Notificación

**Email/Push/Slack/SMS - Formato resumido**:
```
🔔 CIERRE DE TURNO

Operador: Juan Pérez
Sede: Centro
Turno: Mañana (06:00 - 14:00)
Ingresos totales: $610.000
Conciliación: CONCILIADO ✓

📄 Ver reporte: [Enlace al PDF]

Timestamp: DD/MM/YYYY HH:MM:SS
```

**Si hay discrepancia**:
```
⚠️ CIERRE DE TURNO - ALERTA

Operador: Juan Pérez
Sede: Centro
Turno: Mañana (06:00 - 14:00)
Ingresos totales: $608.500
Conciliación: DISCREPANCIA -$1.500

⚠️ Diferencia sin justificar

📄 Ver reporte: [Enlace al PDF]
🚨 Ver justificación: [Enlace]
```

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Admin tiene todos los canales desactivados | No enviar notificación, registrar en log que "admin no tiene canales activos" |
| Email del admin no existe o está mal configurado | Marcar como fallido, reintentar, eventually guardar en cola para reintento manual |
| Admin no está disponible (offline) | Push notification se marca como pendiente, se reintenta cuando esté online |
| Slack no disponible (API down) | Reintentar por otro canal disponible, no bloquear |
| Turno cerrado por admin de otra sede | No notificar a admins de otras sedes |
| Múltiples admins en la misma sede | Enviar notificación a todos los admins de la sede |
| Operador cierra turno pero no se generó PDF | Enviar notificación sin enlace al PDF,注明 "reporte pendiente" |

---

## Criterios de Aceptación

1. La notificación se envía dentro de los 60 segundos posteriores al cierre del turno
2. Todos los canales configurados para el admin reciben la notificación
3. La notificación incluye siempre los datos clave del turno
4. El enlace al PDF es accesible y válido por al menos 30 días
5. El admin puede silenciar notificaciones de cierre de turno (configuración personal)
6. Se mantiene historial de todas las notificaciones enviadas

---

## Métricas de Éxito

- Tiempo promedio entre cierre de turno y recepción de notificación: < 30 segundos
- Porcentaje de notificaciones entregadas exitosamente: > 98%
- Tasa de apertura de notificaciones por email: meta > 70%

---

## Notas

- El admin de sede puede delegar la recepción de notificaciones a otro usuario
- Las notificaciones de cierre con discrepancia tienen prioridad más alta (se envían primero)
- Se puede configurar un "horario silencioso" donde las notificaciones se acumulan y envían al día siguiente
- La notificación por SMS es solo para casos de discrepancia (no para cierres normales) para reducir costos
