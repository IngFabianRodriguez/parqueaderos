# RF-172 — Cierre de Turno con Consolidación de Ingresos y Operaciones

## Metadata

- **Módulo**: Cierre de Turno
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: [turno-service, conciliacion-service, caja-service]

---

## User Story

**Como** operador de caja **quiero** realizar el cierre de mi turno consolidando todos los ingresos, aperturas de talanquera y alertas atendidas **para** dejar el turno correctamente cerrado y permitir que el siguiente operador comience con claridad.

---

## Objetivo

El sistema debe permitir al operador de caja cerrar su turno consolidando y mostrando un resumen completo que incluye: total de ingresos por tipo de transacción, aperturas de talanquera registradas durante el turno, alertas atendidas, y el resultado de la conciliación de efectivo. El cierre solo se completa si la conciliación no tiene discrepancias sin justificar.

---

## Actores

- Operador de caja (principal)
- Administrador de sede (secundario, puede forzar cierre de turno de otro operador)
- Sistema (secundario, valida condiciones de cierre)

---

## Precondiciones

- El operador tiene un turno en estado "abierto"
- El operador tiene permisos para cerrar turnos
- Todas las transacciones del turno están procesadas

---

## Postcondiciones

### En caso de éxito

- El turno pasa a estado "cerrado"
- Se genera reporte de cierre de turno (RF-173)
- Se calcula y almacena el resumen del turno
- El siguiente operador puede abrir un nuevo turno en la misma caja

### En caso de falla

- Si existe discrepancia sin justificar, se impide el cierre hasta que se justifique o se apruebe
- Si hay transacciones pendientes de confirmación, se muestra advertencia

---

## Flujo Funcional

1. Operador selecciona opción "Cerrar Turno" desde su dashboard
2. Sistema verifica que no existan alertas pendientes sin atender del turno
3. Sistema presenta pantalla de resumen con:
   a. Total de cobros de pasaje (cantidad y monto)
   b. Total de recargas (cantidad y monto)
   c. Total de otros ingresos
   d. Lista de aperturas de talanquera del turno
   e. Total de alertas atendidas durante el turno
   f. Resultado de conciliación (diferencia si existe)
4. Operador confirma los datos visualizados
5. Sistema verifica estado de conciliación:
   a. Si hay discrepancia sin justificación: mostrar error, no permitir cierre
   b. Si hay discrepancia con justificación pendiente: mostrar advertencia, permitir cerrar con confirmación adicional
   c. Si todo está conciliado o justificado: continuar con cierre
6. Operador confirma el cierre definitivo
7. Sistema:
   a. Actualiza estado del turno a "cerrado"
   b. Genera reporte de cierre (RF-173)
   c. Notifica al administrador (RF-174)
   d. Libera la caja para el siguiente turno
8. Sistema muestra confirmación de cierre exitoso con número de reporte

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Discrepancia sin justificar | Impedir cierre, mostrar "Debe justificar la diferencia antes de cerrar" |
| Discrepancia con justificación pendiente | Mostrar advertencia, requerir confirmación adicional del operador |
| Hay alertas sin atender | Mostrar lista de alertas, permitir cierre solo con confirmación explícita |
| Turno ya está cerrado | Retornar error "El turno ya se encuentra cerrado" |
| Operador intenta cerrar desde otra sede | Retornar error "No puede cerrar un turno de otra sede" |
| Hay transacciones no confirmadas | Advertir y mostrar transacciones pendientes, permitir cierre con confirmación |
| Admin fuerza cierre de operador ausente | Registrar en audit log que el cierre fue forzado |

---

## Criterios de Aceptación

1. El cierre de turno solo se permite si todas las discrepancias están justificadas o aprobadas
2. El resumen muestra exactamente los totales por tipo de transacción
3. El cierre genera un ID único de reporte que se puede consultar posteriormente
4. El operador puede cancelar la operación de cierre en cualquier momento antes de confirmar
5. El sistema impide que se abra un nuevo turno en la misma caja si hay uno abierto
6. El timestamp de cierre es exacto al segundo y no modificable

- `POST /api/v1/turnos/{turno_id}/cerrar` — Cerrar un turno

---

## Métricas de Éxito

- Porcentaje de turnos cerrados sin discrepancias: meta > 95%
- Tiempo promedio del proceso de cierre (inicio a confirmación): < 5 minutos
- Cantidad de cierres forzados por admin vs cierres normales

---

## Notas

- El operador debe tener posibilidad de imprimir el resumen antes de cerrar (sin ser el cierre oficial)
- El cierre de turno debe ser irrevocable - no se puede "des cerrar" un turno
- El administrador de sede puede cerrar turnos de cualquier operador de su sede
- Se debe garantizar que no haya dinero sin registrar al momento del cierre
