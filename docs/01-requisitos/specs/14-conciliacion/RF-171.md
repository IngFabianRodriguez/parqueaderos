# RF-171 — Registro de Diferencia de Caja con Justificación y Evidencia

## Metadata

- **Módulo**: Conciliación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: [conciliacion-service, notificacion-service]

---

## User Story

**Como** operador de caja **quiero** registrar una diferencia de caja con su justificación y evidencia fotográfica **para** dejar documentado el motivo de la discrepancia y que sea auditado por el administrador.

---

## Objetivo

Cuando se detecta una discrepancia entre el efectivo esperado y el real, el operador debe poder registrar una justificación que explique la diferencia, incluyendo descripción del motivo y evidencia fotográfica del dinero contado al momento de la detección. Este registro queda vinculado a la conciliación del turno para auditoría.

---

## Actores

- Operador de caja (principal, registra la justificación)
- Administrador de sede (secundario, revisa y puede aprobar/rechazar justificación)
- Auditor (secundario, consulta registros históricos)

---

## Precondiciones

- Existe una conciliación con estado "en_discrepancia" para el turno
- El operador tiene acceso al turno correspondiente
- El dispositivo del operador tiene acceso a la cámara (para captura de evidencia)

---

## Postcondiciones

### En caso de éxito

- Se crea registro de justificación asociado a la conciliación
- Se adjunta evidencia fotográfica al registro
- El estado de la conciliación puede pasar a "justificado" o permanecer "en_discrepancia" según configuración

### En caso de falla

- Si no existe conciliación previa, se retorna error
- Si la imagen no se puede subir, se permite guardar la justificación sin imagen pero se marca como "evidencia_pendiente"

---

## Flujo Funcional

1. Operador visualiza la conciliación con discrepancia
2. Operador selecciona opción "Registrar Justificación"
3. Sistema presenta formulario con campos: motivo (dropdown), descripción (texto), evidencia (foto)
4. Operador selecciona motivo de lista predefinida:
   - Error de conteo del operador
   - Billete falso detectado
   - Transacción no registrada en sistema
   - Venta cancelada que no apareció en cierre
   - Otro (requiere descripción)
5. Operador ingresa descripción ampliada del motivo
6. Operador captura foto del dinero contado mostrando el monto total
7. Sistema valida que la imagen sea legible y del tamaño correcto
8. Sistema almacena la imagen y crea el registro de justificación
9. Sistema vincula la justificación a la conciliación del turno
10. Sistema notifica al administrador de sede sobre la justificación registrada
11. Administrador puede aprobar o rechazar la justificación

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| conciliacion_id | UUID | Identificador de la conciliación | Sí |
| operador_id | UUID | Identificador del operador que justifica | Sí |
| motivo | Enum | Código del motivo predefinido | Sí |
| descripcion | Text | Descripción ampliada del motivo | Sí |
| evidencia_foto | File | Imagen JPG/PNG del dinero contado (max 5MB) | No |
| fecha_registro | Timestamp | Fecha y hora del registro | Sí (auto) |

**Motivos predefinidos (enum motivo_diferencia)**:
- `ERROR_CONTEO` — Error de conteo del operador
- `BILLETE_FALSO` — Billete falso detectado y retenido
- `TRANSACCION_OMITIDA` — Transacción no registrada en sistema
- `VENTA_CANCELADA` — Venta cancelada no reflejada en cierre
- `SOBRANTE_CAJA` — Dinero de más sin origen identificado
- `FALTANTE_CAJA` — Dinero de menos sin origen identificado
- `OTRO` — Otro motivo (requiere descripción)

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| justificacion_id | UUID | Identificador de la justificación creada |
| estado | Enum | "pendiente_aprobacion" / "aprobada" / "rechazada" |
| fecha_creacion | Timestamp | Fecha y hora de creación |
| administrador_revisor | UUID | ID del admin que revisó (si aplicacble) |
| fecha_revision | Timestamp | Fecha y hora de revisión (si aplicable) |
| comentario_revision | Text | Comentario del admin al aprobar/rechazar |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Imagen muy grande (>5MB) | Rechazar con mensaje "La imagen debe ser menor a 5MB" |
| Imagen borrosa o ilegible | Advertir al operador pero permitir guardar (con marca) |
| Sin imagen de evidencia | Permitir registro con campo evidencia_pendiente = true |
| Operador intenta justificar dos veces el mismo turno | Rechazar con mensaje "Ya existe una justificación para este turno" |
| Admin rechaza justificación | Notificar al operador, la discrepancia permanece activa |
| Admin aprueba justificación | Notificar al operador, estado pasa a "justificado" |

---

## Criterios de Aceptación

1. El operador puede registrar justificación dentro de las 24 horas posteriores al cierre del turno
2. La foto de evidencia debe mostrar claramente los billetes/contado
3. El administrador recibe notificación inmediata de nueva justificación
4. El registro de justificación es inmutable (solo se añade estado de revisión)
5. Se permite adjuntar hasta 3 fotos como evidencia
6. La justificación queda vinculada al turno y operador específico

## Endpoints API Involucrados

- `POST /api/v1/conciliacion/justificacion` — Crear justificación de diferencia
- `GET /api/v1/conciliacion/justificacion/{justificacion_id}` — Ver detalles de justificación
- `PUT /api/v1/conciliacion/justificacion/{justificacion_id}/revisar` — Aprobar o rechazar justificación (admin)
- `POST /api/v1/conciliacion/justificacion/{justificacion_id}/evidencia` — Subir foto de evidencia

---

## Métricas de Éxito

- Porcentaje de justificaciones revisadas por admin en menos de 24 horas: > 90%
- Cantidad de justificaciones rechazadas vs aprobadas
- Tiempo promedio de resolución de discrepancias justificadas

---

## Notas

- Las justificaciones aprobadas no modifican el monto de la diferencia, solo la documentan
- El admin debe poder rechazar una justificación si el motivo no es válido
- Se debe mantener historial completo de todas las justificaciones para auditorías
- La foto de evidencia se almacena en object storage (S3/MinIO) con referencia en BD
