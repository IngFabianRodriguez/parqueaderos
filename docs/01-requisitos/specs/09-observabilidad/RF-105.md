# RF-105 — Exposición de Métricas en Formato Prometheus

## Metadata

- **Módulo**: Observabilidad — Exposición de Métricas
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: todos los microservicios

---

## User Story

**Como** operador del sistema **quiero** que todos los microservicios expongan sus métricas en formato Prometheus **para** poder integrar con Grafana y visualizar dashboards de monitoreo unificados.

---

## Objetivo

Cada microservicio del sistema debe exponer un endpoint `GET /metrics` que retorne métricas en formato Prometheus (texto plano). Este endpoint debe ser scrapeado por un servidor Prometheus central que recolectará todas las métricas para visualización en Grafana. Las métricas incluyen: health del servicio (RF-100), uso de recursos (RF-103), latencia de APIs (RF-104), y métricas específicas del negocio.

---

## Actores

- **Actor principal**: Sistema (Prometheus server scrapea los endpoints)
- **Actor secundario**: Operador de planta, `tenant_admin`

---

## Precondiciones

- Prometheus server está configurado con la lista de targets (servicios y sus URLs de `/metrics`).
- Cada microservicio tiene instrumentación de métricas (client library Prometheus).

---

## Postcondiciones

### En caso de éxito

- El endpoint `/metrics` retorna un body en texto plano con formato Prometheus correcto.
- Prometheus puede hacer scrape sin errores.
- Las métricas están disponibles en Grafana.

### En caso de falla

- Si el servicio tiene problemas, el endpoint `/metrics` puede responder con errores parciales o timeout.
- Prometheus marca el target como `DOWN` si no puede hacer scrape.

---

## Formato Prometheus

El endpoint `/metrics` retorna texto plano con formato:

```
# HELP gateway_http_requests_total Total HTTP requests to gateway
# TYPE gateway_http_requests_total counter
gateway_http_requests_total{method="GET",path="/api/v1/health",status="200"} 15234

# HELP gateway_http_request_duration_seconds Latency of HTTP requests
# TYPE gateway_http_request_duration_seconds histogram
gateway_http_request_duration_seconds_bucket{method="GET",path="/api/v1/health",status="200",le="0.1"} 15000
gateway_http_request_duration_seconds_bucket{method="GET",path="/api/v1/health",status="200",le="0.5"} 15200
gateway_http_request_duration_seconds_bucket{method="GET",path="/api/v1/health",status="200",le="+Inf"} 15234
gateway_http_request_duration_seconds_sum{method="GET",path="/api/v1/health",status="200"} 1456.78
gateway_http_request_duration_seconds_count{method="GET",path="/api/v1/health",status="200"} 15234
```

### Tipos de Métricas Prometheus Soportados

| Tipo | Descripción | Uso |
|------|-------------|-----|
| `counter` | Valor que solo incrementa | Requests total, errores total |
| `gauge` | Valor que puede subir y bajar | CPU%, memoria usada, conexiones activas |
| `histogram` | Distribucion de valores en buckets | Latencia, tamaño de request |
| `summary` | Percentiles calculados en el cliente | Latencia (alternativa a histogram) |

## Métricas Requeridas en Todos los Servicios

### Estándar (obligatorias en todos los microservicios)

```
# Service identity
service_build_info{version="1.0.0",service_name="billing-service"} 1

# HTTP metrics (si el servicio expone HTTP)
http_requests_total{method="",path="",status=""} counter
http_request_duration_seconds_bucket{...,le=""} histogram

# Process metrics (from Prometheus client library)
process_cpu_seconds_total gauge
process_memory_bytes_bytes gauge
process_resident_memory_bytes gauge
process_start_time_seconds gauge
```

### Específicas por Servicio

Cada servicio expone métricas relevantes para su dominio (ej: `transactions_processed_total`, `rates_calculated_total`, etc.) que se definen en la sección de RFs del módulo correspondiente.

---

## Configuración de Prometheus

```yaml
scrape_configs:
  - job_name: 'parking-observability'
    scrape_interval: 15s
    scrape_timeout: 10s
    static_configs:
      - targets:
        - 'api-gateway:8080'
        - 'auth-service:8080'
        - 'billing-service:8080'
        - 'rate-service:8080'
        - 'device-hub:8080'
        - 'notification-service:8080'
    relabel_configs:
      - source_labels: [__address__]
        target_label: service_name
        regex: '([^:]+):.*'
        replacement: '${1}'
```

## Requisitos del Endpoint

|| Requisito | Valor |
|---|-----------|-------|
| Puerto | El mismo que el servicio (ej: 8080) |
| Path | `/metrics` |
| Autenticación | Ninguna (Prometheus no soporta auth) |
| Timeout de respuesta | < 5s |
| Formato | `text/plain; version=0.0.4` |
| Compresión | Opcional (gzip si `Accept-Encoding: gzip`) |

---

## Casos de Borde y Manejo de Errores

|| Escenario | Comportamiento Esperado |
|---|-----------|--------------------------|
| Prometheus no puede hacer scrape | Target marcado como DOWN en Prometheus; Grafana muestra "No data" |
| Métricas con alta cardinalidad | Limitar labels (path sin IDs, no usar user_id como label) |
| Servicio arrancando (来不及 inicializar) | `/metrics` retorna 503 hasta que esté listo |
| Métricas muy pesadas (> 10MB) | Segmentar en múltiples endpoints (`/metrics`, `/metrics/infra`, `/metrics/app`) |
| Cliente Prometheus malicioso | Rate limit: máximo 1 request cada 10s por cliente |

---

## Criterios de Aceptación

1. Cada microservicio expone `GET /metrics` en el puerto HTTP principal.
2. Las métricas incluyen las estándar de proceso (`process_*`) y las específicas del servicio.
3. El formato es 100% compatible con Prometheus (sin errores de parseo).
4. Prometheus puede hacer scrape cada 15s sin impacto medible en el servicio.
5. Las métricas de un servicio no afectan las de otro (aislamiento).
6. Si el servicio está en mantenimiento y `/health` retorna `DOWN`, `/metrics` también retorna código de error.
7. Las métricas están correctamente etiquetadas con `service_name` y `tenant_id` cuando aplica.
8. Se puede hacer scrape de `/metrics` desde cualquier red interna sin autenticación.

---

## Notas

- Se recomienda usar la librería oficial Prometheus client para el lenguaje del servicio.
- No exponer métricas con labels de alta cardinalidad (user_id, session_id, transaction_id).
- Si el servicio tiene muchos endpoints, usar `path` normalizado (sin IDs) como label.
- Para servicios batch o jobs, exponer `job_batch_last_success_timestamp` para monitoreo de jobs.
- Las métricas `_total` de tipo counter deben usar el sufijo `_total` por convención de nombres.
- Se debe documentar cada métrica con `# HELP` y `# TYPE` para claridad en Prometheus UI.
