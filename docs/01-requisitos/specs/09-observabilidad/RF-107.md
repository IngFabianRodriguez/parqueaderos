# RF-107 — Configuración de Canales de Notificación de Alertas

## Metadata

- **Módulo**: Observabilidad — Alertas Configurables
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: alerting-service, notification-service

---

## User Story

**Como** `tenant_admin` **quiero** elegir cómo recibir las alertas (email, SMS, Slack, webhook) **para** asegurarme de que el equipo correcto sea notificado por el canal apropiado según la severidad de la alerta.

---

## Objetivo

El sistema debe permitir al `tenant_admin` configurar uno o más canales de notificación para recibir alertas. Los canales disponibles son: email, SMS, Slack, y webhook (HTTP POST a una URL). Cada canal tiene su propia configuración (dirección de email, número de teléfono, workspace de Slack, URL de webhook). Se pueden configurar múltiples canales por alerta, y cada canal puede restringirse a ciertas severidades (ej: CRITICAL por SMS, WARNING por email).

---

## Actores

- **Actor principal**: `tenant_admin`
- **Actor secundario**: Sistema (envía notificaciones a través de los canales configurados)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.
- Los canales deseados están disponibles y configurados a nivel de infraestructura (ej: proveedor de SMS, integración con Slack).

---

## Postcondiciones

### En caso de éxito

- Los canales se guardan en la configuración del tenant.
- Cuando se genera una alerta, se envía a todos los canales activos que correspondan a la severidad de la alerta.
- El `tenant_admin` puede agregar, editar, eliminar y probar los canales.

### En caso de falla

- Si un canal está mal configurado (ej: email inválido, URL de webhook inaccesible): se marca como fallido y se reintenta.
- Si todos los canales fallan, la alerta se guarda en cola para reintento.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Observabilidad → Alertas → Canales de Notificación`.
2. El sistema muestra los canales configurados actualmente.
3. El admin puede agregar un nuevo canal:
   - Selecciona el tipo: `email`, `sms`, `slack`, `webhook`.
   - Ingresa la configuración específica (email, teléfono, workspace, URL).
   - Define a qué severidades aplica: `[CRITICAL]`, `[WARNING]`, `[INFO]`, o cualquier combinación.
   - Define si el canal está activo o no.
4. El admin puede probar el canal (enviar mensaje de prueba).
5. El canal se guarda en `notification_channels`.
6. Cuando se genera una alerta, el `alerting-service` consulta los canales activos para la severidad de la alerta y envía la notificación a cada uno.

## Tipos de Canal y Configuración

### Email

|| Campo | Tipo | Descripción |
|---|-------|------|-------------|
| channel_type | `email` | Tipo de canal |
| recipients | array<string> | Lista de emails |
| from_name | string | Nombre del remitente (ej: "Parking Alerts") |
| smtp_config_id | uuid | Configuración del servidor SMTP |
| severity_filter | array | Severidades: `["CRITICAL", "WARNING"]` |
| is_active | boolean | ¿Activo? |

### SMS

|| Campo | Tipo | Descripción |
|---|-------|------|-------------|
| channel_type | `sms` | Tipo de canal |
| phone_numbers | array<string> | Lista de números de teléfono |
| provider | enum | `twilio`, `aws_sns`, `vonage` |
| provider_config | json | Credenciales y config del provider |
| severity_filter | array | Severidades |
| is_active | boolean | ¿Activo? |

### Slack

|| Campo | Tipo | Descripción |
|---|-------|------|-------------|
| channel_type | `slack` | Tipo de canal |
| slack_workspace_id | string | Workspace de Slack |
| slack_channel_id | string | ID del canal (ej: `#alerts-ops`) |
| webhook_url | string | Webhook URL de Slack |
| severity_filter | array | Severidades |
| is_active | boolean | ¿Activo? |

### Webhook

|| Campo | Tipo | Descripción |
|---|-------|------|-------------|
| channel_type | `webhook` | Tipo de canal |
| url | string | URL del endpoint (debe ser HTTPS en producción) |
| method | enum | `POST`, `PUT` (default: POST) |
| headers | json | Headers adicionales (ej: Authorization) |
| auth_type | enum | `none`, `basic`, `bearer`, `api_key` |
| auth_config | json | Credenciales según auth_type |
| severity_filter | array | Severidades |
| is_active | boolean | ¿Activo? |

## Formato de la Notificación

### Email

```
Subject: [CRITICAL] Servicio DOWN - Rate Service

Alerta: SERVICE_DOWN
Severidad: CRITICAL
Servicio: rate-service
Descripción: El servicio Rate Service no ha respondido durante 2 minutos.
Timestamp: 2026-05-08 16:00:00 UTC
Ver detalle: https://admin.parking.com/observability/alerts/abc123
```

### SMS

```
[CRITICAL] rate-service DOWN hace 2 min. Ver: https://admin.parking.com/a/abc123
```

### Slack

```json
{
  "text": "[CRITICAL] *rate-service* está DOWN",
  "attachments": [{
    "color": "danger",
    "fields": [
      { "title": "Servicio", "value": "rate-service", "short": true },
      { "title": "Severidad", "value": "CRITICAL", "short": true },
      { "title": "Desde", "value": "2026-05-08 15:58:00 UTC", "short": true }
    ],
    "actions": [
      { "type": "button", "text": "Ver Detalle", "url": "..." }
    ]
  }]
}
```

### Webhook (HTTP POST body)

```json
{
  "alert_id": "uuid-v4",
  "type": "SERVICE_DOWN",
  "severity": "CRITICAL",
  "timestamp": "2026-05-08T16:00:00Z",
  "service_name": "rate-service",
  "message": "Rate Service no ha respondido durante 2 minutos",
  "details": {
    "down_since": "2026-05-08T15:58:00Z",
    "retry_count": 3
  }
}
```

## Prueba de Canal

El admin puede enviar un mensaje de prueba a cualquier canal configurado. El sistema:
1. Genera una alerta de prueba (tipo `CHANNEL_TEST`, severity `INFO`).
2. Envía la notificación al canal.
3. Muestra "Éxito" o "Falló: [razón]" según el resultado.

---

## Casos de Borde y Manejo de Errores

|| Escenario | Comportamiento Esperado |
|---|-----------|--------------------------|
| Email inválido | Error de validación: "Email no válido" |
| Webhook URL inaccesible | La alerta se marca como "pending_retry"; se reintenta 3 veces con backoff |
| SMS provider down | Guardar en cola; reintentar cuando el provider esté disponible |
| Slack token expirado | Marcar canal como `AUTH_ERROR`; notificar al admin |
| Canal recibe alerta de severidad no configurada | No se envía (silenciamiento implícito) |
| Número de teléfono internacional | Validar formato E.164 (ej: +5215551234567) |
| Máximo de canales por tenant | 20 canales; error si se intenta agregar más |
| Webhook timeout (> 10s) | Considerar fallido, reintentar |

---

## Criterios de Aceptación

1. El `tenant_admin` puede agregar, editar, eliminar y activar/desactivar canales de notificación.
2. Los canales disponibles son: email, SMS, Slack, webhook.
3. Cada canal tiene configuración específica según el tipo.
4. Cada canal puede filtrar por severidad (ej: solo CRITICAL van por SMS).
5. Se puede probar cada canal con un mensaje de prueba.
6. Las notificaciones incluyen: tipo de alerta, severidad, servicio afectado, timestamp, y link al dashboard.
7. Los canales fallidos se reintentan automáticamente con backoff exponencial.
8. El sistema no envía notificaciones duplicadas en menos de 5 minutos para la misma alerta.
9. Todas las credenciales (tokens, API keys) se almacenan encriptadas.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/notification-channels` — Listar canales
- `POST /api/v1/tenants/{tenant_id}/notification-channels` — Crear canal
- `PUT /api/v1/tenants/{tenant_id}/notification-channels/{channel_id}` — Actualizar canal
- `DELETE /api/v1/tenants/{tenant_id}/notification-channels/{channel_id}` — Eliminar canal
- `POST /api/v1/tenants/{tenant_id}/notification-channels/{channel_id}/test` — Probar canal
- `PATCH /api/v1/tenants/{tenant_id}/notification-channels/{channel_id}/activate` — Activar
- `PATCH /api/v1/tenants/{tenant_id}/notification-channels/{channel_id}/deactivate` — Desactivar

---

## Notas

- Se recomienda que el email incluya un link directo al dashboard de alertas para facilitar la investigación.
- Para webhooks, el `auth_type` puede ser `api_key` (envía `X-API-Key` header) o `bearer` (envía `Authorization: Bearer <token>`).
- El log de `alert_notifications` debe incluir: canal, alert_id, timestamp_envio, timestamp_ack, status (`SENT`, `FAILED`, `PENDING_RETRY`).
- Se debe implementar rate limiting por canal para evitar sobrecargar proveedores (ej: máx 100 SMS/hora).
- Las credenciales nunca deben aparecer en logs; usar IDs de credencial en su lugar.
