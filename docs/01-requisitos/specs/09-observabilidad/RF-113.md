# RF-113 — Búsqueda de Logs por Trace ID

## Metadata

- **Módulo**: Observabilidad — Trazabilidad Distributed (Tracing)
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: logging-service, api-gateway, todos los microservicios

---

## User Story

**Como** equipo de operaciones **quiero** poder buscar todos los logs relacionados con un `trace_id` específico **para** reconstruir el flujo completo de una transacción y diagnosticar problemas de manera rápida.

---

## Objetivo

El sistema debe permitir buscar logs por `trace_id` para reconstruir el flujo completo de una transacción. Todos los logs generados durante el procesamiento de un request compartirán el mismo `trace_id`, permitiendo visualizarlos en orden cronológico y entender exactamente qué pasó en cada microservicio.

---

## Actores

- **tenant_admin y equipo de operaciones**: Realizan búsquedas de logs
- **Sistema (logging-service)**: Indexa y sirve las búsquedas de logs

---

## Precondiciones

- El sistema debe tener logs centralizados de todos los microservicios
- Los logs deben incluir el campo `trace_id`

---

## Postcondiciones

### En caso de éxito

- Se devuelve una lista ordenada de todos los logs del trace
- Cada entrada de log incluye: timestamp, servicio, acción, mensaje, trace_id, span_id

### En caso de falla

- Si no se encuentran logs para el trace_id, se devuelve respuesta vacía con mensaje informativo
- Si el servicio de logs no está disponible, se devuelve error 503

---

## Flujo Funcional

1. El usuario (operador/admin) ingresa un `trace_id` en el campo de búsqueda del dashboard de logs
2. El `logging-service` recibe la consulta `GET /api/v1/logs?trace_id={trace_id}`
3. El servicio busca en el índice todos los documentos donde `trace_id` = valor ingresado
4. Los resultados se ordenan por timestamp ascendente (cronológico)
5. Se retorna una lista con cada entrada de log incluyendo: servicio, timestamp, acción, mensaje, metadata
6. El dashboard muestra la traza completa como una lista expandible o como un diagrama de flujo
7. Cada entrada de log tiene un link para ver el detalle (span completo)

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| trace_id | UUID v4 | Identificador del trace a buscar | Si |
| from | timestamp | Fecha/hora inicial del rango (opcional, default: 24h atrás) | No |
| to | timestamp | Fecha/hora final del rango (opcional, default: ahora) | No |
| service | string | Filtrar solo logs de un servicio específico (opcional) | No |
| level | enum | Filtrar por nivel: DEBUG, INFO, WARN, ERROR (opcional) | No |
| limit | integer | Número máximo de resultados (opcional, default: 500, max: 5000) | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| trace_id | UUID v4 | Identificador del trace |
| total_logs | integer | Número total de entradas encontradas |
| logs | array | Lista de entradas de log ordenadas cronológicamente |
| duration_ms | integer | Duración total del trace (primer log a último) |

Cada entrada de log:

| Campo | Tipo | Descripción |
|-------|------|-------------|
| timestamp | ISO8601 | Momento en que se generó el log |
| service | string | Nombre del microservicio que generó el log |
| span_id | UUID | Identificador del span dentro del trace |
| level | enum | Nivel: DEBUG, INFO, WARN, ERROR |
| action | string | Acción/evento que se registra |
| message | string | Mensaje descriptivo del log |
| metadata | JSON | Datos adicionales relevantes (user_id, device_id, etc.) |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| trace_id no existe en los logs | Retorna array vacío con mensaje "No se encontraron logs para este trace_id" |
| trace_id muy antiguo (> 30 días) | Retorna array vacío; se sugiere que los logs pueden haber sido eliminados por retención |
| Logging service no disponible | Retorna error 503 con mensaje de mantenimiento |
| Trace con más de 5000 entradas | Se retorna las primeras 5000 con mensaje "Resultados truncados" |
| Búsqueda sin trace_id | Se retorna error 400 con mensaje "trace_id es requerido" |

---

## Criterios de Aceptación

1. Se puede buscar logs por cualquier `trace_id` generado por el API Gateway
2. Los resultados se muestran en orden cronológico (ascendente por timestamp)
3. Cada resultado incluye: servicio, timestamp, acción, mensaje, span_id
4. La búsqueda responde en menos de 3 segundos para traces de hasta 1000 entradas
5. Se puede filtrar por servicio, nivel de log y rango de fechas
6. Los logs de todos los microservicios están centralizados y consultables

## Endpoints API Involucrados

- `GET /api/v1/logs` — Búsqueda de logs con filtros (trace_id, servicio, nivel, rango)
- `GET /api/v1/traces/{trace_id}` — Endpoint dedicado para consultar un trace completo
- `GET /api/v1/logs/{log_id}` — Detalle de una entrada de log específica

---

## Métricas de Éxito

- Tiempo promedio de respuesta de búsqueda por trace_id (target: < 3 segundos)
- Porcentaje de búsquedas que retornan resultados (target: > 80% para traces de 24h)
- Uso del dashboard de logs por equipo de operaciones (métricas de uso interno)

---

## Notas

- Se recomienda usar Elasticsearch o similar para el indexado y búsqueda eficiente de logs
- El campo `trace_id` debe ser indexado como keyword para búsquedas exactas rápidas
- Para mejorar la experiencia, el dashboard puede mostrar los logs como un "timeline" visual donde cada servicio es una fila
- Las métricas de uso del dashboard de logs pueden ayudar a identificar patrones de errores frecuentes
