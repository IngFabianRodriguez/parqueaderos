# RF-104 — Métricas de Latencia API, Tiempo de Respuesta y Throughput

## Metadata

- **Módulo**: Observabilidad — Métricas de Aplicación
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: api-gateway, todos los microservicios, observability-service

---

## User Story

**Como** operador del sistema **quiero** ver las métricas de latencia del API Gateway, tiempo de respuesta de cada endpoint y throughput (req/min) **para** identificar bottlenecks, optimizar rendimiento y verificar que los SLAs se cumplen.

---

## Objetivo

El sistema debe trackear métricas de rendimiento a nivel de aplicación: latencia del API Gateway (tiempo total de la request), tiempo de respuesta desagregado por endpoint (downstream), y throughput medido en requests por minuto. Estas métricas permiten crear dashboards de rendimiento, establecer baselines de latencia y detectar degradación de rendimiento.

---

## Actores

- **Actor principal**: Sistema (recolecta y almacena métricas)
- **Actor secundario**: Operador de planta, `tenant_admin`, cliente (via portal público)

---

## Precondiciones

- El API Gateway instrumenta todas las requests entrantes.
- Cada microservicio instrumenta sus endpoints.
- El servicio de observabilidad tiene acceso a las métricas.

---

## Postcondiciones

### En caso de éxito

- Las métricas de latencia y throughput se recolectan en tiempo real.
- Los datos se exponen en `/metrics` para consumo por Prometheus.
- Los dashboards muestran p50, p95, p99 de latencia por endpoint.

### En caso de falla

- Si el middleware de métricas falla, el tráfico sigue fluyendo (fail-open).
- Si la DB de métricas está down, se sigue recolectando en memoria local (buffer, hasta 1h).

---

## Flujo Funcional

1. El API Gateway middleware intercepta cada request entrante y registra:
   - `request_timestamp` (inicio)
   - `http_method`, `path`, `status_code`
   - `tenant_id`, `user_id` (si autenticado)
2. Al completar la request, se calcula:
   - `latency_ms`: tiempo total desde que entró al gateway hasta que se respondió
   - `upstream_latency_ms`: tiempo que tomó el microservicio destino
3. El middleware envía la métrica a Prometheus con labels.
4. Cada microservicio también instrumenta sus handlers internos y envía `http_request_duration_seconds` con labels adicionales: `service_name`, `endpoint`, `method`.
5. Se calculan agregaciones: p50, p95, p99, throughput por endpoint.

## Métricas Definidas

### API Gateway

|| Métrica | Tipo | Descripción |
|---|---------|------|-------------|
| `gateway_http_requests_total` | counter | Total de requests al gateway |
| `gateway_http_request_duration_seconds` | histogram | Latencia total (请求 completa) |
| `gateway_http_request_size_bytes` | histogram | Tamaño del request body |
| `gateway_http_response_size_bytes` | histogram | Tamaño del response body |
| `gateway_upstream_latency_seconds` | histogram | Latencia del upstream (sin gateway overhead) |
| `gateway_active_connections` | gauge | Conexiones activas en el gateway |

### Histogramas de Latencia

Los histogramas usan los siguientes buckets (en segundos):
```
[0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10]
```

### Microservicios

|| Métrica | Tipo | Descripción |
|---|---------|------|-------------|
| `service_http_requests_total` | counter | Requests recibidas por el servicio |
| `service_http_request_duration_seconds` | histogram | Latencia del handler del servicio |
| `service_http_requests_in_flight` | gauge | Requests concurrently en ejecución |
| `service_upstream_calls_total` | counter | Llamadas a dependencias externas |
| `service_upstream_call_duration_seconds` | histogram | Latencia de llamadas upstream |

---

## Labels en Métricas

|| Label | Valores de ejemplo | Descripción |
|------|-------|-------------------|-------------|
| `service_name` | `billing-service`, `auth-service` | Nombre del servicio |
| `tenant_id` | `uuid-del-tenant` | Tenant del cliente |
| `method` | `GET`, `POST`, `PUT` | Método HTTP |
| `path` | `/api/v1/rates`, `/api/v1/transactions` | Ruta del endpoint (sin variables) |
| `status_code` | `200`, `404`, `500` | Código de respuesta HTTP |
| `status_class` | `2xx`, `4xx`, `5xx` | Clase del código de respuesta |
| `region` | `us-east-1` | Región del despliegue |

**Nota**: El label `path` no debe incluir IDs de recursos (ej: `/api/v1/users/{id}` no `/api/v1/users/12345`) para permitir agregación.

---

## Throughput (req/min)

El throughput se calcula como:
```
rate(gateway_http_requests_total[1m]) * 60  → req/min
```

O可直接 usar la métrica Prometheus:
```
gateway_http_requests_per_minute (generated from rate over 1m window)
```

---

## Retención y Agregación

| Período | Resolución |
|---------|------------|
| 0 - 6 horas | 15 segundos |
| 6h - 7 días | 1 minuto |
| 7 - 30 días | 5 minutos |

---

## Casos de Borde y Manejo de Errores

|| Escenario | Comportamiento Esperado |
|---|-----------|--------------------------|
| Endpoint con alta cardinalidad (IDs en path) | Normalizar path: reemplazar IDs con `{id}` antes de etiquetar |
| Métricas de un servicio down | El dashboard muestra gap, no 0 ni datos inventados |
| Histograma vacío (primer request) | Inicializar con buckets a 0 |
|throughput > 10000 req/min | Particionar por múltiples metric time series |
| Middleware de métricas crash | Fail-open: request continúa, error de métricas solo en log |
| Buffer de métricas en memoria lleno | Descartar métricas más antiguas, no nuevas |

---

## Criterios de Aceptación

1. La latencia del API Gateway se mide desde el primer byte del request hasta el último byte del response.
2. Cada endpoint tiene métricas de: count, p50, p95, p99, min, max latencia.
3. El throughput por endpoint se muestra en req/min.
4. Los dashboards pueden filtrar por `tenant_id`, `service_name`, `method`, `status_code`.
5. Los histograms permiten calcular percentiles en Grafana con `histogram_quantile`.
6. Las métricas están disponibles en `/metrics` para scraping Prometheus.
7. Se pueden configurar alertas cuando la latencia p99 supera un umbral por endpoint.
8. El overhead del middleware de instrumentación es < 1ms.

## Endpoints API Involucrados

- `GET /metrics` — Métricas Prometheus (incluye todas las métricas de aplicación)
- `GET /api/v1/observability/metrics/application` — API REST para métricas
  - Query params: `service_name`, `endpoint`, `from`, `to`, `percentile` (p50/p95/p99)
- `GET /api/v1/observability/metrics/application/export` — Exportar a CSV/JSON
- `GET /api/v1/observability/metrics/application/top-slow` — Top 10 endpoints más lentos

---

## Notas

- Para evitar alta cardinalidad, el label `path` debe ser la plantilla de ruta (sin IDs). Se normaliza el path en el middleware.
- La métrica `service_http_requests_in_flight` es útil para detectar饱和ación del servicio.
- Se recomienda usar `histogram` en vez de `summary` para latency porque permite agregar percentiles post-scraping.
- Para servicios con múltiples instancias, Prometheus agrega automáticamente las métricas.
- Se deben exponer métricas de errores (`status_code 5xx`) separately para facilitar alertas de error rate.
