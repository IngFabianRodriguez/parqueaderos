# RF-102 — Detección de Servicios Caídos y Latencia Anómala

## Metadata

- **Módulo**: Observabilidad — Detección de Anomalías
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: observability-service, alerting-service, todos los microservicios

---

## User Story

**Como** operador del sistema **quiero** que el sistema detecte automáticamente cuando un microservicio deja de responder o su latencia supera el umbral normal **para** recibir alertas oportunas y tomar acción antes de que los clientes notifiquen la falla.

---

## Objetivo

El sistema debe monitorear continuamente la disponibilidad y latencia de cada microservicio. Cuando un microservicio deja de responder por completo, se genera una alerta **CRITICAL** inmediatamente. Cuando la latencia supera el doble del baseline (p95 histórico), se genera una alerta **WARNING**. Las alertas se envían a través de los canales configurados (email, SMS, Slack, webhook) según lo definido en RF-107.

---

## Actores

- **Actor principal**: Sistema (detecta anomalías y genera alertas)
- **Actor secundario**: Operador de planta, `tenant_admin`

---

## Precondiciones

- Los endpoints `/health` de todos los microservicios están operativos.
- Existe un historial de latencia de al menos 7 días para calcular el baseline.
- Los canales de notificación están configurados (RF-107).

---

## Postcondiciones

### En caso de éxito

- Un microservicio que no responde genera una alerta CRITICAL en menos de 60s.
- Una latencia que supera 2x el baseline genera una alerta WARNING.
- La alerta incluye: servicio afectado, tipo de problema, timestamp, latencia actual vs baseline.
- La alerta se envía por todos los canales configurados.

### En caso de falla

- Si el servicio de observación está caído, las alertas no se envían (fallback: log local).
- Si no hay historial suficiente para baseline, se usa un threshold absoluto configurable.

---

## Flujo Funcional

### Detección de Servicio Caído (CRITICAL)

1. El `observability-service` hace polling al `/health` de cada microservicio cada 30s.
2. Si el endpoint no responde en < 10s (timeout), se reintenta 3 veces con backoff de 2s.
3. Si después de 3 intentos no hay respuesta, se marca el servicio como `DOWN`.
4. Se genera una alerta `CRITICAL` con: `service_name`, `down_since`, `last_latency_ms: null`.
5. La alerta se envía por todos los canales configurados.
6. Cada 30s mientras el servicio esté down, se reintenta el health check y se actualiza `down_since`.
7. Cuando el servicio vuelve, se genera un evento `RECOVERY` y se envía notificación.

### Detección de Latencia Anómala (WARNING)

1. El `observability-service` registra la latencia de cada health check en una serie temporal.
2. Cada 5 minutos se calcula el `baseline_p95` de las últimas 24h de datos.
3. Si la latencia actual supera `2 * baseline_p95`, se genera una alerta `WARNING`.
4. La alerta incluye: `service_name`, `current_latency_ms`, `baseline_p95_ms`, `threshold_ms`.
5. La alerta se reenvía cada 5 minutos mientras persista la condición, pero con deduplicación (mismo evento no reenviado en menos de 5min).

### Recuperación

6. Cuando un servicio `DOWN` vuelve a responder: se genera `SERVICE_RECOVERY` (info).
7. Cuando la latencia vuelve a niveles normales: se genera `LATENCY_RECOVERED` (info).

---

## Tipos de Alerta

| Tipo | Severidad | Gatillo | Tiempo de Detección |
|------|-----------|---------|---------------------|
| `SERVICE_DOWN` | CRITICAL | 3 timeouts consecutivos (>10s cada uno) en health check | < 60s desde el primer timeout |
| `SERVICE_DEGRADED` | WARNING | Latencia actual > 2x baseline_p95 de 24h | < 5 min desde que se excede el umbral |
| `SERVICE_RECOVERY` | INFO | Servicio vuelve a estado UP después de DOWN | Inmediato |
| `LATENCY_RECOVERED` | INFO | Latencia vuelve a estar por debajo de 1.5x baseline | < 5 min |

## Formato de Alerta

```json
{
  "alert_id": "uuid-v4",
  "type": "SERVICE_DOWN",
  "severity": "CRITICAL",
  "service_id": "rate-service",
  "service_name": "Rate Service",
  "timestamp": "2026-05-08T16:00:30Z",
  "payload": {
    "down_since": "2026-05-08T15:58:00Z",
    "last_latency_ms": null,
    "retry_count": 3,
    "affected_endpoints": ["/health", "/api/v1/rates"]
  },
  "channels": ["email", "slack"],
  "acknowledged": false
}
```

| alert_id | uuid | Identificador único de la alerta |

## Casos de Borde y Manejo de Errores

|| Escenario | Comportamiento Esperado |
|---|-----------|--------------------------|
| No hay historial de latencia (< 7 días) | Usar threshold absoluto configurable (ej: 5000ms) |
| Microservicio responde parcialmente (solo liveness, no readiness) | Considerar como `DEGRADED` |
| Latencia spike momentáneo (1 sola vez) | No generar alerta (requiere 2 lecturas consecutivas sobre umbral) |
| Múltiples servicios down simultáneamente | Generar una alerta por servicio, deduplicar en los canales |
| El observability-service se reinicia | Recupera estado desde `service_health_history` |
| Alerta CRITICAL no puede enviarse | Guardar en cola persistente y reintentar con backoff exponencial |

---

## Criterios de Aceptación

1. Una alerta CRITICAL se genera en menos de 60s desde que un servicio deja de responder.
2. Una alerta WARNING se genera en menos de 5 min desde que la latencia supera el umbral.
3. Cada alerta incluye: servicio, timestamp, tipo, severidad y datos de contexto.
4. Las alertas se envían por todos los canales configurados para el tenant.
5. Si un servicio recupera, se envía una notificación de `RECOVERY`.
6. El sistema no genera alertas por spikes momentáneos (requiere consistencia).
7. Las alertas pueden ser silenciadas temporalmente (maintenance windows, RF-108).
8. Existe una vista de historial de alertas para las últimas 24h.
9. El operador puede reconocer (acknowledge) una alerta para indicar que está trabajando en ella.

## Endpoints API Involucrados

- `GET /api/v1/observability/alerts` — Listar alertas activas y recientes
- `POST /api/v1/observability/alerts/{alert_id}/acknowledge` — Reconocer alerta
- `POST /api/v1/observability/alerts/{alert_id}/silence` — Silenciar alerta (con ventana de tiempo)
- `GET /api/v1/observability/alerts/{alert_id}/history` — Historial de cambios de estado
- `GET /api/v1/observability/services/{service_id}/latency-history` — Historial de latencia para dashboards

---

## Notas

- Para evitar falsos positivos por spikes de red, se requiere que la latencia超标 ocurra en 2 de 3 lecturas consecutivas antes de generar WARNING.
- El `baseline_p95` se recalcula cada 5 minutos usando un sliding window de 24h.
- Las alertas de tipo CRITICAL tienen deduplicación de 1min; WARNING de 5min.
- El campo `affected_endpoints` en el payload indica qué endpoints de negocio podrían verse afectados (estimado a partir de la arquitectura del servicio).
