# RF-100 — Health Checks por Microservicio

## Metadata

- **Módulo**: Observabilidad — Health Checks
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: Todos los microservicios

---

## User Story

**Como** operador del sistema **quiero** que cada microservicio expose un endpoint `/health` que retorne el estado de sus dependencias **para** poder detectar rápidamente cuándo un servicio está fallando o degradado.

---

## Objetivo

Cada microservicio del sistema debe exponer un endpoint HTTP `GET /health` que permita verificar su estado operativo. El endpoint debe incluir el estado de la base de datos, dependencias externas (Redis, brokers de mensaje, APIs de terceros) y uso de memoria. Este endpoint será consumido por balanceadores de carga, orquestadores y el dashboard de salud general.

---

## Actores

- **Actor principal**: Sistema (orquestadores, balanceadores, dashboard)
- **Actor secundario**: Operador de planta, `tenant_admin`

---

## Precondiciones

- El microservicio está corriendo y tiene acceso a sus dependencias.

---

## Postcondiciones

### En caso de éxito

- El endpoint `/health` retorna `HTTP 200` con un body JSON que incluye `status: "UP"`, estado de cada dependencia y métricas de memoria.
- Si una dependencia está down pero el servicio puede funcionar en modo degradado, retorna `HTTP 200` con `status: "DEGRADED"`.

### En caso de falla

- Si la base de datos no está alcanzable: retorna `HTTP 503` con `status: "DOWN"`.
- Si más del 50% de las dependencias están down: retorna `HTTP 503` con `status: "DOWN"`.

---

## Flujo Funcional

1. Un cliente (balanceador, orquestador, dashboard) envía `GET /health` al microservicio.
2. El microservicio ejecuta un health check paralelo sobre:
   - **DB principal**: Query `SELECT 1` o equivalente.
   - **Redis** (si aplica): Ping al servidor Redis.
   - **Brokers de mensaje** (si aplica): Conexión activa al broker.
   - **APIs externas críticas**: Request HEAD a la URL configurada con timeout de 3s.
   - **Memoria**: Lectura de `os.Getpid()` RSS o similar.
3. El microservicio compila los resultados en un JSON estructurado.
4. El servicio retorna el JSON con el código HTTP correspondiente.

---

## Formato de Respuesta

```json
{
  "service": "billing-service",
  "version": "1.4.2",
  "timestamp": "2026-05-08T16:00:00Z",
  "status": "UP",
  "dependencies": [
    { "name": "postgres", "status": "UP", "latency_ms": 4 },
    { "name": "redis", "status": "UP", "latency_ms": 1 },
    { "name": "mqtt-broker", "status": "UP", "latency_ms": 8 }
  ],
  "memory": {
    "rss_mb": 128,
    "heap_used_mb": 64,
    "gc_triggered": false
  }
}
```

### Estados Posibles

| `status` | Condición | HTTP Code |
|----------|-----------|-----------|
| `UP` | Todas las dependencias críticas responden | 200 |
| `DEGRADED` | Al menos una dependencia no crítica falla | 200 |
| `DOWN` | DB no responde o +50% dependencias down | 503 |

---

## Datos de Salida

|| Campo | Tipo | Descripción |
|---|-------|------|-------------|
| service | string | Nombre del servicio (ej: "billing-service") |
| version | string | Versión semver del servicio |
| timestamp | datetime | Momento del health check (ISO 8601) |
| status | enum | `UP`, `DEGRADED`, `DOWN` |
| dependencies | array | Lista de dependencias con nombre, estado, latencia |
| memory | object | Métricas de memoria del proceso |

---

## Casos de Borde y Manejo de Errores

|| Escenario | Comportamiento Esperado |
|---|-----------|--------------------------|
| DB no responde | Retorna 503, `status: "DOWN"`, log de error |
| Dependencia no crítica falla | Retorna 200, `status: "DEGRADED"`, incluye la dep en la lista como DOWN |
| Timeout en health check | Considerar DOWN si el timeout supera 5s |
| Memoria agotada (>90% disponible) | Considerar como `DEGRADED` |
| Health check solicitado durante shutdown | Retornar 503 inmediatamente |

---

## Criterios de Aceptación

1. Cada microservicio expone `GET /health` sin autenticación.
2. El endpoint responde en < 100 ms (sin contar timeouts de dependencias).
3. El estado de la DB se verifica con una query real, no solo conexión TCP.
4. Los estados `UP`/`DEGRADED`/`DOWN` se determinan según las reglas definidas.
5. El campo `memory.rss_mb` refleja el uso real de memoria del proceso.
6. Si el servicio tiene múltiples bases de datos (replica), se checkea la primaria.
7. El health check es idempotente (no modifica estado).
8. El dashboard puede hacer polling cada 30s sin impacto en rendimiento.

**RFs relacionados**:

- RF-102 — Detección de microservicios caídos o lentos

**Tablas de base de datos involucradas**: Ninguna (solo lecturas de estado).

**Infraestructura**:
- Los health checks de cada servicio deben poder agregarse desde un balanceador o herramienta de orquestación (Kubernetes liveness/readiness probes).

---

## Endpoints API Involucrados

- `GET /health` — Health check del microservicio (este es el endpoint que cada servicio expone).
- `GET /health/live` — Liveness probe (solo verifica que el proceso está vivo).
- `GET /health/ready` — Readiness probe (verifica que puede recibir tráfico).

---

## Notas

- El endpoint `/health` no requiere autenticación para facilitar los probes de Kubernetes.
- Se recomienda cachear el resultado del health check por 5-10s para evitar sobrecarga en la DB si hay muchos GET simultáneos, pero el cache debe invalidarse inmediatamente si una dependencia cambia de estado.
- El campo `gc_triggered` en `memory` es un booleano que indica si el último health check detectó que se ejecutó un Garbage Collection recientemente (útil para detectar memory pressure).
