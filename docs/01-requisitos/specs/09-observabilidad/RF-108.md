# RF-108 — Ventanas de Silencio (Maintenance Windows)

## Metadata

- **Módulo**: Observabilidad — Alertas Configurables
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: alerting-service, notification-service, observability-service

---

## User Story

**Como** `tenant_admin` **quiero** configurar ventanas de silencio (maintenance windows) durante las cuales las alertas no se envíen **para** realizar mantenimiento planificado sin que se disparen notificaciones falsas que molesten al equipo de operaciones.

---

## Objetivo

El sistema debe permitir al `tenant_admin` crear ventanas de silencio (maintenance windows) que especifiquen: período de inicio y fin, servicios afectados (o "todos"), y motivo del mantenimiento. Durante estas ventanas, si se generan alertas, el sistema las marca como silenciadas, las almacena en historial, pero **no** las envía a los canales de notificación. Cuando la ventana termina, las alertas silenciadas pueden consultarse en un reporte de "alertas silenciadas por mantenimiento".

---

## Actores

- **Actor principal**: `tenant_admin`
- **Actor secundario**: Sistema (silencia alertas durante la ventana configurada)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.
- Los servicios o métricas a silenciar están disponibles.

---

## Postcondiciones

### En caso de éxito

- La ventana de silencio se crea y activa automáticamente en el tiempo especificado.
- Durante la ventana, las alertas de los servicios afectados no se envían.
- Al terminar la ventana, el sistema puede enviar un resumen de alertas silenciadas.
- La ventana se registra en el historial para auditoría.

### En caso de falla

- Si el horario de la ventana se cruza con otra existente: error de conflicto.
- Si la ventana está en el pasado: error de validación.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Observabilidad → Alertas → Ventanas de Silencio`.
2. El sistema muestra las ventanas existentes y un botón "Crear nueva ventana".
3. El admin define la ventana:
   - `window_name`: Nombre descriptivo (ej: "Mantenimiento database server").
   - `start_time`: Fecha y hora de inicio (timezone del tenant).
   - `end_time`: Fecha y hora de fin.
   - `affected_services`: Lista de servicios (o "todos").
   - `affected_metrics` (opcional): Métricas específicas (ej: solo latencia).
   - `severities_to_silence`: `[CRITICAL]`, `[WARNING]`, `[INFO]`, o todas.
   - `notify_on_end`: ¿Enviar resumen al terminar? (boolean).
   - `reason`: Motivo del mantenimiento (para auditoría).
   - `ticket_id` (opcional): ID del ticket de mantenimiento en otro sistema.
4. El sistema valida que no haya conflicto con ventanas existentes.
5. La ventana se guarda en `maintenance_windows`.
6. Un worker background verifica cada minuto si alguna ventana debe activarse o desactivarse.
7. Cuando una ventana se activa:
   - Las alertas de los servicios/métricas afectados se marcan como `SILENCED`.
   - Las alertas no se envían a ningún canal de notificación.
8. Cuando la ventana termina:
   - Si `notify_on_end` = true, se envía un resumen de alertas silenciadas.
   - La ventana se marca como `COMPLETED` en el historial.
9. El admin puede terminar una ventana anticipadamente (opcional).

## Modelo de Datos

### Maintenance Window

|| Campo | Tipo | Descripción |
|---|-------|------|-------------|
| window_id | uuid | Identificador único |
| tenant_id | uuid | Tenant owner |
| window_name | string | Nombre descriptivo |
| start_time | datetime | Inicio de la ventana |
| end_time | datetime | Fin de la ventana |
| affected_services | array<string> | Nombres de servicios (`["all"]` = todos) |
| affected_metrics | array<string> | Métricas específicas (null = todas) |
| severities_to_silence | array<string> | Severidades a silenciar |
| notify_on_end | boolean | ¿Enviar resumen al terminar? |
| reason | string | Motivo del mantenimiento |
| ticket_id | string | ID de ticket externo (opcional) |
| status | enum | `SCHEDULED`, `ACTIVE`, `COMPLETED`, `CANCELLED` |
| created_by | uuid | Admin que creó la ventana |
| created_at | datetime | Fecha de creación |

---

## Estados de la Ventana

```
SCHEDULED → (start_time llega) → ACTIVE → (end_time llega) → COMPLETED
                ↓ (opcional)
           CANCELLED (cancelación manual)
```

---

## Comportamiento de las Alertas Durante la Ventana

1. Se genera una alerta (ej: SERVICE_DOWN en rate-service).
2. El `alerting-service` consulta las ventanas activas para el tenant.
3. Si existe una ventana activa que cubre:
   - El servicio de la alerta (o "todos")
   - La severidad de la alerta
4. Entonces: la alerta se marca como `SILENCED` y se guarda en `alerts` con `silenced: true, silence_window_id: <id>`.
5. La alerta **no** se envía a ningún canal.
6. Si no hay ventana, se sigue el flujo normal (enviar por canales).

### Alertas Críticas Durante Maintenance

|| Escenario | Comportamiento Esperado |
|---|-----------|--------------------------|
| Alerta CRITICAL durante ventana | Se silencia, se guarda en historial |
| Alerta de servicio no incluido en la ventana | Se envía normalmente |
| Alerta de métrica no incluida en la ventana | Se envía normalmente |
| Ventana incluye "todos" pero severity INFO | Se silencian INFO también |
| Ventana se extiende más de lo planeado | El admin puede extender `end_time` mientras está activa |

## Casos de Borde y Manejo de Errores

|| Escenario | Comportamiento Esperado |
|---|-----------|--------------------------|
| Ventana se cruza con otra existente | Error 409: "Ya existe una ventana de mantenimiento que se cruza con este horario" |
| `start_time` en el pasado | Error: "La ventana no puede crearse en el pasado" |
| `end_time` antes de `start_time` | Error: "La fecha de fin debe ser posterior al inicio" |
| Ventana > 24 horas | Warning: "Considere dividir en ventanas más cortas para mejor control" |
| Ventana muy corta (< 5 min) | Warning: "La ventana es muy corta, puede no tener efecto práctico" |
| Alguien intenta crear ventana sin rol de admin | Error 403 |
| Ventana activa se intenta cancelar | Se marca como `CANCELLED` y las alertas nuevas ya no se silencian |
| El alerting-service se reinicia durante ventana activa | Al reiniciar, carga ventanas activas desde BD y continúa normalmente |

## Criterios de Aceptación

1. El `tenant_admin` puede crear, editar, cancelar y consultar ventanas de silencio.
2. Cada ventana especifica: nombre, inicio, fin, servicios afectados, severidades, motivo.
3. Durante la ventana, las alertas de los servicios/severidades afectados se silencian.
4. Las alertas silenciadas se marcan con `silenced: true` y se guardan en historial.
5. Al terminar la ventana, se puede enviar un resumen de alertas silenciadas.
6. Las ventanas se pueden crear con anticipación (ej: programada para mañana a las 10pm).
7. Las ventanas pueden cancelarse anticipadamente.
8. No se permiten ventanas que se crucen en el tiempo para los mismos servicios.
9. Las alertas silenciadas son visibles en el dashboard de alertas con marca de "silenciada".
10. Existe un reporte de "Alertas por Mantenimiento" con historial de ventanas y alertas silenciadas.

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/maintenance-windows` — Listar ventanas
- `POST /api/v1/tenants/{tenant_id}/maintenance-windows` — Crear ventana
- `PUT /api/v1/tenants/{tenant_id}/maintenance-windows/{window_id}` — Editar ventana
- `DELETE /api/v1/tenants/{tenant_id}/maintenance-windows/{window_id}` — Cancelar ventana

- `GET /api/v1/tenants/{tenant_id}/maintenance-windows/history` — Historial de ventanas pasadas

---

## Notas

- Se recomienda que la ventana incluya un buffer de 5-10 min antes del mantenimiento real (para que las alertas de preparación no se envíen).
- El campo `ticket_id` permite vincular el mantenimiento con un sistema de tickets externo (ej: Jira).
- Para servicios críticos, se puede configurar que las alertas CRITICAL nunca se silencien (override a nivel de servicio).
- Las ventanas recurrentes (ej: "todos los domingos 2am-4am") no están en el alcance inicial; pueden agregarse en una versión futura.
- El resumen al terminar la ventana (`notify_on_end`) se envía por los mismos canales configurados para alertas normales.
