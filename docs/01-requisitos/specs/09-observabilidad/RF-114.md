# RF-114 — Cálculo de Uptime por Microservicio

## Metadata

- **Módulo**: Observabilidad — Dashboard SLA
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: monitoring-service, metrics-service, dashboard-service

---

## User Story

**Como** tenant_admin **quiero** ver el uptime real de cada microservicio por mes **para** verificar que el servicio cumple con los niveles pactados en mi contrato y tomar acciones correctivas si hay caídas.

---

## Objetivo

El sistema debe calcular y mostrar el uptime real de cada microservicio por mes, definido como: `(minutos disponibles / total minutos del mes) * 100`. Se considera un microservicio "disponible" cuando responde correctamente a su endpoint `/health` con status 200.

---

## Actores

- **Sistema (monitoring-service)**: Recolecta health checks y calcula uptime
- **tenant_admin**: Visualiza el uptime en el dashboard

---

## Precondiciones

- Los microservicios deben tener endpoints `/health` expuestos (RF-100)
- El job de monitoreo debe estar ejecutándose según la frecuencia configurada

---

## Postcondiciones

### En caso de éxito

- Se calcula el uptime mensual de cada microservicio
- Los datos se muestran en el dashboard de SLA

### En caso de falla

- Si no hay datos de health para un período, ese período se cuenta como no disponible
- Se genera alerta si el uptime del mes cae por debajo del umbral del SLA

---

## Flujo Funcional

1. El servicio `monitoring-service` ejecuta health checks de cada microservicio cada 30 segundos
2. Cada health check registra: `microservicio`, `timestamp`, `status` (ok/fail), `response_time_ms`
3. Al final de cada mes, el job `calculate-monthly-uptime` ejecuta:
   a. Para cada microservicio, cuenta los minutos totales del mes
   b. Cuenta los minutos donde al menos un health check fue exitoso
   c. Calcula: `uptime % = (minutos_ok / total_minutos) * 100`
4. El resultado se almacena en la tabla `monthly_uptime` con: `microservicio_id`, `año`, `mes`, `uptime_percentage`, `total_minutes`, `minutes_available`, `calculated_at`
5. El dashboard de SLA consulta estos datos y los muestra por microservicio y mes
6. Se permite comparar el uptime de diferentes meses

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| microservice_id | string | Identificador del microservicio | Si |
| timestamp | timestamp | Momento del health check | Si |
| status | enum | Resultado: `ok`, `fail` | Si |
| response_time_ms | integer | Tiempo de respuesta en milisegundos | Si |
| error_message | string | Mensaje de error si falló (opcional) | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| microservice_id | string | Identificador del microservicio |
| year | integer | Año del cálculo |
| month | integer | Mes del cálculo (1-12) |
| uptime_percentage | decimal | Porcentaje de uptime (ej: 99.95) |
| total_minutes | integer | Total de minutos en el mes |
| minutes_available | integer | Minutos con al menos un health check exitoso |
| minutes_down | integer | Minutos sin health checks exitosos |
| incidents_count | integer | Número de incidentes únicos en el mes |
| calculated_at | timestamp | Momento del cálculo |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Mes en curso (no finalizado) | Se calcula uptime parcial con los datos disponibles hasta la fecha |
| Health check no ejecuta por outage del monitoring | El período se cuenta como downtime; no se asume uptime |
| Microservicio nuevo (menos de 1 día de datos) | Se muestra el uptime calculado con los datos disponibles,注明 "datos insuficientes" |
| Health check falso positivo (detección de red) | Se permite marcar manualmente un período como "exento" si se comprueba que fue un falso positivo |
| Switchover entre instancias | Se considera disponible si al menos una instancia responde |

---

## Criterios de Aceptación

1. El uptime se calcula como: minutos disponibles / total minutos del mes
2. Se considera "disponible" un minuto donde al menos un health check fue exitoso
3. El uptime se calcula mensualmente y se almacena con histórico
4. El dashboard permite ver uptime de los últimos 12 meses por microservicio
5. Se puede filtrar por sede si el microservicio es específico de una sede
6. El uptime se muestra con 2 decimales (ej: 99.95%)

## Endpoints API Involucrados

- `GET /api/v1/monitoring/uptime?microservice={id}&year={year}&month={month}` — Consulta uptime de un microservicio
- `GET /api/v1/monitoring/uptime/history` — Historial de uptime de todos los microservicios
- `GET /api/v1/health` — Health check del servicio de monitoreo

---

## Métricas de Éxito

- Precisión del cálculo: uptime calculado coincide con datos crudos ± 0.01%
- Disponibilidad del dashboard: uptime del servicio > 99.5%
- Tiempo de generación del reporte mensual: < 5 minutos después de fin de mes

---

## Notas

- El uptime se calcula considerando zonas horarias del tenant (UTC de la sede principal)
- Para contratos Enterprise con SLA de 99.9%, el downtime máximo permitido por mes es ~43 minutos
- Se recomienda guardar los health checks crudos por 30 días para auditoría y recalculo si es necesario
