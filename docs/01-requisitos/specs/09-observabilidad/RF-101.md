# RF-101 вҖ” Dashboard de Salud General de Microservicios

## Metadata

- **MГіdulo**: Observabilidad вҖ” Dashboard
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: observability-service, todos los microservicios

---

## User Story

**Como** operador del sistema **quiero** ver un dashboard que muestre en tiempo real el estado de salud de todos los microservicios **para** detectar rГЎpidamente cuГЎl servicio estГЎ caГӯdo o degradado sin necesidad de consultar cada uno manualmente.

---

## Objetivo

Un dashboard centralizado debe mostrar el estado de salud (`UP`, `DEGRADED`, `DOWN`) de todos los microservicios del sistema. La informaciГіn se actualiza automГЎticamente cada 30 segundos mediante polling a los endpoints `/health` de cada microservicio. El dashboard debe ser accesible desde la interfaz de administraciГіn y debe permitir filtrar por tenant, sede o tipo de servicio.

---

## Actores

- **Actor principal**: Operador de planta, `tenant_admin`
- **Actor secundario**: Sistema (recolecta datos de health checks)

---

## Precondiciones

- Todos los microservicios tienen sus endpoints `/health` operativos.
- El servicio de observabilidad tiene acceso de red a todos los microservicios.

---

## Postcondiciones

### En caso de Г©xito

- El dashboard muestra una lista de servicios con su estado actual.
- Cada entrada se actualiza automГЎticamente cada 30 segundos.
- Un servicio en estado `DOWN` aparece destacado en rojo con una alerta visual.
- Un servicio en `DEGRADED` aparece en amarillo.

### En caso de falla

- Si un microservicio no responde al health check durante mГЎs de 60s: se marca como `UNKNOWN` con Гӯcono gris.
- Si el servicio de observabilidad no puede alcanzar un servicio: sigue mostrando el Гәltimo estado conocido con marca de tiempo.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Observabilidad вҶ’ Dashboard de Salud`.
2. El sistema consulta el endpoint `/health` de todos los microservicios registrados.
3. El sistema compila los resultados y los muestra en una grilla/tarjeta por servicio.
4. Cada 30 segundos, el sistema vuelve a consultar todos los endpoints.
5. Si un servicio cambia de estado, se emite una notificaciГіn visual (toast/badge).
6. El operador puede hacer clic en un servicio para ver el detalle: dependencias, latencia, memoria.

## Criterios de AceptaciГіn

1. El dashboard se actualiza cada 30 segundos automГЎticamente.
2. Todos los servicios registrados aparecen con su estado correcto.
3. Un servicio `DOWN` se muestra con fondo rojo y badge de alerta.
4. El operador puede filtrar por sede, tenant o tipo de servicio.
5. Al hacer clic en un servicio se muestran los detalles de sus dependencias.
6. Se muestra el `uptime_percent_30d` de cada servicio.
7. El operador puede exportar el estado actual a PDF o CSV.
8. El dashboard es accesible desde el menГә principal de `Observabilidad`.
9. Si el operador no ha interactuado por 5 minutos, se sigue actualizando en background.

## Notas

- El polling interval de 30s es configurable por el `tenant_admin` (mГӯnimo 10s, mГЎximo 5min).
- Para mejorar rendimiento, se recomienda usar HTTP HEAD para los health checks.
- Se debe implementar exponential backoff si un servicio estГЎ tardando mГЎs de 5s en responder.
- El campo `down_since` permite saber hace cuГЎnto tiempo estГЎ caГӯdo un servicio.
- La vista de historial debe permitir ver hasta 30 dГӯas de estados para calcular uptime.
