# RF-110 — Monitoreo de Latencia Comando-Respuesta de Talánqueras

## Metadata

- **Módulo**: Observabilidad — Dispositivos IoT
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: device-monitor, talanquera-service, api-gateway

---

## User Story

**Como** sistema de monitoreo **quiero** medir el tiempo entre enviar un comando de apertura a una talanquera y recibir la confirmación de que se abrió **para** detectar dispositivos con latencia anormal que puedan indicar fallas o congestión de red.

---

## Objetivo

El sistema debe trackear la latencia comando-respuesta de cada talanquera: el tiempo transcurrido desde que se envía el comando "abrir" hasta que la talanquera confirma "abierta". Si la latencia excede 3 segundos, se debe generar una alerta de tipo `latency_warning`.

---

## Actores

- **Sistema (device-monitor)**: Registra y evalúa latencias
- **Talanqueras**: Envían confirmación de comandos ejecutados

---

## Precondiciones

- La talanquera debe estar en estado `online` (con heartbeats activos)
- El comando "abrir" debe haber sido enviado a la talanquera

---

## Postcondiciones

### En caso de éxito

- La latencia se registra en la base de datos con timestamp
- Si la latencia es ≤ 3 segundos, no se genera alerta

### En caso de falla

- Si la latencia > 3 segundos, se genera alerta `latency_warning`
- Si la talanquera no confirma el comando en 10 segundos, se genera alerta `command_timeout`

---

## Flujo Funcional

1. El operador o sistema envía comando `POST /api/v1/talanqueras/{id}/abrir`
2. El servicio `talanquera-service` registra en la base de datos el timestamp de envío del comando (`command_sent_at`)
3. La talanquera recibe el comando y ejecuta la apertura
4. La talanquera envía confirmación `POST /api/v1/talanqueras/{id}/confirm` con `command_id` y `status=opened`
5. El servicio `device-monitor` recibe la confirmación, calcula la latencia: `confirm_received_at - command_sent_at`
6. Si latencia ≤ 3s: se registra en tabla `command_latencies` sin alerta
7. Si 3s < latencia ≤ 10s: se genera alerta `latency_warning` con nivel "warning"
8. Si latencia > 10s: se genera alerta `command_timeout` con nivel "critical"
9. El dashboard de latencias muestra en tiempo real el promedio de los últimos 5 comandos por talanquera

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| command_id | UUID | Identificador del comando enviado | Si |
| talanquera_id | UUID | Identificador de la talanquera | Si |
| action | enum | Acción: `open`, `close` | Si |
| sent_at | timestamp | Hora en que se envió el comando | Si |
| confirmed_at | timestamp | Hora en que llegó la confirmación (opcional, se incluye en la confirmación) | Si |
| status | enum | Estado reportado: `opened`, `closed`, `error`, `timeout` | Si |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| talanquera_id | UUID | Identificador de la talanquera |
| latency_ms | integer | Latencia en milisegundos |
| latency_category | enum | `normal` (≤3s), `warning` (3-10s), `critical` (>10s) |
| alert_triggered | boolean | Si se generó alerta |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Confirmación llega con delay de red > 10s | Se genera `command_timeout`, se marca talanquera como `degraded` |
| Talanquera confirma con status `error` | Se genera alerta `device_error` independientemente de la latencia |
| Comando enviado a talanquera offline | Se rechaza inmediatamente con error 503 |
| Confirmación duplicada (reenvío por falla de red) | Se detecta por `command_id` y se ignora la segunda confirmación |
| Latencia normal pero múltiples comandos seguidos con alta latencia | Se genera `latency_warning` cuando el promedio de 5 comandos > 3s |

---

## Criterios de Aceptación

1. La latencia comando-respuesta se calcula con precisión de milisegundos
2. Toda latencia > 3s genera una alerta `latency_warning`
3. Toda latencia > 10s genera una alerta `command_timeout`
4. El dashboard muestra el promedio de latencia de los últimos 5 comandos por talanquera
5. Las alertas se notifican según la configuración de canales del tenant_admin (RF-107)
6. Los datos de latencia se retienen por 30 días para análisis histórico

## Dependencias

**RFs relacionados**:
- RF-109 — Heartbeat de dispositivos (estado online/offline)
- RF-102 — Detección de microservicios con latencia anómala
- RF-106 — Configuración de umbrales de alerta por tenant_admin
- RF-107 — Canales de notificación de alertas

**Tablas de base de datos afectadas**:
- `command_latencies` — Almacena latencia de cada comando enviado a talanquera
- `alerts` — Registro de alertas generadas por latencia excesiva
- `devices` — Se actualiza el campo `latency_avg` con el promedio de los últimos 5 comandos

---

## Endpoints API Involucrados

- `POST /api/v1/talanqueras/{id}/abrir` — Envía comando de apertura
- `POST /api/v1/talanqueras/{id}/confirm` — Recibe confirmación de la talanquera
- `GET /api/v1/talanqueras/{id}/latency` — Consulta historial de latencias de una talanquera

---

## Métricas de Éxito

- Porcentaje de comandos con latencia ≤ 3s (target: > 95%)
- Tiempo promedio de latencia por talanquera
- Número de alertas `latency_warning` y `command_timeout` generadas por día

---

## Notas

- El umbral de 3 segundos es configurable por el tenant_admin a través del módulo de alertas (RF-106)
- Se usa una ventana móvil de los últimos 5 comandos para calcular el promedio y detectar tendencias
- El campo `latency_avg` en la tabla `devices` permite al admin ver rápidamente si una talanquera tiene problemas recurrentes
- Este requerimiento complementa RF-111 (alerta por talanquera sin responder 2 min) — aquí se mide la latencia de comandos exitoso, mientras RF-111 detecta dispositivos completamente sin respuesta
