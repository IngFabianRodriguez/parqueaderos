# RF-109 — Heartbeat de Dispositivos IoT

## Metadata

- **Módulo**: Observabilidad — Dispositivos IoT
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: device-monitor, api-gateway

---

## User Story

**Como** sistema de monitoreo **quiero** recibir heartbeats periódicos de cada talanquera y cámara ANPR **para** detectar oportunamente cuando un dispositivo queda offline y evitar operaciones inconscientes.

---

## Objetivo

El sistema debe recibir un heartbeat de cada talanquera y ANPR cada 60 segundos. Si un heartbeat no llega dentro de ese intervalo, el dispositivo se marca como `offline` automáticamente y se genera una alerta para el tenant_admin.

---

## Actores

- **Sistema (device-monitor)**: Recibe y procesa heartbeats
- **Talanqueras y Cámaras ANPR**: Envían heartbeats periódicos

---

## Precondiciones

- Los dispositivos (talanqueras y ANPR) deben estar configurados y registrados en el sistema
- La conexión de red entre el dispositivo y el servicio device-monitor debe estar activa

---

## Postcondiciones

### En caso de éxito

- El dispositivo permanece en estado `online`
- El timestamp de último heartbeat se actualiza en la base de datos

### En caso de falla

- El dispositivo se marca como `offline`
- Se genera una alerta para el tenant_admin
- Se registra el evento de desconexión en el log de auditoría

---

## Flujo Funcional

1. El dispositivo (talanquera o ANPR) envía un heartbeat `POST /heartbeat` con `device_id`, `device_type`, `timestamp` y `status`
2. El servicio `device-monitor` recibe el heartbeat y actualiza el campo `last_heartbeat_at` del dispositivo en la base de datos
3. El sistema verifica que el heartbeat llegó dentro de la ventana de 60 segundos
4. Si el heartbeat es válido, el dispositivo se mantiene `online`
5. Un job programado (cron) ejecuta cada 60 segundos para verificar dispositivos que no enviaron heartbeat
6. Si un dispositivo no recibe heartbeat en 2 ciclos (120 segundos), se marca como `offline`
7. Se genera automáticamente una alerta de tipo `device_offline`

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| device_id | UUID | Identificador único del dispositivo | Si |
| device_type | enum | Tipo: `talanquera`, `anpr` | Si |
| timestamp | ISO8601 | Fecha y hora del heartbeat en el dispositivo | Si |
| status | enum | Estado actual: `online`, `error`, `maintenance` | Si |
| battery_level | integer | Porcentaje de batería (opcional, solo para dispositivos inalámbricos) | No |
| signal_strength | integer | Porcentaje de señal WiFi/4G (opcional) | No |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| device_id | UUID | Identificador del dispositivo |
| status | enum | Estado actualizado: `online` / `offline` |
| last_heartbeat_at | timestamp | Fecha/hora del último heartbeat recibido |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Heartbeat arrive exactamente a los 60s | Se considera válido; el dispositivo permanece online |
| Dispositivo envía heartbeat pero la red tiene latency > 5s | Se procesa normalmente; se registra la latencia |
| Heartbeat con device_id no registrado | Se rechaza con código 404 y se loguea el intento |
| Dispositivo pasa a modo `maintenance` | Se excluye de la verificación de offline temporalmente |
| Servicio device-monitor caido | Los heartbeats se pierden; al reactivarse, los dispositivos quedaran offline por timeout |

---

## Criterios de Aceptación

1. Todo dispositivo registrado envía un heartbeat al menos cada 60 segundos
2. Un dispositivo se marca como `offline` si no se recibe heartbeat en 120 segundos (2 ciclos)
3. La alerta `device_offline` se genera automáticamente al marcar un dispositivo offline
4. El estado `offline` se refleja en el dashboard de salud del módulo de Observabilidad
5. El campo `last_heartbeat_at` se actualiza en la base de datos para cada heartbeat exitoso

## Endpoints API Involucrados

- `POST /api/v1/devices/heartbeat` — Recibe heartbeat de talanquera o ANPR
- `GET /api/v1/devices/{device_id}/status` — Consulta estado actual de un dispositivo

## Métricas de Éxito

- Porcentaje de heartbeats recibidos a tiempo (target: > 99%)
- Tiempo promedio de detección de dispositivo offline (target: < 90 segundos)
- Número de alertas `device_offline` generadas por día

---

## Notas

- Se usa un scheduler basada en cron (cada 60s) para recorrer todos los dispositivos pendientes de heartbeat
- Los heartbeats se procesan de forma asíncrona para no bloquear la respuesta del dispositivo
- El umbral de 60s es configurable por tenant_admin en la configuración de alertas del módulo de observabilidad
