# RF-106 — Configuración de Umbrales de Alerta por Métrica

## Metadata

- **Módulo**: Observabilidad — Alertas Configurables
- **Prioridad**: Alta
- **Tipo**: Funcional
- **Servicios afectados**: alerting-service, observability-service, todos los microservicios

---

## User Story

**Como** `tenant_admin` **quiero** configurar los umbrales (thresholds) de alerta para cada métrica **para** recibir notificaciones cuando el sistema se comporta fuera de los parámetros normales sin necesidad de contactar al equipo de desarrollo.

---

## Objetivo

El `tenant_admin` debe poder configurar, desde la interfaz de administración, umbrales de alerta personalizados para cualquier métrica disponible (CPU, RAM, latencia, error rate, etc.). Cada umbral define: la métrica objetivo, el valor threshold, el operador de comparación (> , < , >=), la severidad de la alerta (CRITICAL, WARNING, INFO), y el mensaje de notificación. Los umbrales se almacenan por tenant y se aplican a todos los servicios o a servicios específicos.

---

## Actores

- **Actor principal**: `tenant_admin`
- **Actor secundario**: Sistema (evalúa umbrales contra métricas en tiempo real)

---

## Precondiciones

- El actor tiene rol `tenant_admin`.
- La métrica que se desea monitorear está disponible (expuesta en `/metrics`).
- Existe al menos un canal de notificación configurado (RF-107).

---

## Postcondiciones

### En caso de éxito

- El umbral se guarda en la configuración del tenant.
- El sistema comienza a evaluar el umbral contra las métricas entrantes.
- Si una métrica cruza el umbral, se genera la alerta correspondiente.
- El `tenant_admin` puede ver, editar o eliminar sus umbrales.

### En caso de falla

- Si el umbral 配置ado supera límites técnicos (ej: latencia < 0): error de validación.
- Si el servicio de alertas está down: los eventos se guardan en cola para reprocesamiento.

---

## Flujo Funcional

1. El `tenant_admin` accede a `Observabilidad → Alertas → Configurar Thresholds`.
2. El sistema muestra una lista de métricas disponibles organizadas por categoría.
3. El admin puede crear un nuevo umbral:
   - Selecciona la métrica (ej: `process_cpu_percent`).
   - Selecciona el servicio (o "todos los servicios").
   - Define el operador (`>`, `>=`, `<`, `<=`) y el valor threshold.
   - Selecciona la severidad: `CRITICAL`, `WARNING`, `INFO`.
   - Define el mensaje de notificación (template con variables).
   - Define la ventana de evaluación: cada cuánto se evalúa (default: 1min).
   - Define la condición: "se activa si se viola N veces consecutivas" (default: 1).
4. El umbral se guarda en `alert_thresholds` y se activa inmediatamente.
5. El sistema evalúa continuamente las métricas contra los umbrales activos.
6. Cuando se cruza el umbral, se genera una alerta que se envía por los canales configurados.

## Modelo de Datos

### Threshold

|| Campo | Tipo | Descripción |
|---|-------|------|-------------|
| threshold_id | uuid | Identificador único |
| tenant_id | uuid | Tenant owner |
| metric_name | string | Nombre de la métrica Prometheus (ej: `process_cpu_percent`) |
| service_name | string | Nombre del servicio (`"all"` = todos) |
| operator | enum | `gt`, `gte`, `lt`, `lte` |
| value | decimal | Valor del threshold |
| severity | enum | `CRITICAL`, `WARNING`, `INFO` |
| evaluation_interval_seconds | int | Cada cuántos segundos se evalúa (default: 60) |
| consecutive_violations | int | Veces consecutivas que debe violarse (default: 1) |
| message_template | string | Mensaje de la alerta (soporta variables) |
| is_active | boolean | ¿Está activo? |
| created_at | datetime | Fecha de creación |
| updated_at | datetime | Última modificación |

---

## Templates de Mensaje

Los mensajes de alerta pueden usar variables con sintaxis `{{variable}}`:

```
CPU alto en {{service_name}}: {{value}}% (threshold: {{threshold}}%)
Latencia de {{service_name}} en {{tenant_id}} superó {{value}}ms
Error rate en {{service_name}}: {{value}} errores/min (max: {{threshold}})
```

### Variables Disponibles

| Variable | Descripción |
|----------|-------------|
| `{{service_name}}` | Nombre del servicio afectado |
| `{{metric_name}}` | Nombre de la métrica |
| `{{value}}` | Valor actual de la métrica |
| `{{threshold}}` | Valor del threshold configurado |
| `{{tenant_id}}` | ID del tenant |
| `{{timestamp}}` | Momento de la violación |
| `{{duration_seconds}}` | Cuánto tiempo lleva violando |

---

## Categorías de Métricas Disponibles

### Infraestructura

- `node_cpu_usage_percent` — CPU del nodo
- `node_memory_used_percent` — Memoria del nodo
- `node_disk_used_percent` — Disco del nodo
- `process_cpu_percent` — CPU del proceso
- `process_memory_percent` — Memoria del proceso

### Aplicación

- `http_request_duration_seconds_p95` — Latencia p95 (calculada de histogram)
- `http_requests_total` (con filtro `status_class=5xx`) — Error rate
- `gateway_upstream_latency_seconds_p99` — Latencia upstream p99
- `service_http_requests_in_flight` — Requests activas

### Negocio (ejemplos por módulo)

- `transactions_processed_total` — Transacciones procesadas
- `devices_offline_count` — Dispositivos offline
- `alerts_triggered_total` — Alertas generadas (cuidado con recursion)

---

## Casos de Borde y Manejo de Errores

|| Escenario | Comportamiento Esperado |
|---|-----------|--------------------------|
| Threshold para métrica inexistente | Error de validación: "Métrica no encontrada" |
| Threshold con valor negativo (para métricas positivas) | Error: "El valor debe ser >= 0" |
| Threshold duplicado (misma métrica + servicio + operador) | Error 409: "Ya existe un umbral para esta combinación" |
| `consecutive_violations` > 10 | Warning: "Considere un valor menor para respuesta más rápida" |
| `evaluation_interval_seconds` < 15 | Error: "El intervalo mínimo es 15 segundos" |
| Mensaje con variable inexistente | Se deja la variable como `{{unknown}}` en el mensaje |
| Servicio con múltiples thresholds activos | Se evalúan todos; se pueden generar múltiples alertas |

## Criterios de Aceptación

1. El `tenant_admin` puede crear, editar, eliminar y desactivar thresholds desde la UI.
2. Cada threshold tiene: métrica, operador, valor, severidad, mensaje, servicio, intervalo de evaluación.
3. Los thresholds se evalúan contra los valores de métricas en tiempo real.
4. Una alerta se genera cuando la condición se cumple `consecutive_violations` veces seguidas.
5. Los mensajes de alerta usan templates con variables disponibles.
6. Los thresholds son específicos del tenant (aislamiento entre tenants).
7. Los thresholds activos se pueden listar y buscar.
8. Se puede clonar un threshold existente para crear uno similar.
9. Los thresholds son consistentes con la configuración de canales (RF-107) y ventanas de silencio (RF-108).

## Endpoints API Involucrados

- `GET /api/v1/tenants/{tenant_id}/alert-thresholds` — Listar thresholds
- `POST /api/v1/tenants/{tenant_id}/alert-thresholds` — Crear threshold
- `PUT /api/v1/tenants/{tenant_id}/alert-thresholds/{threshold_id}` — Actualizar threshold
- `DELETE /api/v1/tenants/{tenant_id}/alert-thresholds/{threshold_id}` — Eliminar threshold
- `PATCH /api/v1/tenants/{tenant_id}/alert-thresholds/{threshold_id}/activate` — Activar
- `PATCH /api/v1/tenants/{tenant_id}/alert-thresholds/{threshold_id}/deactivate` — Desactivar

---

## Notas

- Se recomienda que los thresholds de producción tengan `consecutive_violations >= 2` para evitar falsos positivos por spikes.
- Para métricas de latencia, el threshold debe definirse en la misma unidad que la métrica (segundos para durations).
- La evaluación de thresholds puede delegarse a Prometheus AlertManager (externo) o implementarse internamente con un worker que haga polling a Prometheus.
- Si el tenant tiene muchos thresholds (> 100), se debe implementar paginación en el listado.
- Los thresholds deben poder importarse/exportarse en JSON para backup y migración.
