# RF-112 — Asignación de Trace ID a Cada Request API

## Metadata

- **Módulo**: Observabilidad — Trazabilidad Distributed (Tracing)
- **Prioridad**: Media
- **Tipo**: Funcional
- **Servicios afectados**: api-gateway, todos los microservicios

---

## User Story

**Como** equipo de operaciones **quiero** que cada request API tenga un identificador único (trace_id) que se propagué a todos los microservicios involucrados **para** poder reconstruir el flujo completo de una transacción cuando algo falla.

---

## Objetivo

El sistema debe asignar un `trace_id` (UUID v4) a cada request API entrante en el API Gateway. Este `trace_id` debe propagarse a todos los microservicios involucrados mediante headers HTTP (`X-Trace-ID`) y almacenarse en los logs de cada servicio para permitir la correlación posterior.

---

## Actores

- **API Gateway**: Genera y propaga el trace_id
- **Todos los microservicios**: Reciben, registran y propagan el trace_id

---

## Precondiciones

- El request debe entrar por el API Gateway
- El header `X-Trace-ID` no debe estar presente (para evitar que un cliente falsifique tracing)

---

## Postcondiciones

### En caso de éxito

- El trace_id se genera y se incluye en todos los logs del request
- El trace_id se propaga a todos los microservicios llamados

### En caso de falla

- Si el API Gateway no puede generar el trace_id, se usa un valor placeholder "unknown"
- Los logs de cada servicio deben indicar si el trace_id estaba ausente

---

## Flujo Funcional

1. Un cliente envía una request HTTP al API Gateway
2. El API Gateway genera un `trace_id` (UUID v4) si no existe en los headers
3. El API Gateway inyecta el header `X-Trace-ID: {trace_id}` en la request
4. El API Gateway registra en su log: `[TRACE] {trace_id} {method} {path} {status} {duration_ms}`
5. El API Gateway reenvía la request al microservicio correspondiente
6. El microservicio recibe la request y lee el header `X-Trace-ID`
7. El microservicio incluye el `trace_id` en todos sus logs: `[TRACE] {trace_id} {service} {action} {details}`
8. Si el microservicio llama a otro servicio (ej: pagos → database), inyecta `X-Trace-ID` en la llamada
9. Todos los spans de un trace comparten el mismo `trace_id`
10. Al completar la request, el trace completo queda disponible para búsqueda

---

## Datos de Entrada

| Campo | Tipo | Descripción | Requerido |
|-------|------|-------------|-----------|
| trace_id | UUID v4 | Identificador único del trace | Generado por API Gateway |
| span_id | UUID v4 | Identificador de cada span individual | Generado por cada servicio |
| parent_span_id | UUID v4 | Span padre del actual (null para el primero) | Opcional |

---

## Datos de Salida

| Campo | Tipo | Descripción |
|-------|------|-------------|
| trace_id | UUID v4 | Propagado en header X-Trace-ID |
| Logs por servicio | JSON | Cada servicio registra sus logs con trace_id |

---

## Casos de Borde y Manejo de Errores

| Escenario | Comportamiento Esperado |
|-----------|--------------------------|
| Cliente envía request con `X-Trace-ID` header | Se ignora el header del cliente y se genera uno nuevo |
| Microservicio no recibe `X-Trace-ID` | Se genera un trace_id nuevo en ese servicio y se marca como "root span" |
| Llamada a servicio externo (webhook, pasarela de pago) | Se inyecta `X-Trace-ID` en la llamada saliente |
| Servicio escribe logs pero no propaga el trace_id | Se detecta en auditoría; se requiere como estándar de logging |
| Trace muy largo (> 100 spans) | Se limita a 100 spans por trace para evitar sobrecarga |

---

## Criterios de Aceptación

1. Todo request HTTP que entre al API Gateway recibe un `trace_id` único
2. El `trace_id` está presente en el 100% de los logs generados durante el procesamiento del request
3. El `trace_id` se propaga a todas las llamadas internas entre microservicios
4. Los logs son consultables por `trace_id` en el sistema centralizado de logs
5. La overhead de propagación del trace_id es < 1ms (no impacto medible en latencia)

## Endpoints API Involucrados

- Todos los endpoints del sistema propagan el header `X-Trace-ID`
- `GET /api/v1/traces/{trace_id}` — Consulta un trace completo (futuro, para dashboard de tracing)

---

## Métricas de Éxito

- Porcentaje de requests con trace_id válido (target: 100%)
- Porcentaje de logs con trace_id presente (target: 100%)
- Tiempo promedio de consulta de logs por trace_id (target: < 2 segundos)

---

## Notas

- Se recomienda usar una biblioteca de tracing como OpenTelemetry para estandarizar la instrumentación
- El trace_id se almacena en logs, no en la base de datos transaccional, para no contaminar las tablas de negocio
- Para traces muy largos (batch jobs), se puede usar un formato alternativo de trace_id con prefijo "batch-"
